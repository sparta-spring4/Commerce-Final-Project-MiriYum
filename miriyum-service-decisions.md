## 1. 프로젝트 기준

### 확정

- 취업·포트폴리오용 프로젝트이지만 실제 운영 가능한 수준으로 설계한다.
- 대규모 트래픽과 특정 예약 오픈 시점의 순간적인 트래픽 폭주를 방어할 수 있어야 한다.
- 다수 서버·다수 애플리케이션 인스턴스 환경에서 모든 핵심 기능이 정상적으로 동작해야 한다.
- 애플리케이션 인스턴스는 가능한 한 무상태로 운영한다.
- 예약·홀드·결제·이벤트 처리는 중복 요청과 순서 역전을 전제로 멱등성을 보장한다.
- 좌석·메뉴·회차의 초과 판매를 DB 제약, 조건부 갱신, 원자적 연산으로 방지한다.
- 서버 또는 작업 인스턴스가 중단·재시작되어도 예약과 비동기 작업이 유실되지 않아야 한다.
- 가상 대기열, 요청 속도 제한, 캐시, 비동기 처리, 장애 격리, 자동 복구 및 부하 테스트를 설계 범위에 포함한다.
- 초기 구현 형태와 관계없이 예약·대기열·알림 등 트래픽이 큰 영역을 독립적으로 확장할 수 있는 도메인 경계를 둔다.

## 2. 지원 매장 범위와 운영 방식

### 확정

- 특정 지역이나 단일 업종으로 제한하지 않는다.
- 일반 식당, 베이커리, 브루어리, 오마카세, 팝업 등 다양한 외식 업종을 지원한다.
- 식당을 1~3단계로 고정 분류하는 기존 방식을 폐기한다.
- 식당이 플랫폼이 제공하는 전체 기능 중 필요한 기능을 직접 선택하는 모듈형 방식을 사용한다.
- 일반 기능은 식당이 즉시 활성화할 수 있다.
- 결제·정산·주류·테이스팅처럼 운영 또는 법적 위험이 큰 기능은 관리자 심사를 거쳐 활성화한다.
- 관리자는 제한 기능에 대해 `승인`, `보완 요청`, `거절`, `일시 중지` 처리를 할 수 있다.
- 기능 활성화, 설정 변경 및 관리자 심사 이력을 감사 로그로 보관한다.
- 기능 간 필수 의존성을 시스템에서 검사하여 올바르지 않은 조합을 차단한다.

### 기능 의존성 예시

```text
대표 메뉴 홀드 → 메뉴 관리 필요
코스 예약 → 회차·정원 관리 필요
선결제 → 결제·환불 정책 및 정산 정보 필요
테이스팅 → 회차 관리 및 참가자 설문 필요
주류 테이스팅 → 연령 확인 필요
```

## 5. 웨이팅 일행 및 중복 등록 정책

### 확정

- 웨이팅 대표자는 매장이 정한 등록 가능 반경 안에 있어야 신규 웨이팅을 생성할 수 있다.
- 대표자는 일행 참여 링크 또는 초대 코드를 공유할 수 있다.
- 초대받은 일행은 현재 위치와 관계없이 기존 웨이팅에 참여할 수 있다.
- 일행 참여는 새로운 대기 팀을 만들지 않고 기존 한 팀에 합류하는 방식이다.
- 일행은 동일한 대기 순번과 예상 대기 시간을 확인한다.
- 위치 정보는 등록 가능 여부 확인에 필요한 최소 범위로 사용하며, 정확한 원본 좌표를 불필요하게 장기 보관하지 않는다.
- 일반 사용자는 가입 완료 전에 휴대전화 번호 본인 확인을 완료한다.

### 2026-07-22 직접 확정된 운영 정책

대표 제품 결정과 표준 운영 세부를 같은 논의 큐에 두지 않는다.

- `확정`: 같은 매장·겹치는 시간대의 중복 팀 방지(`WAIT-007`), 사용자별 동시 웨이팅 제한(`WAIT-008`), 대기 지속 확인·미응답·재등록 처리(`WAIT-012`), 반복 미응답·노쇼·중복 탐지와 오탐 복구(`WAIT-014`)의 상세 안전 규칙
- `당시 확정`: `WAIT-003`은 기본 500m, 매장 설정 300m~1km와 GPS 정확도 부족 시 즉시 탈락시키지 않는 재시도·현장 증명 경로를 적용한다. `WAIT-009`는 매장 선택 빈 테이블에 맞는 첫 FIFO 팀만 호출하고 상업적 우대 없이 호출 전 1회·호환 팀 5팀 뒤로 미루기를 허용한다. `WAIT-011`은 3분 응답 제한의 최대 2회 호출 후 매장 확인 `호출 실패`로 전환한다.
- `당시 팀원 상의 필요`: 결제 마찰과 환급·분쟁 책임을 만드는 웨이팅 보증금 도입 여부(`WAIT-015`)

이 직접 결정은 각 정책 ID의 상세 내용과 확인 기록에 한정된다. `WAIT-003`·`WAIT-011`의 당시 수치와 `WAIT-015`의 미결정 상태는 아래 2026-07-23 결정으로 대체되었고, 호출·도착·만료 판정은 중앙 시각과 버전된 정책을 사용한다.

### 2026-07-23 웨이팅 운영 정책 변경

- `WAIT-003`: 원격 웨이팅의 플랫폼 기본 등록 반경을 3km로, 매장 설정 범위를 1km~5km로 확정하여 2026-07-22의 500m·300m~1km 결정을 대체한다.
- `WAIT-010`: 같은 좌석 조건으로 앞선 활성 호환 팀이 두 팀 이하가 되면 상태·순번을 바꾸지 않는 입장 임박 사건을 웨이팅 팀별 한 번 생성한다.
- `WAIT-011`: 입장 임박 안내와 실제 호출을 분리하고 실제 입장 호출은 팀당 한 번, 중앙 호출 시각부터 현장 도착 제한은 10분으로 확정하여 2026-07-23의 단일 호출·3분 도착 제한 결정을 대체한다.
- `WAIT-012`·`WAIT-016`·`WAIT-017`: 단일 호출·10분 도착 제한, 입장 임박 비상태 사건, 도착 확인·만료 경합과 팀별 1회 알림 정합성을 위 결정에 맞춰 변경한다.
- `WAIT-015`: 현재 서비스와 최종 프로젝트 범위에서는 웨이팅 보증금·선결제를 도입하지 않고 미응답·노쇼를 이유로 금전 청구하지 않는 것으로 확정한다.
- `WAIT-016`: 1차 MVP에서 대표자가 관리하는 매장별 웨이팅 기능 온·오프와 자동·수동·일시중지 접수를 제공한다. 자동 오픈은 영업 구간 시작 60분 전이고 0분~180분 범위로 조정하며, 수동 접수는 해당 영업 구간 시작일의 현지 00:00부터 구간 종료까지 허용한다. 기능 끄기와 마감 일괄 종료는 활성 팀을 조용히 취소하지 않고 영향 팀 확인과 명시적 `매장 종료`를 요구한다.
- `RES-010`·`WAIT-016`·`WAIT-017`: 같은 매장·영업 구간의 예약과 웨이팅 충돌에는 기존 거래 유지 또는 비용 결과 확인 뒤 조건부 상호 전환을 제공한다. 예약에서 웨이팅으로 전환할 때는 대상 가능성을 확보하고 원 예약 종결과 웨이팅 생성을 한 사건으로 확정한다. 웨이팅에서 예약으로 전환할 때는 원 팀·등록 시각·queue sequence를 보존하는 비종결 `예약 전환 중`에서 예약 수용량과 결제 결과를 보호하고, 예약금이 없거나 검증된 결제 성공 뒤에만 예약 확정과 `예약 전환 완료` 종결을 확정한다. 결제 실패·선점 만료에는 같은 sequence로 복귀하고 결과 불명에는 대사 전까지 어느 결과도 추정하지 않는다.

## 6. 2026-07-22 이전 상태 스냅샷

### 6.1 웨이팅 등록 가능 반경 (대체됨)

이 절의 예시 기본값 300m와 100m~1km 범위는 당시 미정 상태를 남긴 역사적 기록이며 현재 정책이 아니다. 현재 값은 `WAIT-003`의 3km 기본값과 매장 설정 1km~5km이다.

비교할 선택지:

1. 모든 매장에 플랫폼 고정 반경을 적용한다.
2. 식당이 반경을 자유롭게 설정한다.
3. 플랫폼이 최소·최대 허용 범위를 정하고 식당이 그 범위 안에서 선택한다.

당시 추천안:

- 3번 방식을 사용한다.
- 예시 기본값은 300m로 둔다.
- 예시 설정 가능 범위는 100m~1km로 둔다.
- 실제 기본값과 허용 범위는 상권 밀도, GPS 오차, 도보 접근성 및 테스트 결과를 근거로 확정한다.

당시 별도 확정 안전 기본선으로 분리한 예외:

- 실내·지하·고층 건물 등 GPS 정확도가 낮은 환경의 현장 QR·매장 코드 대체
- 위치 권한 거부 사용자의 실패 폐쇄와 현장 대체 경로
- 위치 조작 의심 시 추가 확인과 매장 좌표 오류의 감사 가능한 운영자 보정

위 예외는 `WAIT-004`의 표준 실패·복구 규칙으로서 상세화·독립 검토와 정책 ID별 확인 기록을 거쳐 확정되었다. 현재 `WAIT-003`은 이 예외 경로와 함께 3km 기본값·1km~5km 설정 범위를 적용한다.

### 6.2 식당 구독의 무료·유료 기능 범위 (현재 후보, 출시 미정)

상태: **미정 — `SUB-001`~`SUB-003` 팀원 상의 필요, `SUB-004` TODO**

선택된 후보와 남은 조건:

- 핵심 예약·웨이팅 관리는 Free, 실제 집계는 Free, 파생 리포트·규칙 기반 추천과 메뉴·행사 자동화는 Pro라는 후보를 기록한다. 이는 상품·혜택·무료/유료 경계를 정하는 대표 팀 결정이며 현재 권한을 활성화하지 않는다.
- Pro 후보 가격(부가세 별도 월 29,000원 또는 연 290,000원), 7일·3회 유예와 기간 종료 해지는 세무·반복결제 계약·출시 결정이 필요한 `SUB-004`·`SUB-006`의 `TODO`다.
- 사용자 구독도 초기 미출시이며, 광고 제거·리포트·자동화·유료 청구 권한을 현재 제공하지 않는다.

## 8. 공식 문서 체계

### 확정

- `miriyum-service-blueprint.md`: 최초 아이디어 초안이며, 파일명과 본문 브랜드 표기를 MiriYum으로 정렬한다.
- `miriyum-service-decisions.md`: 확정, 변경 및 보류 이력을 누적한다.
- `docs/service-definition.md`: 현재 공식 서비스 전체 구상과 제품 경계를 관리한다.
- `docs/technical-architecture.md`: 현재 공식 기술 방향을 관리한다.
- `docs/service-policies/README.md`: 전체 정책 항목과 진행 상태를 관리한다.
- `docs/service-policies/00-policy-template.md`: 상세 정책의 공통 검토 양식이다.
- `docs/service-policies/01-*.md`부터 `18-*.md`: 도메인별 상세 정책을 관리한다.

행 단위 상태·중요도가 충돌하면 `docs/service-policies/README.md`를 기준으로 한다. 정책 내용은 최신 확정 상세 정책, 공식 서비스 정의·기술 아키텍처, 의사결정 기록, 최초 블루프린트 순으로 적용하되, 미래 자동 추천 후보 또는 `TODO` 행의 남은 범위가 이미 확정된 하위 결정을 덮어쓰지 않고 확정 안전 기본선도 `TODO` 조각을 해소한 것으로 해석하지 않는다.

정책 상태는 `미논의`, `논의 중`, `자동 추천 예정`, `TODO`, `팀원 상의 필요`, `확정`, `변경 검토`로 구분한다. `자동 추천 예정`은 제품 방향을 바꾸지 않는 새 표준 보안·권한·멱등성·동시성·상태 전이·실패·재시도·알림·감사 후보이며, 상세안을 작성하는 동안 이 상태를 유지하고 명시적 위임·세부 완성·독립 검토·결정 기록 없이 확정하지 않는다. 현재 대기 행은 0개지만 미래 후보에 이 상태를 계속 사용한다. `TODO`는 외부 계약·최종 법률·개인정보·약관 검토·미래 기능·실측 수치처럼 지금 확정하지 않을 항목, `팀원 상의 필요`는 제품 방향·주요 흐름·금전 책임·공정성·되돌리기 어려운 약속의 대표 결정을 뜻한다. `확정`은 사용자가 직접 승인했거나 비핵심 추천안이 명시적인 추천 위임 기준 안에서 세부 내용과 인수 조건까지 완성되고 독립 검토를 거쳐 정책 ID별 의사결정 기록에 `확정`으로 남은 항목이다. 자동 추천 후보를 단순히 문서화하거나 구현한 것만으로는 충분하지 않는다. `TODO`의 미정 조각과 확정 안전 기본선은 같은 정책 행 또는 연계 섹션에 공존할 수 있다.
