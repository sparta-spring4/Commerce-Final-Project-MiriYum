# 체크인·노쇼 정책

> **1차 MVP 경계:** 권한 있는 매장 운영자의 예약 방문 완료 처리, 공통 5분 지각 허용시간 경과 뒤 직접 노쇼 처리, 사용자·매장·플랫폼 사유 코드, 예약금 환불 정책 입력과 최소 상태 변경 이력을 제공한다. QR·일회 확인번호 체크인, 매장별 체크인 창·3분~15분 지각시간 설정, 일부 인원 도착, 6시간 노쇼 후보·24시간 자동 확정, 반복 노쇼 경고·제재, 정식 이의 제기와 고급 감사·복구 자동화는 후속 구현 범위다. 웨이팅의 실제 호출 뒤 도착 제한은 CHECK 지각시간을 적용하지 않고 `WAIT-011`의 3분 기준을 따른다.

> 문서 상태: 세부 정책 확정
> 정책 범위: `CHECK-001`~`CHECK-010`
> 최종 변경일: 2026-07-23

체크인 자격과 수단, 지각, 일부 인원 도착, 노쇼 판정, 귀책, 제재 및 이의 제기를 다룬다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 확정 | 핵심 | `CHECK-006` |
| 확정 | 필수 | `CHECK-001`~`CHECK-005`, `CHECK-007`~`CHECK-010` |

- `CHECK-006`: 매장 직접 확인과 방문 시각 후 6시간·24시간 자동 보완 절차로 노쇼를 확정한다.
- `CHECK-005`: 기본 지각 허용 시간은 5분이고 매장은 3분~15분 범위에서 설정하며, 예약 생성 당시 값을 고정한다.
- 지각 허용 시간이 지나도 그 사실만으로 노쇼를 확정하지 않고 `CHECK-006`의 매장 직접 확인 또는 6시간·24시간 보완 절차로 넘긴다.

## 1차 MVP 정책별 적용 범위

| 정책 ID | 1차 MVP 적용 | 후속 구현 경계 |
|---|---|---|
| `CHECK-001` | 권한 있는 매장 운영자가 예약자와 중앙 예약 상태를 확인한 뒤 `방문 완료` 처리 | 회전형 QR·일회 확인번호 발급·검증 |
| `CHECK-002` | 예약 방문 시각과 중앙 시각을 기준으로 방문 완료·지각 여부 판정 | 매장별 체크인 시작·종료 시각 설정 UI·API |
| `CHECK-003` | 현재 상태·권한·멱등성으로 중복 방문 완료 방지 | QR 만료·일회 소비·재사용·캡처 방지 |
| `CHECK-004` | 예약 전체를 하나의 방문 완료 단위로 처리 | 일부 인원 도착·누적 체크인·남은 좌석 처리 |
| `CHECK-005` | 모든 매장에 공통 지각 허용시간 5분 적용 | 매장별 3분~15분 설정 UI·API |
| `CHECK-006` | 지각 허용시간 경과 뒤 권한 있는 매장 운영자의 직접 노쇼 확정과 `PAY-009` 입력 | 방문 후 6시간 노쇼 후보·재알림·24시간 자동 확정 |
| `CHECK-007` | 사용자·매장·플랫폼/외부 장애·불명확 사유 코드와 사용자 귀책 기본값 금지 | 증거 연결·재검토 버전·상세 귀책 검토 도구 |
| `CHECK-008` | 최종 확정된 노쇼 이력 기록; 금전·기능 제한은 적용하지 않음 | 반복 노쇼 스트라이크·경고·회복 자동화와 미래 강화 제재 |
| `CHECK-009` | 별도 상태 변경·제재 자동화 없음 | 사용자·매장의 정식 이의 제출과 운영자 검토·정정 워크플로 |
| `CHECK-010` | 행위자·중앙 시각·이전/이후 상태·사유 코드의 최소 감사 이력 | Outbox 기반 감사 복구·내보내기·보관기간 자동화 |

- 1차 MVP 공개 예약 상태는 `예약 완료`, `방문 완료`, `취소`, `노쇼`로 제한한다. `체크인 요청`, `검증`, `노쇼 후보`, `이의 접수·검토·정정`은 후속 구현 전까지 공개 상태나 상태 진입 경로로 제공하지 않는다.
- `후속 구현`은 정책 폐기나 `TODO` 전환이 아니다. 해당 정책은 확정 상태로 보존하되 1차 MVP의 UI·API·권한과 상태 진입 경로를 제공하지 않는다.
- 부분 적용 정책은 MVP 전용 두 번째 노쇼·환불 규칙을 만들지 않는다. MVP의 방문 완료·노쇼 결과도 `CHECK-006`·`CHECK-007`과 `PAY-009`의 확정 귀책·환불 경계를 따른다.

## CHECK-001 QR·번호·직원 확인 방식

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 예약·웨이팅·테이스팅의 현재 이용 자격이 있는 사용자에게 서버가 회전형 QR 또는 짧은 일회 확인번호를 발급한다. 식당 대표자는 자신의 매장·현재 이행 대상만 검증할 수 있고 플랫폼 운영자는 분쟁·복구 권한과 사건 없이 대신 체크인하지 않는다.
- 스캔/입력은 토큰 자체가 아니라 중앙 거래 ID·버전·매장·회차·유효 상태와 대조한 뒤 `체크인 요청` → `검증` → `체크인 완료`로 전이한다. 토큰 사용이 어려우면 대표자가 예약자 표시명·마스킹 정보와 예약 참조를 대조하는 직원 확인 경로를 사용하되 사유를 남긴다.
- 결제·취소·만료·이미 완료 상태는 체크인할 수 없다. 주류 행사는 `TASTE-005`에 따라 가게가 별도 현장 연령 확인을 수행하며 MiriYum 체크인이 이를 대체하거나 확인 결과를 저장하지 않는다.

### 동시성·실패·알림·감사·개인정보

- 거래 버전·토큰 식별자·체크인 멱등 키와 조건부 전이로 여러 단말·인스턴스의 중복 스캔 중 하나만 성공시킨다. 중앙 상태를 확인할 수 없으면 완료로 추정하지 않고 실패 폐쇄한다.
- 스캔 성공 뒤 응답 유실은 같은 키로 결과를 조회한다. 수동 확인은 기존 거래 소유자·상태를 바꾸지 않으며 오프라인 로컬 목록을 최종 원본으로 사용하지 않는다.
- 결과를 사용자·매장에 표시하고 방식, 거래/상태 버전, 단말이 아닌 행위자 계정, 시각·결과·실패 사유를 감사한다. QR 원문·연락처·신분증은 로그에 남기지 않는다.

### 관측 가능한 인수 조건

- 유효한 세 방식은 같은 중앙 거래를 정확히 한 번 완료하고 타 매장·취소·만료 토큰은 거부된다.
- 병렬 스캔·응답 유실·인스턴스 재시작에도 완료 전이와 알림이 중복되지 않는다.
- 주류 체크인이 현장 신분증 자료나 확인 완료 상태를 MiriYum에 수집하지 않는다.

## CHECK-002 체크인 가능 시간 범위

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 식당 대표자는 상품·회차별 체크인 시작 시각과 체크인 창 종료 시각을 버전된 공개 운영 설정으로 직접 게시한다. 두 값은 매장 운영에 맞춘 구성값이며 이 정책은 경험적 고정 숫자를 정하지 않는다. 중앙 시각·매장 시간대·거래 당시 설정 버전으로 적용한다.
- 사용자는 유효한 거래 상태와 게시된 체크인 창 안에서만 일반 체크인을 완료한다. 창 시작 전에는 `아직 불가`, 창 종료 뒤에는 `일반 체크인 창 종료`로 표시한다. `CHECK-005`는 이 종료 경계 뒤 별도로 적용할 지각 허용 시간만 정하며 체크인 시작·종료 설정의 소유권이나 값을 정하지 않는다.
- 설정 변경은 이후 생성 거래에 적용하고 기존 거래 스냅샷을 소급 축소하지 않는다. 기존 이행 충돌은 영향 사건으로 처리하며 매장·사용자 취소 책임을 새로 정하지 않는다.
- 중앙 시각·정책 버전·체크인 멱등 키로 경계 경합을 판정한다. 시각 원본 장애에는 신규 완료를 실패 폐쇄하고, 작업 지연에도 명령 시점에 범위를 재검증한다.
- 범위·정책 변경과 차단 사유를 당사자에게 알리고 판정 시각·버전·결과를 감사한다. 추가 개인정보인 위치·기기정보를 수집하지 않는다.

### 관측 가능한 인수 조건

- 경계 병렬 요청은 중앙 시각과 한 정책 버전으로 동일 판정되고 클라이언트 시각 조작이 통하지 않는다.
- 매장이 게시한 시작·종료 설정 변경이 기존 거래의 체크인 창을 소급 변경하지 않고 창 경과만으로 노쇼가 확정되지 않는다.
- 매장 소유의 체크인 창과 `CHECK-005`의 창 종료 뒤 지각 허용 시간이 서로 다른 정책 입력으로 저장·판정되며, 이 정책은 체크인 창의 고정 시간을 정하지 않는다.

## CHECK-003 QR 만료·재사용·캡처 방지

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- QR은 추측 불가능한 짧은 수명의 서명 토큰 또는 서버 참조로 거래·매장·회차·토큰 버전에 묶고 화면에서 주기적으로 교체한다. 캡처 방지를 보조하되 기기 지문·정밀 위치·얼굴정보를 수집하지 않는다.
- 최신 유효 토큰만 검증하고 교체·로그아웃·거래 취소·체크인 완료 시 이전 토큰을 무효화한다. 확인번호도 제한된 시도와 일회 소비 상태를 중앙 관리하며 실패 횟수의 수치는 버전 설정으로 조정한다.
- 토큰 ID·거래 버전·소비 원장의 유일성으로 복제 캡처의 동시 사용 중 한 요청만 성공한다. 키 회전은 이전/신규 검증 기간을 명시하고 혼합 배포 중에도 허용 버전을 중앙 정책으로 일치시킨다.
- 검증 저장소 장애에는 실패 폐쇄하고 반복 실패는 거래를 자동 취소·제재하지 않은 채 단계적 제한과 직원 확인을 안내한다. 의심 사건은 개인정보를 배제한 마스킹 토큰 ID·요청 결과만 감사한다.

### 관측 가능한 인수 조건

- 만료·구버전·타 거래·타 매장 토큰과 캡처 동시 재사용이 체크인 완료를 만들지 않는다.
- 키 회전·혼합 버전·중앙 장애 중 허용 토큰 정책이 일관되고 로컬 검증 우회가 없다.
- 방어를 위해 생체정보·정밀 위치·지속 기기 지문이 수집되지 않고 오탐은 직원 확인으로 복구 가능하다.

## CHECK-004 일부 인원 도착 처리

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 예약 대표자는 예약 총인원 안에서 자신이 주장하는 도착 인원으로 체크인 요청만 제출할 수 있으며 그 값은 증거나 도착 원장으로 직접 기록되지 않는다. 도착 인원 증거는 각 인원에 대해 `CHECK-001`의 서버 검증 체크인이 완료되거나, 권한 있는 해당 매장 대표자가 현장에서 확인하고 확인 방식·사유를 기록한 뒤에만 추가 전용 원장에 반영한다.
- 상태는 검증된 도착 원장을 기준으로 `미도착`, `일부 도착`, `전원 도착/체크인 완료`를 구분하고, 사용자 자기 주장만으로 상태를 전이하지 않으며 일부 도착을 전체 완료나 남은 인원의 노쇼로 자동 해석하지 않는다.
- 인원 증가는 예약 변경 정책으로 새 자원 가용성을 검증해야 하며 현장에서 임의 초과 입력하지 않는다. 남은 인원 처리·좌석 유지·지각 판단은 거래 당시 공개 정책과 `CHECK-005`·`CHECK-006` 결과를 따른다.
- 거래 수정 번호·검증 사건 ID·도착 원장·멱등 키로 병렬 입력의 합계가 예약 인원을 넘지 않게 한다. 검증되지 않은 자기 주장·위조 토큰·권한 없는 확인은 증거 추가 전에 거부하고, 잘못 입력한 도착 인원은 원본을 삭제하지 않고 권한 있는 매장·운영자의 사유 있는 정정 사건으로 보정한다.
- 사용자·매장에 현재 도착 수와 필요한 행동만 알리고 인원 변화·행위자·정정 근거를 감사한다. 동행자 이름·연락처·개별 신분을 요구하지 않는다.

### 관측 가능한 인수 조건

- 병렬 부분 체크인의 검증된 누적 인원이 예약 총인원을 넘지 않고 일부 도착만으로 전체 완료·노쇼가 생성되지 않는다.
- 예약 대표자의 자기 주장, 위조된 체크인 자료와 권한 없는 확인은 도착 증거·원장·상태 전이를 만들지 않는다.
- 정정 전후를 재현할 수 있고 구 버전 요청이 최신 도착 상태를 덮어쓰지 않는다.
- 동행자 개인정보 없이 처리되며 노쇼·금전 책임은 `CHECK-006` 등 별도 결과를 따른다.

## CHECK-005 지각 허용 시간

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 예약 등 예정 방문 거래의 지각 허용 시간 플랫폼 기본값은 5분이다. 식당 대표자는 자신의 매장에 한해 `CHECK-002` 체크인 창 종료 뒤 적용할 3분 이상 15분 이하의 값을 버전된 공개 운영 설정으로 게시할 수 있으며, 범위 밖 값·사용자별 예외·운영자의 직접 변경은 허용하지 않는다. 웨이팅의 실제 호출 뒤 도착 제한은 이 지각 허용 시간을 중복 적용하지 않고 `WAIT-011`의 3분을 따른다.
- 예약 등 예정 방문 거래를 생성할 때 적용된 지각 허용 시간·체크인 창 종료·정책 버전을 스냅샷으로 보존한다. 웨이팅은 실제 호출 사건에 중앙 호출 시각·도착 제한 시각·`WAIT-011` 정책 버전을 스냅샷으로 보존한다. 이후 매장 설정 변경은 새 거래 또는 새 호출에만 적용하며 기존 경계를 소급 변경하지 않는다.
- 중앙 시각으로 창 종료와 스냅샷 지각 허용 시간을 판정한다. 시간이 지나면 거래를 `CHECK-006` 노쇼 확정 절차에 넘길 뿐, 이 정책만으로 노쇼·수수료·제재를 자동 확정하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 체크인·지각 경과·`CHECK-006` 인계는 거래 버전·정책 버전·멱등 키의 조건부 전이로 처리한다. 지각 경계의 체크인과 인계가 경합하면 먼저 중앙에서 유효하게 확정된 전이 하나만 성공한다.
- 중앙 시각·스냅샷·거래 상태를 확인할 수 없으면 노쇼 인계를 실패 폐쇄하고 기존 거래를 유지한다. 지연된 작업은 명령 시점에 다시 검증하며 복구는 같은 인계 사건을 재실행한다.
- 적용 설정·스냅샷·판정 시각·전이·행위자·멱등 키를 감사하고, 정확 위치·동행자 신원·결제수단 원문을 추가 수집하지 않는다.

### 관측 가능한 인수 조건

- 1차 MVP에서는 매장별 설정 없이 모든 매장에 공통 5분을 적용한다. 후속 구현에서 매장별 설정을 활성화하면 3분·15분 값은 게시·적용되고 2분·16분은 모든 인스턴스에서 거부된다.
- 예약 생성 뒤 매장 설정을 바꿔도 기존 거래는 생성 당시 스냅샷으로, 새 거래는 새 값으로 판정되며 창 종료와 지각 경계가 중앙 시각으로 재현된다.
- 지각 경계의 체크인·노쇼 인계를 병렬 실행해도 체크인 또는 `CHECK-006` 인계 하나만 남고, 시간 경과만으로 자동 노쇼·수수료·제재가 생성되지 않는다.

## CHECK-006 노쇼 후보와 확정 절차

> 정책 상태: 확정

### 확정 상태·행위자·절차

- 예약 방문 시각에 매장에 방문 확인 알림을 보낸다. 유효한 QR 체크인 또는 해당 매장의 `방문 완료` 확인이 있으면 정상 방문으로 확정하고 `PAY-009`에 예약금 전액 환불 입력을 전달한다.
- 체크인 창과 예약 생성 시 고정된 지각 허용 시간이 지난 뒤 해당 매장은 노쇼를 직접 확인할 수 있다. 권한·현재 예약 상태·중앙 시각을 검증한 매장 직접 확인은 노쇼를 즉시 확정한다.
- 방문 시각부터 6시간 동안 방문 완료와 노쇼 확인이 모두 없으면 `노쇼 후보`로 전환하고 매장과 사용자에게 재알림한다.
- 방문 시각부터 24시간 안에 방문 완료 정정 또는 사용자 이의가 접수되면 자동 확정을 중단한다. 24시간까지 아무 확인·이의가 없을 때만 노쇼를 자동 확정한다.
- 매장 취소·휴점, MiriYum 체크인·알림 장애처럼 사용자 귀책으로 볼 수 없는 사건이 있거나 중앙 증거가 불명확하면 자동 확정하지 않고 검토 상태로 격리한다.

### 동시성·실패·정정·감사

- 예약 버전, 후보/확정 사건 ID와 멱등 키로 체크인·매장 확인·6시간 후보·24시간 자동 확정·이의 접수 경합 중 하나의 유효 전이만 허용한다.
- 다중 작업 인스턴스는 중앙 임대 또는 조건부 획득을 사용하되 작업 자체도 멱등하다. 지연 작업은 실행 시 현재 방문·이의·귀책 상태를 다시 검증한다.
- 오판은 `CHECK-009`의 추가 전용 정정 사건으로 반전하고 `CHECK-008` 제재와 `PAY-009` 예약금 환불을 재계산한다. 판정·증거·알림·정정 결과는 `CHECK-010`에 감사한다.

### 관측 가능한 인수 조건

- 매장 직접 확인은 유효한 권한과 지각 경계 뒤 한 번만 노쇼를 확정한다.
- 6시간에는 후보와 재알림만 생성되고, 24시간 전 방문 정정·이의·비사용자 귀책 사건은 자동 확정을 막는다.
- 24시간 경계의 체크인·이의·자동 작업을 병렬 실행해도 최종 상태와 환불·제재 입력이 하나로 수렴한다.

## CHECK-007 사용자·매장 귀책 구분

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 체크인 실패·미이행 사건은 `사용자 사유 후보`, `매장 사유 후보`, `플랫폼/외부 장애 후보`, `불명확`으로만 분류하고, 증거 검토 전 귀책·노쇼·수수료를 확정하지 않는다. `CHECK-006`과 `PAY-009`가 최종 절차·금전 결과를 정한다.
- 중앙 거래 상태, 체크인 시도, 매장 영업·취소, 플랫폼 장애, 발송 결과처럼 이미 보유한 사건을 시간순으로 연결한다. 위치·통화 녹음·신분증 등 새 개인정보를 자동 수집하거나 알림 열람을 도착 증거로 보지 않는다.
- 사건 버전과 원인 사건 ID의 조건부 갱신으로 중복·동시 분류 중 하나만 확정하고, 새 증거에는 기존 판단을 덮어쓰지 않는 재검토 버전을 만든다. 데이터 누락·서비스 장애 시 사용자 귀책으로 기본값을 두지 않는다.
- 당사자에게 후보 분류·근거 요약·이의 경로를 알리고 조회·분류·변경·최종 참조 정책을 감사한다. 상대방의 내부 메모와 개인정보는 노출하지 않는다.

### 관측 가능한 인수 조건

- 같은 사실에 중복 귀책 사건이 생기지 않고 증거 부족·플랫폼 장애가 사용자 귀책으로 자동 확정되지 않는다.
- 후보 분류가 직접 환불·수수료·제재를 실행하지 않고 `CHECK-006`·`PAY-009` 결과를 소비한다.
- 판단에 사용된 최소 사건과 변경 이력을 재현할 수 있으며 추가 민감정보 수집이 없다.

## CHECK-008 반복 노쇼 제재와 회복

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 반복 판단에는 향후 `CHECK-006` 절차로 최종 확정된 노쇼 원인 사건만 사용한다. 후보·불명확·취소·매장/플랫폼 귀책 사건은 스트라이크를 만들지 않으며, 원인 사건 하나당 계정에 추가 전용 스트라이크 사건 하나만 생성한다.
- 현재 안전 기본선에서 하나 이상의 유효 스트라이크가 있으면 사용자에게 노쇼 예방 안내와 이의 제기 경로가 포함된 경고·교육만 표시한다. 금전 청구·보증금, 영구 정지, 추천·검색·신뢰 랭킹 불이익, 예약·웨이팅·홀드·테이스팅 기능 제한은 적용하지 않는다.
- 더 강한 제재가 필요하면 적용 조건·단계·기간·회복·이의 절차를 가진 별도의 미래 버전 정책으로 확정·활성화해야 한다. 이 행은 그 정책을 만들거나 활성화하지 않으며, 미래 정책이 없거나 비활성이면 모든 인스턴스가 경고·교육 기본선만 적용한다.
- 계정-원인 사건 유일성, 스트라이크 버전·조건부 갱신·멱등 키로 같은 노쇼의 중복 누적과 다중 인스턴스 중복 경고를 막는다. `CHECK-009`로 원인 노쇼가 정정·취소되면 연결된 스트라이크를 자동 반전 사건으로 무효화하고 남은 유효 스트라이크에서 경고 상태를 멱등 재계산한다.
- 사용자에게 원인 사건, 경고·교육, 자동 반전과 이의 방법을 알리고 원인-스트라이크 연결·정책 버전·경고 전후·정정/반전 작업을 감사한다. 개인정보인 기기 지문이나 가족·동행자 계정을 연결해 확대하지 않는다.

### 관측 가능한 인수 조건

- 후보·중복 노쇼는 스트라이크를 만들지 않고 `CHECK-006`으로 최종 확정된 같은 원인 사건은 병렬·반복 처리에도 스트라이크 하나만 만든다.
- 원인 노쇼가 정정되면 연결 스트라이크와 경고 상태가 자동 반전·재계산되고 모든 인스턴스에서 같은 감사 가능한 결과가 적용된다.
- 미래의 별도 버전 정책이 명시적으로 활성화되지 않은 동안 경고·교육 외에 금전·보증금·영구 정지·랭킹 불이익·예약 기능 제한이 생성되지 않는다.

## CHECK-009 오판 정정과 이의 제기

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 예약 대표자와 해당 매장 대표자는 자신의 사건에 이의를 제출하고 이미 보유한 거래·체크인·운영 증거를 연결할 수 있다. 사건 권한이 있는 플랫폼 운영자만 검토하며 최초 처리자는 결정 근거를 남기고 이해충돌이 있으면 재배정한다.
- 상태는 `접수`, `검토`, `보완 요청`, `유지`, `정정`, `종결`이며 정정은 원 노쇼·귀책 기록을 삭제하지 않고 반대 보정 사건으로 연결한다. 검토 중에는 금전·제재를 임의 확대하지 않으며 적용 중단 여부는 `CHECK-006`·PAY 정책을 따른다.
- 사건당 활성 이의 하나, 제출·결정 멱등 키와 상태 버전으로 중복·동시 결정을 막는다. 정정 확정 후 `CHECK-008` 제재와 관련 금전 대사를 내구성 작업으로 재계산한다.
- 접수·보완·결정을 당사자에게 알리고 제출자·검토자·근거·상태 전후·후속 보정 결과를 감사한다. 증거 접근은 사건 범위로 제한하고 새 신분증·정밀 위치를 요구하지 않는다.

### 관측 가능한 인수 조건

- 병렬 이의·동시 유지/정정 중 하나의 결정만 유효하고 원 기록과 보정 연결이 모두 보존된다.
- 정정 뒤 제재·관련 원장이 중복 없이 재계산되며 후속 작업 실패가 운영자 큐에서 추적된다.
- 권한 없는 조회·결정이 차단되고 이의를 위해 불필요한 개인정보를 추가 수집하지 않는다.

## CHECK-010 체크인·노쇼 감사 증거

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 체크인 발급·검증·완료, 일부 인원, 시간 판정, 영업·취소, 알림 발송, 노쇼 후보/확정 참조, 귀책·이의·정정의 사건 ID·중앙 시각·행위자·정책/거래 버전·결과를 추가 전용 감사로 연결한다.
- 감사 증거는 누가 무엇을 언제 어떤 원본 상태로 판단했는지 재현하는 최소 메타데이터이며 QR 원문, 신분증, 정확한 위치, 연락처, 알림 내용 원문은 저장하지 않는다. 해시·무결성 값은 원본을 불필요하게 복제하지 않는 범위에서 사용한다.
- 업무 상태와 핵심 감사 사건을 같은 트랜잭션 또는 유실 없는 Outbox로 기록하고 사건 ID 유일성으로 중복 이벤트를 막는다. 감사 기록 실패 시 중요한 수동 판정은 확정하지 않는다.
- 권한 있는 사건 담당자만 조회·내보내며 모든 접근도 감사한다. 보관 기간은 개인정보·법률 TODO를 따르고 만료 전 임의 수정·삭제하지 않는다.

### 관측 가능한 인수 조건

- 각 체크인·노쇼·이의 결과를 연결된 사건과 정책 버전에서 재현할 수 있고 상태 변경만 있고 감사 증거가 없는 확정 판정을 만들 수 없다.
- 중복 이벤트·작업 재생 뒤 감사 사건이 중복되지 않으며 권한 없는 접근·삭제가 차단된다.
- QR·신분증·정밀 위치·연락처 원문 없이도 분쟁 검토에 필요한 최소 시계열이 유지된다.
