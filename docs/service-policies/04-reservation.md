# 방문 예약 정책

> 문서 상태: 세부 정책 확정
> 정책 범위: `RES-001`~`RES-015`
> 최종 변경일: 2026-07-23

예약 접수, 서비스 구간별 예약 수용량, 임시 선점, 변경·취소, 중복 예약 및 동시성 규칙을 다룬다. 1차 MVP는 즉시 확정 예약만 지원하고, 취소·환불은 `RES-009`의 공통 기준과 PAY 정책을 함께 적용한다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 확정 | 핵심 | `RES-001`, `RES-009` |
| 확정 | 필수 | `RES-002`~`RES-008`, `RES-010`, `RES-011`, `RES-013`~`RES-015` |
| 확정 | 권장 | `RES-012` |

- `RES-001`: 1차 MVP는 즉시 확정 예약만 지원한다. 매장 승인제는 향후 범위이며 현재 예약 상태나 권한으로 활성화하지 않는다.
- `RES-009`: 사용자 취소 시점과 매장·플랫폼 귀책을 MiriYum 공통 기준으로 판정하고 실제 환불은 `PAY-009`를 적용한다.
- `RES-006`: 임시 선점은 10분이며 만료 2분 전에 경고한다. 사용자 임의 연장은 허용하지 않는다.

## RES-001 즉시 확정·승인제 지원 범위

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 1차 MVP의 방문 예약은 즉시 확정 방식만 지원한다. 로그인 사용자가 방문 시각·인원을 제출하면 현재 운영·자격·중복·수용량 정책을 검증하고 `RES-006`의 10분 선점을 만든다.
- 메뉴 홀드를 선택하면 `HOLD-004`에 따라 같은 선점 그룹에 메뉴 수량을 결합한다. 예약금이 필요한 매장은 `PAY-001`의 검증된 결제 성공 뒤, 필요하지 않은 매장은 결제 불필요 판정 뒤 수용량과 선택 메뉴를 함께 최종 확정한다.
- `요청 중`, `수용량 선점`, `결제 처리 중`, `결제 확인 중`은 확정 예약으로 표시하지 않는다. 매장 승인·거절 대기 상태와 승인 권한은 1차 MVP에서 생성하지 않는다.
- 매장 승인제는 향후 정책 버전이 별도로 확정·활성화되기 전까지 공개 설정·예약 선택지·운영 권한으로 노출하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약 버전·선점 그룹·멱등 키와 중앙 조건부 전이로 다중 인스턴스의 재전송 중 하나만 확정한다. 결제·만료 경합은 `PAY-007`과 `RES-006`의 중앙 상태를 따른다.
- 일부 자원만 확정되거나 결제 성공 뒤 자원 확정이 실패하면 예약을 성공으로 표시하지 않고 멱등 재시도 또는 승인 취소·전액 환불 보상으로 수렴한다.
- 확정·실패·만료 결과를 사용자와 매장에 알리고 적용 정책·수용량·선점·결제 참조와 전이 결과를 감사한다. 승인제를 위해 추가 개인정보를 수집하지 않는다.

### 관측 가능한 인수 조건

- 유효 요청은 결제 성공 또는 결제 불필요 판정 뒤 즉시 확정되고 매장 승인 대기 상태가 생성되지 않는다.
- 중복 요청·결제 응답 유실·만료 경합에도 확정 예약과 자원 차감은 최대 한 번만 남는다.
- 향후 승인제 정책이 활성화되지 않은 상태에서 매장 승인·거절 API나 화면 권한을 전제로 하지 않는다.

## RES-002 예약 가능 기간과 슬롯 시각

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 유효한 매장 운영 권한을 가진 식당 대표자만 매장별 예약 가능 시작·종료 기간, 슬롯 시작 간격, 서비스 소요 시간, 전환 시간과 적용 시점을 정책 버전으로 게시할 수 있다. 서비스 구간별 수용량은 `RES-005`에서 관리하며 수치는 운영 데이터로 조정 가능한 설정이므로 이 정책에서 고정하지 않는다.
- 슬롯의 전체 서비스 구간은 게시된 영업시간 안에 있고 휴무·브레이크타임·휴점·폐점과 겹치지 않아야 한다. 사용자는 로그인 후 공개된 슬롯만 요청할 수 있고 플랫폼 운영자는 임의의 슬롯을 만들지 않는다.
- 매장 IANA 시간대와 중앙 기준 시각을 사용하며, 일광절약시간·자정 경계에서 존재하지 않거나 중복되는 현지 시각은 명시적으로 거부하거나 오프셋까지 식별한다.

### 정상 흐름·변경·취소·만료

- 조회는 현재 효력이 있는 운영·슬롯 정책 버전으로 후보를 만들고, 예약 요청과 최종 확정 단계에서 같은 버전과 `RES-005`의 수용량 가용성을 다시 검증한다. 슬롯 노출은 예약 성공 보장이 아니다.
- 새 정책은 효력 시각 이후 생성되는 요청에만 적용한다. 기존 확정 예약은 소급 이동·취소하지 않고, 충돌이 생기면 `RES-013` 영향 사건으로 처리한다.
- 게시 예약은 효력 전에 철회할 수 있고 활성 버전은 새 버전으로만 교체한다. 예약 가능 종료를 지난 미사용 슬롯은 요청 시점 판정으로 즉시 닫히며 스케줄러 완료에 의존하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 정책 게시와 예약 요청은 정책 버전·수용량 버전·멱등 키를 사용한다. 예약은 시작 시각부터 서비스 소요 시간과 전환 시간을 합산한 점유 구간과 겹치는 모든 서비스 구간의 수용량을 사용하며, 중앙 조건부 갱신으로 잔여 인원·팀 수 경합을 직렬화한다. 유일성 제약은 슬롯 전체를 예약 한 건으로 제한하는 데 사용하지 않고 동일 요청의 중복 생성을 막는 데 사용한다. 캐시나 인스턴스 로컬 시각은 확정 근거로 쓰지 않는다.
- 운영·정책 원본을 확인할 수 없으면 신규 확정을 실패 폐쇄한다. 검색·캐시 갱신 실패는 중앙 정책을 되돌리지 않고 재처리하며 기존 확정 예약은 유지한다.
- 일반 슬롯 게시에는 사용자 알림을 보내지 않는다. 기존 예약 충돌만 당사자에게 알리고, 행위자·정책 버전·효력 시각·충돌 예약 식별자·처리 결과를 감사한다. 추가 개인정보나 결제 정보는 수집하지 않는다.

### 관측 가능한 인수 조건

- 영업 외·휴무·브레이크타임 슬롯과 모호한 현지 시각이 게시·확정되지 않고 모든 인스턴스가 같은 결과를 반환한다.
- 정책 전환과 예약 요청이 경합해도 예약은 정확히 한 정책 버전을 참조하며 재전송이 중복 예약을 만들지 않는다.
- 새 정책이 기존 확정 예약을 자동 변경하지 않고 충돌 사건·알림·감사 이력이 한 번만 생성된다.

## RES-003 예약 인원 최소·최대와 영유아 포함 기준

> 정책 상태: 개정 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 예약 상품별 최소 인원과 최대 일행 인원, 좌석을 차지하지 않는 영유아의 허용 여부를 버전된 설정으로 게시한다. 구체적 수치는 매장 설정이며 플랫폼이 일률적으로 정하지 않는다.
- 사용자는 성인·아동·영유아 인원수를 구분해 입력하고 총 동행 인원을 숨기지 않는다. 안전 및 서비스 구간 수용량 판정에는 영유아를 포함한 전체 동행 인원을 사용한다.
- 생년월일·아동 이름 같은 추가 개인정보는 수집하지 않는다. 유아용 의자 등 설비 요청은 수용량 예약과 분리된 선택 요청으로 관리하고 보장 여부를 명확히 표시한다.

### 정상 흐름·변경·취소·상태

- 서버는 현재 정책으로 `최소 인원 ≤ 요청 인원 ≤ 최대 일행 인원`을 검증한 뒤 `RES-005`의 모든 관련 서비스 구간에서 요청 인원과 팀 수를 선점한다. 검증 실패는 허용 범위와 재입력 방법만 안내한다.
- 최대 일행 인원을 초과한 요청을 여러 독립 예약으로 자동 분할하거나 인원을 임의로 줄이지 않는다. 별도 문의 경로를 이 정책이 약속하지 않는다.
- 인원 변경은 `RES-008`에 따라 새 총인원으로 모든 관련 서비스 구간의 수용량을 다시 확인하며, 성공 전까지 원래 예약과 점유량을 유지한다.
- 설정 변경은 이후 요청에 적용하고 기존 예약의 인원 스냅샷을 유지한다. 실제 이행 불가 충돌은 `RES-013`으로 보내며 이 정책이 취소 책임을 정하지 않는다.

### 동시성·실패·복구·알림·감사

- 인원 변경과 관련 서비스 구간의 수용량 전환은 예약 버전·멱등 키와 하나의 조건부 처리 경계에서 확정한다. 일부 구간이나 일부 인원만 변경된 부분 성공을 남기지 않는다.
- 수용량 검증 장애에는 변경·신규 예약을 실패 폐쇄하고 원래 예약을 유지한다. 재시도는 동일 요청 결과를 반환한다.
- 인원·설비 변경 결과는 예약 대표자와 매장에 알리고, 이전·새 인원 구분, 정책·수용량 버전, 판정과 사유를 감사하되 동행자의 신원은 기록하지 않는다.

### 관측 가능한 인수 조건

- 최소·최대 경계와 영유아 포함 기준이 게시된 정책대로 판정되고 범위 밖 요청은 수용량을 점유하지 않는다.
- 동시 인원 변경에도 예약 인원과 모든 관련 서비스 구간의 점유량이 일치하며 실패 시 원래 인원·점유량이 그대로 남는다.
- 설정 변경 뒤에도 기존 예약 스냅샷이 변하지 않고 동행자 추가 개인정보가 로그·알림에 나타나지 않는다.

## RES-004 서비스 구간 수용량 자원 모델

> 정책 상태: 개정 확정

### 확정 규칙·행위자·권한·사전 조건

- 플랫폼의 방문 예약 자원은 개별 테이블이나 좌석이 아니라 안정 식별자가 있는 서비스 구간별 예약 수용량이다. 각 구간은 매장 시간대, 시작·종료 시각, 전체 수용 인원, 플랫폼 예약 제공 인원, 최대 팀 수, 수용량 버전과 운영 상태를 가진다.
- 플랫폼은 개별 테이블, 테이블 번호, 좌석 위치, 물리적 배치와 테이블 조합을 등록·선점·배정하거나 고객에게 보장하지 않는다. 실제 착석 위치와 테이블 구성은 매장이 현장에서 결정한다.
- 식당 대표자만 수용량 설정을 게시·중지할 수 있고, 사용자는 공개된 시간과 인원을 요청한다. 식당 대표자는 실제 테이블 구성, 일행별 착석 가능성, 서비스 소요 시간과 전환 시간을 고려해 플랫폼 제공 인원·최대 팀 수·최대 일행 인원을 설정할 책임이 있다.
- 수용량은 `예약 가능`, `임시 선점`, `확정 점유`, `운영 중지`로 구분해 계산하며 예약 상태는 `RES-014`를 따른다. 1차 MVP에서는 `RES-001`의 즉시 확정 흐름만 수용량 점유로 전환한다.

### 정상 흐름·변경·취소·만료

- 요청은 시작 시각부터 서비스 소요 시간과 전환 시간을 합산한 점유 구간과 겹치는 모든 서비스 구간에서 요청 인원과 팀 수 1건을 하나의 수용량 선점 기록으로 확보하고, 선택된 확정 방식의 성공 시 점유로 전환한다. 실패·거절·유효한 만료에는 전체 선점을 해제한다.
- 서비스 구간은 매장이 게시한 고정 시간 단위로 생성하고, 예약 점유 구간과 조금이라도 겹치는 모든 구간을 동일 인원과 팀 수 1건으로 점유한다.
- 게시된 수용량 설정은 직접 덮어쓰지 않고 새 버전으로 바꾼다. 기존 예약이 참조하는 버전과 당시 스냅샷은 삭제하지 않으며 운영 중지는 신규 선점만 차단한다.
- 임시 선점의 10분 만료와 만료 2분 전 경고는 `RES-006`을 따른다. 만료 후 뒤늦은 확정은 현재 정책과 모든 관련 구간의 가용성 재검증 없이 허용하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 모든 관련 서비스 구간의 인원과 팀 수는 가능한 경우 하나의 DB 트랜잭션과 조건부 갱신으로 전부 확보하거나 전부 실패한다. 외부 장애로 일부 구간만 선점되면 예약을 확정하지 않고 보상 작업으로 전체를 해제한다.
- 중앙 수용량 상태를 확인할 수 없으면 신규 확정을 실패 폐쇄한다. 해제가 확인되지 않은 점유량은 다시 판매하지 않으며 복구 작업이 고아 선점과 구간별 원장을 대사한다.
- 수용량 설정·선점·점유·해제·중지의 행위자, 예약·구간·버전과 결과를 감사한다. 테이블·좌석·내부 배치 정보와 결제·추가 개인정보는 이 모델에 저장하거나 사용자에게 노출하지 않는다.

### 관측 가능한 인수 조건

- 같은 구간의 동시 요청 중 게시된 제공 인원과 최대 팀 수가 허용하는 예약만 성공하며 초과 점유나 음수 가용량이 없다.
- 여러 구간을 사용하는 요청의 중간 실패와 작업 재시작 뒤 일부 구간 점유만 남지 않고 중앙 상태가 전부 점유 또는 전부 해제로 수렴한다.
- 수용량 설정 변경 후에도 기존 예약은 당시 버전으로 재현되고 운영 중지 구간에는 신규 점유가 생기지 않는다.

## RES-005 서비스 구간별 예약 수용량 관리

> 정책 상태: 개정 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 서비스 구간별 전체 수용 인원, 플랫폼 예약 제공 인원, 플랫폼 예약 최대 팀 수와 예약 한 건당 최대 일행 인원을 버전된 설정으로 게시한다. `0 ≤ 플랫폼 예약 제공 인원 ≤ 전체 수용 인원`이고 최대 일행 인원은 플랫폼 예약 제공 인원을 넘을 수 없다.
- 전체 수용 인원에서 플랫폼 예약 제공 인원을 뺀 수량은 워크인과 현장 운영을 위해 사전에 제외한 기준 몫이다. 다만 제공 인원을 기존 온라인 점유보다 낮춘 경우 그 차이가 즉시 현장 가용량으로 늘어나는 것은 아니며, 전체 수용 인원에서 기존 온라인 수용 의무를 먼저 뺀 범위 안에서만 현장에 사용한다. 온라인과 현장 몫을 독립된 재고처럼 중복 판매하지 않는다.
- 플랫폼은 개별 테이블, 테이블 번호, 물리적 배치 및 테이블 조합을 예약 자원으로 관리하거나 고객에게 보장하지 않는다. 실제 착석 위치와 테이블 구성은 매장이 현장에서 결정한다.
- 예약 요청은 `RES-004`가 계산한 모든 관련 서비스 구간에서 요청 인원과 팀 수 1건을 수용할 수 있고 `RES-003`의 최대 일행 인원을 충족할 때만 후보가 된다.

### 정상 흐름·변경·취소·만료

- 예약 요청 인원과 팀 수 1건은 모든 관련 서비스 구간에서 하나의 수용량 선점 기록으로 원자 선점한다. 같은 일행을 여러 독립 예약으로 자동 분할하지 않는다.
- 구간별 신규 예약 가능 인원은 `max(0, 현재 플랫폼 예약 제공 인원 - 유효 선점 인원 - 확정 대기 인원 - 확정 예약 인원)`, 신규 예약 가능 팀 수는 `max(0, 현재 최대 팀 수 - 유효 선점 팀 수 - 확정 대기 팀 수 - 확정 예약 팀 수)`로 계산한다.
- 예를 들어 플랫폼 예약 제공 인원 18명 중 11명이 점유된 경우, 최대 팀 수와 최대 일행 인원 조건도 충족하면 6명 요청은 선점할 수 있지만 8명 요청은 선점할 수 없다.
- 수용량 설정 변경은 이후 요청에 적용한다. 이미 확정되거나 유효하게 선점된 예약을 소급 취소·축소하지 않으며, 현재 제공 인원 또는 최대 팀 수를 기존 점유 인원·팀 수보다 낮추면 해당 신규 예약 가능량을 0으로 처리하고 매장에 알린다.
- 선점 만료, 예약 인원 변경 또는 유효한 취소가 발생하면 해당 예약의 점유 인원과 팀 수를 한 번만 제거하고 현재 설정과 남은 유효 점유를 기준으로 가용량을 다시 계산한다. 과거 제공량에서 차감했던 수량을 가용량에 그대로 더하지 않는다.

### 워크인과 현장 운영

- 워크인은 매장이 별도로 확보한 현장 운영 몫에서 수용한다. 플랫폼은 모든 워크인의 입장과 퇴장을 실시간으로 등록해야만 예약 정합성이 유지되는 구조에 의존하지 않는다.
- 매장이 아직 온라인 예약에 제공 중인 몫을 워크인에게 사용하려면 워크인을 수용하기 전에 해당 서비스 구간의 플랫폼 예약 제공 인원을 먼저 축소하고 그 변경이 중앙 시스템에 확정됐는지 확인해야 한다. 확정 전까지 해당 인원은 온라인 예약에 제공되는 것으로 본다. 축소 뒤에도 기존 온라인 점유를 포함한 전체 수용 의무를 뺀 실제 잔여 인원만 워크인에게 사용할 수 있다.
- 제공 인원을 축소해도 이미 확정되거나 유효하게 선점된 예약의 수용 의무는 유지된다. 실제 현장 사정으로 확정 예약을 수용할 수 없으면 `RES-013`의 변경 제안·영향 처리로 전환한다.

### 동시성·실패·복구·알림·감사·개인정보

- 동시에 여러 예약 요청이 접수되더라도 모든 관련 서비스 구간의 현재 수용량 버전과 잔여 인원·팀 수를 조건부 갱신하여 플랫폼 예약 제공량과 최대 팀 수를 초과하지 않게 한다.
- 멱등 키가 같은 요청은 인원과 팀 수를 반복 차감하지 않고 기존 수용량 선점 기록과 같은 결과를 반환한다.
- 일부 서비스 구간에서만 선점된 경우 예약을 확정하지 않고 보상 작업으로 모든 구간의 선점을 해제한다. 해제가 완료되지 않은 인원·팀 수는 신규 예약 가능량으로 다시 판매하지 않는다.
- 수용량 설정을 읽지 못하거나 모든 관련 구간의 잔여 인원·팀 수를 확정할 수 없으면 신규 예약을 실패 폐쇄한다.
- 수용량 설정·변경, 예약 인원·팀 수 선점과 해제, 초과 시도 및 실패 복구를 감사한다. 고객에게 테이블 번호나 내부 배치 정보를 노출하지 않으며 이 정책을 위해 개인정보를 추가 수집하지 않는다.

### 관측 가능한 인수 조건

- 예약 제공 인원이 10명인 구간에 합계 12명의 요청을 동시에 보내도 점유 인원은 10명을 초과하지 않고, 최대 팀 수를 넘는 동시 요청도 초과 확정되지 않는다.
- 동일한 멱등 키를 재전송해도 예약 인원과 팀 수가 중복 차감되지 않는다.
- 여러 서비스 구간을 사용하는 예약의 일부 선점에 장애가 발생해도 부분 예약은 확정되지 않는다.
- 제공 인원이나 최대 팀 수를 기존 점유량보다 낮춰도 기존 예약은 유지되고 해당 신규 예약 가능량만 0이 된다. 이후 취소·만료가 발생하면 현재 설정을 기준으로 가용량이 다시 계산된다.
- 워크인을 플랫폼에 실시간 입력하지 않아도 별도로 확보한 현장 운영 몫은 온라인 예약으로 판매되지 않는다. 플랫폼 몫 전용은 제공 인원 축소가 먼저 확정되고 기존 온라인 수용 의무를 제외한 실제 현장 잔여 인원이 있는 경우에만 허용된다.
- 고객 응답과 일반 로그에 특정 테이블·조합·내부 배치 정보가 나타나지 않는다.

## RES-006 예약 임시 선점과 만료

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 예약 생성·변경의 임시 선점은 중앙 시각 기준 정확히 10분간 유효하며, 선점 소유자·관련 서비스 구간·점유 인원·팀 수·정책 및 수용량 버전·만료 시각·멱등 키를 함께 기록한다. 만료 2분 전에는 선점 소유자에게 경고를 한 번 생성하되, 경고 발송 성공 여부는 선점의 유효성을 바꾸지 않는다.
- 사용자·매장·운영자는 임의로 선점 만료를 연장하거나 만료 시각을 직접 수정할 수 없다. 필요한 새 요청은 기존 선점이 유효하게 종결된 뒤 현재 가용성과 정책을 다시 검증한다.
- 결제 결과가 불명확하거나 중앙 처리 장애가 난 경우 사용자의 취소·실패 책임으로 기록하지 않는다. 해당 선점은 `결과 대사/복구 필요`로 격리하고 결제 원본과 서비스 구간별 수용량 원장을 대사한 뒤에만 확정 또는 해제한다.

### 정상 흐름·동시성·실패·복구

- 선점 생성 → 만료 2분 전 경고 → 결제·확정 또는 중앙 만료 → 모든 관련 서비스 구간의 점유 인원·팀 수 정확히 한 번 해제 순으로 처리한다. 만료는 스케줄러 실행뿐 아니라 모든 확정·변경 명령 시점에도 중앙 시각으로 재검사한다.
- 선점·결제 확정·만료·해제는 같은 선점 및 수용량 버전과 조건부 갱신으로 처리한다. 여러 인스턴스의 만료 작업·재전송이 겹쳐도 한 전이만 성공하고, 뒤늦은 작업은 최신 결과를 반환한다.
- 해제·대사·복구에는 선점·관련 서비스 구간·점유 인원·팀 수·정책 및 수용량 버전·기준 시각·결제 참조·멱등 키·결과를 감사한다. 결제수단 원문이나 추가 개인정보는 이 정책 데이터에 기록하지 않는다.

### 관측 가능한 인수 조건

- 중앙 만료 시각 직전에는 확정이 가능하고 정확히 만료 시각 및 직후에는 새 가용성 검증 없는 확정이 거부되며, 만료 2분 전 경고는 최대 한 번 생성된다.
- 동일 선점의 결제 확정과 다중 인스턴스 만료를 병렬 실행해도 확정 또는 정확히 한 번 해제 중 하나만 남고 인원·팀 수가 이중 반환되지 않는다.
- 결제 결과 불명확·서버 중단을 주입해도 사용자 책임 상태나 자동 재판매가 생기지 않고 대사 뒤 감사 가능한 확정 또는 단일 해제로 수렴한다.

## RES-007 예약자 연락처 정보

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 예약 대표자는 로그인한 일반 사용자이며 계정에서 소유 확인된 연락 채널을 예약 알림에 사용한다. 별도 자유 입력 전화번호나 동행자 연락처를 필수 수집하지 않는다.
- 식당 대표자는 해당 매장의 유효한 예약 이행에 필요한 예약자 표시명, 마스킹 연락 경로와 인원·요청사항만 조회할 수 있다. 플랫폼 운영자는 고객지원·분쟁 권한과 사건이 있을 때만 최소 범위를 조회한다.
- 예약에는 계정 연락처 원문 대신 계정 식별자, 알림 대상 참조와 거래 당시 연락 가능 상태를 연결한다. 연락처 변경은 과거 감사 기록을 덮어쓰지 않는다.

### 정상 흐름·변경·취소·만료

- 생성 전 사용 가능한 확인 채널을 검증하고 없으면 계정 설정 경로를 안내한다. 예약 상태 알림은 `RES-014`의 확정 전이를 기준으로 내구성 있는 작업으로 생성한다.
- 계정 연락처가 바뀌면 이후 발송은 현재 확인된 채널을 사용하되 이미 발송한 내용과 당시 대상 참조는 유지한다. 예약 양도·대리 범위는 `RES-011`을 따른다.
- 예약 종료·취소 뒤 식당의 운영 화면에서는 연락처 접근을 제거하고, 정확한 보관·파기 기간은 개인정보 정책 TODO를 따르며 이 행에서 새 기간을 정하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약 전이와 알림 작업 생성은 같은 사건 식별자로 연결하고 소비자는 멱등하게 발송한다. 발송 실패가 예약 상태를 되돌리지 않으며 제한 재시도 후 운영 경보·수동 재발송으로 남긴다.
- 권한 또는 계정 연락 상태를 확인할 수 없으면 원문 연락처를 우회 노출하지 않는다. 필수 알림 실패는 앱 내 최신 상태 조회를 유지한다.
- 조회·내보내기·발송에는 예약, 목적, 행위자, 마스킹된 대상과 결과를 감사하고 로그·분석·URL에는 연락처 원문을 남기지 않는다.

### 관측 가능한 인수 조건

- 확인된 연락 채널이 없는 사용자는 대체 개인정보 입력 없이 안전한 계정 설정 안내를 받고 예약 정보가 생성되지 않는다.
- 권한 없는 매장·운영자·다른 사용자가 연락처나 예약 상세를 조회할 수 없고 종료 후 매장 화면 접근이 제거된다.
- 같은 상태 이벤트의 중복 소비가 알림 작업을 중복 생성하지 않으며 발송 장애가 예약 확정·취소 결과를 바꾸지 않는다.

## RES-008 예약 변경 가능 항목과 마감

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 예약 대표자와 해당 매장 권한이 유효한 식당 대표자만 예약 변경을 요청할 수 있다. 변경 가능한 필드·요청 마감은 매장별 버전된 공개 정책으로 관리하되 구체적 시간 수치는 이 행에서 고정하지 않는다.
- 시간·인원·메뉴 등 가용성에 영향을 주는 변경은 새 요청과 같은 서비스 구간별 수용량·재고 검증을 거친다. 표시 메모처럼 가용성에 영향 없는 필드도 허용 목록과 입력 제한을 적용한다.
- 변경은 원본 예약 ID 아래 새 수정 버전으로 만들고, 원본 확정 스냅샷·가격·정책 버전을 덮어쓰지 않는다. 변경 가능 여부는 서버 중앙 시각으로 판정한다.

### 정상 흐름·변경·취소·만료

- 요청은 `변경 검증 중`에서 새 시간의 모든 관련 서비스 구간 또는 인원 증가분을 임시 선점하고 필요한 동의·확정 방식을 거쳐 한 번에 새 버전으로 전환한다. 성공 전까지 원래 예약과 수용량 점유를 유지한다.
- 사용자는 확정 전 변경 요청을 철회할 수 있고 후보 수용량 선점은 해제된다. 변경 실패·거절·만료에도 원래 예약은 유지하며 이를 취소로 해석하지 않는다.
- 매장 제안 변경은 `RES-013`을 따르고 사용자 동의 없이 확정하지 않는다. 변경 중 취소 요청의 우선관계와 비용은 확정된 `RES-009` 공통 기준을 적용한다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약 수정 번호·멱등 키·조건부 갱신으로 동시에 하나의 활성 변경만 허용한다. 구 버전 변경 요청은 충돌로 거부하고 최신 예약을 반환한다.
- 새 구간 또는 증가분 확보 뒤 원본 점유나 감소분 해제에 실패하면 관련 가용량의 신규 사용을 잠시 차단한 채 중앙 전환 사건을 멱등 재실행한다. 어느 시점에도 같은 점유량을 둘 이상의 예약 가능량으로 계산하지 않는다.
- 요청·확정·거절·철회·만료를 당사자에게 알리고 이전·새 스냅샷, 행위자, 이유, 동의와 결과를 감사한다. 변경에 필요 없는 새 개인정보는 수집하지 않는다.

### 관측 가능한 인수 조건

- 가용성 영향 변경이 실패하거나 만료되면 원래 예약·점유량이 유지되고 후보 수용량 선점은 해제된다.
- 동시 변경 중 하나만 확정되며 재전송은 같은 수정 버전을 반환하고 인원·팀 수의 이중 해제가 없다.
- 마감 정책 변경은 기존 예약의 당시 공개 스냅샷을 보존하고 매장 제안은 사용자 동의 전에 적용되지 않는다.

## RES-009 사용자·식당 취소 정책

> 정책 상태: 확정

### 확정 규칙·행위자·기준

- 사용자는 자신의 확정 예약을 취소할 수 있고, 매장 사유 취소는 해당 매장의 유효한 운영 권한을 가진 대표자가 근거와 함께 요청한다. 플랫폼 운영자는 장애·분쟁 권한과 사건 없이 대신 취소하지 않는다.
- 사용자 취소의 환불 입력은 예약 확정 후 10분 이내이면서 방문 예정 시각 전이면 100%, 방문 예정 시각 48시간 전까지 100%, 48시간 미만부터 24시간 전까지 50%, 24시간 미만이면 0%다.
- 매장 귀책, MiriYum 장애, 중복 결제, 결제만 성공하고 예약 확정에 실패한 경우는 100% 환불 입력으로 판정한다. 확정 노쇼의 환불 입력은 `CHECK-006`과 `PAY-009`에 따라 0%다.
- 판정에는 예약 생성 당시 공개된 취소 정책 버전, 확정 시각, 방문 예정 시각, 취소 접수 중앙 시각과 귀책 사건을 사용한다. 이 정책은 최종 약관·법률 검토를 완료한 것으로 해석하지 않는다.

### 정상 흐름·동시성·복구

- 취소 요청은 자격·현재 상태·기준 시각을 검증한 뒤 예약을 한 번 종결하고, 점유 수용량과 결합 메뉴 수량을 각각의 원장에 정확히 한 번 반환하며 환불 입력을 `PAY-009`에 전달한다.
- 취소·변경·체크인·노쇼 판정이 경합하면 예약 버전과 명령 멱등 키의 조건부 전이 하나만 확정한다. 이미 종결된 예약의 재요청은 기존 결과를 반환한다.
- 자원 반환이나 환불 실행이 실패해도 취소 판정을 되돌리지 않고 별도 복구 상태로 추적한다. 결과 불명확 건은 재판매·중복 환불하지 않고 대사한다.

### 알림·감사·개인정보와 인수 조건

- 취소 주체·접수 시각·적용 구간·환불 비율·귀책 요약·자원 반환·환불 상태를 당사자에게 알리고 감사하되 내부 메모나 결제수단 원문을 노출하지 않는다.
- 각 경계 시각에서 공개된 비율이 결정적으로 적용되고 재전송이 취소·수량 반환·환불을 중복 생성하지 않는다.
- 매장·플랫폼 귀책은 사용자 취소 구간과 무관하게 100% 환불 입력으로 전달되며 실제 환불은 PAY 원장과 대사 정책으로 수렴한다.

## RES-010 중복 예약과 예약·웨이팅 충돌 판정

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 같은 일반 사용자 계정이 같은 매장에 겹치는 서비스 구간으로 가진 활성 요청·보류·확정 예약을 중복 후보로 본다. 동행자 이름·전화번호·기기 지문을 추가 수집해 동일인을 추정하지 않는다.
- 정확히 같은 의도의 재전송은 멱등 키와 요청 지문으로 기존 결과를 반환한다. 서로 다른 요청이 중복 후보이면 자동 취소·병합하지 않고 신규 확정을 차단하며 기존 예약 확인·변경 경로를 안내한다.
- 다른 매장 예약이나 겹치지 않는 구간을 일률적으로 금지하지 않는다. 더 넓은 이용 제한이나 제재는 별도 정책·근거 없이 적용하지 않는다.
- 같은 사용자·매장·겹치는 영업 구간의 활성 웨이팅과 예약 요청이 충돌하면 강제 차단 화면만 제공하지 않고 기존 웨이팅 유지 또는 웨이팅을 종결한 예약 전환 선택지를 표시한다. 반대 방향은 `WAIT-017`의 예약 유지 또는 예약 취소 후 웨이팅 전환을 적용한다.
- 예약 취소가 포함된 전환은 `RES-009`·`PAY-009`로 계산한 취소 수수료·환불 결과와 계산 정책 버전을 먼저 표시하고 사용자의 명시적 확인을 받아야 한다.

### 정상 흐름·변경·취소·만료

- 신규·변경 요청 직전에 활성 예약 구간을 중앙 조회하고, 후보 선점과 중복 방지 표식을 같은 처리 경계에서 확정한다.
- 기존 요청이 유효하게 취소·거절·만료되어 수용량 점유가 해제된 뒤에만 새 요청을 허용한다. 취소 가능 여부와 책임은 `RES-009` 결과를 따른다.
- 중복 판정 오탐 신고는 플랫폼 운영자가 근거와 권한을 갖고 검토할 수 있으나 기존 예약을 삭제하거나 새 예약을 강제 확정하지 않는다.
- 웨이팅에서 예약으로 전환할 때는 예약 수용량을 조건부 선점한 뒤 웨이팅 종결과 예약 생성·선점을 하나의 전환 사건으로 확정한다. 전환을 완료할 수 없으면 기존 웨이팅과 순번을 유지하고 예약을 부분 생성하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 사용자·매장·겹침 구간에 대한 중앙 제약과 조건부 갱신으로 여러 인스턴스의 동시 요청 중 하나만 활성화한다. 캐시만으로 중복 없음이라고 판정하지 않는다.
- 중복 검증 저장소 장애 시 신규 확정을 실패 폐쇄하고 기존 예약은 유지한다. 고아 보류 표식은 상태 대사로 제거한다.
- 차단 결과에는 기존 예약의 최소 식별·시간만 본인에게 표시한다. 요청 지문, 충돌 예약, 판정·해제·운영자 조치를 감사하되 연락처 원문은 기록하지 않는다.
- 상호 전환은 원 거래 버전, 대상 자원·접수 상태 버전, 전환 멱등 키와 사용자가 확인한 비용 계산 버전을 조건부로 검사한다. 외부 결제·환불은 Outbox 후속 작업으로 실행하고, 결과 불명이나 부분 장애에는 원 거래를 먼저 잃지 않도록 중앙 전환 원장과 PAY 상태를 대사한다.

### 관측 가능한 인수 조건

- 같은 사용자·매장·겹치는 구간의 동시 요청에서 활성 예약은 하나만 남고 동일 멱등 키는 같은 결과를 반환한다.
- 다른 매장 또는 비겹침 요청은 이 규칙만으로 차단되지 않으며 동행자 개인정보 없이 판정된다.
- 검증 장애에는 신규 예약이 확정되지 않고 복구·대사 후 고아 표식 없이 재시도할 수 있다.
- 예약·웨이팅 상호 전환에 실패하면 원 예약 또는 원 웨이팅이 유지되며, 예약 취소만 성공하거나 웨이팅 종결만 성공한 부분 결과가 남지 않는다.

## RES-011 대리 예약과 예약 양도

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 로그인한 예약 생성자가 예약 대표자와 책임 주체로 남으며 자신의 일행을 위해 예약할 수 있다. 동행자 계정·연락처 등록은 요구하지 않는다.
- 현재 안전 기본선에서는 예약 소유권을 다른 계정에 직접 양도하지 않는다. 참석자 구성이 바뀌어도 대표자 변경으로 처리하지 않으며, 대표자가 참석할 수 없는 경우 이용 가능한 변경·취소 절차를 안내한다.
- 식당 대표자나 플랫폼 운영자는 사용자 대신 소유자를 바꾸거나 자신에게 예약을 이전할 수 없다. 고객지원은 본인 확인과 사건 권한 안에서 상태를 조회·재전송할 뿐이다.

### 정상 흐름·변경·취소·만료

- 예약 생성·알림·변경 요청은 대표자 계정에 귀속한다. 입장 시 필요한 참석 확인 방식은 체크인 정책을 따르며 이 행이 새로운 신원 수집을 만들지 않는다.
- 인원 변경은 `RES-003`·`RES-008`, 취소는 확정된 `RES-009`, 매장 귀책 변경은 `RES-013`을 따른다. 양도 불가가 취소 수수료나 보상 책임을 새로 정하지 않는다.
- 계정 탈퇴·정지 등으로 대표자 권한이 사라지면 계정 정책과 거래 정리 절차를 따르며 다른 일행에게 자동 이전하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약의 소유 계정 식별자는 불변이며 변경 명령은 소유자·예약 버전·멱등 키를 검사한다. 여러 인스턴스에서 위조된 대표자 교체 요청을 동일하게 거부한다.
- 인증·권한 저장소 장애에는 변경·취소 명령을 실패 폐쇄하고 기존 예약을 유지한다. 고객지원 복구도 소유자 값을 직접 수정하지 않는다.
- 대표자에게만 상태 알림을 보내고 동행자 연락처를 수집·노출하지 않는다. 대리 이용 관련 요청과 거부, 행위자·이유를 감사하되 일행 신원을 기록하지 않는다.

### 관측 가능한 인수 조건

- 다른 계정·매장·운영자가 대표자 변경을 요청해도 예약 소유자가 바뀌지 않으며 모든 인스턴스가 같은 권한 결과를 낸다.
- 일행 예약은 동행자 계정·연락처 없이 생성 가능하고 인원 변경이 대표자 소유권을 바꾸지 않는다.
- 권한 장애와 재전송에도 기존 예약은 유지되고 취소·금전 책임은 `RES-009` 결정을 벗어나 새로 적용되지 않는다.

## RES-012 단체 예약과 별도 승인

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 예약 상품별로 단체로 분류할 인원 조건과 필요한 추가 운영 정보 항목을 버전된 공개 정책으로 설정할 수 있다. 단체 기준 숫자와 최대 일행 인원은 매장 운영 설정이며 플랫폼이 일률적으로 정하지 않는다.
- 단체 요청도 1차 MVP에서는 `RES-001`의 즉시 확정 범위를 그대로 따른다. 단체라는 이유만으로 별도 승인제를 만들지 않으며 향후 승인제는 별도 정책 버전 없이는 활성화하지 않는다.
- 추가 정보는 접근성·메뉴 준비에 필요한 구조화된 항목으로 최소화하고 주민번호, 동행자 연락처·이름 목록 같은 불필요한 개인정보를 요구하지 않는다. 플랫폼은 단체 요청에도 특정 좌석·테이블 배치를 보장하지 않는다.

### 정상 흐름·변경·취소·만료

- 요청은 인원·서비스 구간별 수용량·영업·중복 검증 후 단체 표시와 적용 정책 버전을 고정하고 선택된 `RES-001` 흐름으로 전환한다. 최대 일행 인원을 초과한 요청은 거절하고 여러 독립 예약으로 자동 분할하거나 별도 문의 경로를 약속하지 않는다. 중간 상태를 확정으로 표시하지 않는다.
- 인원 변경으로 단체 기준을 넘나들면 새 정책과 모든 관련 서비스 구간의 수용량을 재검증하고 사용자에게 흐름 변화가 있으면 확정 전에 안내한다. 기존 예약을 자동 취소하지 않는다.
- 거절·철회·만료·취소는 일반 예약 상태와 `RES-009` 결과를 따르며 단체라는 이유로 별도 금전 책임·우선권을 만들지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 단체 요청도 같은 예약·수용량 멱등성 및 조건부 갱신을 적용하고 별도 수동 목록이나 단일 인스턴스 메모에 의존하지 않는다.
- 필수 검증이나 선택된 승인 흐름의 상태를 확인할 수 없으면 실패 폐쇄하고 모든 관련 구간의 수용량 선점을 해제한다. 복구는 중앙 예약 사건으로 재개한다.
- 요청·추가 정보 보완·선택된 확정 결과를 대표자와 매장에 알리고, 인원·정책 버전·결정자·사유를 감사한다. 운영 정보의 접근은 해당 예약 이행 권한으로 제한한다.

### 관측 가능한 인수 조건

- 단체 경계 전후 요청이 버전된 매장 정책과 `RES-001` 선택 결과를 그대로 따르고 별도 확정 방식을 암묵적으로 만들지 않는다.
- 중복 제출·다중 인스턴스 처리에도 단체 예약 사건과 수용량 점유는 하나만 생성된다.
- 불필요한 일행 개인정보 없이 처리되고 실패·인원 변경 시 원래 예약과 점유량이 일관되게 유지된다.

## RES-013 매장 규칙 변경·취소

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 해당 매장의 유효한 운영 권한을 가진 식당 대표자만 기존 예약의 이행 변경을 제안하거나 매장 사유의 취소 처리를 시작할 수 있다. 플랫폼 운영자는 장애·분쟁 권한과 기록된 사유 없이 대신 처리하지 않는다.
- 영업시간·수용량·메뉴 정책 변경은 기존 예약을 소급 수정하지 않는다. 제공 인원이나 최대 팀 수를 기존 점유보다 낮춘 사실만으로 영향 사건을 만들지 않고 해당 신규 예약 가능량을 0으로 처리한다. 실제 이행 불가가 확인된 예약에는 영향 사건을 만들며, 변경 제안에는 원래·대안 조건, 이유, 응답 경로와 적용될 `RES-009`·PAY 정책 참조를 명확히 표시한다.
- 사용자의 명시적 수락 전에는 원래 예약을 유지하며 침묵·알림 열람·알림 실패를 동의로 보지 않는다. 안전상 이행 불가 시 신규 이용만 차단하고 최종 취소 책임은 확정된 `RES-009` 공통 기준을 따른다.

### 정상 흐름·변경·취소·만료

- 영향 탐지 → 매장 검토 → 변경 제안 또는 취소 요청 → 사용자 응답 → 확정 전이 순으로 처리한다. 변경 수락은 `RES-008` 수용량 재검증과 원자 전환을 거친다.
- 사용자가 거절하거나 제안이 버전된 유효기간을 지나면 변경은 적용하지 않는다. 이후 취소·원래 이행 여부는 공개 정책과 `RES-009` 결과로 처리하며 임의 기본값을 만들지 않는다.
- 같은 충돌 원인의 여러 예약은 각각 독립 사건으로 추적한다. 운영 정책을 되돌려도 이미 확정된 사용자 선택·취소를 자동 복원하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약별 활성 영향 사건을 하나만 허용하고 예약 버전·제안 버전·멱등 키로 중복 제안·응답을 막는다. 취소와 수락 경합은 중앙 조건부 전이 중 하나만 성공한다.
- 알림 실패는 변경 동의가 아니며 원래 상태를 유지한다. 수용량 전환의 일부 구간 실패는 `RES-008` 보상 흐름으로 복구하고 자동 복구 불가 건은 운영자 큐로 보낸다.
- 제안·응답·취소 결과를 당사자에게 알리고 행위자, 사유, 이전·대안 스냅샷, 정책 참조와 처리 결과를 감사한다. 알림·로그에는 필요한 최소 예약 정보만 사용한다.

### 관측 가능한 인수 조건

- 운영 정책 변경만으로 기존 예약의 시간·인원·가격·상태가 바뀌지 않고 사용자 수락 전 변경 제안은 미적용 상태다.
- 동시 수락·거절·취소 요청 중 하나의 최종 전이만 성공하고 중복 알림·수용량 전환이 없다.
- 알림 장애와 정책 되돌림이 사용자 동의나 취소를 만들어내지 않으며 금전 결과는 `RES-009`·`PAY-009`의 확정 기준을 넘지 않는다.

## RES-014 예약 상태와 상태 전이

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 공통 예약 상태는 `요청 중`, `수용량 선점`, `확정`, `변경 대기`, `취소 대기`, `취소`, `거절`, `만료`, `이행 완료`로 구분한다. 1차 MVP에서는 매장 승인을 기다리는 `확정 대기`를 사용하지 않으며 취소 대기·취소 조건과 책임은 `RES-009`를 적용한다.
- 사용자·식당 대표자·플랫폼 운영자는 각 정책이 허용한 명령만 제출하고 상태를 직접 덮어쓰지 않는다. 모든 전이는 현재 상태·예약 버전·권한·사전 조건을 검증한 서버 명령으로만 수행한다.
- 중간 상태는 확정으로 표시하지 않고, 각 예약은 적용된 영업·슬롯·인원·수용량·가격·확정·취소 정책 버전을 참조한다.

### 정상 흐름·변경·취소·만료

- 유효 요청은 `요청 중` → `수용량 선점` 후 예약금 결제 또는 결제 불필요 판정을 거쳐 `확정`으로 간다. `수용량 선점`과 `확정`은 관련 구간의 인원과 팀 수를 점유하며 매장 승인·거절 전이는 생성하지 않는다.
- 변경은 `확정` → `변경 대기` → `확정`의 새 버전 또는 철회·실패 후 원래 `확정`으로 복귀한다. 취소는 `RES-009`가 허용한 경우에만 `취소 대기` 또는 `취소`로 전이한다.
- 변경 대기 중에는 기존 예약의 점유를 유지하고 인원 증가분 또는 새 구간만 추가 선점한다. 선점 기한이 지나면 요청 시점에도 `만료`로 판정하고 관련 구간의 인원·팀 수를 해제한다. `취소`, `거절`, `만료`, `이행 완료`는 종결 상태이며 관리자도 과거 상태를 되돌리지 않고 새 보정 사건을 만든다.

### 동시성·실패·복구·알림·감사·개인정보

- 허용 전이표, 조건부 갱신, 수정 번호와 명령 멱등 키로 중복·순서 역전 이벤트를 무해하게 한다. 여러 인스턴스의 경합에서 하나의 다음 상태만 확정된다.
- DB 전이와 Outbox 후속 작업을 같은 처리 경계로 기록한다. 메시지·알림 지연은 중앙 상태를 되돌리지 않고 소비자는 예약·전이 식별자로 재처리한다.
- 사용자에게 현재 상태와 필요한 행동을 표시하고 확정·변경·거절·취소·만료를 알린다. 전이 전후, 행위자, 이유, 정책·수용량 버전과 상관관계 ID를 감사하며 불필요한 개인정보를 이벤트에 넣지 않는다.

### 관측 가능한 인수 조건

- 허용되지 않은 상태 건너뛰기·종결 상태 재개·구 버전 명령이 거부되고 중간 상태가 확정으로 표시되지 않는다.
- 중복·역순 이벤트와 동시 승인·취소 처리에도 최종 상태와 수용량 결과가 하나로 수렴한다.
- `RES-001`의 즉시 확정과 `RES-009`의 취소 기준이 전이표에 일관되게 적용되고 승인 대기나 임의 취소 책임이 생성되지 않는다.

## RES-015 동시 예약과 초과 판매 방지

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 모든 예약·변경·확정 명령은 중앙 수용량 원본에서 실제 점유 구간과 각 서비스 구간의 잔여 인원·팀 수를 검증한다. 검색 결과·캐시·클라이언트 표시를 판매 가능성의 최종 근거로 쓰지 않는다.
- 유효 선점·확정 대기·확정 예약을 합산한 결과가 각 구간의 플랫폼 예약 제공 인원과 최대 팀 수를 넘지 않도록 조건부 원자 갱신한다. 모든 관련 구간에서 요청 인원과 팀 수 1건을 전부 확보할 때만 성공한다.
- 사용자·식당 대표자·운영자에게 제약을 우회하는 강제 초과 점유 권한을 주지 않는다. 운영상 예외는 수용량 설정을 새 버전으로 수정한 뒤 동일 검증을 거친다.

### 정상 흐름·변경·취소·만료

- 요청 검증 → 모든 관련 구간의 원자 선점 → 선택된 확정 흐름 → 최종 점유 순으로 처리한다. 최종 확정 직전에 수용량 버전·선점 소유권·만료와 현재 점유 한도를 다시 확인한다.
- 경합에서 실패한 요청은 `마감` 또는 `가용성 변경`으로 종료하고 임의의 다른 시간이나 인원으로 확정하지 않는다. 사용자가 새 대안을 선택해야 한다.
- 유효한 변경·거절·만료·취소로 감소하는 점유 인원·팀 수는 해당 전이가 중앙 확정된 뒤 정확히 한 번 수용량 원장에 반영하고, 현재 제공량과 남은 유효 점유를 기준으로 가용량을 다시 계산한다. 취소 자격은 `RES-009`를 따른다.
- 점유 감소와 같은 중앙 처분 사건에서 자동 승계가 비활성이면 감소분은 `일반 가용` 계산에 반영한다. 자동 승계가 활성이고 `TRANSFER-002`가 모든 관련 구간의 잔여 인원·팀 수와 후보 일행 인원을 충족하는 유효 후보를 산출하면 정확히 한 후보의 `승계 제안 중(proposed)` 또는 `일반 가용` 중 하나만 원자적으로 선택한다. 유효 후보가 없으면 해당 감소분을 `일반 가용` 계산에 반영한다.
- 후보·우선순위는 `TRANSFER-002` 결과만 소비한다. 제안 수락은 `승계 선점(claimed)`, 유효기간 경과는 `승계 제안 만료(expired)`로 기록하고, 이후 재제안·일반 공개·취소 경합은 `TRANSFER-004`·`TRANSFER-005`·`TRANSFER-009`의 확정 결과를 따른다. 승계 제안·선점에 사용 중인 인원·팀 수를 일반 가용량에 동시에 포함하지 않는다.
- 승계 후보가 최종 확정되면 해당 팀 인원을 제외한 반환 잔여 인원은 일반 가용량으로 반영하고, 같은 점유 감소 사건으로 다른 승계 대기 팀에게 추가 제안하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- DB가 최종 정합성 원본이며 짧은 캐시 선점은 보조 수단이다. 멱등 키·수용량 버전·조건부 갱신·요청 유일성 제약으로 다중 인스턴스와 재시도에서도 같은 요청을 한 번만 반영한다. 매장·날짜·슬롯만으로 유일성 제약을 두지 않으며 같은 슬롯의 서로 다른 예약은 수용 한도 안에서 함께 존재할 수 있다.
- 타임아웃 후 결과가 불명확하면 새 예약을 만들지 않고 멱등 키로 중앙 결과를 조회한다. 일부 구간 실패·고아 선점·가용량 불일치는 대사 작업으로 격리·복구하고 자동 해결 불가 건은 경보한다.
- 점유 감소 사건별 처분과 활성 승계 제안을 각각 하나만 허용한다. 수용량 반영·일반 공개·제안·선점·만료 전이는 같은 수용량 버전을 조건부 갱신하며, 제안·취소·만료 재전송과 경합은 `TRANSFER-004`·`TRANSFER-005`·`TRANSFER-009`의 멱등·순서 규칙으로 한 결과에 수렴시킨다.
- 성공·마감 결과만 요청자에게 알리고 다른 예약자의 정보나 경쟁 요청 수를 노출하지 않는다. 수용량 원장 변화, 요청·선점·확정·해제·복구를 상관관계 ID로 감사한다.

### 관측 가능한 인수 조건

- 가용량을 넘는 병렬 요청·변경·만료 경합 부하 테스트에서 각 구간의 점유 인원과 팀 수가 중앙 한도를 넘지 않고 음수 가용량이 없다.
- 응답 유실 후 같은 멱등 키 재시도는 기존 결과를 반환하며 새 예약이나 인원·팀 수 차감을 만들지 않는다.
- 일부 구간 장애와 작업 중복 뒤 수용량 원장과 활성 예약의 인원·팀 수 합계가 대사되고 다른 사용자의 정보 없이 실패가 안내된다.
- 하나의 점유 감소 사건은 자동 승계 비활성 시에만 일반 가용량에 즉시 반영되고, 활성 시에는 `TRANSFER-002`가 정한 적합한 후보 한 명에게만 제안된다. 제안 수락·만료·취소를 동시에 처리해도 `일반 가용`, `승계 제안 중`, `승계 선점`, `승계 제안 만료` 중 하나의 중앙 처분만 유효하며 일반 가용량과 승계 점유가 겹치지 않는다.
- 4명 점유 감소에 1명 후보가 전환 완료되면 1명만 승계 점유에 반영되고 잔여 3명은 일반 가용량으로 한 번 반환되며 추가 승계 제안은 생성되지 않는다.
