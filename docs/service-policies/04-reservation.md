# 방문 예약 정책

> 문서 상태: 세부 정책 확정(팀원 상의 필요·TODO 경계 보존)
> 정책 범위: `RES-001`~`RES-015`
> 최종 변경일: 2026-07-22

예약 접수, 테이블·좌석 자원, 임시 선점, 변경·취소, 중복 예약 및 동시성 규칙을 다룬다. `RES-001`의 확정 방식과 `RES-009`의 사용자 취소 책임은 팀 결정 전까지 선택하지 않으며, 아래 확정 세부는 어느 결과에서도 안전하게 작동하는 검증·상태·복구 기준이다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 팀원 상의 필요 | 핵심 | `RES-001`, `RES-009` |
| 확정 | 필수 | `RES-002`~`RES-008`, `RES-010`, `RES-011`, `RES-013`~`RES-015` |
| 확정 | 권장 | `RES-012` |

- `RES-001`: 즉시 확정과 승인제의 지원 범위는 예약의 주 흐름과 매장 운영 방식을 바꾸므로 팀 합의가 필요하다.
- `RES-009`: 취소 권한·마감·책임은 사용자 약속과 금전 분쟁에 직접 연결되므로 팀 합의가 필요하다.
- `RES-006`: 임시 선점은 10분이며 만료 2분 전에 경고한다. 사용자 임의 연장은 허용하지 않는다.

## RES-002 예약 가능 기간과 슬롯 시각

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 유효한 매장 운영 권한을 가진 식당 대표자만 매장별 예약 가능 시작·종료 기간, 슬롯 시작 간격, 서비스 소요 구간과 적용 시점을 정책 버전으로 게시할 수 있다. 수치는 운영 데이터로 조정 가능한 설정이며 이 정책에서 고정하지 않는다.
- 슬롯의 전체 서비스 구간은 게시된 영업시간 안에 있고 휴무·브레이크타임·휴점·폐점과 겹치지 않아야 한다. 사용자는 로그인 후 공개된 슬롯만 요청할 수 있고 플랫폼 운영자는 임의의 슬롯을 만들지 않는다.
- 매장 IANA 시간대와 중앙 기준 시각을 사용하며, 일광절약시간·자정 경계에서 존재하지 않거나 중복되는 현지 시각은 명시적으로 거부하거나 오프셋까지 식별한다.

### 정상 흐름·변경·취소·만료

- 조회는 현재 효력이 있는 운영·슬롯 정책 버전으로 후보를 만들고, 예약 요청과 최종 확정 단계에서 같은 버전과 자원 가용성을 다시 검증한다. 슬롯 노출은 예약 성공 보장이 아니다.
- 새 정책은 효력 시각 이후 생성되는 요청에만 적용한다. 기존 확정 예약은 소급 이동·취소하지 않고, 충돌이 생기면 `RES-013` 영향 사건으로 처리한다.
- 게시 예약은 효력 전에 철회할 수 있고 활성 버전은 새 버전으로만 교체한다. 예약 가능 종료를 지난 미사용 슬롯은 요청 시점 판정으로 즉시 닫히며 스케줄러 완료에 의존하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 정책 게시와 예약 요청은 정책 버전·자원 버전·멱등 키를 사용한다. 중앙 조건부 갱신과 유일성 제약으로 같은 슬롯의 경합을 직렬화하며, 캐시나 인스턴스 로컬 시각을 확정 근거로 쓰지 않는다.
- 운영·정책 원본을 확인할 수 없으면 신규 확정을 실패 폐쇄한다. 검색·캐시 갱신 실패는 중앙 정책을 되돌리지 않고 재처리하며 기존 확정 예약은 유지한다.
- 일반 슬롯 게시에는 사용자 알림을 보내지 않는다. 기존 예약 충돌만 당사자에게 알리고, 행위자·정책 버전·효력 시각·충돌 예약 식별자·처리 결과를 감사한다. 추가 개인정보나 결제 정보는 수집하지 않는다.

### 관측 가능한 인수 조건

- 영업 외·휴무·브레이크타임 슬롯과 모호한 현지 시각이 게시·확정되지 않고 모든 인스턴스가 같은 결과를 반환한다.
- 정책 전환과 예약 요청이 경합해도 예약은 정확히 한 정책 버전을 참조하며 재전송이 중복 예약을 만들지 않는다.
- 새 정책이 기존 확정 예약을 자동 변경하지 않고 충돌 사건·알림·감사 이력이 한 번만 생성된다.

## RES-003 인원 최소·최대와 영유아 포함 기준

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 예약 상품·슬롯별 최소·최대 총인원과 좌석을 차지하지 않는 영유아의 허용 여부를 버전된 설정으로 게시한다. 구체적 수치는 매장 설정이며 플랫폼이 일률적으로 정하지 않는다.
- 사용자는 성인·아동·영유아 인원수를 구분해 입력하고 총 동행 인원을 숨기지 않는다. 안전·수용 인원 판정에는 영유아를 총인원에 포함하되, 좌석 자원 계산은 게시된 영유아 좌석 규칙을 적용한다.
- 생년월일·아동 이름 같은 추가 개인정보는 수집하지 않는다. 유아용 의자 등 설비 요청은 재고 보장이 아닌 선택 요청으로 분리하고 확정 여부를 표시한다.

### 정상 흐름·변경·취소·상태

- 서버가 현재 정책으로 인원 범위와 필요한 좌석 용량을 검증한 뒤 자원 배정을 요청한다. 검증 실패는 허용 범위와 재입력 방법만 안내한다.
- 인원 변경은 `RES-008`에 따라 새 총인원으로 전체 자원 가용성을 다시 확인하며, 성공 전까지 원래 예약을 유지한다. 범위를 벗어난 변경을 자동 분할하거나 임의로 인원을 줄이지 않는다.
- 설정 변경은 신규 요청에만 적용하고 기존 예약의 인원 스냅샷을 유지한다. 이행 불가 충돌은 `RES-013`으로 보내며 이 정책이 취소 책임을 정하지 않는다.

### 동시성·실패·복구·알림·감사

- 인원 변경과 좌석 배정은 예약 버전·멱등 키와 하나의 조건부 처리 경계에서 확정한다. 일부 좌석만 변경된 부분 성공을 남기지 않는다.
- 자원 검증 장애에는 변경·신규 예약을 실패 폐쇄하고 원래 예약을 유지한다. 재시도는 동일 요청 결과를 반환한다.
- 인원·설비 변경 결과는 예약 대표자와 매장에 알리고, 이전·새 인원 구분, 정책 버전, 자원 판정과 사유를 감사하되 동행자의 신원은 기록하지 않는다.

### 관측 가능한 인수 조건

- 최소·최대 경계와 영유아 포함·좌석 미점유 조합이 게시된 정책대로 판정되고 범위 밖 요청은 자원을 점유하지 않는다.
- 동시 인원 변경에도 예약과 좌석 수가 일치하며 실패 시 원래 인원·자원이 그대로 남는다.
- 설정 변경 뒤에도 기존 예약 스냅샷이 변하지 않고 동행자 추가 개인정보가 로그·알림에 나타나지 않는다.

## RES-004 테이블·좌석·정원 모델

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 매장은 예약 자원을 안정 식별자가 있는 테이블, 좌석 또는 정원 버킷으로 정의하고 용량·사용 가능 시간·접근성 등 배정에 필요한 속성만 관리한다. 같은 실제 좌석을 여러 활성 자원으로 중복 등록하지 않는다.
- 식당 대표자만 자원 구성을 게시·중지할 수 있으며, 사용자는 인원·시간·공개된 요청 조건만 선택한다. 특정 좌석 선택 제공 여부를 이 정책이 새로 약속하지 않는다.
- 자원은 `사용 가능`, `임시 선점`, `배정`, `운영 중지` 상태를 가지며, 최종 예약 상태는 `RES-014`를 따른다. 자원 배정은 `RES-001`의 즉시 확정·승인제 선택을 대신하지 않는다.

### 정상 흐름·변경·취소·만료

- 요청 시 필요한 전체 서비스 구간과 인원에 맞는 자원을 선택해 하나의 선점 그룹으로 확보하고, 선택된 확정 방식의 성공 시 배정으로 전환한다. 실패·거절·유효한 만료에는 그룹 전체를 해제한다.
- 게시된 자원의 용량·구성을 직접 덮어쓰지 않고 새 버전으로 바꾼다. 기존 예약이 참조하는 자원 버전은 삭제하지 않으며 운영 중지는 신규 배정만 차단한다.
- 임시 선점의 존재·10분 만료·만료 2분 전 경고는 `RES-006`을 따르며, 만료 후 뒤늦은 확정은 새 가용성 검증 없이 허용하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 겹치는 시간 구간에 같은 자원이 두 예약에 배정되지 않도록 DB 제약·조건부 갱신·원자적 그룹 연산을 최종 방어선으로 둔다. 여러 인스턴스가 같은 후보를 선택해도 하나만 성공한다.
- 일부 자원만 선점·배정되면 전체 그룹을 보상 해제한다. 중앙 자원 상태를 확인할 수 없으면 확정을 실패 폐쇄하고 복구 작업이 고아 선점을 대사한다.
- 사용자에게 내부 테이블 식별자나 운영 메모를 노출하지 않는다. 배정·해제·중지의 행위자, 자원·예약·그룹·버전과 결과를 감사하며 결제·추가 개인정보는 이 모델에 저장하지 않는다.

### 관측 가능한 인수 조건

- 같은 자원·겹치는 구간의 동시 요청 중 허용 수량만 성공하고 초과 배정이나 음수 가용량이 없다.
- 그룹 중간 실패와 작업 재시작 뒤 일부 자원만 남지 않고 중앙 상태가 전부 배정 또는 전부 해제로 수렴한다.
- 자원 구성 변경 후에도 기존 예약은 당시 버전으로 재현되고 중지 자원에는 신규 배정이 생기지 않는다.

## RES-005 테이블 조합과 분리 정책

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 실제로 함께 사용할 수 있는 테이블 조합과 조합 용량을 자원 버전으로 명시한다. 플랫폼은 거리·형태만으로 임의 조합하지 않고, 같은 테이블이 서로 양립할 수 없는 활성 조합에 동시에 배정되지 않게 한다.
- 조합은 구성 테이블 전체가 같은 서비스 구간에 사용 가능하고 인원 정책을 충족할 때만 후보가 된다. 분리는 기존 결합 배정의 모든 예약이 종료·변경·취소돼 구성 자원이 해제된 뒤에만 허용한다.
- 조합 설정은 신규 배정 규칙이며 `RES-001`의 확정 방식, `RES-009`의 취소 책임이나 대규모 일행 우선권을 결정하지 않는다.

### 정상 흐름·변경·취소·만료

- 예약 요청은 단일 자원 또는 허용된 조합 하나를 하나의 배정 그룹으로 원자 선점한다. 사용자에게 조합이 필요하다는 이유로 여러 독립 예약을 자동 생성하지 않는다.
- 조합 버전 변경은 이후 요청에만 적용한다. 기존 예약의 구성은 유지하고, 물리적으로 이행할 수 없으면 `RES-013` 변경 제안·영향 처리로 전환한다.
- 선점 만료·예약 변경·유효한 취소 시 구성 자원을 그룹 단위로 한 번만 해제한다. 일부 구성만 먼저 재판매하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 구성 테이블별 겹침 제약과 그룹 조건부 갱신으로 단일 배정과 조합 배정의 경합을 막는다. 멱등 키 재전송은 같은 그룹을 반환한다.
- 부분 선점·부분 해제 실패는 보상 작업으로 전부 해제될 때까지 신규 사용을 차단한다. 중앙 조합 버전을 읽지 못하면 신규 조합을 실패 폐쇄한다.
- 조합 변경과 예약 영향만 매장에 알리고 사용자에게 내부 배치도를 노출하지 않는다. 조합 버전·구성 자원·예약·전이·실패 복구를 감사하며 개인정보는 추가 수집하지 않는다.

### 관측 가능한 인수 조건

- 단일 테이블 예약과 겹치는 조합 예약을 동시에 보내도 구성 테이블은 한 그룹에만 배정된다.
- 조합 요청의 일부 단계에 장애를 주입해도 부분 예약이 확정되지 않고 재시도 뒤 한 결과로 수렴한다.
- 조합 설정 변경이 기존 배정을 소급 분리하지 않고 내부 배치 정보가 사용자 응답·로그에 불필요하게 노출되지 않는다.

## RES-006 예약 임시 선점과 만료

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 예약 생성·변경의 임시 선점은 중앙 시각 기준 정확히 10분간 유효하며, 선점 소유자·자원 그룹·정책 버전·만료 시각·멱등 키를 함께 기록한다. 만료 2분 전에는 선점 소유자에게 경고를 한 번 생성하되, 경고 발송 성공 여부는 선점의 유효성을 바꾸지 않는다.
- 사용자·매장·운영자는 임의로 선점 만료를 연장하거나 만료 시각을 직접 수정할 수 없다. 필요한 새 요청은 기존 선점이 유효하게 종결된 뒤 현재 가용성과 정책을 다시 검증한다.
- 결제 결과가 불명확하거나 중앙 처리 장애가 난 경우 사용자의 취소·실패 책임으로 기록하지 않는다. 해당 선점은 `결과 대사/복구 필요`로 격리하고 결제 원본과 자원 원장을 대사한 뒤에만 확정 또는 해제한다.

### 정상 흐름·동시성·실패·복구

- 선점 생성 → 만료 2분 전 경고 → 결제·확정 또는 중앙 만료 → 자원 그룹 정확히 한 번 해제 순으로 처리한다. 만료는 스케줄러 실행뿐 아니라 모든 확정·변경 명령 시점에도 중앙 시각으로 재검사한다.
- 선점·결제 확정·만료·해제는 같은 선점/자원 버전과 조건부 갱신으로 처리한다. 여러 인스턴스의 만료 작업·재전송이 겹쳐도 한 전이만 성공하고, 뒤늦은 작업은 최신 결과를 반환한다.
- 해제·대사·복구에는 선점·자원 그룹·정책 버전·기준 시각·결제 참조·멱등 키·결과를 감사한다. 결제수단 원문이나 추가 개인정보는 이 정책 데이터에 기록하지 않는다.

### 관측 가능한 인수 조건

- 중앙 만료 시각 직전에는 확정이 가능하고 정확히 만료 시각 및 직후에는 새 가용성 검증 없는 확정이 거부되며, 만료 2분 전 경고는 최대 한 번 생성된다.
- 동일 선점의 결제 확정과 다중 인스턴스 만료를 병렬 실행해도 확정 또는 정확히 한 번 해제 중 하나만 남고 자원이 이중 반환되지 않는다.
- 결제 결과 불명확·서버 중단을 주입해도 사용자 책임 상태나 자동 재판매가 생기지 않고 대사 뒤 감사 가능한 확정 또는 단일 해제로 수렴한다.

## RES-007 예약자 연락처 정보

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 예약 대표자는 로그인한 일반 사용자이며 계정에서 소유 확인된 연락 채널을 예약 알림에 사용한다. 별도 자유 입력 전화번호나 동행자 연락처를 필수 수집하지 않는다.
- 식당 대표자는 해당 매장의 유효한 예약 이행에 필요한 예약자 표시명, 마스킹 연락 경로와 인원·요청사항만 조회할 수 있다. 플랫폼 운영자는 고객지원·분쟁 권한과 사건이 있을 때만 최소 범위를 조회한다.
- 예약에는 계정 연락처 원문 대신 계정 식별자, 알림 대상 참조와 거래 당시 연락 가능 상태를 연결한다. 연락처 변경은 과거 감사 기록을 덮어쓰지 않는다.

### 정상 흐름·변경·취소·만료

- 생성 전 사용 가능한 확인 채널을 검증하고 없으면 계정 설정 경로를 안내한다. 예약 상태 알림은 `RES-014`의 확정 전이를 기준으로 내구성 있는 작업으로 생성한다.
- 계정 연락처가 바뀌면 이후 발송은 현재 확인된 채널을 사용하되 이미 발송한 내용과 당시 대상 참조는 유지한다. 예약 양도·대리 범위는 `RES-011`을 따른다.
- 예약 종료·취소 뒤 식당의 운영 화면에서는 연락처 접근을 제거하고, 정확한 보관·파기 기간은 개인정보 정책 TODO를 따르며 이 행에서 새 기간을 정하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약 전이와 알림 작업 생성은 같은 사건 식별자로 연결하고 소비자는 멱등하게 발송한다. 발송 실패가 예약 상태를 되돌리지 않으며 제한 재시도 후 운영 경보·수동 재발송으로 남긴다.
- 권한 또는 계정 연락 상태를 확인할 수 없으면 원문 연락처를 우회 노출하지 않는다. 필수 알림 실패는 앱 내 최신 상태 조회를 유지한다.
- 조회·내보내기·발송에는 예약, 목적, 행위자, 마스킹된 대상과 결과를 감사하고 로그·분석·URL에는 연락처 원문을 남기지 않는다.

### 관측 가능한 인수 조건

- 확인된 연락 채널이 없는 사용자는 대체 개인정보 입력 없이 안전한 계정 설정 안내를 받고 예약 정보가 생성되지 않는다.
- 권한 없는 매장·운영자·다른 사용자가 연락처나 예약 상세를 조회할 수 없고 종료 후 매장 화면 접근이 제거된다.
- 같은 상태 이벤트의 중복 소비가 알림 작업을 중복 생성하지 않으며 발송 장애가 예약 확정·취소 결과를 바꾸지 않는다.

## RES-008 예약 변경 가능 항목과 마감

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 예약 대표자와 해당 매장 권한이 유효한 식당 대표자만 예약 변경을 요청할 수 있다. 변경 가능한 필드·요청 마감은 매장별 버전된 공개 정책으로 관리하되 구체적 시간 수치는 이 행에서 고정하지 않는다.
- 시간·인원·자원·메뉴 등 가용성에 영향을 주는 변경은 새 요청과 같은 검증을 거친다. 표시 메모처럼 자원에 영향 없는 필드도 허용 목록과 입력 제한을 적용한다.
- 변경은 원본 예약 ID 아래 새 수정 버전으로 만들고, 원본 확정 스냅샷·가격·정책 버전을 덮어쓰지 않는다. 변경 가능 여부는 서버 중앙 시각으로 판정한다.

### 정상 흐름·변경·취소·만료

- 요청은 `변경 검증 중`에서 후보 자원을 임시 선점하고 필요한 동의·확정 방식을 거쳐 한 번에 새 버전으로 전환한다. 성공 전까지 원래 예약과 자원을 유지한다.
- 사용자는 확정 전 변경 요청을 철회할 수 있고 후보 선점은 해제된다. 변경 실패·거절·만료에도 원래 예약은 유지하며 이를 취소로 해석하지 않는다.
- 매장 제안 변경은 `RES-013`을 따르고 사용자 동의 없이 확정하지 않는다. 변경 중 취소 요청의 우선관계와 비용은 `RES-009` 팀 결정 결과를 소비하며 이 정책이 선결정하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약 수정 번호·멱등 키·조건부 갱신으로 동시에 하나의 활성 변경만 허용한다. 구 버전 변경 요청은 충돌로 거부하고 최신 예약을 반환한다.
- 새 자원 확보 뒤 원본 해제에 실패하면 신규 사용을 잠시 차단한 채 중앙 전환 사건을 멱등 재실행한다. 어느 시점에도 원본·새 자원이 서로 다른 예약에 판매되지 않게 한다.
- 요청·확정·거절·철회·만료를 당사자에게 알리고 이전·새 스냅샷, 행위자, 이유, 동의와 결과를 감사한다. 변경에 필요 없는 새 개인정보는 수집하지 않는다.

### 관측 가능한 인수 조건

- 가용성 영향 변경이 실패하거나 만료되면 원래 예약·자원이 유지되고 후보 자원은 해제된다.
- 동시 변경 중 하나만 확정되며 재전송은 같은 수정 버전을 반환하고 이중 자원 해제가 없다.
- 마감 정책 변경은 기존 예약의 당시 공개 스냅샷을 보존하고 매장 제안은 사용자 동의 전에 적용되지 않는다.

## RES-010 중복 예약 판정

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 같은 일반 사용자 계정이 같은 매장에 겹치는 서비스 구간으로 가진 활성 요청·보류·확정 예약을 중복 후보로 본다. 동행자 이름·전화번호·기기 지문을 추가 수집해 동일인을 추정하지 않는다.
- 정확히 같은 의도의 재전송은 멱등 키와 요청 지문으로 기존 결과를 반환한다. 서로 다른 요청이 중복 후보이면 자동 취소·병합하지 않고 신규 확정을 차단하며 기존 예약 확인·변경 경로를 안내한다.
- 다른 매장 예약이나 겹치지 않는 구간을 일률적으로 금지하지 않는다. 더 넓은 이용 제한이나 제재는 별도 정책·근거 없이 적용하지 않는다.

### 정상 흐름·변경·취소·만료

- 신규·변경 요청 직전에 활성 예약 구간을 중앙 조회하고, 후보 선점과 중복 방지 표식을 같은 처리 경계에서 확정한다.
- 기존 요청이 유효하게 취소·거절·만료되어 자원이 해제된 뒤에만 새 요청을 허용한다. 취소 가능 여부와 책임은 `RES-009` 결과를 따른다.
- 중복 판정 오탐 신고는 플랫폼 운영자가 근거와 권한을 갖고 검토할 수 있으나 기존 예약을 삭제하거나 새 예약을 강제 확정하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 사용자·매장·겹침 구간에 대한 중앙 제약과 조건부 갱신으로 여러 인스턴스의 동시 요청 중 하나만 활성화한다. 캐시만으로 중복 없음이라고 판정하지 않는다.
- 중복 검증 저장소 장애 시 신규 확정을 실패 폐쇄하고 기존 예약은 유지한다. 고아 보류 표식은 상태 대사로 제거한다.
- 차단 결과에는 기존 예약의 최소 식별·시간만 본인에게 표시한다. 요청 지문, 충돌 예약, 판정·해제·운영자 조치를 감사하되 연락처 원문은 기록하지 않는다.

### 관측 가능한 인수 조건

- 같은 사용자·매장·겹치는 구간의 동시 요청에서 활성 예약은 하나만 남고 동일 멱등 키는 같은 결과를 반환한다.
- 다른 매장 또는 비겹침 요청은 이 규칙만으로 차단되지 않으며 동행자 개인정보 없이 판정된다.
- 검증 장애에는 신규 예약이 확정되지 않고 복구·대사 후 고아 표식 없이 재시도할 수 있다.

## RES-011 대리 예약과 예약 양도

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 로그인한 예약 생성자가 예약 대표자와 책임 주체로 남으며 자신의 일행을 위해 예약할 수 있다. 동행자 계정·연락처 등록은 요구하지 않는다.
- 현재 안전 기본선에서는 예약 소유권을 다른 계정에 직접 양도하지 않는다. 참석자 구성이 바뀌어도 대표자 변경으로 처리하지 않으며, 대표자가 참석할 수 없는 경우 이용 가능한 변경·취소 절차를 안내한다.
- 식당 대표자나 플랫폼 운영자는 사용자 대신 소유자를 바꾸거나 자신에게 예약을 이전할 수 없다. 고객지원은 본인 확인과 사건 권한 안에서 상태를 조회·재전송할 뿐이다.

### 정상 흐름·변경·취소·만료

- 예약 생성·알림·변경 요청은 대표자 계정에 귀속한다. 입장 시 필요한 참석 확인 방식은 체크인 정책을 따르며 이 행이 새로운 신원 수집을 만들지 않는다.
- 인원 변경은 `RES-003`·`RES-008`, 취소는 `RES-009`의 최종 팀 결정, 매장 귀책 변경은 `RES-013`을 따른다. 양도 불가가 취소 수수료나 보상 책임을 새로 정하지 않는다.
- 계정 탈퇴·정지 등으로 대표자 권한이 사라지면 계정 정책과 거래 정리 절차를 따르며 다른 일행에게 자동 이전하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약의 소유 계정 식별자는 불변이며 변경 명령은 소유자·예약 버전·멱등 키를 검사한다. 여러 인스턴스에서 위조된 대표자 교체 요청을 동일하게 거부한다.
- 인증·권한 저장소 장애에는 변경·취소 명령을 실패 폐쇄하고 기존 예약을 유지한다. 고객지원 복구도 소유자 값을 직접 수정하지 않는다.
- 대표자에게만 상태 알림을 보내고 동행자 연락처를 수집·노출하지 않는다. 대리 이용 관련 요청과 거부, 행위자·이유를 감사하되 일행 신원을 기록하지 않는다.

### 관측 가능한 인수 조건

- 다른 계정·매장·운영자가 대표자 변경을 요청해도 예약 소유자가 바뀌지 않으며 모든 인스턴스가 같은 권한 결과를 낸다.
- 일행 예약은 동행자 계정·연락처 없이 생성 가능하고 인원 변경이 대표자 소유권을 바꾸지 않는다.
- 권한 장애와 재전송에도 기존 예약은 유지되고 취소·금전 책임은 `RES-009` 결정을 벗어나 새로 적용되지 않는다.

## RES-012 단체 예약과 별도 승인

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 예약 상품별로 단체로 분류할 인원 조건과 필요한 추가 운영 정보 항목을 버전된 공개 정책으로 설정할 수 있다. 단체 기준 숫자는 매장 운영 설정이며 이 정책에서 고정하지 않는다.
- 단체 요청도 `RES-001`에서 팀이 선택한 즉시 확정·승인제 범위를 그대로 따른다. 이 행은 단체라는 이유만으로 별도 승인제를 강제하거나 즉시 확정을 약속하지 않는다.
- 추가 정보는 좌석 배치·접근성·메뉴 준비에 필요한 구조화된 항목으로 최소화하고 주민번호, 동행자 연락처·이름 목록 같은 불필요한 개인정보를 요구하지 않는다.

### 정상 흐름·변경·취소·만료

- 요청은 인원·자원·영업·중복 검증 후 단체 표시와 적용 정책 버전을 고정하고 선택된 `RES-001` 흐름으로 전환한다. 중간 상태를 확정으로 표시하지 않는다.
- 인원 변경으로 단체 기준을 넘나들면 새 정책과 자원을 재검증하고 사용자에게 흐름 변화가 있으면 확정 전에 안내한다. 기존 예약을 자동 취소하지 않는다.
- 거절·철회·만료·취소는 일반 예약 상태와 `RES-009` 결과를 따르며 단체라는 이유로 별도 금전 책임·우선권을 만들지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 단체 요청도 같은 예약·자원 멱등성 및 조건부 갱신을 적용하고 별도 수동 목록이나 단일 인스턴스 메모에 의존하지 않는다.
- 필수 검증이나 선택된 승인 흐름의 상태를 확인할 수 없으면 실패 폐쇄하고 선점 자원을 해제한다. 복구는 중앙 예약 사건으로 재개한다.
- 요청·추가 정보 보완·선택된 확정 결과를 대표자와 매장에 알리고, 인원·정책 버전·결정자·사유를 감사한다. 운영 정보의 접근은 해당 예약 이행 권한으로 제한한다.

### 관측 가능한 인수 조건

- 단체 경계 전후 요청이 버전된 매장 정책과 `RES-001` 선택 결과를 그대로 따르고 별도 확정 방식을 암묵적으로 만들지 않는다.
- 중복 제출·다중 인스턴스 처리에도 단체 예약 사건과 자원 배정은 하나만 생성된다.
- 불필요한 일행 개인정보 없이 처리되고 실패·인원 변경 시 원래 예약과 자원이 일관되게 유지된다.

## RES-013 매장 규칙 변경·취소

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 해당 매장의 유효한 운영 권한을 가진 식당 대표자만 기존 예약의 이행 변경을 제안하거나 매장 사유의 취소 처리를 시작할 수 있다. 플랫폼 운영자는 장애·분쟁 권한과 기록된 사유 없이 대신 처리하지 않는다.
- 영업시간·자원·메뉴 정책 변경은 기존 예약을 소급 수정하지 않고 영향 사건을 만든다. 변경 제안에는 원래·대안 조건, 이유, 응답 경로와 적용될 `RES-009`·PAY 정책 참조를 명확히 표시한다.
- 사용자의 명시적 수락 전에는 원래 예약을 유지하며 침묵·알림 열람·알림 실패를 동의로 보지 않는다. 안전상 이행 불가 시 신규 이용만 차단하고 최종 취소 책임은 `RES-009` 팀 결정 결과를 따른다.

### 정상 흐름·변경·취소·만료

- 영향 탐지 → 매장 검토 → 변경 제안 또는 취소 요청 → 사용자 응답 → 확정 전이 순으로 처리한다. 변경 수락은 `RES-008` 자원 재검증과 원자 전환을 거친다.
- 사용자가 거절하거나 제안이 버전된 유효기간을 지나면 변경은 적용하지 않는다. 이후 취소·원래 이행 여부는 공개 정책과 `RES-009` 결과로 처리하며 임의 기본값을 만들지 않는다.
- 같은 충돌 원인의 여러 예약은 각각 독립 사건으로 추적한다. 운영 정책을 되돌려도 이미 확정된 사용자 선택·취소를 자동 복원하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 예약별 활성 영향 사건을 하나만 허용하고 예약 버전·제안 버전·멱등 키로 중복 제안·응답을 막는다. 취소와 수락 경합은 중앙 조건부 전이 중 하나만 성공한다.
- 알림 실패는 변경 동의가 아니며 원래 상태를 유지한다. 자원 전환 부분 실패는 `RES-008` 보상 흐름으로 복구하고 자동 복구 불가 건은 운영자 큐로 보낸다.
- 제안·응답·취소 결과를 당사자에게 알리고 행위자, 사유, 이전·대안 스냅샷, 정책 참조와 처리 결과를 감사한다. 알림·로그에는 필요한 최소 예약 정보만 사용한다.

### 관측 가능한 인수 조건

- 운영 정책 변경만으로 기존 예약의 시간·인원·가격·상태가 바뀌지 않고 사용자 수락 전 변경 제안은 미적용 상태다.
- 동시 수락·거절·취소 요청 중 하나의 최종 전이만 성공하고 중복 알림·자원 전환이 없다.
- 알림 장애와 정책 되돌림이 사용자 동의나 취소를 만들어내지 않으며 금전 결과는 미정인 `RES-009`를 넘지 않는다.

## RES-014 예약 상태와 상태 전이

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 공통 예약 상태는 `요청 중`, `자원 선점`, `확정 대기`, `확정`, `변경 대기`, `취소 대기`, `취소`, `거절`, `만료`, `이행 완료`로 구분한다. 실제로 `확정 대기`를 사용하는 범위는 `RES-001`, 취소 대기·취소 조건과 책임은 `RES-009` 결과에 의해 활성화한다.
- 사용자·식당 대표자·플랫폼 운영자는 각 정책이 허용한 명령만 제출하고 상태를 직접 덮어쓰지 않는다. 모든 전이는 현재 상태·예약 버전·권한·사전 조건을 검증한 서버 명령으로만 수행한다.
- 중간 상태는 확정으로 표시하지 않고, 각 예약은 적용된 영업·슬롯·인원·자원·가격·확정·취소 정책 버전을 참조한다.

### 정상 흐름·변경·취소·만료

- 유효 요청은 `요청 중` → `자원 선점` 후 `RES-001`의 선택에 따라 `확정` 또는 `확정 대기`로 간다. 승인·거절 전이는 선택된 정책만 허용하며 이 행이 승인 주체나 범위를 정하지 않는다.
- 변경은 `확정` → `변경 대기` → `확정`의 새 버전 또는 철회·실패 후 원래 `확정`으로 복귀한다. 취소는 `RES-009`가 허용한 경우에만 `취소 대기` 또는 `취소`로 전이한다.
- 선점 기한이 지나면 요청 시점에도 `만료`로 판정하고 자원을 해제한다. `취소`, `거절`, `만료`, `이행 완료`는 종결 상태이며 관리자도 과거 상태를 되돌리지 않고 새 보정 사건을 만든다.

### 동시성·실패·복구·알림·감사·개인정보

- 허용 전이표, 조건부 갱신, 수정 번호와 명령 멱등 키로 중복·순서 역전 이벤트를 무해하게 한다. 여러 인스턴스의 경합에서 하나의 다음 상태만 확정된다.
- DB 전이와 Outbox 후속 작업을 같은 처리 경계로 기록한다. 메시지·알림 지연은 중앙 상태를 되돌리지 않고 소비자는 예약·전이 식별자로 재처리한다.
- 사용자에게 현재 상태와 필요한 행동을 표시하고 확정·변경·거절·취소·만료를 알린다. 전이 전후, 행위자, 이유, 정책·자원 버전과 상관관계 ID를 감사하며 불필요한 개인정보를 이벤트에 넣지 않는다.

### 관측 가능한 인수 조건

- 허용되지 않은 상태 건너뛰기·종결 상태 재개·구 버전 명령이 거부되고 중간 상태가 확정으로 표시되지 않는다.
- 중복·역순 이벤트와 동시 승인·취소 처리에도 최종 상태와 자원 결과가 하나로 수렴한다.
- `RES-001`·`RES-009`의 서로 다른 가능한 팀 결과를 대입해도 전이표가 그 결과를 소비할 뿐 자체적으로 확정·취소 책임을 선택하지 않는다.

## RES-015 동시 예약과 초과 판매 방지

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 모든 예약·변경·확정 명령은 중앙 자원 원본에서 실제 겹침 구간과 가용량을 검증한다. 검색 결과·캐시·클라이언트 표시를 판매 가능성의 최종 근거로 쓰지 않는다.
- 테이블·좌석에는 겹침 배정 금지 제약, 정원·수량에는 0 미만이 되지 않는 조건부 원자 갱신을 적용한다. 자원 그룹은 전부 확보될 때만 성공한다.
- 사용자·식당 대표자·운영자에게 제약을 우회하는 강제 초과 배정 권한을 주지 않는다. 운영상 예외는 자원 구성을 새 버전으로 수정한 뒤 동일 검증을 거친다.

### 정상 흐름·변경·취소·만료

- 요청 검증 → 원자 선점 → 선택된 확정 흐름 → 최종 배정 순으로 처리한다. 최종 확정 직전에 자원 버전·선점 소유권·만료를 다시 확인한다.
- 경합에서 실패한 요청은 `마감` 또는 `가용성 변경`으로 종료하고 임의의 다른 시간·자원으로 확정하지 않는다. 사용자가 새 대안을 선택해야 한다.
- 유효한 변경·거절·만료·취소로 반환되는 자원은 해당 전이가 중앙 확정된 뒤 정확히 한 번 자원 원장에 복구한다. 취소 자격은 `RES-009`를 따른다.
- 원장 복구와 같은 중앙 처분 사건에서 자동 승계가 비활성이면 `일반 가용`, 활성이고 `TRANSFER-002`가 유효 후보를 산출하면 정확히 한 후보의 `승계 제안 중(proposed)` 중 하나만 원자적으로 선택한다. 유효 후보가 없다는 `TRANSFER-002` 결과는 해당 반환 건을 `일반 가용`으로 보낸다.
- 후보·우선순위는 `TRANSFER-002` 결과만 소비한다. 제안 수락은 `승계 선점(claimed)`, 유효기간 경과는 `승계 제안 만료(expired)`로 기록하고, 이후 재제안·일반 공개·취소 경합은 `TRANSFER-004`·`TRANSFER-005`·`TRANSFER-009`의 확정 결과를 따른다. `승계 제안 중`·`승계 선점` 자원을 일반 가용으로 동시에 게시하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- DB가 최종 정합성 원본이며 짧은 캐시 선점은 보조 수단이다. 멱등 키·조건부 갱신·유일성 제약으로 다중 인스턴스와 재시도에서도 같은 요청을 한 번만 반영한다.
- 타임아웃 후 결과가 불명확하면 새 예약을 만들지 않고 멱등 키로 중앙 결과를 조회한다. 부분 실패·고아 선점·가용량 불일치는 대사 작업으로 격리·복구하고 자동 해결 불가 건은 경보한다.
- 반환 원장 ID별 처분 사건과 활성 승계 제안을 각각 하나만 허용한다. 원장 복구·일반 공개·제안·선점·만료 전이는 같은 자원 버전을 조건부 갱신하며, 제안·취소·만료 재전송과 경합은 `TRANSFER-004`·`TRANSFER-005`·`TRANSFER-009`의 멱등·순서 규칙으로 한 결과에 수렴시킨다.
- 성공·마감 결과만 요청자에게 알리고 다른 예약자의 정보나 경쟁 요청 수를 노출하지 않는다. 자원 원장 변화, 요청·선점·확정·해제·복구를 상관관계 ID로 감사한다.

### 관측 가능한 인수 조건

- 가용량을 넘는 병렬 요청·변경·만료 경합 부하 테스트에서 확정 수량이 중앙 용량을 넘지 않고 음수 재고가 없다.
- 응답 유실 후 같은 멱등 키 재시도는 기존 결과를 반환하며 새 예약·자원 차감을 만들지 않는다.
