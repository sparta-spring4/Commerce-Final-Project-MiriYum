# 알림 정책

> 문서 상태: 세부 정책 확정(TODO 혼재)
> 정책 범위: `NOTI-001`~`NOTI-010`
> 최종 변경일: 2026-07-23

필수·선택 알림, 채널, 수신 동의, 발송 시점, 중복 방지, 재시도, 대체 채널 및 공급자 장애를 다룬다. 알림은 원 도메인의 확정 상태를 전달하는 파생 기능이며 상태 전이나 권리의 성립 조건이 아니다.

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 확정 | 필수 | `NOTI-001`~`NOTI-008`, `NOTI-010` |

- `NOTI-009`의 발송 이력 보관기간과 항목별 법률 문구는 개인정보·분쟁 목적과 최종 법률 검토 후 정한다. 다만 이력 최소화, 접근 통제, 조회 감사와 만료 정책을 적용할 수 있는 구조는 확정 안전 기본선이다.
- 채널 제공자와 유료 계약, 발송량·속도·재시도 횟수의 운영 수치는 이 문서에서 새로 확정하지 않는다. 채널 어댑터와 중앙 정책 버전으로 교체·조정한다.

## 알림 공통 불변식

- 원 도메인은 확정된 사건과 수신자 계정 ID, 알림 목적, 자원 ID·버전, 발생 시각만 Outbox에 기록한다. 전화번호·이메일 같은 실제 주소와 완성된 메시지 본문을 도메인 사건에 복제하지 않는다.
- 알림 작업은 `생성`, `동의·대상 검증`, `발송 대기`, `발송 중`, `전달`, `재시도 대기`, `대체 채널 대기`, `취소`, `최종 실패` 상태를 가진다. 상태·시도·결과는 중앙 작업 저장소에 내구성 있게 기록하고 인스턴스 로컬 큐만으로 성공을 판정하지 않는다.
- 발송 실패·지연은 이미 확정된 예약·웨이팅·결제·제재 상태를 되돌리지 않는다. 사용자가 행동해야 하는 기능은 화면/API에서 중앙 상태와 남은 조건을 직접 조회할 수 있어야 하며 알림 수신만을 권리 행사 조건으로 삼지 않는다.
- 로그·경보·감사에는 알림 ID, 목적, 템플릿·정책 버전, 마스킹한 채널 유형, 제공자 응답 분류와 상관관계 ID만 남기고 주소·본문·인증번호·결제정보를 남기지 않는다.

## 마이페이지 알림 이력 표시

- 마이페이지 알림 이력에는 현재 실제로 보관 중인 논리 알림의 목적·안전한 제목, 관련 상태, 생성·전달 시각과 전달/실패 상태만 표시한다. 원문 수신 주소, 메시지 본문 전체, 제공자 payload와 내부 재시도·감사 메모는 노출하지 않는다.
- 사용자는 자신의 알림만 최신 발생 시각 내림차순, 동률이면 알림 이력 ID 내림차순으로 조회한다. 커서 페이징은 기본 20개·최대 50개이며 다른 방문자·매장 내부 수신자의 이력을 결합하지 않는다.
- 이 화면은 보관 중인 기록만 표시하며 새 보관기간을 만들지 않는다. 정확한 보관 항목·기간과 파기는 `NOTI-009`, `PRIV-005`·`PRIV-006`의 기존 `TODO`가 확정된 뒤 그 버전을 따른다.
- 표시 이력의 유실·지연은 원 거래 상태를 바꾸지 않고, 보관 정책을 확인할 수 없으면 과거 기록이 영구 보존되는 것으로 추정하지 않는다.

## NOTI-001 필수·선택 알림 구분

> 정책 상태: 확정

### 확정 분류·권한·상태

- 알림 목적은 `보안`, `거래 상태`, `서비스 운영`, `선택 정보`, `광고·판촉` 가운데 하나와 정책 버전을 가져야 한다. 보안·거래 상태처럼 서비스 수행과 계정 보호에 필요한 알림만 필수로 분류하고, 편의 정보와 광고·판촉은 선택으로 분리한다.
- 원 도메인이 임의로 필수 여부를 지정하지 못한다. 승인된 알림 목적 카탈로그가 목적별 법적·서비스 근거, 허용 수신자, 허용 채널과 사용자 설정 가능 여부를 제공하며 누락·미확정 목적은 발송하지 않는다.
- 선택 알림 거부는 해당 목적의 미래 작업을 생성하지 않게 하고 이미 대기 중인 작업도 발송 직전 다시 확인해 취소한다. 필수 알림 거부 설정을 만들지 않되 해당 알림을 광고·추천 내용과 결합하지 않는다.
- 목적 분류 변경은 새 정책 버전부터 적용하고 과거 선택 동의를 필수 또는 다른 선택 목적으로 소급 재사용하지 않는다.

### 동시성·실패·감사·개인정보

- 작업 생성과 발송 직전에 중앙 목적 카탈로그·동의 버전을 확인한다. 다중 인스턴스의 동시 생성은 수신자-목적-원 사건의 멱등 키로 하나의 논리 작업에 수렴한다.
- 분류 저장소를 확인할 수 없으면 선택 알림은 생성·발송하지 않고 필수 알림도 임의 분류하지 않은 채 검증 대기로 둔다. 분류·거부·취소·예외 발송을 감사한다.
- 분류 카탈로그에는 목적 수행에 필요한 템플릿 필드만 허용하고 필수라는 이유로 추가 PII를 수집하거나 다른 목적으로 재사용하지 않는다.

### 관측 가능한 인수 조건

- 같은 거래 사건이 필수 거래 알림과 선택 홍보 알림으로 동시에 분류되지 않으며 선택 동의를 철회하면 대기 중 선택 작업이 발송 전 취소된다.
- 미등록 목적이나 분류 저장소 장애에는 메시지가 발송되지 않고 검증 대기·경보가 남는다.
- 필수 알림 본문과 데이터에 광고 문구나 해당 목적에 불필요한 개인정보가 포함되지 않는다.

## NOTI-002 문자·알림톡·푸시·이메일 우선순위

> 정책 상태: 확정

### 확정 채널 선택·사전 조건

- 채널 우선순위는 알림 목적, 긴급성, 수신자가 확인한 채널, 채널별 동의·수신 가능 상태와 현재 장애 상태를 입력으로 하는 중앙 버전 정책으로 결정한다. 코드에 특정 제공자나 유료 채널을 필수값으로 고정하지 않는다.
- 선택 알림은 그 목적·채널에 유효한 동의가 있는 채널만 후보가 된다. 필수 알림도 확인되지 않은 주소나 다른 계정의 주소로 보내지 않으며, 사용 가능한 채널이 없으면 최종 실패 상태와 서비스 화면의 확인 경로를 남긴다.
- 하나의 논리 알림에서 최초 채널과 대체 채널을 구분한다. 대체 채널은 동일 목적에 허용되고 수신 가능성과 동의가 다시 검증된 경우에만 사용하며 채널을 바꿔 광고 동의를 우회하지 않는다.

### 다중 인스턴스·실패·감사·개인정보

- 선택 결과에는 정책 버전과 채널 후보 순서를 고정한다. 채널 상태가 바뀌면 다음 시도 전에 새 버전으로 재평가하되 이미 전달된 알림을 다른 채널로 다시 보내지 않는다.
- 모든 작업자는 중앙 채널 상태와 수신자 채널 버전을 사용한다. 인스턴스별 로컬 장애 판단이나 설정 차이로 서로 다른 채널을 동시에 선택하지 않는다.
- 채널 선택·제외·대체 사유를 주소 없이 감사하고, 더 많은 데이터를 담을 수 있는 채널로 바뀌어도 `NOTI-005`의 최소 템플릿을 유지한다.

### 관측 가능한 인수 조건

- 같은 정책·동의·장애 입력에는 어느 인스턴스에서도 같은 채널 순서가 선택된다.
- 선택 채널 동의가 없거나 주소가 미확인 상태이면 해당 채널 시도가 생성되지 않는다.
- 최초 채널 전달 성공 뒤 늦은 실패 응답이나 재시도 작업이 도착해도 대체 채널 발송이 생성되지 않는다.

## NOTI-003 수신 동의와 야간 발송

> 정책 상태: 확정

### 확정 동의·철회·시간 정책

- 선택 알림 동의는 목적과 채널을 분리하여 명시적으로 받고, 동의 문구·정책 버전·획득 경로·시각을 중앙 상태로 관리한다. 미응답·사전 선택·다른 목적 동의를 동의로 간주하지 않는다.
- 사용자는 같은 경로에서 선택 동의를 철회할 수 있다. 철회는 새 작업 생성 차단과 대기 작업 취소에 즉시 사용하고, 전달 완료 이력이나 원 거래 상태를 변경하지 않는다.
- 야간 여부와 허용 예외는 적용 가능한 중앙 발송 시간 정책의 시간대·정책 버전으로 판정한다. 선택 알림은 금지 구간이면 허용 구간까지 연기하고, 보안·거래 알림도 명시적으로 허용된 목적만 예외로 즉시 발송한다.
- 법적 시간대와 예외 문구를 코드나 이 정책에 추측하여 숫자로 고정하지 않는다. 유효한 시간 정책이 없거나 지역·시간대를 판정할 수 없으면 선택 알림은 실패 폐쇄한다.

### 동시성·실패·감사·개인정보

- 동의 변경은 중앙 버전의 조건부 갱신으로 순서를 확정한다. 발송 작업은 발송 직전 최신 버전을 다시 읽어 철회와 동시에 경합해도 철회 확정 뒤 새 전송을 시작하지 않는다.
- 동의 획득·철회·정책 변경·야간 연기·예외 사용을 감사하되 동의 증거에는 문구 버전과 최소 계정 참조만 저장한다.
- 시간대 판정을 위해 정밀 위치나 상시 위치를 수집하지 않고 계정·서비스에 이미 필요한 지역 설정만 사용한다.

### 관측 가능한 인수 조건

- 목적 A 동의가 목적 B나 다른 채널의 발송을 허용하지 않고 철회 후 대기 작업이 취소된다.
- 발송 금지 구간의 선택 알림은 연기되며 유효한 예외 목적이나 시간 정책이 없으면 즉시 발송되지 않는다.
- 동의 철회와 작업 획득을 병렬 실행해도 철회 확정 뒤 제공자 호출이 새로 시작되지 않는다.

## NOTI-004 예약·웨이팅 알림 시점

> 정책 상태: 확정

### 확정 사건·시점·우선순위

- 예약·웨이팅 알림은 원 도메인의 확정 상태 전이, 상태 버전과 예정 시각을 기준으로 생성한다. 웨이팅 입장 임박 안내는 예외적으로 `WAIT-010`의 확정된 순번 원장과 팀별 1회 사건을 기준으로 생성하되 상태 전이·실제 호출로 표시하지 않는다. 화면 요청 수신, 임시 선점, 미확정 결제처럼 되돌릴 수 있는 중간 상태를 확정 알림으로 보내지 않는다.
- 알림 종류별 기준 시점과 사전 안내 시점은 중앙 정책 버전과 원 자원의 시간대에서 계산한다. 근거 없는 횟수·시간 숫자를 이 문서에서 정하지 않고 매장이나 작업 인스턴스가 임의 변경하지 못하게 한다.
- 웨이팅은 앞선 호환 팀이 두 팀 이하가 될 때 입장 임박 안내를 한 번 보내고, 실제 입장 호출에는 중앙 호출 시각과 3분 도착 제한 시각을 함께 안내한다. 입장 임박·호출·미응답 종료·체크인 완료는 서로 다른 알림 목적으로 구분한다.
- 취소·변경·완료 같은 더 최신 상태가 도착하면 이전 상태의 아직 미발송 알림을 취소한다. 이미 전달된 정보가 중요하게 바뀌면 새 상태 버전의 변경 알림을 만들고 이전 메시지를 조용히 재사용하지 않는다.

### 동시성·실패·감사·개인정보

- 자원 ID-수신자-알림 목적-상태 또는 순번 원장 버전의 멱등 키와 중앙 시각을 사용한다. 입장 임박은 같은 웨이팅·수신자에게 한 번만 발송 작업을 만들고, 중복·역순 사건과 여러 스케줄러에도 최신 유효 버전의 작업만 발송 가능하다.
- 스케줄러 지연 시 원 상태와 현재 시각을 다시 검증하여 행동 시점이 지난 알림은 발송하지 않고 만료·취소한다. 지연·만료·상태 대체를 관측하고 감사한다.
- 메시지에는 이용에 필요한 매장 표시명, 상태, 예정·호출·도착 제한 시각과 안전한 화면 링크만 포함하고 동행자·다른 대기 팀·결제수단·알레르기 정보를 넣지 않는다.

### 관측 가능한 인수 조건

- 임시 선점이나 결제 결과 불명 상태에는 예약 확정 알림이 생성되지 않는다.
- 앞선 호환 팀이 두 팀 이하가 된 웨이팅에는 입장 임박 안내가 정확히 한 번 생성되고, 이 안내의 재시도는 실제 호출이나 상태 전이를 만들지 않는다.
- 호출 알림과 취소 사건을 역순·중복 주입해도 최신 취소 뒤 오래된 호출이 발송되지 않는다.
- 실제 호출 알림에는 중앙 호출 시각과 정확히 3분 뒤의 도착 제한 시각이 표시되고 사용자는 알림 수신 여부와 관계없이 최신 제한 시각을 상태 조회로 확인할 수 있다.
- 지연 작업이 행동 가능 시점 뒤 실행되면 만료되고, 어떤 기준 사건·상태 버전으로 취소됐는지 조회할 수 있다.

## NOTI-006 중복 알림 방지

> 정책 상태: 확정

### 확정 멱등 키·전달 판정

- 논리 알림은 원 사건 ID, 수신자 계정 ID, 목적, 자원·상태 버전으로 결정되는 멱등 키를 가진다. 채널별 시도는 같은 논리 알림 아래 별도 식별자로 기록하여 대체 채널을 새 알림으로 중복 계산하지 않는다.
- 중앙 저장소는 멱등 키당 활성 논리 작업 하나와 채널 시도별 유일성을 강제한다. 제공자가 멱등 키를 지원하면 같은 시도 키를 전달하되 내부 원장을 최종 중복 방어선으로 유지한다.
- 전달 성공이 확인되면 논리 작업을 완료하고 예약된 재시도·대체 채널을 취소한다. 응답이 불명확하면 새 시도를 즉시 만들지 않고 같은 시도의 결과 조회 또는 안전한 재시도 규칙을 따른다.

### 다중 인스턴스·실패·감사

- 작업 획득은 임대와 조건부 상태 전이를 사용하며 임대 만료 후 다른 작업자가 이어받아도 같은 시도 식별자를 재사용한다. 처리 인스턴스의 로컬 성공 기록만으로 완료하지 않는다.
- 중복 억제, 제공자 응답 충돌과 늦은 성공을 관측한다. 운영자 수동 재발송은 원 알림 참조·사유·권한을 가진 새 감사 작업으로 만들고 기존 멱등 키를 몰래 우회하지 않는다.
- 중복 판정을 위해 주소·기기 지문을 새로 저장하지 않고 중앙 계정과 원 사건 식별자만 사용한다.

### 관측 가능한 인수 조건

- 동일 사건을 여러 생산자·작업자가 병렬로 제출해도 논리 알림과 최초 채널 시도는 각각 하나다.
- 제공자 성공 응답 뒤 작업자가 중단되어 재개해도 새 대체 채널이나 다른 메시지가 발송되지 않는다.
- 수동 재발송은 원 알림·사유·행위자가 감사되고 일반 재시도와 구분된다.

## NOTI-007 실패 재시도와 대체 채널

> 정책 상태: 확정

### 확정 오류 분류·복구 흐름

- 제공자 결과는 `전달`, `일시 실패`, `영구 실패`, `속도 제한`, `결과 불명`으로 정규화한다. 네트워크·일시 장애만 제한된 지연 재시도 대상으로 하고 잘못된 주소·동의 없음·차단 같은 영구 실패는 같은 채널로 반복하지 않는다.
- 재시도 횟수·간격은 채널별 중앙 운영 정책으로 제한하며 무제한·즉시 반복하지 않는다. 결과 불명은 제공자 조회가 가능하면 같은 시도 키로 확인하고, 확인할 수 없으면 중복 위험을 표시한 채 정책이 허용하는 복구 경로로 전환한다.
- 대체 채널은 `NOTI-002`의 목적·동의·수신 가능성 검증을 다시 통과해야 한다. 모든 후보가 소진되면 최종 실패로 확정하고 사용자가 중앙 상태를 확인할 화면 경로와 운영자 경보를 제공한다.

### 다중 인스턴스·격리·감사·개인정보

- 다음 실행 시각과 시도 횟수를 중앙 상태에서 원자 갱신하고 작업 임대로 하나의 작업자만 호출한다. 공급자별 회로 차단과 동시 실행 격리로 한 채널 장애가 다른 채널이나 예약·결제 처리 스레드를 고갈시키지 않게 한다.
- 재시도·대체·최종 실패·수동 재발송을 감사하고 성공률, 지연, 오류 분류와 대기열 적체를 채널·목적별로 관측한다. 경보 임계치는 실측 설정으로 둔다.
- 대체 채널에도 원 템플릿보다 더 많은 개인정보를 넣지 않고 실패 응답의 주소·본문을 로그에 남기지 않는다.

### 관측 가능한 인수 조건

- 영구 실패에는 같은 채널 재시도가 생성되지 않고 일시 실패만 중앙 정책 한도 안에서 재시도된다.
- 한 제공자의 지연·오류를 주입해도 다른 채널 작업과 원 거래 API의 작업 풀이 고갈되지 않는다.
- 대체 채널 동의가 없으면 최종 실패로 남으며, 최종 실패 뒤에도 원 예약·결제 상태가 되돌아가지 않는다.

## NOTI-008 잘못된 대상 발송 방지

> 정책 상태: 확정

### 확정 수신자 결속·검증

- 원 사건은 수신자 계정·역할·자원 관계와 목적만 지정하고 주소를 직접 제공하지 않는다. 발송 직전에 중앙 계정에서 최신 확인 채널과 소유 버전을 조회하고 원 자원에 대한 수신 자격을 다시 검증한다.
- 연락처 변경·계정 삭제·권한 회수·예약 양도·매장 소속 변경으로 수신 관계가 바뀌면 이전 주소 작업을 취소한다. 새 수신자 알림이 필요하면 새 관계 버전과 멱등 키로 별도 생성한다.
- 대량·관리자 발송은 대상 질의 버전, 목적, 예상 대상 범위와 샘플 검증을 고정하고 실행 직전 대상 수·권한 변화 검사를 통과해야 한다. 자유 입력 주소 목록이나 다른 환경의 데이터를 붙여 넣어 발송하지 않는다.
- 테스트·개발·운영 환경의 채널 자격증명과 대상 저장소를 분리하고 비운영 환경은 실제 사용자 주소 발송을 기본 차단한다.

### 동시성·실패·감사·개인정보

- 주소 해석과 작업 전이를 같은 수신자 버전에 조건부 결속한다. 검증 뒤 주소가 바뀌면 제공자 호출 전 전이를 거부하고 재검증한다.
- 발송 대상 결정·제외·관계 변경·대량 대상 승인과 실제 채널 참조 버전을 감사하되 원 주소를 감사 로그에 복제하지 않는다.
- 대상 저장소 장애·관계 불일치·검증 결과 불명에는 발송하지 않고 격리한다. 오류 조사 화면도 주소를 기본 마스킹하고 업무 권한 있는 운영자만 최소 범위를 조회한다.

### 관측 가능한 인수 조건

- 연락처 변경과 발송을 경합시키면 이전 소유 버전 주소로 새 제공자 호출이 시작되지 않는다.
- 예약 양도·매장 소속 회수 뒤 이전 당사자의 대기 작업은 취소되고 새 관계 작업과 섞이지 않는다.
- 비운영 환경이나 자유 입력 주소 목록으로 실제 대상 발송을 시도하면 차단·감사된다.

## NOTI-010 대량 발송 속도와 공급자 장애

> 정책 상태: 확정

### 확정 대량 작업·우선순위·격리

- 대량 발송은 목적·대상 질의 버전·템플릿 버전·정책 버전으로 불변 실행 계획을 만들고 검증·승인된 계획만 중앙 작업으로 분할한다. 특정 제공자, 유료 상품, 숫자 용량을 이 정책에 고정하지 않는다.
- 보안·시간 민감 거래, 일반 거래, 선택 정보·광고를 별도 우선순위와 격리된 대기열로 처리한다. 선택 대량 발송 적체가 보안·거래 알림의 작업자·연결·제공 한도를 점유하지 못하게 한다.
- 채널·제공자별 중앙 속도 정책과 회로 차단을 공유하고 `속도 제한` 응답에는 발송 속도를 낮춘다. 장애 중에는 내구성 대기열에 보관하고 새 대량 작업을 보류할 수 있지만 이미 확정된 논리 알림을 조용히 폐기하지 않는다.
- 중단·취소는 아직 획득되지 않은 작업에 적용하고 이미 전달된 알림을 되돌리지 않는다. 재개는 마지막 중앙 체크포인트에서 수행한다.

### 다중 인스턴스·관측·감사·개인정보

- 파티션별 작업 임대, 수신자 멱등 키와 중앙 체크포인트로 여러 작업자가 분담해도 누락·중복 없이 수렴한다. 한 인스턴스 중단 뒤 다른 작업자가 미완료 파티션만 이어받는다.
- 대기 깊이, 가장 오래된 작업 지연, 처리율, 속도 제한·회로 상태, 실패·대체 채널 비율을 목적·채널별 관측한다. 수치 임계치는 실측 운영 설정으로 두고 공급자 장애 런북에 중지·격리·재개·대사를 포함한다.
- 대량 계획·승인·취소·재개·속도 정책 변경과 수동 개입을 감사한다. 작업 파티션과 지표에는 주소·본문·사용자 프로필을 넣지 않는다.

### 관측 가능한 인수 조건

- 선택 대량 발송을 적체시켜도 보안·거래 알림이 독립 대기열과 작업 용량으로 처리된다.
- 공급자 속도 제한·장애와 작업자 중단을 주입한 뒤 중앙 체크포인트에서 재개하여 대상별 전달은 최대 한 번의 논리 결과로 수렴한다.
- 배포 인스턴스 수가 달라도 공유 속도 정책이 지켜지고 작업·대상·승인·취소·재개 이력이 감사된다.
