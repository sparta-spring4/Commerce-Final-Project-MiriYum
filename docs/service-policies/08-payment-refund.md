## PAY-001 결제 대상과 결제 시점

> 정책 상태: 확정

### 확정 규칙·상태·흐름

- 매장은 예약금 사용 여부를 선택한다. 예약금을 사용하려면 현재 게시된 유효한 대표 메뉴가 하나 이상 있어야 하며, 사용하지 않는 매장은 결제 불필요 판정 뒤 예약을 확정한다.
- 예약금이 필요한 흐름은 예약 수용량과 선택 메뉴 수량을 10분 선점한 뒤 결제를 요청하고, 서버가 검증한 결제 성공만 `RES-001`·`HOLD-004` 최종 확정의 근거로 사용한다.
- 명확한 결제 실패에는 원래 만료 시각까지 같은 선점 그룹으로 재결제를 허용하되 만료를 연장하지 않는다. 사용자 포기나 10분 내 미성공에는 전체 자원을 반환한다.
- 결과 불명확·Webhook 지연에는 `결제 확인 중`으로 격리하고 PG 조회·검증된 Webhook 전에는 재청구·재판매·실패 확정을 하지 않는다.

### 동시성·실패·인수 조건

- 거래·결제 시도·선점 그룹 멱등 키와 상태 버전으로 중복 청구와 중복 확정을 막는다. 성공 뒤 자원 확정 실패는 멱등 재시도하고 불가능하면 승인 취소·전액 환불한다.
- 사용자 성공 화면만으로 예약을 확정하지 않으며, 재전송·Webhook 중복·만료 경합 뒤 결제와 자원은 완료 또는 보상 완료로 수렴한다.

## PAY-002 예약금·선결제·보증금 구분

> 정책 상태: 확정

### 확정 금액·성격·표시

- 예약금은 노쇼 방지용 보증금이다. 메뉴를 선택해도 메뉴 가격이나 별도 홀드 수수료를 선결제하지 않고, 실제 식사 금액은 매장에서 별도로 결제하며 MiriYum 예약금과 상계하지 않는다.
- 대표 메뉴 평균값은 현재 게시된 대표 메뉴 기본 판매가의 단순 산술평균이다. 일시 품절 대표 메뉴는 포함하고 숨김·판매 종료·삭제 메뉴와 옵션 추가금은 제외한다.
- 최솟값은 `(대표 메뉴 평균값 × 예약 인원수) × 10%`, 최댓값은 `(대표 메뉴 평균값 × 예약 인원수) × 30%`이며 둘 다 1,000원 단위로 올림한다. 매장은 이 범위 안에서 1,000원 단위 금액을 선택한다.
- 정상 체크인에는 예약금을 전액 환불한다. 예약금은 실제 손해액을 확정·보장하지 않으며 메뉴 대금·코스 선결제·식당 정산금으로 표시하지 않는다.

### 버전·감사·인수 조건

- 계산에는 예약 생성 당시 대표 메뉴·가격·인원·예약금 정책 버전을 고정하고, 이후 가격 변경으로 기존 금액을 소급 변경하지 않는다.
- 계산 입력·범위·선택 금액·표시와 환불 연결을 감사하되 결제수단 원문을 저장하지 않는다. 경계값·올림·포함/제외 메뉴가 같은 버전에서 결정적으로 재현되어야 한다.

## PAY-004 금액 계산·세금·반올림

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 서버는 승인된 상품·옵션·수량·가격·세금 정책 버전과 `PAY-001`·`PAY-002`의 거래 유형으로 통화별 최소 단위의 정수 금액을 계산한다. 클라이언트 총액, 부동소수점 계산, 운영자 자유 입력을 승인 근거로 사용하지 않는다.
- 주문에는 항목별 단가·수량·할인·과세 구분·세액·반올림·총액·정책 버전의 불변 스냅샷을 저장한다. 반올림 위치와 방식은 적용 법규·계약에 따른 버전 설정을 사용하며 이 행에서 세율이나 방식을 발명하지 않는다.
- 금액 영향 변경은 새 계산 스냅샷을 만들고 사용자에게 차이를 확인시킨 뒤 필요한 결제 조정을 요청한다. 실패·철회·만료에는 원래 거래를 유지하고 자동 추가 청구·환불하지 않는다.

### 동시성·실패·알림·감사·개인정보

- 주문 수정 번호·계산 지문·멱등 키로 동시에 한 금액 버전만 결제 요청에 연결한다. 가격 원본이나 세금 설정을 확인할 수 없으면 신규 승인 요청을 실패 폐쇄한다.
- 결제 후 재계산 차이는 거래를 임의 수정하지 않고 불일치로 격리해 `PAY-014` 대사·보정 절차로 보낸다. 다른 통화끼리 합산하거나 암묵 환전하지 않는다.
- 사용자와 매장에는 항목별 계산과 최종 금액을 표시하고 계산 버전·원장 전후·행위자·사유를 감사한다. 결제수단 원문이나 불필요한 개인정보를 계산 로그에 넣지 않는다.

### 관측 가능한 인수 조건

- 동일 스냅샷은 모든 인스턴스에서 같은 최소단위 금액을 만들고 클라이언트 금액 변조·부동소수점 경계가 승인으로 이어지지 않는다.
- 가격 변경과 결제 경합에도 결제는 정확히 한 금액 버전을 참조하고 실패 시 원래 거래가 유지된다.
- 계산 내역을 원장과 감사 기록에서 재현할 수 있으며 `PAY-001`·`PAY-002`·`PAY-009`의 금전 결정을 새로 만들지 않는다.

## PAY-005 결제 요청 멱등성

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 로그인 사용자의 결제 시작, 식당·운영자의 허용된 취소·환불 명령은 작업 유형과 거래별 고유 멱등 키를 요구한다. 같은 키와 같은 요청 지문은 기존 결과를 반환하고, 같은 키의 다른 지문은 충돌로 거부한다.
- 중앙 결제 시도는 `생성`, `외부 처리 중`, `결과 확인`, `성공`, `실패`, `결과 불명확` 상태와 주문·금액 버전, 외부 참조를 가진다. 응답 타임아웃에는 새 시도를 만들지 않고 같은 키로 서버 조회·외부 상태 조회 후 전이한다.
- 키는 인증된 행위자·거래·명령 범위에 묶고 다른 사용자나 거래에서 재사용할 수 없다. 종결 결과도 대사·분쟁에 필요한 동안 재현 가능하게 유지하되 보관 숫자는 개인정보·법률 TODO를 따른다.

### 동시성·실패·알림·감사·개인정보

- DB 유일성 제약과 조건부 갱신으로 여러 인스턴스의 동시 요청 중 외부 명령 하나만 발행한다. 외부 승인·취소 명령은 일반 자동 재시도하지 않고 제공자 멱등 참조와 상태 재조회로 복구한다.
- 결과 저장 전 장애가 발생하면 `결과 불명확`으로 격리하고 신규 결제·자원 확정을 막는다. 복구 작업과 `PAY-014` 대사가 최종 상태를 확인하며 운영자도 새 외부 참조로 중복 실행하지 않는다.
- 진행·성공·실패의 최종 결과만 사용자에게 알리고, 멱등 키·요청 지문·외부 참조·상태 전이·복구를 감사한다. 키나 로그에 카드·계좌 원문을 넣지 않는다.

### 관측 가능한 인수 조건

- 병렬·반복 요청과 응답 유실에도 외부 결제 명령·내부 금전 원장·알림이 각각 한 번만 유효하다.
- 같은 키의 다른 요청은 거부되고 결과 불명확 상태에서 새 청구가 생성되지 않는다.
- 작업 인스턴스 재시작 후 같은 키로 기존 결과를 조회·복구할 수 있고 단일 인스턴스 메모리에 의존하지 않는다.

## PAY-006 웹훅 검증·중복·순서 역전

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 웹훅은 `PAY-003`에서 선택될 제공자의 공식 서명·타임스탬프·이벤트 식별 검증과 허용된 전송 경로를 모두 통과해야 수신 사건이 된다. 검증 비밀은 서버 비밀 저장소에서 관리하고 클라이언트나 로그에 노출하지 않는다.
- 검증 후에도 이벤트 금액·통화·가맹 참조·외부 거래 ID를 내부 주문 스냅샷 및 서버-대-서버 조회 결과와 대조한다. 웹훅 본문만으로 결제 성공·환불 완료·예약 확정을 만들지 않는다.
- 원본 수신 → 검증/격리 → 제공자 상태 조회 → 현재 결제 버전에 허용된 전이 → Outbox 후속 처리 순으로 진행한다. 이전 상태 이벤트는 원본만 보존하고 현재 종결 상태를 되돌리지 않는다.

### 동시성·실패·알림·감사·개인정보

- 제공자 이벤트 ID와 업무 전이 ID에 유일성 제약을 두고 중복 소비·순서 역전을 멱등하게 무시한다. 여러 인스턴스가 처리해도 결제 상태 버전의 조건부 전이 하나만 성공한다.
- 서명·대조 실패는 성공 응답과 별개로 보안 격리·경보 대상이며 거래를 변경하지 않는다. 제공자 조회 장애에는 `검증 대기`로 두고 제한 재시도·대사하며 성공으로 추정하지 않는다.
- 실제 상태 전이가 있을 때만 사용자·매장 후속 알림을 생성한다. 수신 시각·검증 결과·외부/내부 참조·전이·격리 사유를 감사하되 웹훅 원문의 결제수단·개인정보는 마스킹한다.

### 관측 가능한 인수 조건

- 위조·변조·잘못된 금액/통화 웹훅이 결제·예약 상태를 바꾸지 않고 격리·경보된다.
- 동일 이벤트 반복과 성공 뒤 과거 실패 이벤트 순서 역전에도 최종 상태·원장·알림이 중복되지 않는다.
- 제공자 조회 장애가 복구되면 중앙 원본으로 상태가 수렴하고 모든 인스턴스에서 같은 검증 결과가 적용된다.

## PAY-007 결제 상태와 예약 상태 연결

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 결제와 예약·홀드·테이스팅은 각자 상태를 소유하고 거래 연결 ID와 버전된 명령·이벤트로만 연동한다. `PAY-001`·`PAY-002`가 결제가 필요하다고 정한 흐름에서만 검증된 결제 성공이 자원 최종 확정의 사전 조건이 된다.
- 기본 연결 상태는 `결제 불필요/대기`, `결제 처리 중`, `결제 검증 성공`, `자원 확정 중`, `완료`, `보상 필요`, `보상 중`, `복구 필요`로 구분한다. 결제 성공 전 자원을 확정하거나 예약 실패만으로 결제 성공을 취소된 것으로 표시하지 않는다.
- 검증된 결제 성공 뒤 자원 확정이 실패하면 같은 자원 명령을 멱등 재시도한다. 중앙 자원이 만료·소진되어 확정 불가능하면 거래를 `보상 필요`로 두고 `PAY-009`의 매장·플랫폼 귀책 전액 환불로 연결한다.

### 동시성·실패·알림·감사·개인정보

- 거래 연결 버전, 결제·자원 멱등 키와 조건부 전이로 결제/취소/만료 경합 중 하나의 다음 상태만 허용한다. DB 상태와 Outbox를 같은 처리 경계에 기록해 후속 작업 유실을 막는다.

- 진행 상태와 사용자의 필요한 행동, 최종 확정·실패·보상 결과를 알리고 거래 연결, 두 도메인의 전이 전후, 원장과 복구를 감사한다. 도메인 이벤트에는 결제수단 원문을 포함하지 않는다.

### 관측 가능한 인수 조건

- 결제 성공/예약 확정 사이 각 장애 지점에서 재시도 후 완료 또는 보상 필요로 수렴하며 성공 결제와 미확정 자원이 방치되지 않는다.
- 결제·만료·취소 동시 처리에도 자원 확정과 보상 명령이 동시에 유효하지 않고 중복 청구·이중 복구가 없다.
- `PAY-001`의 결제 시점과 `PAY-002`의 예약금 성격을 적용해 결제 성공 전 자원을 확정하지 않고 메뉴 대금을 청구하지 않는다.

## PAY-008 전체·부분 환불 계산

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 환불 가능 여부·귀책·수수료 입력은 `PAY-009`와 원 거래 정책에서 받고, 서버는 원 결제의 항목·세금·할인·기환불 원장 스냅샷으로 환불 가능 잔액을 최소 통화 단위로 계산한다. 총 누적 환불은 원 승인 금액을 넘을 수 없다.
- 전체 환불은 남은 환불 가능액 전부, 부분 환불은 지정된 항목·수량 또는 승인된 금액 근거와 계산 내역을 가진다. 할인·세금 배분과 반올림은 원 거래 정책 버전을 사용하며 운영자가 결과 금액을 근거 없이 덮어쓰지 않는다.

### 동시성·실패·알림·감사·개인정보

- 원 결제-환불 요청 유일성, 환불 멱등 키와 잔액 조건부 갱신으로 병렬 부분 환불의 합계를 제한한다. 외부 결과 불명확 시 같은 참조로 조회하고 별도 환불을 발행하지 않는다.
- 외부 완료 후 내부 기록 실패는 `PAY-014` 대사로 복구하고, 내부 예약·수량 상태를 임의 되돌리지 않는다. 계산 불일치는 환불 실행 전 실패 폐쇄한다.
- 사용자와 매장에 원 금액·이번 환불·누적·남은 금액과 상태를 알리고 계산 버전·승인 근거·외부 참조·원장 전후를 감사한다. 결제수단 원문은 저장·알림하지 않는다.

### 관측 가능한 인수 조건

- 전체·부분·병렬 환불 조합에서도 누적 완료액이 원 승인액을 넘지 않고 계산을 원 스냅샷에서 재현할 수 있다.
- 응답 유실·중복 웹훅이 환불 원장이나 외부 요청을 중복 생성하지 않고 결과 불명확 중 새 환불이 차단된다.
- 환불 금액은 `PAY-009` 입력을 적용할 뿐 취소 책임·수수료 규칙을 이 행에서 선택하지 않는다.

- 사용자와 매장에 적용 기준·비율·금액·상태와 이의 경로를 알리고 원 사건·정책·계산·보정 관계를 감사하되 내부 메모와 결제수단 원문을 노출하지 않는다.

- 경계 시각, 매장·플랫폼 귀책, 확정 노쇼와 정정 사례가 공개된 비율대로 재현되고 중복 환불이 없다.
- 노쇼 후보나 결과 불명확 사건만으로 환불 없음이 확정되지 않으며 정정 뒤 필요한 환불이 원장과 함께 수렴한다.

## PAY-010 환불 실패와 재시도

> 정책 상태: 확정

### 관측 가능한 인수 조건

- 일시 장애·작업 재시작·중복 실행에도 외부 환불과 완료 원장은 한 번만 유효하고 무제한 재시도가 없다.
- 결과 불명확 상태에서 새 환불이 발행되지 않고 제공자 복구 후 조회·대사로 완료 또는 실패가 확정된다.
- 최종 실패가 사라지지 않고 운영자 큐·경보·사용자 상태에서 추적 가능하다.

### 관측 가능한 인수 조건

- 권한 없는 접근·내보내기가 차단되고 추가 금융·신분 개인정보 없이 당사자에게 상태가 안내된다.

## PAY-015 운영자 수동 복구와 승인

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- `결제 복구` 권한이 있고 사건에 배정된 플랫폼 운영자만 `PAY-014` 불일치 사건에서 허용된 보정안을 제안할 수 있다. 사용자의 결제수단으로 새 청구를 만들거나 근거 없이 금액·귀책·정산 상태를 직접 덮어쓸 수 없다.
- 실행 전 최신 내부 원장·외부 서버 조회·자원 상태, 예상 원장 변화와 중복 여부를 다시 검증한다. 금전 이동이 없는 멱등 재처리와 검증된 PG·외부 정산 결과를 누락 없이 반영하는 원장 보정은 `결제 복구` 권한이 있고 사건에 배정된 운영자 1명이 실행할 수 있다. 이 원장 보정은 과거 항목을 수정·삭제하지 않고 반대·보정 항목을 추가하며 새 외부 금전 이동이나 재량적인 금액·귀책·정산 결정을 만들지 않는다.
- 새 환불·보상의 승인 경계는 `ADMIN-006`을 따른다. 원 결제금액 이하이면서 건당 200,000원 이하이면 사건에 배정되고 `결제 복구`와 `결제 관리자` 권한을 모두 가진 운영자 1명이 제안·승인·실행할 수 있다. 원 결제금액 이하에서 200,000원을 초과하면 결제 관리자의 요청과 그 요청자와 다른 슈퍼관리자의 추가 승인을 요구하고, 원 결제금액을 초과하는 별도 보상은 금액과 무관하게 요청자와 다른 슈퍼관리자의 승인을 요구한다.
- 복구 명령은 사건 버전·승인 버전·업무 멱등 키로 한 번만 실행하며 결과가 불명확하면 재발행하지 않고 조회·대사로 돌아간다. 실패한 승인안은 원본 상태를 유지하고 새 버전으로 재검토한다.
- 당사자에게 상태 변경과 문의 경로를 알리고 제안자·승인자·실행자·승인 유형·근거·조회 결과·명령·원장 전후·실패를 변경 불가능한 감사 이력으로 남긴다. 200,000원 이하 단독 승인에는 요청자·승인자·실행자로 같은 계정을 기록하고 `단독 승인` 유형을 명시한다. 증거 접근은 최소 권한이며 결제수단 원문을 운영 화면·로그에 노출하지 않는다.

### 관측 가능한 인수 조건

- 권한이 없거나 사건에 배정되지 않은 운영자·구 버전 복구·직접 상태 덮어쓰기가 차단된다. 원 결제금액 이하 200,000원은 필요한 두 권한을 가진 담당 운영자 1명이 단독 승인·실행할 수 있고, 200,001원과 원 결제금액 초과 보상은 `ADMIN-006`이 요구하는 서로 다른 두 행위자를 가진다.
