# 코스·테이스팅 정책

> 문서 상태: 세부 정책 확정(팀원 상의 필요 경계·기존 확정 하위 범위 보존)
> 정책 범위: `TASTE-001`~`TASTE-012`
> 최종 변경일: 2026-07-22

회차·정원, 참가 방식, 주류와 연령, 알레르기 설문, 선결제, 평가 및 행사 취소를 다룬다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 팀원 상의 필요 | 핵심 | `TASTE-003`, `TASTE-008`, `TASTE-012` |
| 확정 | 필수 | `TASTE-001`, `TASTE-002`, `TASTE-004`~`TASTE-006`, `TASTE-009` |
| 확정 | 권장 | `TASTE-007`, `TASTE-010`, `TASTE-011` |

- `TASTE-003`: 선착순·추첨·승인제는 희소한 참가 기회의 배분 원칙이므로 팀 합의가 필요하다.
- `TASTE-008`: 선결제 범위와 부분 환불은 참가 흐름과 금전 책임을 결정하므로 팀 합의가 필요하다.
- `TASTE-012`: 행사 취소·변경 시 보상 약속은 재무·고객 책임을 만드므로 팀 합의가 필요하다.
- `TASTE-001`의 가게별 테이스팅 기능 승인과 행사·버전별 재승인 범위는 그대로 보존하고, 일반 코스와 테이스팅의 제품·용어·상태 경계만 아래 위임 기준으로 확정했다.
- `TASTE-005`의 미성년자 차단, 가게의 현장 확인 책임, MiriYum의 신분증·확인 상태 미수집은 그대로 보존하고, 현장 확인 시점·실패 처리만 아래 위임 기준으로 확정했다. 환불은 `TASTE-008` 팀 결정에 종속된다.

## TASTE-001 코스와 테이스팅의 구분

> 정책 상태: 확정. 아래 규칙은 기존 STORE-008 연계 승인 범위를 변경하거나 `STORE-007`의 일반 기능 활성화 결정을 대신하지 않는다.

### 확정 규칙·행위자·상태·흐름

- `코스`는 게시된 구성과 회차 자원을 예약하는 일반 상품, `테이스팅`은 참가자 설문·평가 또는 주류 등 별도 행사 운영과 행사별 승인이 필요한 경험으로 구분한다. 유효한 해당 매장 권한을 가진 식당 대표자만 코스와 테이스팅 중 상품 유형을 선택·변경할 수 있다.
- 플랫폼 운영자는 상품 유형을 코스에서 테이스팅으로 또는 그 반대로 변경할 수 없다. 기존 STORE-008 권한 안에서 테이스팅 제한 기능에는 정확히 `승인`, `보완 요청`, `거절`, `일시 중지`만 수행할 수 있고, 개별 행사 버전 심사에는 아래의 `승인`, `수정 요청`, `거절` 상태 전이만 수행한다. 어느 심사 결과도 상품 유형을 바꾸지 않으며, 별도로 권한과 절차가 확정된 복구 정책이 생기기 전에는 장애·고객지원 사유로도 유형을 대신 변경하지 않는다.
- 코스의 생성·게시는 향후 `STORE-007`이 코스를 일반 기능으로 활성화하도록 정한 경우에만 허용한다. `STORE-007` 결과가 미확정이거나 코스 활성화를 허용하지 않으면 코스 초안의 게시 전이를 실패 폐쇄한다. 테이스팅은 `STORE-007`을 사용하지 않고, 기존 STORE-008에 따른 해당 매장의 제한 기능이 `승인` 상태이며 `일시 중지`되지 않았고 해당 행사 버전도 승인된 경우에만 게시할 수 있다. 기능이 `일시 중지`되면 신규 행사 게시·모집·예약과 새 행사 버전 활성화를 모든 인스턴스에서 차단하고, 기존 신청을 임의 취소하지 않은 채 영향 사건으로 처리한다. 이 정책 자체는 어느 기능도 활성화하지 않는다.
- 코스 상태는 `초안` → `게시` → `마감` 또는 `취소`로 전이한다. 게시 전 수정은 같은 초안 버전에서, 게시 뒤 안전한 표현 수정은 변경 이력을 가진 새 표시 버전에서 처리하고, 회차·가격·정원·주요 구성처럼 신청 조건을 바꾸는 수정은 새 불변 상품 버전으로 게시한다. 코스에는 테이스팅 심사 상태를 사용하지 않는다.
- 테이스팅 상태는 `초안` → `심사 대기` → `승인`·`수정 요청`·`거절` 중 하나로 전이하고, `승인`된 행사 버전만 `게시` → `마감` 또는 `취소`로 갈 수 있다. `수정 요청`은 새 버전 보완 뒤 다시 `심사 대기`로 제출하고 `거절` 버전을 게시하지 않는다. 기존 확정대로 가게별 테이스팅 기능 승인과 행사 버전별 승인이 모두 있어야 하며 어느 하나도 다른 단계를 대체하지 않는다.
- 코스에서 테이스팅 또는 테이스팅에서 코스로 유형을 바꾸는 행위는 기존 버전의 상태를 덮어쓰지 않고 대상 유형의 새 초안에서 해당 상태 기계를 처음부터 따른다.
- 사용자는 로그인 후 게시된 버전만 신청한다. 가격·회차·정원·주요 구성·주류 여부 등 중요 변경은 기존 신청에 소급하지 않고 영향 사건과 `TASTE-012` 결과로 처리한다. 철회·취소·종료 뒤 회차 자원 복구는 연결된 예약·홀드 정책을 따른다.

### 동시성·실패·알림·감사·개인정보

- 코스는 `STORE-007` 일반 기능 결과, 테이스팅은 STORE-008 제한 기능·행사 승인 원본을 각각 중앙에서 검증한다. 유형·상품/행사 버전·승인 상태·명령 멱등 키를 조건부 갱신하고 동일 버전의 활성 심사와 게시 상태를 하나만 허용하며, 해당 유형의 게이트 원본을 확인할 수 없으면 신규 공개·모집·확정을 실패 폐쇄한다.
- 검색·알림 갱신 실패는 승인·모집 상태를 되돌리지 않고 내구성 작업으로 재처리한다. 잘못 게시된 미승인 버전은 즉시 신규 신청을 차단하고 기존 신청은 삭제하지 않은 채 운영자 검토와 영향 사건으로 격리한다.
- 유형·버전·승인·게시·중지·영향 대상을 당사자에게 필요한 범위로 알리고 행위자·이전/새 버전·심사 사건·사유·결과를 감사한다. 유형 구분을 위해 추가 개인정보를 수집하지 않는다.

### 관측 가능한 인수 조건

- 코스와 테이스팅의 필수 필드·상태가 혼용되지 않고, 코스는 `STORE-007`이 일반 기능 활성화를 허용한 경우에만 심사 상태 없이 게시되며 테이스팅은 STORE-008의 매장 제한 기능 승인과 행사 버전 승인이 모두 없으면 어느 인스턴스에서도 공개·모집되지 않는다.
- 식당 대표자가 아닌 플랫폼 운영자·다른 사용자·다른 매장 대표자의 유형 변경은 모두 거부되고, 운영자의 테이스팅 기능 `승인`·`보완 요청`·`거절`·`일시 중지` 또는 행사 버전 심사 결과가 상품 유형 자체를 바꾸지 않는다.
- 테이스팅 기능 `일시 중지`와 신규 게시·모집·예약·행사 버전 활성화 요청이 경합해도 중지 후 신규 활성화는 하나도 성공하지 않고 기존 신청은 자동 취소되지 않는다.
- `STORE-007`이 미확정·비활성인 코스와 STORE-008 기능 또는 행사 승인이 없는 테이스팅은 게시로 전이하지 않으며, TASTE-001만으로 어느 기능도 활성화되지 않는다.
- 중요 변경과 게시 요청이 경합해도 승인된 한 버전만 사용되고 기존 신청의 스냅샷이 소급 변경되지 않는다.
- 중복 명령·검색 장애·작업 재시작 뒤에도 게시 상태와 감사 이력이 하나로 수렴하며 기존 확정 하위 범위가 유지된다.

## TASTE-002 회차·정원·좌석 구성

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 식당 대표자는 행사 버전별 회차 시작·종료, 총정원과 좌석/정원 버킷을 게시한다. 회차는 승인된 행사 버전, 영업 상태와 모순되지 않아야 하며 같은 실제 좌석·정원을 겹치는 회차에 중복 산입하지 않는다.
- 신청은 공개된 회차에 대해 참가 인원과 필요한 자원 전체를 하나의 선점 그룹으로 확보한 뒤 `TASTE-003`이 정한 참가 방식으로 전이한다. 중간 선점·심사 상태를 참가 확정으로 표시하지 않는다.
- 회차·정원 변경은 새 버전으로 이후 신청에만 적용한다. 기존 확정 참가의 이행이 어려우면 임의 취소하지 않고 영향 사건과 `TASTE-012` 처리로 넘기며 유효한 거절·취소·만료에만 자원을 한 번 복구한다.

### 동시성·실패·알림·감사·개인정보

- 중앙 DB의 겹침 제약, 0 미만 금지 조건부 갱신, 회차 버전과 멱등 키로 다중 인스턴스 경합을 직렬화한다. 일부 자원만 확보되면 그룹 전체를 보상 해제한다.
- 자원 원본 장애에는 신규 확정을 실패 폐쇄하고 고아 선점·정원 불일치는 대사 작업으로 격리·복구한다. 복구 전 수량을 판매 가능으로 노출하지 않는다.
- 생성·변경·확정·마감·영향을 참가자와 매장에 알리고 회차/자원 버전, 원장 전후, 행위자와 상관관계 ID를 감사한다. 좌석 내부 메모나 다른 참가자의 신원은 노출하지 않는다.

### 관측 가능한 인수 조건

- 병렬 신청 성공 합계가 회차 정원을 넘지 않고 같은 좌석이 겹치는 회차에 배정되지 않는다.
- 부분 장애·중복 이벤트 후 그룹은 전부 확보 또는 전부 해제로 수렴하며 원장과 참가 상태가 대사된다.
- 회차 변경이 기존 참가를 소급 이동하지 않고 `TASTE-003`의 가능한 모든 결과를 기계적으로 소비한다.

## TASTE-004 참가 자격과 초대

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 로그인한 일반 사용자만 신청할 수 있고 행사에 공개된 계정 상태·연령·정원·신청 기간 조건을 서버가 검증한다. 식당은 운영에 필요한 자격 항목만 버전으로 게시하며 임의의 민감정보나 동행자 연락처를 요구하지 않는다.
- 초대는 행사 버전에 연결된 일회성·목적 제한 초대 참조이며 참가 확정이나 우선권을 뜻하지 않는다. 초대받은 사용자도 `TASTE-003`의 동일 배분 규칙과 자격 검증을 통과해야 하고 식당·운영자는 숨은 우선 승인을 만들지 않는다.
- 자격 상실·초대 철회는 신규 신청만 차단하고 기존 확정 건은 자동 취소하지 않는다. 이행 불가 여부는 영향 사건으로 검토하며 취소·환불·보상은 `TASTE-008`·`TASTE-012` 결과를 따른다.

### 동시성·실패·알림·감사·개인정보

- 행사-사용자 활성 신청 유일성, 초대 소비 원자성, 상태 버전과 멱등 키로 중복 사용을 막는다. 인증·자격 원본 장애에는 신규 신청을 실패 폐쇄한다.
- 초대 발송 실패는 자격이나 참가 상태를 바꾸지 않으며 앱 내 상태를 유지하고 제한 재시도한다. 오사용 의심은 초대를 폐기하되 참가자 계정을 자동 제재하지 않고 검토·이의 경로를 둔다.
- 신청·초대·철회·자격 판정 결과를 해당 사용자와 매장에 알리고 정책 버전·최소 판정값·행위자·사유를 감사한다. 생년월일 원본, 신분증, 동행자 명단은 수집하지 않는다.

### 관측 가능한 인수 조건

- 미인증·자격 미충족·중복 신청은 자원을 점유하지 않고 초대만으로 참가 확정·우선권이 생기지 않는다.
- 초대 동시 사용과 재전송에도 하나의 소비 결과만 남고 자격 저장소 장애 시 실패 폐쇄한다.
- 기존 참가의 금전·취소 결과가 자격 변경만으로 결정되지 않고 불필요한 개인정보가 로그·알림에 없다.

## TASTE-006 알레르기·기피 재료 설문

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 식당 대표자는 준비에 실제 필요한 구조화된 알레르기·기피 항목과 선택적 자유 메모를 행사 버전으로 게시하고 목적·열람 주체를 안내한다. 사용자는 참가 신청 또는 허용된 변경 기간에 본인과 동행 인원의 조리 참고 정보를 선택 제출한다.
- 제출은 의료 진단이나 안전 보증이 아니며, 매장은 교차접촉 가능성과 대응 불가 여부를 명확히 답한다. 대응 불가가 확인돼도 시스템이 임의 취소·환불하지 않고 사용자 선택과 `TASTE-008`·`TASTE-012` 결과로 처리한다.
- 설문은 `미제출`, `제출`, `변경`, `확인 필요`, `매장 응답` 상태를 참가 버전과 연결한다. 참가 변경·취소 후 매장 운영 접근을 제거하고 정확한 보관·파기는 개인정보 TODO를 따른다.

### 동시성·실패·알림·감사·개인정보

- 참가 수정 번호와 설문 버전·멱등 키로 동시 편집을 제어하고 매장은 최신 제출본만 운영 화면에서 보되 과거 변경 감사는 덮어쓰지 않는다.
- 설문 저장이나 권한 확인 실패 시 제출 완료로 표시하지 않는다. 필수로 게시된 질문이 미완료면 신청 확정을 실패 폐쇄하고 원래 참가 상태를 유지한다.
- 새 제출·중요 변경·대응 불가 응답을 사용자와 해당 매장에 최소 내용으로 알린다. 접근·내보내기·변경을 감사하고 분석·검색·일반 이벤트에는 원문을 복제하지 않는다.

### 관측 가능한 인수 조건

- 권한 없는 사용자·타 매장·운영자가 설문 원문을 조회하지 못하고 취소 후 매장 운영 접근이 제거된다.
- 동시 수정·재전송에도 최신 한 버전이 적용되고 저장 실패가 제출 완료나 참가 확정을 만들지 않는다.
- 대응 불가 표시가 금전 책임을 독자 결정하지 않으며 설문이 안전 보증으로 표현되지 않는다.

## TASTE-007 옵션·페어링 선택

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 식당 대표자는 승인된 행사 버전 안에서 선택 가능한 옵션·페어링, 호환 조건, 가용 수량과 가격 표시를 게시한다. 주류 옵션은 `TASTE-005` 연령 제한을 적용하고 선택하지 않은 옵션을 기본 청구하지 않는다.
- 사용자는 신청·허용된 변경 단계에서 명시적으로 선택한다. 수량 영향 변경은 새 가용성 검증과 원자 교체를 거치며 성공 전까지 원래 선택을 유지한다. 금액 확정·추가 청구·환불은 `TASTE-008`과 PAY 정책 결과를 따른다.
- 옵션 버전·참가 수정 번호·원장·멱등 키로 경합을 제어한다. 부분 실패에는 원래 선택 또는 전부 해제로 수렴하고 중앙 원본 장애에는 신규 선택을 실패 폐쇄한다.
- 선택·변경·품절 영향을 당사자에게 알리고 옵션/가격/수량 버전과 원장 전이를 감사한다. 선택 사유나 새 취향·건강정보를 요구하지 않는다.

### 관측 가능한 인수 조건

- 호환되지 않거나 품절·연령 제한된 옵션은 선택·차감되지 않고 기본 선택이 자동 결제를 만들지 않는다.
- 동시 변경에도 재고 초과·이중 차감이 없고 실패 시 원래 선택이 유지된다.
- 금액 결과는 미정인 `TASTE-008`을 넘지 않으며 추가 개인정보 없이 선택 이력을 재현할 수 있다.

## TASTE-009 참가자 변경·양도

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 신청한 일반 사용자 계정을 참가 대표자로 유지하고 직접 계정 소유권 양도는 허용하지 않는다. 동행 인원·옵션 변경은 공개 정책과 자격·정원·설문을 전체 재검증하며 다른 사람의 연락처·신분증을 등록하게 하지 않는다.
- 변경 요청은 후보 자원 선점 후 한 번에 새 참가 버전으로 전환하며 성공 전까지 원래 참가와 자원을 유지한다. 대표자가 참석할 수 없으면 허용된 변경·취소 경로를 안내하고 식당·운영자가 임의 계정으로 이전하지 않는다.
- 참가 수정 번호·멱등 키·조건부 갱신으로 동시에 하나의 변경만 허용한다. 새 자원 확보 후 전환 실패는 원래 상태 유지 또는 보정 사건으로 복구하고 이중 참가를 만들지 않는다.
- 변경·거부·취소 영향을 당사자에게 알리고 이전/새 인원·옵션·정책 버전과 행위자를 감사한다. 금전 결과는 `TASTE-008`·`TASTE-012`를 따르고 동행자 신원은 감사하지 않는다.

### 관측 가능한 인수 조건

- 다른 계정·매장·운영자의 소유자 교체 요청이 거부되고 인원 변경이 소유권을 바꾸지 않는다.
- 동시 변경·취소 중 유효한 한 전이만 남으며 실패 시 원래 참가·자원이 유지된다.
- 양도 불가 규칙이 별도 취소 수수료·환불 책임을 만들지 않고 동행자 개인정보를 수집하지 않는다.

## TASTE-010 블라인드 평가와 공개 시점

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 식당 대표자는 행사 버전별 평가 문항, 익명 표시 방식, 제출·수정·공개 상태를 게시한다. 참가 이행이 확인된 사용자만 자신의 평가를 제출하며 공개 전 작성자와 다른 참가자의 답변을 볼 수 없다.
- 상태는 `비공개 수집`, `공개 대기`, `공개`로 단방향 전이한다. 공개 시점은 행사 정책에 기록된 조건으로 중앙 판정하고, 공개 뒤 원문을 덮어쓰지 않으며 정정·숨김은 별도 사건으로 남긴다.
- 행사-사용자 제출 유일성, 평가 버전·멱등 키와 조건부 전이로 중복·조기 공개를 막는다. 작업 지연에도 조회 시 중앙 공개 조건을 재검증하고 권한 원본 장애에는 비공개로 실패 폐쇄한다.
- 제출·공개·숨김을 당사자에게 필요한 범위로 알리고 접근·변경·공개 행위자를 감사한다. 블라인드 화면·이벤트에서 계정 식별자와 추론 가능한 불필요한 메타데이터를 제거한다.

### 관측 가능한 인수 조건

- 공개 조건 전에는 다른 참가자·매장이 비공개 답변을 조회할 수 없고 중복 작업이 조기 공개를 만들지 않는다.
- 재전송은 하나의 제출 버전을 반환하고 공개 후 정정도 원본 감사 이력을 보존한다.
- 블라인드 응답·로그·내보내기에 불필요한 작성자 식별정보가 없으며 장애에는 비공개가 유지된다.

## TASTE-011 출시 투표와 결과 이용

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 투표는 참가 이행이 확인된 계정당 행사·문항별 한 표로 집계하며, 결과는 참고 신호일 뿐 메뉴 출시·가격·재고나 보상을 자동 확정하지 않는다. 식당은 문항·선택지·이용 목적과 공개 범위를 행사 버전으로 사전 고지한다.
- 제출 기간과 공개 조건은 버전된 행사 설정에서 가져오며 종료 후 표를 소급 변경하지 않는다. 명백한 중복·무효 표는 원본을 삭제하지 않고 근거가 있는 제외 사건으로 처리하고 사용자 이의 경로를 둔다.
- 중앙 유일성 제약, 투표 버전·멱등 키와 재집계 가능한 추가 전용 사건으로 중복 표와 다중 인스턴스 오차를 막는다. 집계 장애는 원표를 유지한 채 재처리하며 부분 결과를 최종으로 표시하지 않는다.
- 제출 완료와 결과 공개만 알리고 개인별 선택을 다른 참가자에게 노출하지 않는다. 집계·제외·재집계와 조회 권한을 감사하고 결과 이용을 위해 새 개인 프로필을 만들지 않는다.

### 관측 가능한 인수 조건

- 같은 계정의 병렬·반복 제출에도 유효 표는 하나이고 비참가·미이행 계정 표는 집계되지 않는다.
- 재집계 결과가 원표 사건에서 재현되며 실패 중 부분 집계가 최종 공개되지 않는다.
- 투표 결과가 출시·혜택·참가 우선권을 자동 생성하지 않고 개인별 선택과 불필요한 개인정보가 공개되지 않는다.

## STORE-008·TASTE-001 연계 — 테이스팅 기능·행사 승인

> 행 단위 상태: 확정. 아래의 기존 승인 절차 하위 범위를 보존하고, 남은 일반 코스·테이스팅 경계는 위 세부 정책으로 확정했다.

### 확정 사항

- 가게는 먼저 가게별 테이스팅 기능 승인을 받아야 한다.
- 테이스팅 기능이 승인된 뒤에도 개별 행사를 생성할 때마다 별도의 관리자 승인을 받아야 한다.
- 행사 승인이 완료되기 전에는 행사를 공개하거나 참가자 모집·예약을 받을 수 없다.
- 테이스팅 기능 승인과 개별 행사 승인은 서로 대체하지 않으며 다른 가게나 다른 행사로 전파되지 않는다.
- 일정·장소, 가격·결제·환불 조건, 정원, 코스·메뉴 주요 구성, 주류 제공 여부와 참가 연령 조건을 바꾸면 새 행사 버전으로 재승인받는다. 오탈자·표현과 주요 내용을 바꾸지 않는 이미지 수정은 기존 승인을 유지한다.
- 수정본이 승인되기 전에는 기존 승인본을 계속 사용자에게 적용하고 미승인 변경을 공개·모집·예약에 사용하지 않는다.


## TASTE-005 주류 행사와 연령 확인

> 행 단위 상태: 확정. 아래 연령 차단·현장 책임·최소 수집 범위는 그대로 보존한다.

### 확정 사항

- 플랫폼의 연령 기준을 충족하지 못한 사용자는 주류 행사에 예약하거나 참여할 수 없다.
- 주류 행사에서는 예약자뿐 아니라 실제 참가자 전원의 나이와 본인 여부를 가게가 현장에서 확인한다.
- 현장 확인은 가게의 법적·운영 책임이며 MiriYum은 이를 직접 수행하거나 완료를 보증하지 않는다.
- MiriYum의 가입 시 연령 판정과 온라인 참여 차단은 가게의 현장 확인을 대체하지 않는다.
- MiriYum은 참가자별 현장 확인 완료 상태나 현장에서 제시된 신분증 자료를 수집·기록하지 않는다.

### 남은 범위 — 확정 세부

- 가게는 주류 제공 또는 주류 구역 입장 전에 실제 참가자 전원의 나이와 본인 여부를 확인한다. 확인 전에는 주류를 제공하지 않고, 확인 실패·거부·기준 미충족 참가자는 주류 이용과 주류 행사 체크인을 실패 폐쇄하되 비주류 동행자의 상태를 자동 취소하지 않는다.
- MiriYum은 온라인 연령 자격만 중앙에서 재검증하고 현장 신분증 번호·이미지·확인 결과를 저장하지 않는다. 체크인 거부 사유는 `현장 연령 확인 실패` 범주와 행위자·시각만 감사하며 상세 신분정보를 기록하지 않는다.
- 체크인·취소 경합은 참가 버전과 멱등 키로 하나의 상태만 확정한다. 현장 네트워크 장애에는 MiriYum이 확인 완료를 추정하지 않고 가게가 법적 책임 아래 현장 확인을 수행하며, 플랫폼 상태 복구 후 최소 거부 사건만 동기화한다.
- 확인 실패 사실과 가능한 문의·변경 경로를 참가 대표자와 매장에 알리되 환불·보상은 `TASTE-008`·`TASTE-012` 결과를 그대로 소비한다.

### 관측 가능한 인수 조건

- 연령 미충족·미확인 참가자에게 주류 이용 또는 주류 행사 체크인이 열리지 않고 비주류 동행자 상태는 자동 변경되지 않는다.
- 중복 체크인과 네트워크 복구 뒤에도 현장 신분증 자료·개별 확인 상태가 MiriYum 저장소·로그·알림에 남지 않는다.
- 확인 실패가 임의 환불·수수료·보상을 만들지 않고 `TASTE-008`·`TASTE-012`의 가능한 결과를 따른다.
