# 대규모 트래픽·분산 환경·장애 복구 정책

> 문서 상태: 세부 정책 확정(TODO 혼재)
> 정책 범위: `SCALE-001`~`SCALE-016`
> 최종 변경일: 2026-07-22

대규모 트래픽과 다중 서버·다중 인스턴스 대응을 필수 설계 기준으로 삼는다. 분산 상태 공유·메시지 멱등성·배치 중복 방지·부하 및 장애 주입 테스트는 기존 확정 내용을 유지하며, 사용자 직접 결정으로 정한 초기 용량·SLO·속도 제한·복구 목표를 모든 인스턴스에 일관되게 적용한다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 확정 | 필수 | `SCALE-001`~`SCALE-016` |

- `SCALE-001`~`SCALE-003`, `SCALE-005`, `SCALE-013`은 2026-07-22 사용자 직접 결정으로 확정했다. 목표와 인수 조건은 아래 각 정책에서 관리한다.
- 암호화 백업, 운영계와 분리된 접근 경계, 복원 전 무결성·삭제 원장 재적용, 정기 복원 검증과 실패 추적은 수치와 독립된 확정 안전 기본선이다.
- 특정 유료 관리형 상품이나 단일 인스턴스 전용 구조는 이 문서에서 새로 선택하지 않는다. 아래 수치는 제공자·상품·최소 인프라 용량의 선택이 아니라 서비스 목표다.

## 분산 운영 공통 불변식

- 세션·홀드·작업 소유권·속도 제한·배포 상태를 애플리케이션 로컬 메모리만으로 판정하지 않는다. 중앙 원본·버전·공통 시각을 사용하고 캐시는 권위 있는 상태를 대신하지 않는다.
- 모든 명령·작업·이벤트는 멱등 키와 허용 상태 전이를 가지며 DB 제약·조건부 갱신·원자 연산을 최종 방어선으로 사용한다. 중복·역순·지연과 인스턴스 중단을 정상 장애 모델로 다룬다.
- 서비스별 용량·경보·재시도·TTL은 버전된 중앙 설정으로 관리한다. 확정 목표를 변경하려면 새 사용자 결정과 정책 버전이 필요하며, 무제한 대기·재시도·락·보존은 허용하지 않는다.
- 상관관계 ID, 상태 버전, 작업 임대·체크포인트, 의존성 결과와 배포 버전을 구조화 관측에 남기되 개인정보·비밀정보·결제수단 원문은 로그·지표 라벨로 사용하지 않는다.

## SCALE-001 초기 용량 목표

> 정책 상태: 확정. 2026-07-22 사용자 직접 결정

### 확정 용량·다중 인스턴스 기준

- 초기 운영 목표는 월간 활성 사용자(MAU) 100,000명, 입점 매장 5,000개, 예약·웨이팅 등 핵심 거래 일 50,000건이다.
- 이 목표는 단일 인스턴스의 처리 약속이 아니다. 모든 API·worker 인스턴스는 무상태로 동작하고 세션, 속도 제한, 작업 소유권과 거래 상태를 중앙 원본으로 공유한다.
- 제공자·유료 상품·인스턴스 수·자동 확장 임계치는 이 정책으로 선택하지 않으며, 위 목표를 충족하는 실제 구성은 별도 비용·운영 승인으로 정한다.

### 관측 가능한 인수 조건

- 부하 시험에서 100,000 MAU·5,000개 매장·일 50,000건 핵심 거래를 대표하는 데이터와 요청 분포를 사용하고, 단일 인스턴스 중단 뒤에도 중앙 상태의 거래가 중복·유실되지 않는다.

## SCALE-002 정상·피크 트래픽 목표

> 정책 상태: 확정. 2026-07-22 사용자 직접 결정

### 확정 트래픽 기준

- 전체 API의 정상 처리 목표는 200 RPS이며, 피크 시 1,000 RPS를 최소 10분간 처리한다.
- 모든 인스턴스는 같은 중앙 정책 버전과 공통 시각을 사용한다. 일부 인스턴스의 로컬 카운터·고정 세션·개별 우회가 피크 판단이나 허용량을 바꾸지 않는다.

### 관측 가능한 인수 조건

- 다중 인스턴스 부하 시험에서 정상 200 RPS와 1,000 RPS 10분 피크를 각각 재현하고, 인스턴스 추가·중단 중에도 요청 합계와 오류·지연 지표가 중앙 집계로 연속 관측된다.

## SCALE-003 가용성·응답시간 SLO

> 정책 상태: 확정. 2026-07-22 사용자 직접 결정

### 확정 SLO

- 월간 가용성 목표는 99.9%다.
- 조회 API p95는 500ms, 상태 변경 API p95는 1초, 예약·선점 등 핵심 API p99는 2초 이내를 목표로 한다.
- SLO 측정은 모든 API 인스턴스의 공통 시각·배포 버전·오류 분류를 합산하고, 개인정보·결제수단 원문을 지표 라벨에 넣지 않는다.

### 관측 가능한 인수 조건

- 정상·피크 부하와 인스턴스 장애 주입에서 조회 p95 500ms, 상태 변경 p95 1초, 핵심 API p99 2초 및 월간 99.9% 가용성 산정에 필요한 원시 지표가 관측된다.

## SCALE-005 요청 속도 제한

> 정책 상태: 확정. 2026-07-22 사용자 직접 결정

### 확정 제한 등급·응답

- 공개 조회는 IP당 분당 60회, 로그인된 조회는 계정당 분당 300회, 상태 변경은 계정당 분당 60회로 제한한다.
- 예약·웨이팅 생성은 계정당 분당 20회, 로그인·인증번호 요청은 10분당 5회로 제한한다.
- 속도 제한 상태·카운터·정책 버전은 모든 인스턴스가 공유하는 중앙 상태에서 원자적으로 갱신한다. 제한을 넘으면 `429`와 다음 허용까지의 재시도 시각(`Retry-After`)을 반환하며, 개별 인스턴스가 로컬 우회하거나 서로 다른 재시도 시간을 주지 않는다.

### 관측 가능한 인수 조건

- 각 등급에서 한도 직전 요청은 허용되고 바로 다음 요청은 `429`와 재시도 시각을 받는다. 여러 인스턴스에 분산한 동시 요청 합계도 동일 한도를 넘지 않는다.
- 중앙 속도 제한 상태의 장애에는 인증·상태 변경·예약·웨이팅 생성 요청을 실패 폐쇄하고, 공개 조회는 정책상 허용된 안전한 제한 모드 외에 우회하지 않는다. 복구 뒤 중앙 카운터·정책 버전에서 재개한다.

## SCALE-013 백업·복구 목표

> 정책 상태: 확정. 2026-07-22 사용자 직접 결정

### 확정 RPO·RTO·검증

- 거래 데이터의 RPO는 5분, RTO는 1시간이다.
- 설정·문서 데이터의 RPO는 24시간, RTO는 4시간이다.
- 분석 데이터의 RPO와 RTO는 각각 24시간이다.
- 암호화 백업과 운영계 분리 접근 경계를 유지하고, 복원 전 무결성 확인·삭제 원장 재적용·정기 복원 검증을 수행한다. 복원 검증 실패 또는 목표를 넘긴 복구는 중앙 장애 사건에 실패 원인·영향·다음 조치와 함께 보고하며 성공으로 종료하지 않는다.

### 관측 가능한 인수 조건

- 거래 복구 훈련은 최대 5분의 데이터 손실과 1시간 이내 복구, 설정·문서는 최대 24시간 손실과 4시간 이내 복구, 분석은 최대 24시간 손실·복구를 각각 증명한다.
- 복구 작업자·인스턴스가 중단되거나 동시에 재개되어도 중앙 체크포인트와 멱등 키로 한 번만 전환하며, 무결성·삭제 원장·대사 검증이 실패하면 실패 보고와 미해결 상태가 남는다.

## SCALE-004 가상 대기열 적용 조건

> 정책 상태: 확정

### 확정 적용·입장·상태

- 가상 대기열은 예약·한정 자원 쓰기 경로의 현재 처리 여력, 오류·지연, DB/캐시 의존성 상태와 `SCALE-001`~`SCALE-003`의 확정 목표를 입력으로 하는 중앙 정책으로 켜고 끈다. 페이지 조회 수 하나나 개별 인스턴스 판단만으로 전환하지 않는다.
- 대기열은 원 도메인의 자격·우선순위·자원 배분 결정을 바꾸지 않고 시스템 입장 속도만 제어한다. 입장 뒤에도 인증, 정책 자격과 최종 가용성을 다시 검증하며 대기 번호가 예약·구매 성공을 보장하지 않는다.
- 대기 항목은 서버 발급 불투명 식별자, 대상 오픈·계정 또는 제한 주체, 발급 순서·상태와 만료 정책 버전만 가진다. 정밀 위치·기기 지문·추가 연락처를 수집하지 않는다.
- 입장 토큰은 대상·주체·정책 버전에 결속하고 서명·만료·한 번 사용을 검증한다. 양도·변조·재사용·다른 오픈 사용을 거부하고 새로고침은 같은 유효 대기 항목으로 멱등 연결한다.

### 다중 인스턴스·장애·복구·감사

- 대기 순서, 상태와 입장 허용량은 중앙 저장소의 원자 연산·공통 시각으로 관리한다. 모든 게이트 인스턴스가 같은 토큰·정책 버전을 검사하고 로컬 대기열이나 고정 세션 라우팅에 의존하지 않는다.
- 중앙 대기열 상태를 확인할 수 없으면 새 쓰기 입장을 실패 폐쇄하고 이미 확정된 거래 조회·공개 읽기는 가능한 범위에서 유지한다. 복구 뒤 중앙 체크포인트와 토큰 상태에서 재개하며 대기 항목을 임의 재정렬하지 않는다.
- 전환 사유, 정책 버전, 입장·거부·만료·재사용 탐지와 대기 지연을 감사·관측한다. 경보와 적용 임계치는 `SCALE-001`~`SCALE-003`의 확정 목표와 별도 버전 설정으로 관리한다.

### 관측 가능한 인수 조건

- 여러 게이트 인스턴스에서 같은 사용자가 동시에 새로고침해도 활성 대기 항목과 유효 입장 토큰은 하나다.
- 만료·변조·다른 오픈 토큰이 최종 쓰기 API에서 거부되고 입장 토큰만으로 예약 가용성 검증을 우회할 수 없다.
- 중앙 대기열 장애 시 새 입장이 중단되고 복구 뒤 기존 순서·상태에서 재개되며 원 예약·결제 상태는 변경되지 않는다.

## SCALE-007 예약 선점의 원자성과 TTL

> 정책 상태: 확정

### 확정 선점·만료·권한

- 선점은 자원 ID·수량, 소유 계정, 원 요청 멱등 키, 생성·만료 시각, 정책 버전과 `활성`, `확정 중`, `확정`, `해제`, `만료` 상태를 가진다. 유효한 인증·자격과 중앙 가용성 검증 뒤에만 생성한다.
- 동일 자원의 수량 확인과 감소, 선점 레코드 생성은 짧은 원자 연산 또는 DB 조건부 갱신으로 수행한다. 좌석·메뉴·회차를 묶을 때 일부만 성공하면 전체 그룹을 확정하지 않고 이미 잡힌 부분을 멱등 해제한다.
- TTL 수치는 `RES-006`·`HOLD-007` 등 해당 도메인의 확정/TODO 정책 버전에서 공급받고 이 행이 임의 숫자를 정하지 않는다. 모든 인스턴스는 서버·DB 공통 시각으로 만료를 판정하며 클라이언트 시각을 신뢰하지 않는다.
- 확정 명령은 소유권·상태·만료·원 거래 버전을 다시 확인하고 조건부로 한 번만 전환한다. 만료·해제된 선점을 되살리거나 다른 계정에 이전하지 않는다.

### 다중 인스턴스·실패·복구·감사

- 생성·확정·해제와 만료 작업은 같은 선점 ID와 멱등 키를 사용한다. 만료 스케줄러가 늦어도 모든 쓰기 명령이 만료를 재검증하며 여러 만료 작업자가 수량을 중복 복구하지 않는다.
- 캐시 원자 선점을 사용할 때도 내구성 원장·최종 DB 제약과 대사 경로를 둔다. 캐시 장애나 결과 불명에는 추가 선점을 낙관 승인하지 않고 `SCALE-011` 복구 뒤 원장과 자원을 재조정한다.
- 선점 상태 전이, 소유권 충돌, 만료 지연, 보상 실패와 초과 판매 탐지를 감사·관측한다. 로그에는 사용자 연락처나 결제정보를 남기지 않는다.

### 관측 가능한 인수 조건

- 마지막 한 자원에 여러 인스턴스가 동시에 요청해도 활성·확정 선점 합계가 가용량을 넘지 않는다.
- 확정과 만료를 경합시키면 유효한 최종 상태 하나만 남고 자원 수량이 이중 복구되지 않는다.
- 스케줄러를 중단해 TTL을 넘긴 뒤 확정 요청을 보내도 거부되며 복구 작업은 같은 결과로 멱등 수렴한다.

## SCALE-008 DB 최종 제약과 락 전략

> 정책 상태: 확정

### 확정 제약·트랜잭션·락

- 데이터 모델은 계정·연락처·사업자번호·예약 자원·활성 사건·멱등 키의 유일성, 수량 비음수와 허용 상태 관계를 DB 제약으로 보호한다. 애플리케이션 사전 조회만을 최종 방어선으로 사용하지 않는다.
- 경합 쓰기는 현재 버전·상태·수량을 조건으로 한 원자 갱신을 우선하고, 여러 행을 잠글 때는 도메인별 고정 순서를 사용한다. 비관적 락은 실제 충돌이 큰 짧은 구간, 낙관적 버전은 충돌을 사용자/작업자가 안전하게 재시도할 수 있는 구간에 한정한다.
- 트랜잭션 안에서 외부 PG·알림·파일·긴 계산을 기다리지 않는다. 상태와 Outbox를 함께 확정하고 외부 처리는 트랜잭션 밖에서 멱등하게 수행한다.
- 분산 락은 DB 제약을 대체하지 않으며 소유자·임대·fencing token과 장애 시나리오가 명확한 최소 구간에서만 보조로 사용한다.

### 교착·실패·배포·감사

- 교착·버전 충돌은 명령의 멱등성과 상태 재조회가 보장된 경우에만 제한 재시도한다. 결과 불명이나 결제 명령은 새 쓰기를 추측 실행하지 않고 원 상태를 대사한다.
- 제약 추가는 기존 데이터 검증과 정리, expand 단계, 애플리케이션 호환 배포, 제약 활성 순으로 수행한다. 혼합 버전 중 어느 버전도 새 제약을 우회하는 쓰기를 하지 못하게 한다.
- 제약 위반, 락 대기, 교착, 재시도와 충돌 자원을 구조화 관측하고 운영자 수동 수정은 전용 권한·사유·감사 절차를 거친다.

### 관측 가능한 인수 조건

- 애플리케이션 검사를 우회한 동시 삽입에도 DB 유일·수량 제약이 중복 계정·초과 판매를 거부한다.
- 락 교착을 주입해도 제한된 재시도 또는 명시적 충돌로 종료하며 외부 호출을 트랜잭션 락 동안 반복하지 않는다.
- 이전·신규 앱 버전을 동시에 실행해도 새 제약 위반 쓰기가 없고 롤백 가능한 expand 단계가 검증된다.

## SCALE-011 캐시 장애와 데이터 재조정

> 정책 상태: 확정

### 확정 원본·장애 모드·복구

- MySQL의 확정 거래·권한·Outbox가 내구성 원본이며 캐시는 버전·만료를 가진 파생 조회 또는 제한된 임시 상태다. 캐시 적중·부재만으로 예약 확정, 결제 성공, 권한 부여나 제재 해제를 판정하지 않는다.
- 캐시 연결 실패·타임아웃·데이터 형식 오류에는 쓰기 경로별 안전 모드로 전환한다. 권한·세션·선점처럼 안전하게 검증할 수 없는 요청은 실패 폐쇄하고, 공개 읽기는 원본 보호 범위에서 제한적으로 우회할 수 있다.
- 복구는 원본 체크포인트와 버전으로 새 네임스페이스에 재구축하고 표본·합계·무결성을 검증한 뒤 원자 전환한다. 장애 중의 변경 사건과 삭제 원장을 재적용하며 오래된 캐시를 정상으로 표시하지 않는다.
- 재구축은 키별 임대·요청 병합·지연 갱신으로 스탬피드와 핫키를 제한하고 원 DB가 과부하되면 속도를 낮춘다. 단일 대량 삭제·전체 동시 재적재에 의존하지 않는다.

### 다중 인스턴스·실패·감사·개인정보

- 모든 인스턴스가 중앙 캐시 상태·네임스페이스 버전과 회로 상태를 공유한다. 로컬 캐시가 있다면 짧은 파생 읽기만 담고 권한 회수·삭제·상태 변경 사건으로 무효화한다.
- 재조정 작업은 파티션 체크포인트와 멱등 키로 중단 뒤 재개하고 일부 실패를 관측·재처리한다. 원본과 캐시 불일치, 우회율, 재구축 진척, 스탬피드와 DB 부하를 경보한다.
- 캐시에 민감 원문·비밀·결제수단을 복제하지 않고 필요한 식별자는 내부 ID와 최소 필드로 제한한다. 조회·재구축 로그도 PII를 마스킹한다.

### 관측 가능한 인수 조건

- 캐시 장애·오염을 주입하면 세션·권한·선점 쓰기가 오래된 값으로 승인되지 않고 공개 읽기 우회가 원본 용량을 보호한다.
- 재구축 중 작업자 중단 뒤 체크포인트에서 재개해 원본 버전·합계와 일치한 뒤에만 새 캐시가 활성화된다.
- 삭제·권한 회수 사건을 장애 중 발생시켜도 복구 캐시에 삭제된 PII나 회수 전 권한이 부활하지 않는다.

## SCALE-012 외부 PG·알림 장애 격리

> 정책 상태: 확정

### 확정 경계·격리·상태

- PG와 알림은 제공자 중립 포트·어댑터 뒤에서 별도 연결 제한 시간, 회로 차단, 동시 실행 격리와 내구성 작업 큐를 사용한다. 어느 외부 제공자 장애도 API 공용 작업 풀·DB 연결 풀·다른 외부 연동을 고갈시키지 않게 한다.
- 결제 승인·취소는 결과 불명 상태를 명시하고 새 거래로 맹목 재시도하지 않는다. 같은 멱등 키의 제공자 조회·웹훅 검증·정기 대사로 확정하며 불가능하면 권한 있는 운영자 복구 큐로 보낸다.
- 알림 실패는 `NOTI-007`에 따라 제한 재시도·대체 채널·최종 실패로 처리하고 원 예약·결제 상태를 되돌리지 않는다. 브로커 장애에도 원 거래와 Outbox를 함께 확정하여 복구 뒤 후속 작업을 재개한다.
- 제공자·채널과 재시도 숫자는 계약·실측 설정으로 교체 가능하게 두고 이 행에서 유료 서비스나 무제한 자동 전환을 선택하지 않는다.

### 다중 인스턴스·복구·감사·개인정보

- 회로·속도 정책, 작업 상태와 시도 횟수는 중앙에서 공유하고 작업 임대로 중복 외부 호출을 막는다. 인스턴스 중단 뒤 같은 멱등 키·상태에서 이어받는다.
- 의존성별 지연, 오류 분류, 회로 상태, 대기열 적체, 결과 불명과 대사 미해결 건을 관측한다. 장애 격리·수동 재개·제공자 응답 보정은 권한·사유와 함께 감사한다.
- 외부 요청에는 계약된 목적의 최소 데이터만 보내고 응답·로그에서 결제수단 원문, 연락처와 메시지 본문을 마스킹한다. 테스트 환경 자격증명과 데이터를 운영과 분리한다.

### 관측 가능한 인수 조건

- PG 지연·오류를 주입해도 알림·예약 조회와 다른 외부 연동의 실행 자원이 고갈되지 않는다.
- 결제 응답 유실과 중복 웹훅 뒤 같은 멱등 키 대사로 하나의 최종 결제 상태에 수렴하고 이중 승인·취소가 없다.
- 알림 제공자 장애 중 원 거래가 확정 상태를 유지하며 복구 뒤 Outbox 작업만 재개된다.

## SCALE-015 관측 지표·경보·운영 런북

> 정책 상태: 확정

### 확정 관측·경보·권한

- API·도메인 명령·DB·캐시·메시지·스케줄러·외부 연동은 요청/사건 상관관계 ID, 서비스·환경·배포 버전, 상태·오류 분류와 지연을 구조화 로그·지표·추적에 일관되게 기록한다.
- 핵심 지표는 가용성, 오류·지연, 경합·제약 위반, 초과 판매, 결제 불일치, 알림 실패, 큐 적체·오래된 작업, 캐시 불일치, 배포 상태와 백업 복원 검증을 포함한다. 사용자 ID·연락처·매장 자유 텍스트를 고카디널리티 라벨로 넣지 않는다.
- 경보는 사용자가 겪는 증상과 복구 가능한 원인에 연결하고 중복 사건을 하나로 묶는다. SLO·속도 제한 숫자는 `SCALE-001`~`SCALE-003`·`SCALE-005`의 확정 목표를 따르고, 경보 발화 세부값은 별도 버전 설정으로 관리한다.
- 운영 런북은 탐지 조건, 영향·데이터 위험 확인, 권한 있는 중지·격리, 복구·대사, 사용자/내부 알림, 성공 검증, 에스컬레이션과 사후 조치를 버전 관리한다. 중요한 실행과 break-glass 사용은 관리자 정책에 따라 감사한다.

### 다중 인스턴스·장애·개인정보

- 모든 인스턴스가 공통 필드·시각·배포 버전을 사용하고 중앙 집계 장애가 서비스 쓰기 성공을 좌우하지 않게 비동기·제한 버퍼를 사용한다. 관측 장애 자체를 별도 상태로 경보하되 애플리케이션 메모리를 무제한 점유하지 않는다.
- 경보 전송 실패는 원 장애를 해제하지 않고 중앙 사건에 남겨 재시도·수동 확인한다. 런북 실행자는 단계별 결과와 우회 조치를 기록하며 자동 복구도 같은 사건 ID에 연결한다.
- 로그·트레이스·덤프는 허용 필드 기반 마스킹, 최소 접근과 환경별 보존 정책을 적용한다. 비밀·세션 ID·본문·정밀 위치·결제수단 원문은 수집하지 않는다.

### 관측 가능한 인수 조건

- 한 사용자 흐름을 API, DB/캐시, Outbox·작업과 외부 연동까지 PII 없이 같은 상관관계 ID로 추적할 수 있다.
- 같은 장애가 여러 인스턴스에서 발생해도 하나의 운영 사건으로 묶이고 담당 런북·복구 검증·감사 이력이 연결된다.
- 관측 백엔드 장애를 주입해도 서비스 메모리·스레드가 고갈되지 않고 관측 공백 자체가 표시된다.

## SCALE-016 무중단 배포와 스키마 호환

> 정책 상태: 확정

### 확정 배포·호환·전환

- 배포 산출물은 변경 불가능한 버전으로 식별하고 테스트·스키마 검증·이미지 무결성 검사를 통과해야 한다. 준비 상태 검사를 통과한 새 인스턴스만 트래픽을 받고 종료 인스턴스는 새 작업 획득을 멈춘 뒤 진행 중 요청·임대를 안전하게 넘긴다.
- DB 변경은 expand, migrate, contract 순서로 수행한다. 먼저 이전·신규 버전이 함께 읽고 쓸 수 있는 추가 변경을 배포하고 데이터 이관·검증 뒤 모든 이전 버전이 제거된 경우에만 구 필드·제약을 삭제한다.
- API·이벤트·작업 payload는 호환 가능한 스키마 버전과 기본값·알 수 없는 필드 처리 규칙을 가진다. 소비자가 배포되기 전에 생산자가 필수 필드를 제거하거나 의미를 바꾸지 않는다.
- 중앙 기능 정책·플래그는 환경·버전·권한과 결속하고 기본값은 안전하게 꺼진 상태다. 롤백해도 새 데이터가 이전 코드에 손상되지 않는지 검증하며 되돌릴 수 없는 이관은 별도 승인·복구 계획 없이는 실행하지 않는다.

### 다중 인스턴스·실패·감사·개인정보

- 롤링 중 이전·신규 인스턴스와 API·worker 조합을 동시에 검증한다. 스케줄러·작업 임대와 메시지 멱등성은 재시작·축소에도 작업 유실·중복 결과를 만들지 않는다.
- 마이그레이션·준비 상태·핵심 지표가 실패하면 새 트래픽 확대를 중단하고 호환 가능한 직전 버전으로 롤백한다. 부분 배포 상태와 수동 승인·중단·롤백을 중앙 배포 사건에 감사한다.
- 마이그레이션·배포 로그에 실제 PII나 비밀을 넣지 않고, 삭제·암호화·접근 통제 스키마 변경은 개인정보 검증과 복원 시험을 포함한다.

### 관측 가능한 인수 조건

- 이전·신규 API와 worker를 함께 실행한 계약·통합 테스트에서 같은 사건을 처리해도 스키마 오류·중복 결과가 없다.
- expand 단계 롤백, 작업자 강제 종료와 재기동 뒤 요청·Outbox·예약 작업이 중앙 상태에서 멱등 수렴한다.
- contract 변경은 이전 버전 부재와 데이터 이관·삭제·복원 검증 증거 없이는 실행되지 않고 실패 배포는 준비 상태에서 차단된다.
