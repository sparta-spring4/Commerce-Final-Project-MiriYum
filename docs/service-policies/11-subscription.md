## 구독 공통 불변식

- 혜택 종류·대상·무료/유료 경계·가격·갱신·중도 해지·환불 책임은 각각 `SUB-001`~`SUB-004`, `SUB-006`, `SUB-007`의 결과가 소유한다. 아래 정책은 그 결과를 생성하거나 확대하지 않는다.
- 모든 인스턴스는 중앙 구독 상태, 혜택 원장과 버전된 정책을 사용한다. 로컬 캐시·결제 화면·알림 성공만으로 혜택을 부여하거나 회수하지 않는다.
- 구독 명령, 결제 결과 소비, 혜택 발급·사용·회수와 스냅샷에는 서로 다른 멱등 키와 조건부 상태 전이를 적용하고, 상태 변경과 Outbox 사건을 유실 없는 처리 경계에 기록한다.

### 사용자 구독 후보

- 단일 사용자 구독 후보의 표시 가격은 부가가치세 포함 월 3,900원 또는 연 39,000원이다. 무료 사용자는 매장·메뉴·행사·취소 자리 알림을 최대 5개까지 활성화하고, 구독자는 앱·웹 알림을 개수 제한 없이 활성화하는 후보이다. 생성 속도 제한과 중복 알림 방지 규칙은 동일하게 적용한다.
- 후보 혜택은 사용자가 명시한 취향과 MiriYum 이용 이력에 기반한 강화 추천, MiriYum 내부 통합 캘린더의 일정 충돌 경고·맞춤 리마인더·방문 기록 정리, 구독자 전용 행사 요청이다. 외부 캘린더 연동이나 새로운 개인정보 수집은 이 후보에 포함하지 않는다.
- 참여 매장이 선택한 공개 행사에 한해 전체 정원의 최대 40%를 구독자에게 최대 24시간 먼저 공개할 수 있고, 나머지 최소 60%는 처음부터 일반 사용자에게 공개하는 후보이다. 일반 예약, 일반 웨이팅, 취소 자리 배정에는 구독 우선순위·선공개·별도 재고를 부여하지 않는다.
- 광고 제거는 광고 상품과 사용자 구독 플랜이 모두 실제 출시되고 해당 구독 권한이 유효할 때만 가능한 후보 혜택이다. 현재 광고 상품이 없거나 사용자 구독이 미출시인 동안에는 제거할 광고도, 활성 광고 제거 권한도 없다.
- 결제 실패 후보 유예 기간은 3일이며, 유예 중 기존 거래의 효력은 유지하되 구독자 전용 행사 요청과 선공개 혜택은 제한한다. 복구되면 즉시 정상화하고, 유예 종료 시 자동 해지한다. 이는 `SUB-006`의 운영·계약 `TODO`가 해소되기 전에는 실행하지 않는다.
- 해지는 다음 자동결제를 중단하고 현재 이용 기간 종료일까지 효력을 유지하는 후보이다. 단순 변심에 따른 일할 환불을 제공하지 않는다는 안내 후보는 법정 청약철회, 중복·오결제, 서비스 미제공, MiriYum 귀책의 강행 예외를 제한하지 않는다.

### 매장 Free·Pro 후보

- 매장 Free 후보는 예약·웨이팅의 등록·호출·관리 기능과 실제 발생한 예약·웨이팅·행사 운영의 기본 집계를 포함한다. 이 범위는 매장에 현재 Pro 권한이나 리포트 조회 권한을 부여하지 않는다.

- 매장 Pro 출시 시점과 운영 반복결제 계약은 아직 정해지지 않았다. 따라서 이 후보는 현재 리포트·자동화·추천·유료 청구·권한을 활성화하지 않는다.

## SUB-005 무료 체험·쿠폰

> 정책 상태: 확정

### 확정 규칙·권한·상태·흐름

- 무료 체험이나 쿠폰은 `SUB-001`~`SUB-004`가 허용하고 권한 있는 플랫폼 운영자가 활성화한 버전된 캠페인이 있을 때만 발급한다. 이 정책은 캠페인 도입, 혜택·할인액·대상·기간·중복 가능 여부를 정하지 않으며 안전한 기본값은 미발급이다.
- 발급 전 서버가 계정 유형·활성 상태, 캠페인 자격, 동일 캠페인 기존 발급·사용·취소 이력을 확인한다. 사용자는 자신의 발급분만 조회·사용하고, 식당 대표자는 자기 매장 부담 여부가 명시된 캠페인 결과만 보며 임의 발급하지 못한다.
- 발급 상태는 `발급`, `사용 가능`, `사용 예약`, `사용 완료`, `만료`, `취소`로 관리한다. 거래 확정 때만 `사용 예약`에서 `사용 완료`로 전이하고 실패·취소 때는 캠페인 규칙이 허용하는 범위에서 원자적으로 복구한다.

### 동시성·실패·알림·감사·개인정보

- 계정-캠페인 자격 키, 발급 원장 유일성, 거래별 사용 멱등 키와 원장 버전으로 병렬 발급·중복 사용을 막는다. 캠페인 정책 누락·버전 불일치·중앙 상태 장애에는 신규 발급과 사용을 실패 폐쇄한다.
- 부분 실패는 금액이나 유료 혜택을 추정하지 않고 `복구 필요`로 격리해 거래·혜택 원장을 대사한다. 재처리는 완료 상태를 되돌리지 않고 누락된 발급·복구 사건만 보완한다.
- 사용자에게 발급·사용·만료·취소와 적용 정책 버전을 알리고, 캠페인·행위자·자격 판정·거래·상태 전후를 감사한다. 판정에는 기존 계정·구독·거래 상태만 사용하며 연락처, 기기 지문이나 외부 프로필을 추가 수집하지 않는다.

### 관측 가능한 인수 조건

- 활성 캠페인이 없거나 팀 결정 입력이 불완전하면 무료 체험·쿠폰이 발급되지 않는다.
- 병렬 발급·사용·취소와 작업 재시작에도 같은 자격에 허용 수량을 넘는 발급이나 한 쿠폰의 중복 사용이 없다.
- 거래 실패 후 원장이 대사되고, 어떤 결과도 가격·혜택·중복 규칙을 이 정책이 독자 결정한 것으로 표시하지 않는다.

## SUB-008 혜택 스냅샷과 기존 예약

> 정책 상태: 확정

### 확정 규칙·데이터·흐름

- 예약·홀드·행사·리포트 접근 등 구독 혜택을 소비하는 명령은 최종 확정 직전에 중앙 구독 상태와 `SUB-001`~`SUB-003`의 혜택 정책 버전을 검증한다. 적용 결과에는 구독 ID, 혜택 ID·버전, 판정 시각, 입력 거래 버전과 적용 결과만 불변 스냅샷으로 남긴다.
- 스냅샷은 당시의 서버 판정을 재현하기 위한 근거이며 새 혜택이나 환불 권리를 만들지 않는다. 이미 확정된 거래는 이후 플랜 변경·만료만으로 소급 재계산하지 않고, 변경·취소·환불은 해당 거래 정책과 `SUB-007`의 향후 결정에 따른다.
- 구독 자격이 필요한 새 명령에는 항상 현재 상태를 다시 검사한다. 기존 거래를 변경해 새로운 자원·기간·수량을 얻는 부분은 새 판정을 받고, 변경되지 않은 확정 부분의 원 스냅샷은 보존한다.

### 동시성·실패·복구·감사·개인정보

- 거래 버전과 혜택 소비 원장에 조건부 유일성 제약을 적용해 같은 혜택을 두 번 소비하거나 서로 다른 스냅샷으로 확정하지 않는다. 구독 변경과 거래 확정이 경합하면 중앙에서 먼저 확정된 버전 하나만 사용하고 패자는 최신 상태로 재검증한다.
- 정책·구독 원본을 확인할 수 없으면 혜택을 성공으로 추정하지 않고 일반 접근이 가능한 범위만 제공하거나 명령을 보류한다. 스냅샷 누락·불일치는 거래를 소급 변경하지 않고 복구 큐에서 원장·정책 버전으로 대사하며 정정은 별도 보정 기록으로 남긴다.
- 확정·변경 시 사용자와 해당 매장에 적용된 혜택의 명칭·효력만 알리고 다른 플랜 내부 규칙은 노출하지 않는다. 스냅샷 생성·재검증·충돌·보정을 감사하되 결제수단 원문이나 새로운 개인정보를 저장하지 않는다.

### 관측 가능한 인수 조건

- 동일 거래 재전송과 다중 인스턴스 경합에도 거래당 적용 스냅샷은 하나이고 해당 정책 버전으로 결과를 재현할 수 있다.
- 플랜 만료·변경이 기존 확정 거래를 자동 취소·재가격화하지 않고, 새 혜택 사용은 현재 자격 없이는 승인되지 않는다.
- 구독 원본 장애·스냅샷 누락 뒤에도 추정 혜택이 부여되지 않고 감사 가능한 대사 결과로 수렴한다.

## SUB-009 선공개 물량과 공정성

> 정책 상태: 확정

### 확정 기본값·정책 소비 경계

- `SUB-001`~`SUB-003`이 선공개 또는 구독자 전용 물량을 명시적으로 허용하기 전에는 모든 좌석·수량·회차를 일반 공개와 같은 시각·가용 원장·획득 규칙으로 제공한다. 구독 상태를 정렬, 대기열 입장, 재고 분할, 오픈 시각이나 승계 우선순위에 사용하지 않는다.
- 팀 결정이 허용한 뒤에도 이 정책은 버전된 결정의 대상 플랜·자원·수량 구분·공개 구간만 기계적으로 소비한다. 입력에 없는 우대, 추가 가중치, 암묵적 얼리 액세스나 영구 우선권을 만들지 않는다.
- 각 자원 버전은 `일반`, 또는 팀 결정이 명시한 경우에만 `제한 구간`과 이후 `일반 전환` 상태를 가진다. 제한 구간 종료 뒤 남은 자원은 같은 원장에서 일반 가용으로 원자 전환하며 이 정책이 수량이나 시간을 정하지 않는다.

### 동시성·실패·복구·투명성·개인정보

- 정책 버전 누락·불일치·구독 판정 장애에는 제한 구간을 만들거나 우선 배분하지 않고 안전 기본값인 일반 공개만 사용한다. 다만 이미 유효하게 시작된 제한 구간의 자원을 중복 공개하지 않고 중앙 원장 대사 후 전환한다.
- 자원-정책 버전 유일성, 중앙 기준 시각과 조건부 갱신으로 여러 인스턴스가 같은 공개 상태를 적용한다. 재처리는 이미 확정된 획득을 소급 회수하지 않고 누락된 일반 전환만 멱등하게 보완한다.
- 화면에는 현재 일반 공개인지 명시적으로 승인된 제한 구간인지와 적용 정책 버전을 설명한다. 공개 전환·자격 판정·경합·복구를 감사하며 다른 사용자의 구독 여부·순서나 추가 행동 데이터를 노출·수집하지 않는다.

### 관측 가능한 인수 조건

- `SUB-001`~`SUB-003` 미확정 또는 비허용 버전에서는 구독자·비구독자의 공개 시각, 재고 풀과 획득 순서가 동일하다.
- 허용 버전을 주입했을 때만 정확히 그 버전의 구간·대상에 제한이 적용되고 입력에 없는 우대가 생기지 않는다.
- 정책 배포 중 혼합 버전·동시 오픈·장애 복구에도 한 자원이 제한·일반 양쪽에 중복 노출되지 않고 최종 원장 상태가 일치한다.

## SUB-010 환불·차지백 후 혜택 회수

> 정책 상태: 확정

### 확정 규칙·상태·흐름

- 회수는 PAY 도메인이 서명·대사로 확정한 환불 또는 차지백 상태와 그 원인 구독 거래 ID만 입력으로 받는다. 이 정책은 환불 가능성·귀책·수수료·손실 부담을 정하지 않고, 결과 불명확·분쟁 중 상태에서는 신규 혜택 사용만 보류하며 회수를 확정하지 않는다.
- 혜택 원장은 `사용 가능`, `사용 예약`, `사용 완료`, `회수 대기`, `회수 완료`, `복구`를 구분한다. 확정된 환불·차지백에는 아직 다른 거래에 확정 사용되지 않은 미래 혜택만 `회수 완료`로 전이하고, `SUB-008` 스냅샷이 있는 완료·기존 확정 거래를 소급 취소하거나 현금·채권을 만들지 않는다.
- PAY 정정으로 환불·차지백이 취소되면 동일 원인 사건을 참조하는 보정 원장으로 회수 가능한 혜택을 복구한다. 이미 별도 거래에서 정당하게 확정된 결과를 자동 취소하지 않고 충돌은 운영자 검토 큐에 격리한다.

### 동시성·실패·오남용·알림·감사

- 결제 사건 ID-혜택 원장 유일성, 상태 버전과 멱등 키로 중복 웹훅·순서 역전·병렬 회수/사용 중 하나만 확정한다. 부분 실패는 PAY·구독·혜택 원장을 대사하고 단일 인스턴스 작업 소유권에 의존하지 않는다.
- 반복 차지백 등 위험 신호는 별도 신뢰 사건으로 기록할 수 있지만 자동 영구 제재나 추가 금전 책임을 만들지 않으며, 오탐 정정·이의 제기와 감사 가능한 수동 복구를 제공한다.
- 사용자에게 회수·보류·복구의 대상 혜택과 사유 요약, 이의 경로를 알린다. PAY 사건 참조, 상태 전후, 자동·수동 행위자와 보정 관계를 감사하되 결제수단 원문·외부 분쟁 자료나 추가 개인정보를 복제하지 않는다.

### 관측 가능한 인수 조건

- 중복·역순 환불/차지백/정정 사건에도 혜택 원장이 한 번만 회수·복구되고 PAY 최종 상태와 수렴한다.
- 완료된 방문·기존 확정 거래는 자동 취소되지 않으며 이 정책에서 수수료·환불 책임·채권이 생성되지 않는다.
- 결과 불명확 중 신규 혜택은 성공으로 승인되지 않고, 대사·이의 처리 후 중앙 상태와 사용자 안내가 일치한다.
