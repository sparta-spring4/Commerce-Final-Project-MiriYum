## 개인정보 공통 불변식

- 모든 개인정보는 데이터 항목, 목적 ID, 처리 근거/동의, 원본 시스템, 허용 처리자·수신자, 보유 정책 참조, 민감도와 삭제 전파 대상을 가진 데이터 목록에 등록한 뒤에만 처리한다. 등록되지 않은 필드와 목적 외 복제를 실패 폐쇄한다.
- 일반 사용자·식당 대표자·운영자 계정과 공개 정보, 인증·결제·심사·분쟁 자료를 권한·저장·로그 경계에서 분리한다. 내부 ID도 목적 없는 결합·교차 추적에 재사용하지 않는다.
- 개인정보 상태 변경은 원 사건 ID와 버전, 멱등 키로 중앙 원본·검색·캐시·분석·파일·복제본에 전파한다. 일부 실패는 완료로 표시하지 않고 대상별 체크포인트에서 재처리한다.
- `PRIV-012` 안전 기본선에 따라 침해 의심 시 영향 자격증명·세션·접근 경로를 즉시 격리하고, 원본을 훼손하지 않는 증거 참조와 중앙 사건을 만든 뒤 최소 권한 대응자가 영향 범위·삭제/유출 여부를 분석한다. 복구는 권한·키 회수, 취약점 차단, 데이터·감사 무결성 검증 뒤 확정하며 법정 외부 통지 시점·대상·문구는 TODO를 따른다.
- 운영 로그·감사·경보에는 연락처·정밀 위치·알레르기 원문·신분/사업자 증빙·결제수단·비밀을 넣지 않고 마스킹한 내부 참조와 상관관계 ID만 사용한다.

## PRIV-001 연령 판정을 위한 최소 개인정보

> 정책 상태: 확정. 아래 주류 연령 판정의 최소 수집 하위 범위는 기존 확정으로 보존한다.

### 확정 사항

- 가입 본인확인 결과에서 법적 기준에 따른 주류 이용 가능일과 확인 시점만 파생해 저장한다.
- 주류 연령 판정 목적으로 생년월일 원본을 저장하지 않는다.
- 가게 현장에서 제시된 신분증 자료와 현장 확인 완료 상태를 수집·기록하지 않는다.
- 연령 판정 정보는 일반 프로필과 분리하고 필요한 기능에서만 최소 권한으로 조회한다.

### 법적 기준과 근거

- 개인정보 보호법 제16조는 처리 목적에 필요한 최소한의 개인정보를 수집하도록 규정한다.
- 같은 법 제21조는 보유기간 경과 또는 처리 목적 달성 등으로 불필요해진 개인정보를 지체 없이 파기하도록 규정한다.
- 국가법령정보센터 제16조: https://www.law.go.kr/LSW/lsLawLinkInfo.do?chrClsCd=010202&lsJoLnkSeq=900079387
- 국가법령정보센터 제21조: https://www.law.go.kr/LSW/lsLinkCommonInfo.do?ancYnChk=&chrClsCd=010202&lsJoLnkSeq=1020398651

### 다중 서버·다중 인스턴스 원칙

- 모든 인스턴스는 중앙의 동일한 주류 이용 가능일과 확인 시점을 조회하여 권한을 판정한다.
- 연령 판정 정보는 일반 프로필 캐시나 로그에 불필요하게 복제하지 않는다.

### 남은 범위

- 주류 이용 가능일·확인 시점의 정확한 보유·파기 기간은 `PRIV-005` TODO에서 정한다.

- 각 개인정보 목적은 대상 기능, 허용 처리, 공개/제공 범위, 보유 정책 참조와 삭제 사건을 독립적으로 가진다. 한 목적의 수집을 추천·광고·분석·제재 판단에 자동 재사용하지 않는다.

## PRIV-002 필수·선택 동의

> 정책 상태: 확정

### 확정 동의·근거·상태

- 동의 화면은 처리 목적, 데이터 항목, 이용·제공 범위, 보유 정책 참조, 거부·철회 영향과 문구 버전을 목적별로 분리한다. 서비스 수행에 필수인 처리와 선택 개인화·광고·민감정보 처리를 하나의 포괄 동의로 묶거나 선택 항목을 사전 선택하지 않는다.
- 동의가 아닌 계약 이행·법적 의무 등 다른 근거를 사용하는 처리는 데이터 목록에 그 근거와 범위를 명시하고 선택 동의처럼 표시하지 않는다. 근거가 없거나 정책 버전을 확인할 수 없는 처리는 시작하지 않는다.
- 동의 상태는 `미요청`, `동의`, `거부`, `철회`, `만료/재동의 필요`와 목적·문구 버전·획득 경로·시각을 가진다. 새 목적·중대한 범위 변경은 새 동의를 요구하고 과거 동의를 소급 확장하지 않는다.
- 철회는 미래 수집·선택 처리·관련 대기 작업을 차단하고 목적상 불필요한 파생 데이터 삭제를 요청한다. 이미 확정된 거래와 별도 유효 근거로 보존되는 자료는 임의 삭제하지 않고 활성 이용에서 분리한다.

### 다중 인스턴스·실패·감사·개인정보

- 동의 변경은 중앙 버전 조건부 갱신으로 순서를 확정하고 모든 API·worker가 처리 직전 최신 상태를 검증한다. 철회와 비동기 처리 경합에는 철회 확정 뒤 새 선택 처리를 시작하지 않는다.
- 동의 저장소 장애·서명 버전 누락에는 선택 처리를 실패 폐쇄한다. 획득·거부·철회·재동의 요구와 차단된 처리 목적을 감사하되 화면 입력·연락처 원문을 증거에 복제하지 않는다.
- 동의 관리를 위해 정밀 위치·기기 지문·추가 연락처를 수집하지 않고 중앙 계정과 문구 버전만 사용한다.

### 관측 가능한 인수 조건

- 필수 처리 동의와 알레르기·취향·광고 선택 동의를 각각 독립적으로 거부·철회할 수 있고 선택 거부가 핵심 비관련 기능 가입을 막지 않는다.
- 이전 문구 동의로 새 목적 처리를 시도하면 재동의 없이는 거부된다.
- 철회와 여러 작업 인스턴스의 처리를 경합시켜도 철회 확정 뒤 새 파생 처리·외부 제공이 시작되지 않는다.

## PRIV-003 알레르기·취향 정보 취급

> 정책 상태: 확정

### 확정 수집·공개·사용 경계

- 알레르기와 기피 재료는 현재 검색·추천 요청 또는 예약·행사별 안전 요청에 사용자가 명시적으로 입력한 경우에만 처리하고, 취향 정보는 개인화 목적의 별도 선택 동의가 있을 때만 처리한다. 두 정보를 의료 진단, 연령·신원 확인, 제재·가격·희소 자원 배분에 사용하지 않는다.
- 검색·추천의 제외 성분은 현재 요청 처리 동안만 사용하고 계정 프로필·추천 이력에 영구 저장하지 않는다. 예약/행사에서 가게에 전달할 때는 사용자가 범위를 선택하며, 가게에는 이행에 필요한 현재 거래의 최소 정보만 제공한다.
- 알레르기 입력은 가게의 확인과 안전 보장을 대신하지 않는다. 메뉴 상태 `함유`, `혼입 가능`, `정보 미등록` 중 위험을 배제할 수 없는 후보는 안전 우선 추천에서 제외하고 입력 누락·전달 실패를 `알레르기 없음`으로 해석하지 않는다.
- 자유 텍스트는 가능하면 구조화 선택과 짧은 보충 설명으로 제한하고 검색·분석·추천 학습·광고 이벤트에 복제하지 않는다. 공개 리뷰나 동행자에게 자동 공개하지 않는다.

### 권한·동시성·실패·알림·감사

- 사용자 본인과 현재 예약을 처리하는 권한 있는 가게 대표자만 유효 기간의 정보를 조회하며 운영자는 배정된 지원·사고 사건에 필요한 경우에만 마스킹 범위를 단계적으로 본다.
- 예약 버전·동의 버전과 정보 스냅샷을 결속하여 수정·취소와 전달을 경합해도 가게가 어느 버전을 받았는지 추적한다. 전달 실패에는 안전 정보를 전달 완료로 표시하지 않고 사용자·가게에 확인 필요 상태를 알린다.
- 조회·전달·수정·삭제·동의 철회를 감사하고 저장·전송 암호화와 `PRIV-009` 로그 마스킹을 적용한다. 정확한 보유기간은 `PRIV-005` TODO를 따른다.

### 관측 가능한 인수 조건

- 알레르기 입력 없이도 일반 탐색은 가능하고, 현재 요청의 제외 성분은 해당 안전 우선 필터 외에 광고·제재·가격 입력이나 외부 AI 입력으로 전달되지 않는다.
- 예약 취소·동의 철회 뒤 가게 화면에서 해당 정보가 더 이상 조회되지 않고 파생 삭제 작업이 추적된다.
- 전달 장애나 미입력을 `알레르기 없음`으로 표시하지 않으며 사용자·가게가 확인 필요 상태와 전달 버전을 볼 수 있다.

## PRIV-004 위치 정보 이용·보관

> 정책 상태: 확정

### 확정 수집 목적·상태·최소화

- 위치는 사용자가 주변 매장 조회 또는 원격 웨이팅 등록처럼 명시적 기능을 실행한 동안에만 요청한다. 백그라운드·상시 추적, 이동 경로, 방문 이력 생성과 광고·신뢰 점수용 위치 수집을 하지 않는다.
- 주변 검색·거리 정렬·거리 추천은 사용자가 명시적으로 실행할 때 현재 좌표를 요청 처리 동안만 사용하고, 공용 캐시에는 사용자 ID·원 좌표 대신 익명화된 지역 셀을 사용한다. 권한 거부·GPS 오류에는 시·군·구 또는 동 단위 직접 선택을 제공한다.
- 원격 웨이팅은 현재 좌표·정확도·측정 시각을 짧은 검증 세션에서만 사용하고 매장 반경 판정 뒤 장기 거래 상태에는 통과/실패, 검증 시각, 정책·매장 좌표 버전과 필요한 최소 오차 분류만 남긴다. 구체 반경은 확정된 `WAIT-003`의 플랫폼 기본 500m·매장 설정 300m~1km를 따른다.
- 권한 거부·부정확·오래된 측정은 실패 폐쇄하고 `WAIT-004`의 현장 QR·매장 코드 같은 대체 경로를 안내한다. 운영자가 좌표를 임의 입력해 통과시키거나 정확한 사용자 위치를 요청하지 않는다.
- 매장 좌표는 공개 주소와 운영에 필요한 지점 정보로 별도 관리하고 사용자 원 좌표와 결합한 장기 기록을 만들지 않는다.
- 위치정보 외부 제공·장기 저장과 최종 법률 검토는 `PRIV-010` 및 관련 개인정보 `TODO` 경계를 유지하며, 현재 요청 처리 규칙이 이를 해소한 것으로 보지 않는다.

### 다중 인스턴스·오남용·감사·삭제

- 검증 세션은 서버 발급 만료 식별자와 계정·매장·목적에 결속하고 중앙 시각·정책 버전으로 한 번만 소비한다. 여러 인스턴스의 동시 요청과 재전송에도 한 웨이팅 사건에 같은 판정만 적용한다.
- 비정상 이동·재사용·변조 신호는 추가 확인 또는 현장 대체로 전환하되 지속 기기 지문이나 유료 위치 탐지를 기본 요구하지 않는다. 매장 좌표 보정은 전용 권한·근거·변경 이력을 요구한다.
- 원 좌표는 검증 세션 만료·목적 완료 시 활성 저장소와 파생 로그에서 삭제하고 실패 대상을 재처리한다. 조회·판정·거부·대체·매장 좌표 변경을 원 좌표 없이 감사한다.

### 관측 가능한 인수 조건

- 기능 실행 전·백그라운드에는 위치 요청이 없고 광고·분석 이벤트에 원 좌표·이동 경로가 남지 않는다.
- 같은 검증 세션을 다른 계정·매장·웨이팅이나 만료 뒤 재사용하면 거부된다.
- 판정 완료·세션 만료 뒤 DB·로그·이벤트에서 원 좌표를 찾을 수 없고 권한 거부에는 현장 대체 경로만 제공된다.

## 검색·추천 외부 AI 요청 최소화

- 개발·시연용 자연어 조건 해석 어댑터에는 사용자 입력 문장과 허용된 MiriYum 지역·카테고리·태그 코드만 전달한다. 사용자 ID·연락처·예약 원문·예약 이력·메뉴 홀드 이력·결제정보·정확한 위치·알레르기 제외 성분은 전달하지 않는다.
- 원문 프롬프트와 외부 AI 응답은 운영 로그·분석 이벤트에 보관하지 않는다. 호출 수, 입력 길이와 토큰을 제한하고 로그에는 결과 분류·지연·실패 코드와 비식별 상관관계만 남긴다.
- 규칙으로 해석하지 못한 표현만 외부 어댑터에 보내며, 실패·시간 초과·형식 오류에는 규칙 기반 결과 또는 일반 키워드 검색으로 축소한다. 운영용 AI 제공자·모델·비용 한도와 외부 처리의 최종 법률 검토는 기존 `TODO` 경계로 남긴다.
- 이전 예약과 최근 메뉴 홀드 이력은 MiriYum 내부 재정렬에만 사용하고 외부 AI 입력이나 공용 캐시 키에 포함하지 않는다. 사용자가 이력 기반 개인화를 철회하면 이후 재정렬에서 제외하고 관련 파생 캐시를 무효화한다.

## PRIV-007 사용자 열람·정정·이동 요청

> 정책 상태: 확정

### 확정 요청·권한·처리 상태

- 인증된 사용자는 자기 계정의 등록된 개인정보 목적, 보유 항목 범주, 출처, 이용·제공 범위와 정정 가능한 원본을 조회하고 열람·정정·이동 요청을 만들 수 있다. 운영자는 본인 대신 요청을 생성하지 않고 AUTH-006 복구 전에는 우회 신원 확인을 하지 않는다.
- 요청은 `접수`, `본인·범위 확인`, `자료 수집`, `제외 검토`, `사용자 제공`, `정정 전파`, `완료`, `보류`, `실패` 상태를 가진다. 요청 ID, 대상 계정, 범위, 정책 버전과 각 원본·파생 대상 체크포인트를 기록한다.
- 제공 자료는 이해 가능한 구조와 기계 판독 가능한 형식으로 만들되 다른 사용자·동행자·신고자, 운영 비밀·보안 토큰·내부 위험 규칙·결제수단 원문을 제외하거나 마스킹한다. 법적 제외 범위와 처리 기한은 최종 법률 정책을 따르고 이 행이 임의 문구·숫자를 만들지 않는다.
- 정정은 권위 있는 원 도메인에서 검증·적용하고 검색·캐시·분석·파일에 정정 사건을 전파한다. 거래 사실·감사 원장을 덮어쓰지 않고 오류 정정·보정 이력으로 남긴다.

### 다중 인스턴스·실패·알림·감사

- 계정-요청 유형-활성 범위별 중복 요청을 같은 사건에 멱등 연결하고 중앙 작업 임대·체크포인트로 자료를 수집한다. 일부 원본 실패를 전체 완료로 표시하지 않고 실패 대상만 재개한다.
- 결과 파일은 암호화된 비공개 객체와 짧은 수명의 한 번 사용 접근으로 제공하고 요청자 재인증을 다시 요구한다. 생성·열람·만료·재발급·정정 전파를 감사하며 파일 URL·내용을 로그에 남기지 않는다.
- 완료·보류·실패와 필요한 다음 행동을 확인된 채널로 알리되 알림 실패가 요청 상태를 바꾸지 않는다. 접근 만료 뒤 파일을 일반 첨부 저장소에 남기지 않는다.

### 관측 가능한 인수 조건

- 다른 사용자·신고자·운영 비밀이 내보내기에서 제외되고 결과 링크는 재인증·만료·한 번 사용을 검증한다.
- 같은 요청을 여러 인스턴스가 처리해도 결과 버전 하나와 대상별 체크포인트가 남고 일부 실패가 완료로 오표시되지 않는다.
- 원본 정정 뒤 검색·캐시·분석에 보정 사건이 반영되며 거래·감사 사실을 제자리 삭제하지 않는다.

## PRIV-008 암호화·키 관리의 남은 범위

> 정책 상태: 확정. 위 사업자등록증 이미지의 저장 암호화·단기 접근 기존 확정 사항을 그대로 적용한다.

### 확정 전송·저장·키 생명주기

- 외부·내부 네트워크의 개인정보·세션·관리 API는 인증된 TLS로 전송하고 평문 우회 경로를 두지 않는다. DB·객체·백업의 민감 데이터는 아키텍처가 승인한 암호화 기능과 키 저장소로 저장하며 애플리케이션 코드·이미지·Terraform 상태·로그에 비밀값을 넣지 않는다.
- 환경별 데이터 키와 비밀을 분리하고 서비스 역할에는 필요한 키 사용 권한만 부여한다. 운영 비밀을 개발자·개발 환경에 배포하지 않고 테스트에는 합성 자격증명을 사용한다.
- 암호문은 키 버전·알고리즘/형식 버전과 무결성 정보를 가져 회전 중 이전·신규 버전을 읽을 수 있게 한다. 새 쓰기는 활성 키를 사용하고 이전 데이터는 체크포인트 작업으로 재암호화한 뒤 검증 전 키를 폐기하지 않는다.
- 비밀 변경·권한 회수는 새 배포 또는 동적 갱신으로 모든 인스턴스에 반영하고 이전 세션·연결을 필요 범위에서 무효화한다. 키 저장소 장애에는 새 민감 쓰기·복호화를 실패 폐쇄하고 평문/로컬 기본키로 우회하지 않는다.

### 다중 인스턴스·복구·감사

- 모든 인스턴스가 중앙 키 별칭·버전과 권한 정책을 사용하고 로컬 디스크에 장기 키를 저장하지 않는다. 회전 작업은 대상-키 버전 멱등 키와 임대·체크포인트로 중복 재암호화를 막는다.
- 암호화 백업도 운영계와 분리된 최소 권한으로 접근하고 복원 시 키 가용성·무결성·삭제 원장 재적용을 검증한다. 키 분실·오사용·권한 거부는 중앙 보안 사건과 `PRIV-012` 격리 절차로 연결한다.
- 키 생성·사용 권한 변경·회전·폐기·복호화 실패와 비밀 조회를 감사하되 키·비밀 원문은 기록하지 않는다.

### 관측 가능한 인수 조건

- 저장소·백업·네트워크를 검사해 민감 필드 평문과 코드·로그·상태 파일의 비밀을 찾을 수 없다.
- 회전 중 이전·신규 인스턴스가 각 키 버전 암호문을 안전하게 읽고 새 쓰기는 활성 키만 사용하며 재실행에도 중복 손상이 없다.
- 키 저장소 장애 시 평문 우회 없이 민감 작업이 실패 폐쇄되고 복구·권한 변경이 감사된다.

## PRIV-009 로그·분석 데이터 마스킹

> 정책 상태: 확정

### 확정 허용 필드·마스킹·격리

- 로그·지표·추적·분석 이벤트는 목적별 허용 필드 스키마를 사용한다. 요청/사건 상관관계 ID, 오류 코드, 상태 버전과 비식별 내부 참조만 기본 기록하고 연락처·실명·정밀 위치·알레르기/리뷰 원문·사업자 증빙·결제수단·세션/토큰·비밀을 제외한다.
- 이메일·전화번호 등 제한적으로 필요한 운영 표시도 원문 대신 일관된 마스킹을 사용하고 단순 해시를 익명정보로 간주하지 않는다. 자유 형식 요청/응답 본문, HTTP 헤더·쿠키와 외부 제공자 payload 전체 로깅을 금지한다.
- 예외 디버그는 중앙 보안 사건, 승인된 필드, 대상 서비스·환경과 만료를 결속하고 기본 비활성화한다. 원문 덤프·운영 DB 복사·개발 환경 전송을 허용하지 않고 종료 시 접근을 회수한다.
- 분석은 `ANALYTICS-001`의 기존 최소 사건만 사용하고 새 추적 식별자·기기 지문·교차 목적 프로필을 만들지 않는다.

### 다중 인스턴스·실패·삭제·감사

- 공통 로깅 라이브러리와 스키마 버전을 API·worker·배치에 동일 적용하고 이전·신규 버전 혼재 시 더 제한적인 규칙을 적용한다. 마스킹 실패·알 수 없는 필드는 기록 전에 제거하거나 이벤트를 격리한다.
- PII 탐지 검사와 샘플링으로 유출을 경보하고 발견 즉시 접근 격리, 인덱스·캐시 삭제 작업과 `PRIV-012` 영향 분석을 시작한다. `PRIV-005` 보유 정책과 삭제 원장을 로그·분석 파생물에도 적용한다.
- 로그 정책 변경·예외 디버그 승인·조회·내보내기·삭제와 유출 탐지를 감사한다. 관측 시스템 접근도 최소 역할과 환경 분리를 적용한다.

### 관측 가능한 인수 조건

- PII·토큰을 요청 본문·헤더·외부 오류에 주입해도 로그·지표·추적·분석 이벤트에 원문이 남지 않는다.
- API·worker 여러 버전을 동시에 실행해도 동일 금지 필드가 제거되고 마스킹 실패는 격리·경보된다.
- 로그 PII 탐지 사건 뒤 접근 격리와 파생 삭제 대상·재처리·완료 상태가 중앙 사건에서 추적된다.

## PRIV-011 관리자 접근 통제의 남은 범위

> 정책 상태: 확정. 위 사업자등록증의 배정 심사자 단기 접근 기존 확정 사항을 그대로 적용한다.

### 확정 최소 권한·사건 결속·이중 통제

- 개인정보 접근 권한은 심사, 고객지원, 결제 복구, 보안 사고, 감사처럼 업무별로 분리하고 계정 역할만 아니라 대상 사건 배정·매장/계정 범위·필드 민감도·목적과 만료를 함께 검증한다.
- 기본 화면은 마스킹 정보만 제공하고 원문 공개는 사건 처리에 필요한 필드별 명시 작업과 현재 비밀번호 재인증을 요구한다. 비밀번호·인증번호·세션 토큰·결제수단 원문·암호화 키는 어떤 일반 관리자 권한으로도 조회하지 않는다.
- 대량 조회·내보내기, 감사 전체 열람, break-glass와 광범위 민감정보 접근은 요청자와 다른 승인자의 이중 승인, 불변 대상 계획과 만료를 요구한다. 일반 업무 API를 우회한 직접 DB·객체 저장소 접근은 금지하고 장애 절차의 제한된 긴급 접근만 허용한다.
- 계정 정지·퇴사·역할 회수·사건 종결 시 접근과 미사용 승인·세션을 중앙에서 즉시 회수한다. 정기 권한 검토는 실제 사용·미사용 권한·이해충돌을 확인하고 필요 없는 권한을 제거한다.

### 다중 인스턴스·실패·알림·감사

- 모든 서비스가 중앙 권한·사건·승인 버전을 요청마다 검증하고 권한 회수 사건으로 캐시를 무효화한다. 권한 저장소 장애나 버전 불일치에는 민감정보 접근을 실패 폐쇄한다.
- 허용·거부된 조회, 공개한 필드 등급, 내보내기, 승인, break-glass, 권한 변경을 변경 불가능한 `ADMIN-009` 안전 감사 원장에 남긴다. 감사 조회 자체도 별도 권한과 감사 대상이다.
- 중대한 비정상 접근은 `PRIV-012` 중앙 사건·격리로 연결하고 필요한 경우 영향 대상 통지는 최종 법률 정책을 따른다. 경보·감사에는 조회한 원문 PII를 복제하지 않는다.

### 관측 가능한 인수 조건

- 사건 배정이나 목적·필드 권한 없이 역할만 가진 관리자는 원문 PII를 조회할 수 없다.
- 권한 회수와 여러 인스턴스 조회를 경합시켜도 회수 확정 뒤 새 원문 접근이 없고 캐시도 실패 폐쇄한다.
- 대량 내보내기·break-glass는 서로 다른 승인자, 대상 계획·만료·조회 감사 없이는 실행되지 않는다.

## PRIV-013 백업 데이터의 삭제 반영

> 정책 상태: 확정

### 확정 삭제 원장·복제 전파·복원

- 최종 삭제·익명화 사건은 대상 내부 ID·데이터 범주, 원 사건 버전, 효력 시각, 보존 예외 참조와 파생 대상 목록을 가진 변경 불가능한 삭제 원장에 기록한다. 삭제 원장에는 원 연락처·민감 원문을 넣지 않는다.
- 활성 DB·객체·검색·캐시·분석·메시지 파생물은 대상별 멱등 키와 상태 `대기`, `처리 중`, `검증`, `완료`, `실패`로 삭제 또는 비식별화한다. 일부 대상 실패를 전체 완료로 표시하지 않고 실패 대상만 재시도한다.
- 시점 백업처럼 제자리 수정할 수 없는 복제본은 일반 서비스·분석 조회에서 격리한다. 복원은 격리 환경에서 무결성을 확인한 뒤 백업 생성 이후까지의 삭제 원장·권한 회수·키 변경을 먼저 재적용하고 검증 전 운영 트래픽에 연결하지 않는다.
- 보존기간과 법적 예외는 `PRIV-005`·`PRIV-006` TODO의 유효한 정책을 참조하며 이 행이 기간을 정하지 않는다. 예외 자료도 활성 계정과 분리하고 목적·접근을 제한하며 만료 사건을 삭제 원장으로 처리한다.

### 다중 인스턴스·실패·감사·개인정보

- 삭제 작업은 파티션 임대·체크포인트와 대상 버전 조건을 사용하고 재실행에도 이미 삭제된 데이터가 부활하거나 집계가 중복 감소하지 않는다. 새 파생 소비자는 처리 전 삭제 원장을 확인한다.
- 복원·재색인·재집계 작업은 삭제 체크포인트를 산출물 메타데이터에 기록한다. 원장이나 키에 접근할 수 없으면 복원 공개를 실패 폐쇄하고 운영자·보안 경보를 생성한다.
- 삭제 요청·대상별 처리·예외 적용·복원·재적용·검증·실패와 수동 개입을 감사한다. 테스트는 합성 식별자를 사용하고 백업 검증을 위해 실제 PII를 별도 복제하지 않는다.

### 관측 가능한 인수 조건

- 같은 삭제 사건을 여러 작업자가 반복 처리해도 대상별 완료는 한 번이고 집계·참조가 중복 변경되지 않는다.
- 삭제 전 백업을 복원해도 삭제 원장 재적용·검증 전에는 운영 연결이 차단되고 삭제된 식별정보가 검색·캐시·분석에 나타나지 않는다.
- 일부 파생 저장소 장애 뒤 전체 완료로 오표시되지 않으며 복구 후 실패 대상만 재처리되어 모두 검증 상태로 수렴한다.
