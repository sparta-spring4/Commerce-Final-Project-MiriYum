# MiriYum 서비스 정책 마스터 체크리스트

> 문서 상태: 운영 중
> 기준일: 2026-07-23
> 목적: 구현 전에 결정해야 할 정책을 빠짐없이 추적한다.

## 상태 정의

- `미논의`: 아직 선택지와 영향을 검토하지 않음
- `논의 중`: 현재 대화에서 검토 중
- `자동 추천 예정`: 제품 방향을 바꾸지 않는 안전한 표준 기술·운영 기본안을 개별 팀 논의 없이 제안할 후보 상태. 상세안을 작성하는 동안 이 상태를 유지하며, 명시적 위임·세부 완성·독립 검토·결정 기록 없이 상태만으로 `확정`되지 않는다. 현재는 기존 139개 후보가 이 절차를 모두 마쳐 대기 행이 0개지만, 미래 후보에 이 상태를 계속 사용할 수 있다.
- `TODO`: 유료·외부 사업자 계약, 최종 법률·개인정보·약관 검토, 미래 확장, 실측이 필요한 수치 또는 현재 범위를 넘는 복잡성 때문에 후속 결정으로 미룬 항목. 필요한 경우 미정 부분과 경계를 분리해 안전 기본선을 먼저 상세화·확정할 수 있다.
- `팀원 상의 필요`: 제품 방향, 주요 사용자·가게 흐름, 금전 책임, 희소 기회의 공정한 배분 또는 되돌리기 어려운 약속을 바꾸며 안전한 단일 기본값을 정하기 어려운 항목
- `확정`: 사용자가 직접 승인했거나, 비핵심 추천안이 명시적인 추천 위임 기준 안에서 세부 내용과 인수 조건까지 완성되고 독립 검토를 거쳐 정책 ID별 의사결정 기록에 `확정`으로 남은 항목. 자동 추천 후보를 문서화하거나 구현한 사실만으로는 충분하지 않다.
- `변경 검토`: 기존 확정 내용을 다시 검토 중

중요도와 상태는 서로 독립적이다. 중요도 `필수`는 구현 전에 빠뜨릴 수 없다는 뜻일 뿐, 그 자체로 팀원 논의가 필요하다는 뜻은 아니다. 중요도 `핵심`은 이번 분류에서 대표 팀 결정에만 사용한다.

## 운영 규칙

- 정책 ID는 이름이 바뀌어도 변경하지 않는다.
- `팀원 상의 필요` 항목만 한 번에 하나씩 선택지·장단점·추천 근거를 검수한다. 대표 결정에 종속된 새 세부 규칙은 별도 팀 의제로 늘리지 않고 먼저 `자동 추천 예정` 후보로 분류하며, 대표 결정의 결과만 기계적으로 소비하게 한다.
- 표준 보안·권한·멱등성·동시성·상태 전이·실패·재시도·알림·감사 규칙은 제품 방향을 바꾸지 않는 새 후보라면 `자동 추천 예정`으로 시작할 수 있다.
- `자동 추천 예정` 후보는 무료로 운영 가능한 단순한 구조, 개인정보 최소 수집, 실패 폐쇄, 멱등성, 중앙 상태와 다중 인스턴스 정합성을 우선한다. 상세 정책과 관측 가능한 인수 조건을 완성하고 독립 검토를 통과한 뒤, 명시적 추천 위임 기준에 따른 승인 사실을 정책 ID별 결정 기록에 남겨야 `확정`으로 전환한다.
- 2026-07-22에는 기존 139개 후보가 위 절차를 모두 완료해 `확정`으로 전환되었다. 이 집합의 ID별 상세 근거와 확인 기록은 각 도메인 문서에 있으며, `TODO` 행이나 행 안의 계약·법률·미래·실측 수치 조각은 이 전환에 포함되지 않는다.
- 유료·외부 사업자 계약, 최종 법률·개인정보·약관 검토, 미래 기능, 실데이터 기반 수치 조정 또는 과도한 복잡성이 필요한 항목은 추측하지 않고 `TODO`로 남긴다.
- `TODO`에 미정 부분이 있어도 최소 권한, 중복 처리 방지, 암호화 백업, 감사 기록 같은 안전 기본선은 경계를 분리해 함께 명시하고 별도 절차로 확정할 수 있다.
- 대규모 트래픽과 다중 서버·다중 인스턴스를 전제로 하되, 유료 서비스 도입·불필요하게 복잡한 인프라·추가 개인정보 수집은 자동 기본값으로 선택하지 않는다.
- 상세 정책이 끝나면 예외, 악용, 동시성, 장애 및 테스트 관점으로 재검토한다.
- 제품 범위 변경은 `docs/01-product-vision.md`, 기술 방향 변경은 `docs/06-system-architecture.md`에 반영한다.
- 결정 및 변경 이력은 `miriyum-service-decisions.md`에 남긴다.
