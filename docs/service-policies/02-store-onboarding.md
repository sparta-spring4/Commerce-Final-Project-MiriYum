## STORE-002 사업자등록 단위와 매장 관계

### 확정 사항

- 플랫폼의 사업자 확인 단위는 제출된 사업자등록증과 사업자등록번호를 기준으로 한다.
- MiriYum에서는 보통 사업자등록번호 1개를 매장 1개로 본다.
- 같은 개인 또는 법인이 여러 지점을 운영하더라도 사업자등록번호가 다르면 각각 별도 매장으로 등록한다.
- 하나의 매장 대표자 계정은 서로 다른 사업자등록번호를 가진 여러 매장을 관리할 수 있다.
- 매장은 특정 시점에 하나의 사업자등록 단위에만 소속되며, 양도나 운영 주체 변경 시 과거 관계를 덮어쓰지 않고 변경 이력을 보존한다.

### 근거

- 사업장마다 사업자등록을 하는 일반 원칙을 서비스의 기본 입점 모델에 반영한다.
- 고객이 예약·웨이팅·방문하는 매장과 계약·정산·법적 책임의 기준이 되는 사업자등록 정보를 명확히 연결한다.
- 지점별 정산, 제재, 폐업 및 양도 이력이 다른 매장으로 전파되는 것을 방지한다.

- 같은 법인 또는 개인이 운영하는 여러 매장을 하나의 본사·브랜드 그룹으로 묶는 방식
- 본사·지점 그룹의 조회, 정산 및 운영 권한 범위

## STORE-003 매장 추가 시 사업자등록증 인증

> 정책 상태: 확정

### 확정 사항

- 매장 대표자는 최초 매장을 포함해 매장을 추가할 때마다 해당 매장의 사업자등록증 인증 절차를 완료해야 한다.
- 계정 가입 전 휴대전화 본인 확인을 완료했더라도 매장별 사업자등록증 인증을 생략할 수 없다.
- 사업자등록증 인증이 완료되기 전에는 해당 매장을 활성화하거나 운영 기능을 사용할 수 없다.
- 여러 매장을 추가하는 경우 각 매장의 인증 상태와 이력을 서로 독립적으로 관리한다.
- 신청자는 사업자등록증의 사업자등록번호·상호·대표자명·개업일자·사업장 주소·업태·종목을 글로 입력하고 같은 사업자등록증 이미지를 필수로 제출한다.
- 신청자가 입력한 사업자등록번호·개업일자·대표자명은 국세청의 무료 사업자등록정보 진위확인 API 결과와 비교하고, 사업자 상태도 함께 확인한다.
- 진위확인 정보가 일치하고 정책상 허용된 사업자 상태이면 사업자 정보 자동 검사를 통과한 것으로 기록한다. 최종 사업자등록증 인증과 입점은 STORE-005의 이미지·입력 정보 및 업종 관리자 확인을 통과해야 한다.
- 외부 API 장애·응답 지연·신규 사업자 정보 미반영·입력 정보 불일치가 발생하면 자동 승인하거나 즉시 최종 거절하지 않고 `확인 보류`로 전환하여 재시도 또는 관리자 보완 검토 대상으로 처리한다.
- 현재 범위에서는 휴대전화·이메일 소유 확인을 마친 신청자가 자신이 사업자등록증상 대표자 본인임을 확약하고 허위 등록의 책임에 동의하도록 한다. 이 확약은 공공 API가 확인한 대표자명과 실제 신청자가 동일인이라는 강한 실명 확인을 대신하지 않는다.
- 신분증을 추가 수집하거나 PASS 등 유료 본인확인 사업자를 사용하지 않는다. 실제 신청자와 사업자등록증상 대표자의 강한 동일인 확인은 운영 전 `TODO`로 남긴다.

### 근거

- 대표자 개인의 신원 확인과 개별 매장의 실제 사업자·운영 주체 확인은 목적과 증빙이 다르므로 분리한다.
- 하나의 대표자 계정이 여러 매장을 관리하더라도 잘못된 소유권 전파와 사업자등록번호 중복 등록을 방지한다.
- 무료 공공 조회로 사업자 정보의 진위와 상태를 자동 확인하여 민간 인증 비용과 전체 관리자 심사 병목을 피한다.
- 대표자 자기확약의 검증 한계를 명시하여 현재 팀 프로젝트가 제공하지 않는 강한 실명 확인을 완료한 것처럼 취급하지 않는다.

### 다중 서버·다중 인스턴스 원칙

- 매장별 사업자등록증 인증 상태는 중앙 저장소를 기준으로 판정한다.
- 같은 신청 또는 외부 인증 결과가 중복 전달되어도 사업자등록번호와 인증 시도 식별자를 기준으로 한 번만 반영한다.
- 동일한 사업자등록번호의 동시 등록 시도는 최종 저장 제약과 원자적 상태 전이로 중복 확정을 방지한다.
- 외부 조회 요청은 중앙 작업 상태와 요청 식별자로 관리하고, 제공 한도 안에서 묶음 처리·속도 제한·재시도를 적용한다. 여러 작업 인스턴스가 같은 신청을 동시에 조회하거나 중복 승인하지 않게 한다.
- 외부 API의 장애·시간 초과·한도 초과는 인증 성공이나 정보 불일치로 해석하지 않는다. 중앙 상태를 `확인 보류`로 유지하고 지수형 지연을 둔 재시도와 최종 운영자 경보를 적용한다.

## STORE-004 매장 소유권 확인

> 정책 상태: 확정

현재 1차 MVP에서는 STORE-004의 소유권 충돌 처리 경로를 구현 범위에 포함한다. STORE-003의 사업자등록증 인증, STORE-005의 기본 입점 승인, STORE-004의 중복 사업자등록번호 충돌 재확인 경로를 모두 현재 MVP 입점 경로의 일부로 운영하며 아래 정책 방향을 기준으로 설계한다.

### 확정 사항

- 이미 다른 가게 계정에 귀속된 사업자등록번호는 일반 매장 추가 절차로 중복 등록하거나 자동 이전할 수 없다.
- 중복 신청이 발생해도 기존 가게의 운영 권한·세션·공개 상태를 중지하거나 변경하지 않는다. 신규 신청자에게 기존 계정의 이메일·휴대전화·대표자 정보와 내부 매장 정보를 공개하지 않는다.
- 동일한 사업자등록번호로 새 신청이 들어오면 자동 승인하지 않는다.
- 해당 신청은 `매장 소유권 충돌 심사` 건으로 전환하고, 담당 관리자가 사업자 정보와 기존 매장 귀속 이력을 확인한 뒤 충돌 원인을 분류하고 후속 절차를 결정한다.
- 관리자는 충돌 원인과 필요한 후속 절차를 분류할 수 있지만 STORE-004 심사만으로 매장을 다른 계정에 직접 이전할 수 없다.
- 현재 MVP에서는 소유권 충돌을 자동 이전 절차로 처리하지 않고, 계정 복구 또는 관리자 수동 검토 대상으로 남긴다. 실제 양도·대표자 변경은 STORE-014의 후속 정책 범위로 다룬다. 단순 중복·허위·증빙 불충분 신청은 거절한다.
- 관리자의 분류 결과, 확인 근거, 신청자·기존 귀속 식별자와 처리 시각은 수정·삭제할 수 없는 감사 이력으로 남긴다.

### 정책 근거

- 허위 중복 신청만으로 정상 가게가 정지되거나 다른 계정으로 이전되는 서비스 거부·소유권 탈취를 막는다.
- 계정 복구와 실제 사업 양도·대표자 변경을 신규 매장 등록에서 분리하여 기존 예약·결제·정산·제재 이력의 귀속을 보존한다.
- 관리자는 사실관계와 올바른 절차만 재확인하게 하여 자기확약만으로 직접 소유권을 바꾸는 위험을 제한한다.

### 다중 서버·다중 인스턴스 원칙

- 사업자등록번호의 활성 매장 귀속은 중앙 유일성 제약으로 보호하고, 중복 등록과 소유권 충돌 사건 생성을 하나의 일관된 처리 경계에서 확정한다.
- 같은 사업자등록번호에는 활성 소유권 충돌 심사를 하나만 허용한다. 여러 인스턴스의 동시·반복 신청은 기존 사건에 멱등하게 연결하고 신청자·IP·요청 경로별 속도 제한을 적용한다.
- 심사 중에도 모든 서비스 인스턴스는 기존 중앙 귀속과 권한을 계속 사용한다. 관리자 분류 결과는 후속 AUTH-006 또는 STORE-014 사건을 생성할 뿐 기존 귀속을 직접 변경하지 않는다.

## STORE-005 기본 입점 심사 기준

> 정책 상태: 확정


### 확정 사항

- 기본 입점은 글로 입력한 사업자 정보의 자동 검사와 제출한 사업자등록증 이미지에 대한 관리자 확인을 모두 통과해야 확정한다.
- 자동 검사는 신청자가 입력한 사업자 정보가 맞는지, 현재 운영 중인 사업자인지, 이미 같은 사업자등록번호로 등록된 매장이 없는지 먼저 확인한다.
- 또한 필수 가게 정보 입력, 대표자 확인 체크, 필수 약관 동의가 모두 완료됐는지도 함께 확인한다.
- 자동 검사에 실패하거나 외부 조회가 완료되지 않은 신청은 관리자 승인 단계로 넘기지 않고 `확인 보류` 상태에서 재입력·재조회·보완 절차를 먼저 진행한다.
- 자동 검사를 통과하면 `기본 입점 심사` 권한을 가진 관리자 1명이 제출 이미지의 사업자등록번호·상호·대표자명·개업일자·사업장 주소·업태·종목이 입력 정보와 같은지 확인한다.
- 관리자는 이미지의 업태·종목이 MiriYum이 지원하는 외식업 범위인지 확인하고, 이미지와 글 정보가 일치하며 지원 업종이면 승인한다. 이미지 식별 불가·정보 불일치·업종 모호성은 보완 요청하고 명백한 위조·허위 또는 지원 제외 업종은 사유를 기록해 거절한다.
- 관리자는 국세청이 이미 확인한 결과를 재판정하거나 재량으로 우회할 수 없다. 계속사업자가 아니거나 자동 검사 미완료인 신청은 이미지가 정상이어도 승인할 수 없다.
- 기본 입점이 승인되기 전에는 가게를 공개하거나 운영 기능을 사용할 수 없다. 승인 후에는 STORE-007과 STORE-008에서 정한 현재 MVP 기능을 사용할 수 있다.
- 승인·보완 요청·거절에는 신청 정보 버전, 자동 검사 결과, 이미지 비교 항목, 업종 판정, 담당 관리자, 결정 사유와 처리 시각을 수정·삭제할 수 없는 감사 이력으로 남긴다.

### 정책 근거

- 공공 API가 확인할 수 있는 사업자 진위·상태는 자동화하고, API가 제공하지 않는 업태·종목과 이미지·입력값 일치 여부만 관리자가 확인하여 심사 범위를 줄인다.
- 고정 체크리스트로 관리자별 판단 편차를 줄이면서도 음식점이 아닌 사업자의 자동 입점을 막는다.
- 글 정보와 원본 이미지 증빙을 함께 보존하여 보완·분쟁·감사 시 어떤 자료를 기준으로 승인했는지 추적한다.

### 다중 서버·다중 인스턴스 원칙

## STORE-007 일반 기능의 즉시 활성화 범위

> 정책 상태: 확정

현재 1차 MVP에서는 아래 정책을 기준으로 일반 기능 범위를 구현한다.

- 기본 입점 승인이 완료된 매장은 별도의 기능별 관리자 심사 없이 일반 기능을 바로 사용할 수 있다.
- 일반 기능은 금전 결제가 직접 연결되지 않는 매장 정보 관리와 기본 예약 운영 기능으로 한정한다.
- 일반 기능에는 매장 정보 등록 및 수정, 운영 시간 관리, 메뉴 초안 작성 및 수정, 매장 이미지 업로드, 예약 가능 시간대 관리, 예약 가능 인원 수 관리, 웨이팅 가능 여부 설정, 방문 예약, 현장 방문자가 우리 웹에서 직접 등록하는 현장 웨이팅, 예약·웨이팅 현황 조회, QR 체크인, 미응답 종료 확정 후 다음 호출, 취소 자리·메뉴 수량 자동 승계를 포함한다.
- 메뉴 홀드는 일반 매장에서는 방문 예약에 연결된 경우에만 사용할 수 있다. 사용자는 예약 과정에서 메뉴를 미리 선택할 수 있으며, 매장은 해당 예약 건에 대해 필요한 수량을 준비할 수 있다. 다만 베이커리는 방문 예약 개념을 두지 않으므로 단독 픽업 홀드를 사용할 수 있고, 이 경우 사용자는 픽업 주문에 대해 메뉴를 홀드할 수 있다.
- QR 체크인은 예약 또는 웨이팅 사용자가 매장 도착 후 우리 웹에서 자신의 방문 여부를 확인하는 기능으로 본다.
- 미응답 종료 확정 후 다음 호출은 WAIT-011 기준을 따라, 웨이팅 호출 후 만료된 팀이 매장 대표자의 현장 확인을 거쳐 미응답 종료로 확정된 뒤에만 다음 순번을 호출하는 기능으로 본다.
- 취소 자리·메뉴 수량 자동 승계는 예약 취소, 홀드 해제, 또는 정책상 반환 처리로 다시 가용 상태가 된 자리·메뉴 수량이 생기면 조건에 맞는 다음 대기자에게 제안하는 기능으로 본다.
- 예약금 결제, 선결제, 정산, 구독, 주류·테이스팅, 코스 예약, 행사성 모집 기능, 수요 분석은 일반 기능에 포함하지 않는다.
- 기본 입점 승인 전에는 매장 정보, 메뉴, 이미지, 운영 시간 등의 초안 작성만 가능하며, 사용자에게 공개되거나 실제 예약·웨이팅·결제가 발생하는 기능은 사용할 수 없다.
- 기본 입점 승인 후에는 STORE-007과 STORE-008에서 정한 현재 MVP 기능을 사용할 수 있다.

## STORE-014 폐업·양도·사업자 변경

> 정책 상태: 확정

현재 1차 MVP에서는 STORE-014의 별도 폐업·양도·대표자 변경 절차를 구현 범위에 포함하지 않는다. 현재 MVP의 신규 매장 입점과 기본 운영 경로는 STORE-003, STORE-005, STORE-007, STORE-008 기준으로 처리하며, 추가로 STORE-014를 구현할 경우 아래 정책 방향을 기준으로 설계한다.

### 확정 정책과 적용 범위

- `폐업`은 해당 사업자등록 단위의 영업 종료, `양도`는 매장의 운영 주체 또는 사업자등록 단위가 바뀌는 경우, `대표자 변경`은 같은 사업자등록 단위에서 사업자등록증상 대표자가 바뀌는 경우로 구분한다.
- 이 정책은 MiriYum의 현재 매장 운영 권한과 매장 운영 상태를 안전하게 바꾸는 절차만 정한다. 과거 거래의 세무·법적 책임, 정산 귀속, 계약 승계 여부와 STORE-013의 본사·지점·사업자 단위 과세 예외를 판정하지 않는다.
- 매장과 사업자등록 단위·대표자 계정의 관계는 유효 시작·종료 시각이 있는 이력으로 추가한다. 기존 관계와 과거 예약·결제·환불·정산·제재·감사 기록의 당시 귀속을 새 대표자나 새 사업자에게 덮어쓰거나 소급 이전하지 않는다.
- 양도 후에도 같은 물리 매장을 계속 운영하는 경우 매장 식별자는 유지할 수 있지만, 변경 전후 운영 주체 관계와 거래 책임 스냅샷은 별도 버전으로 보존한다. 새 운영 주체를 기존 사업자등록번호의 단순 정보 수정으로 처리하지 않는다.

### 행위자·권한과 사전 조건

- 현재 권한이 유효한 사업자등록증상 대표자는 자신의 매장에 대해 폐업·양도·대표자 변경 신청을 생성·철회할 수 있다. 철회는 최종 전환 전까지만 허용한다.
- 변경 후 대표자는 별도의 활성 매장 대표자 계정으로 휴대전화·이메일 확인을 마치고, STORE-003의 변경 후 사업자등록증 입력·이미지 제출·무료 공공 조회 및 STORE-005의 관리자 확인을 통과해야 한다. 기존 대표자 계정의 세션·인증정보·공용 계정은 승계 수단으로 사용할 수 없다.
- `매장 소유권 변경 심사` 권한을 가진 플랫폼 운영자만 증빙 충족 여부와 전환을 승인·보완 요청·거절할 수 있다. 신청자 자기확약만으로는 기존 귀속을 변경할 수 없고, STORE-004 충돌 심사 권한만으로도 전환할 수 없다.
- 정상 양도·대표자 변경은 현재 대표자와 변경 후 대표자의 확인을 모두 요구한다. 현재 대표자의 협조가 불가능하거나 양측 주장이 충돌하면 자동 전환하지 않고 하나의 소유권 충돌 사건으로 보류하며, 관리자는 공공 조회 결과·변경된 사업자등록증·기존 귀속 이력 등 MiriYum이 이미 요구하는 최소 증빙만 재확인한다. 추가 신분증이나 유료 본인확인 서비스는 기본 요건으로 수집하지 않는다.
- 폐업 신청은 공공 사업자 상태 조회 또는 현재 대표자의 신청으로 접수할 수 있다. 조회 장애·정보 미반영·불일치는 폐업 확정이나 권한 이전으로 간주하지 않고 `확인 보류`로 둔다.

### 상태와 정상 전이

- 변경 사건은 `신청` → `증빙 확인` → `양측 확인` → `전환 예약` → `전환 완료`로 진행한다. 보완이 필요하면 `보완 요청`, 증빙이 부족하거나 충돌이 해소되지 않으면 `거절`, 최종 전환 전 유효한 철회는 `철회`로 종결한다.
- `보완 요청`은 새 자료 버전이 접수되면 `증빙 확인`으로 돌아간다. `거절`·`철회`·`전환 완료` 사건은 다시 열지 않고 새 요청 식별자로 재신청한다.
- `전환 예약`에는 중앙 기준의 효력 시각, 변경 전·후 사업자 관계 버전, 변경 후 대표자 계정, 처리할 기존 미래 거래 목록의 기준 시각을 고정한다. 효력 시각 전에는 기존 대표자의 권한을 유지하며 새 대표자에게 운영 권한이나 과거 거래 조회 권한을 미리 부여하지 않는다.
- 효력 시각에는 기존 대표자의 해당 매장 소속과 세션 권한을 먼저 종료하고, 같은 원자적 전환에서 변경 후 대표자에게 현재 운영 권한을 부여한다. 변경 후 대표자가 다시 로그인하기 전까지 기존 세션을 재사용할 수 없다.
- 완료 시점 이후 생성되는 거래에는 새 운영 주체 관계 버전을 기록한다. 완료 전 생성된 거래는 원래의 관계·가격·정책 스냅샷을 유지하며, 새 대표자는 아직 이행해야 할 미래 예약·홀드 등 매장 운영에 꼭 필요한 정보만 처리할 수 있다. 완료된 과거 거래의 개인정보와 변경 전 정산 자료는 새 대표자에게 자동 공개하지 않는다.

### 폐업·기존 거래·변경 실패 처리

- 폐업이 확인되거나 양도 과정에서 영업을 이어갈 수 없으면 OPER-009에 따라 매장을 `휴점` 또는 `폐점`으로 전환하고 신규 예약·웨이팅·홀드·결제를 차단한다. 기존 확정 거래를 조용히 취소·이전하지 않고 OPER-010과 각 거래의 매장 귀책 변경·취소·환불 정책에 따라 개별 처리한다.
- 전환 사건 생성이나 심사만으로 기존 가게의 공개 상태·세션·기능을 중지하지 않는다. 명백한 안전 위험이 있으면 별도 권한과 사유를 갖춘 OPER-010 긴급 영업 중단을 사용한다.
- 변경 후 대표자의 인증·심사가 만료·철회되거나 효력 시각 전 필수 조건이 깨지면 전환 예약을 실패 폐쇄하고 기존 귀속을 유지한 채 `보완 요청`으로 되돌린다.
- 권한 종료에는 성공했지만 새 권한 부여가 확인되지 않는 부분 실패가 발생하면 새 거래를 차단하고 사건을 `전환 복구 중`으로 표시한다. 중앙 전환 기록을 기준으로 멱등 재실행하며, 어느 쪽에도 중복 활성 권한을 주지 않는다. 자동 복구가 끝나지 않으면 권한 있는 운영자 작업 큐와 경보로 보낸다.

### 동시성·멱등성·다중 인스턴스

- 매장 하나에는 활성 STORE-014 사건을 하나만 허용한다. 동일 요청 식별자의 재전송은 기존 사건을 반환하고, 서로 다른 동시 요청은 중앙 유일성 제약으로 하나만 생성한다.
- 승인·거절·철회·전환 요청은 사건 버전과 멱등 키를 확인한다. 심사 상태, 관계 이력 종료·생성, 매장 권한 버전, 감사 이력과 후속 작업 생성은 하나의 일관된 처리 경계에서 확정한다.
- 모든 API·작업 인스턴스는 중앙 관계·권한 버전을 기준으로 판정한다. 캐시가 오래됐거나 중앙 권한을 확인할 수 없으면 소유권 변경과 매장 관리 쓰기를 실패 폐쇄하며, 인스턴스 로컬 상태나 고정 라우팅으로 이전 권한을 인정하지 않는다.
- 전환 직전·직후 거래 생성은 같은 매장 권한 버전을 조건부로 검사한다. 거래가 먼저 확정되면 변경 전 관계 스냅샷을 갖는 기존 거래로 처리하고, 전환이 먼저 확정되면 이전 권한의 거래 생성 요청을 거부한다.

### 알림·감사·개인정보

- 신청·보완 요청·전환 예약·철회·거절·완료 시 변경 전·후 대표자의 확인된 연락처로 사건 상태와 필요한 행동만 알린다. 상대방의 전체 연락처, 사업자등록증 이미지, 심사 메모와 과거 거래 정보는 알림에 포함하지 않는다.
- 전환 완료 알림 실패는 확정된 권한 전환을 되돌리지 않는다. 내구성 있는 알림 작업을 제한적으로 재시도하고 최종 실패는 운영자 경보와 감사 가능한 수동 재발송 대상으로 남긴다.
- 사건 식별자, 매장, 변경 유형, 변경 전·후 관계 식별자, 제출 자료 버전·무결성 값, 공공 조회 결과, 양측 확인 시각, 심사 관리자, 결정 사유, 효력 시각, 권한 버전과 복구 작업 결과를 수정·삭제 불가능한 감사 이력으로 남긴다. 인증정보 원문·사업자등록증 원본 URL·불필요한 개인정보는 로그나 이벤트에 기록하지 않는다.
- 제출 증빙은 STORE-003·STORE-005의 비공개 암호화·최소 권한 기준을 따르며, 정확한 보관·파기 기간은 기존 PRIV-005 법률·개인정보 TODO를 따르고 이 정책에서 새 기간을 정하지 않는다.

### 결제·정산 및 예외

- 이 정책만으로 금액을 청구·환불·정산하거나 변경 전 거래의 정산 수취인을 바꾸지 않는다. 기존 거래의 결제·환불·정산은 거래 생성 당시 스냅샷과 PAY 도메인의 확정 정책 및 남은 팀/TODO 결정을 따른다.
- 공동대표·각자대표, 사업자 단위 과세와 본사·지점 그룹은 기존 STORE-003·STORE-013의 TODO 범위를 유지한다. 해당 결정이 나중에 확정되면 STORE-014의 증빙 요건과 변경 후 관계 생성에 그 결과를 입력하되, 과거 관계 불변 원칙은 유지한다.

### 관측 가능한 인수 조건

- 같은 매장에 동시 변경 신청을 보내도 활성 사건은 하나만 생기고 기존 대표자의 권한·공개 상태는 심사만으로 바뀌지 않는다.
- 유효한 양측 확인과 변경 후 STORE-003·STORE-005 검증 없이 전환이 확정되지 않으며, 충돌·외부 조회 장애는 `확인 보류` 또는 `보완 요청`으로 남는다.
- 전환 효력 시각 경합 테스트에서 변경 전·후 대표자가 동시에 활성 권한을 갖지 않고, 이전 권한의 전환 후 쓰기 요청은 모든 인스턴스에서 거부된다.
- 전환 전 거래의 관계·가격·정산 스냅샷과 감사 이력은 전환 후에도 바뀌지 않으며, 전환 후 거래만 새 관계 버전을 참조한다.
- 권한 부여 부분 실패를 주입하면 신규 거래가 차단되고 중앙 사건을 기준으로 중복 권한 없이 복구되며, 알림 장애가 권한 전환 결과를 되돌리지 않는다.
- 새 대표자는 처리할 미래 거래의 최소 정보만 조회할 수 있고 완료된 과거 거래의 개인정보·정산 자료와 제출 증빙 원본은 권한 없이 조회할 수 없다.
