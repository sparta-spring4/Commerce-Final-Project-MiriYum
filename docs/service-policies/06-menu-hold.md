# 메뉴 홀드 정책

> 문서 상태: 세부 정책 확정
> 정책 범위: `HOLD-001`~`HOLD-013`
> 최종 변경일: 2026-07-22

메뉴 수량 규칙, 방문 예약 수용량과의 결합 선점, 현장 판매, 품절, 취소 복구, 대체 메뉴 및 메뉴 대기를 다룬다. 방문 예약 수용량과 선택 메뉴 수량은 하나의 10분 선점 그룹으로 결합하며, 단독 픽업 홀드는 독립된 기존 흐름을 유지한다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 확정 | 핵심 | `HOLD-004` |
| 확정 | 필수 | `HOLD-001`~`HOLD-003`, `HOLD-006`~`HOLD-010`, `HOLD-012`, `HOLD-013` |
| 확정 | 권장 | `HOLD-005`, `HOLD-011` |

- `HOLD-004`: 방문 예약 수용량과 선택 메뉴 수량을 동일한 10분 선점 그룹에 결합하고 전부 확정하거나 전부 반환한다.
- `HOLD-007`: 메뉴 임시 선점은 10분이며 방문 예약 수용량과 결합할 때 하나의 만료 시각과 전부 확정·반환을 적용한다.

## HOLD-001 홀드 가능한 메뉴의 자격

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 해당 매장 운영 권한이 유효한 식당 대표자만 게시된 메뉴 버전을 `홀드 가능`으로 설정할 수 있다. 메뉴 관리 기능, 수량 원장, 제공 시간·수령 조건과 공개 설명이 준비된 게시 메뉴가 대상이며, 대표 메뉴 여부는 홀드 자격이 아니다.
- 식당 대표자는 현재 게시된 모든 메뉴를 한 번에 홀드 가능으로 설정할 수 있다. 일괄 설정도 메뉴별 자격·제공 시간·수량을 검증하며, 이후 개별 메뉴를 별도로 중지할 수 있다.
- 메뉴는 판매·노출 상태가 유효하고 현재 적용되는 홀드 정책 버전과 가용 수량이 있어야 신규 홀드를 받는다. 품절·판매 중지·숨김·매장 휴점·폐점에는 신규 생성을 차단한다.
- 로그인한 일반 사용자만 홀드를 생성한다. 플랫폼 운영자는 장애 복구 권한 없이 홀드를 대신 만들거나 재고를 임의로 늘리지 않는다. 모든 메뉴의 POS 수준 재고나 새 개인정보는 요구하지 않는다.

### 정상 흐름·변경·취소·만료

- 대표자는 `초안` 설정을 검증해 효력 시각이 있는 버전으로 게시한다. 사용자는 현재 메뉴·가격·홀드 정책·수량 버전을 확인하고 요청하며 최종 선점 직전에 자격을 다시 검증한다.
- 홀드 가능 설정 변경은 이후 요청에 적용하고 기존 활성·확정 홀드를 자동 취소·대체하지 않는다. 이행 위험은 `HOLD-009` 사건으로 처리한다.
- 게시 철회·판매 종료에는 신규 요청만 닫고, 활성 선점은 `HOLD-007` 만료 또는 유효한 취소·이행 전이로 정리한다. 메뉴를 삭제해 과거 거래 참조를 끊지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 설정 게시와 홀드 생성은 메뉴·정책·수량 버전 및 멱등 키를 조건부 검증한다. 캐시가 아닌 중앙 판매·재고 상태를 최종 근거로 사용한다.
- 원본 상태를 확인할 수 없으면 신규 홀드를 실패 폐쇄한다. 검색·캐시 갱신 실패는 중앙 게시 결과를 되돌리지 않고 재처리한다.
- 일반 설정 변경은 전체 사용자에게 알리지 않고 기존 홀드 영향만 당사자에게 알린다. 행위자·메뉴/설정 버전·효력·영향 홀드·결과를 감사하며 연락처·결제정보를 재고 로그에 넣지 않는다.

### 관측 가능한 인수 조건

- 품절·중지·숨김·휴점 메뉴와 수량 원장이 없는 메뉴에는 모든 인스턴스가 신규 홀드를 거부한다.
- 설정 전환과 요청 경합에도 홀드는 정확히 한 메뉴·정책·수량 버전을 참조하고 재전송이 중복 홀드를 만들지 않는다.
- 설정 철회가 기존 활성 홀드를 자동 취소하지 않고 영향 사건·알림·감사가 한 번만 생성된다.

## HOLD-002 시간대별 수량 규칙

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 메뉴별 제공 날짜·시간 구간과 각 구간의 판매 가능 수량을 불변 정책 버전으로 게시한다. 구체적 시간 폭·수량은 매장 생산·운영 설정이며 이 정책에서 고정하지 않는다.
- 하나의 물리 수량이 겹치는 시간 구간이나 여러 버킷에 중복 산입되지 않게 공급 단위와 버킷 관계를 명시한다. 자정을 넘는 구간은 매장 시간대와 종료 날짜를 함께 저장한다.
- 사용자 요청은 정확히 하나의 제공 구간을 선택하고 수령·방문 시간이 그 구간과 영업 상태에 유효해야 한다. 오래된 화면의 수량은 확보 약속이 아니다.

### 정상 흐름·변경·취소·만료

- 조회 → 구간·메뉴·수량 정책 검증 → 해당 버킷 원자 선점 → 선택된 확정 흐름 순으로 처리한다. 홀드에는 공급·버킷·정책 버전과 수량을 고정한다.
- 새 정책은 효력 이후 신규 홀드에 적용하고 기존 홀드의 구간·수량을 소급 변경하지 않는다. 기존 이행이 불가능하면 `HOLD-009`로 처리한다.
- 종료 구간에는 신규·증량을 허용하지 않고 유효한 취소·만료·이행 결과로 수량을 정리한다. 스케줄러가 늦어도 요청 중앙 시각으로 종료를 판정한다.

### 동시성·실패·복구·알림·감사·개인정보

- 버킷 가용량은 0 미만이 되지 않는 조건부 원자 갱신과 홀드 멱등 키로 차감한다. 여러 인스턴스가 같은 마지막 수량을 요청해도 허용량만 성공한다.
- 일부 차감·홀드 생성 실패는 같은 사건으로 보상 복구하고 고아 차감은 원장 대사로 격리한다. 중앙 시간·수량 원본 장애에는 실패 폐쇄한다.
- 구간 변경의 기존 홀드 영향만 알리고, 공급·버킷·정책·홀드·차감/복구 원장과 상관관계 ID를 감사한다. 개인정보는 필요하지 않다.

### 관측 가능한 인수 조건

- 겹치는 시간 구간과 자정 경계가 버전 정책대로 판정되고 동일 물리 수량이 두 버킷에서 판매되지 않는다.
- 가용 수량을 넘는 병렬 요청에도 성공 합계가 버킷 수량을 넘지 않고 음수 재고가 없다.
- 정책 변경·작업 지연이 기존 홀드 스냅샷을 바꾸지 않으며 부분 실패 뒤 원장과 활성 홀드가 대사된다.

## HOLD-003 사용자별 최소·최대 수량

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 메뉴·제공 구간별 1회 최소 수량과 사용자별 활성·확정 홀드 합산 최대 수량을 버전된 공개 정책으로 설정한다. 구체 수치는 메뉴 생산 데이터에 맞춘 설정이며 이 행에서 정하지 않는다.
- 확인된 일반 사용자 계정 기준으로 계산하고 같은 계정의 중복 요청·변경을 합산한다. 동행자 연락처, 기기 지문이나 위치 이력을 추가 수집해 우회 계정을 추정하지 않는다.
- 최소·최대 정책은 모든 일반 사용자에게 동일하게 적용하며 운영자가 특정 사용자만 우회 승인하지 못한다. 별도 상품·구독 우대나 공정성 배분을 만들지 않는다.

### 정상 흐름·변경·취소·만료

- 생성·증량 전 현재 사용자-메뉴-구간의 활성·확정 수량과 새 요청을 합산 검증하고 성공과 함께 중앙 수량 관계를 점유한다.
- 감량은 최소 수량 이상이며 아직 변경 가능한 상태일 때만 원자 반영한다. 유효한 취소·만료·이행으로 제한 점유를 한 번 해제한다.
- 정책 변경은 이후 요청·변경에 적용하고 기존 유효 홀드를 자동 취소하지 않는다. 기존 보유량이 새 최대를 넘으면 신규 증량만 차단한다.

### 동시성·실패·복구·알림·감사·개인정보

- 사용자별 중앙 합산 값과 재고 차감을 같은 조건부 처리 경계로 갱신한다. 다중 인스턴스 요청이 각각 최대 검증을 통과하지 못하게 한다.
- 합산 저장소 장애에는 신규·증량을 실패 폐쇄하고 기존 홀드는 유지한다. 원장 대사 전에는 로컬 계산으로 우회하지 않는다.
- 제한 차단에는 적용 정책과 자신의 보유 수량만 안내한다. 사용자·메뉴·구간·정책 버전·점유/해제와 복구를 감사하되 연락처·동행자 정보는 기록하지 않는다.

### 관측 가능한 인수 조건

- 최소 미만과 사용자별 최대 초과 요청이 수량을 차감하지 않고, 동시 요청 합계도 현재 최대를 넘지 않는다.
- 정책 축소가 기존 홀드를 취소하지 않으며 취소·만료 뒤 합산 값과 실제 활성 수량이 일치한다.
- 권한 있는 운영자도 사용자별 예외 수량을 직접 입력할 수 없고 추가 개인 식별정보 없이 판정된다.

## HOLD-004 방문 예약 수용량과 메뉴 홀드 결합

> 정책 상태: 확정

### 확정 규칙·상태·흐름

- 방문 예약에서 메뉴를 선택하면 `RES-006`의 예약 수용량과 모든 선택 메뉴 수량을 하나의 선점 그룹·소유자·만료 시각에 결합한다. 그룹의 유효시간은 최초 생성 시점부터 10분이며 메뉴 추가·결제 재시도로 연장하지 않는다.
- 메뉴를 건너뛰면 예약금 결제 또는 결제 불필요 판정 뒤 예약 수용량만 확정한다. 메뉴를 선택하면 같은 판정 뒤 예약 수용량과 모든 메뉴 수량을 함께 확정한다.
- 사용자 포기·명확한 실패·10분 만료·최종 확정 불가에는 그룹에 포함된 예약 수용량과 메뉴 수량을 모두 반환한다. 일부만 확정하거나 일부만 판매 가능으로 복구하지 않는다.
- 결제 결과가 불명확하면 그룹을 `결제 확인 중`으로 격리하고 PG 조회·검증된 Webhook으로 판정할 때까지 확정이나 재판매로 추정하지 않는다.

### 동시성·실패·복구·감사

- 선점 그룹 ID, 예약·메뉴 원장 버전과 멱등 키로 모든 자원 변경을 조건부 처리한다. 일부 선점 실패는 이미 확보한 자원을 같은 그룹의 보상 작업으로 반환한다.
- 결제 성공 후 자원 확정 실패는 멱등 재시도하고 최종 확정이 불가능하면 `PAY-007`에 따라 승인 취소·전액 환불로 수렴한다.
- 그룹 생성·증감·결제 판정·확정·반환과 각 원장 결과를 감사하되 결제수단 원문이나 추가 개인정보를 수량 사건에 넣지 않는다.

### 관측 가능한 인수 조건

- 예약 수용량과 복수 메뉴를 함께 요청해도 하나의 10분 만료 시각과 하나의 최종 결과만 가진다.
- 자원·결제·만료 경합과 작업 재시작 뒤 그룹이 전부 확정 또는 전부 반환으로 수렴하고 부분 확정·이중 판매가 없다.
- 단독 픽업 홀드는 방문 예약 수용량을 만들거나 결합하지 않고 `HOLD-005`의 독립 흐름을 유지한다.

## HOLD-005 단독 픽업 홀드 지원

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자가 메뉴별로 `단독 픽업 홀드`를 명시적으로 활성화하고 픽업 제공 구간·장소·수령 조건을 게시한 경우, 로그인 사용자는 방문 예약 없이 메뉴만 홀드할 수 있다.
- 단독 홀드는 독립 거래·자원 식별자를 가지며 방문 예약의 생성·확정·취소를 암묵적으로 만들지 않는다. 방문 예약 흐름에서는 `HOLD-004`의 결합 선점을 적용하지만 단독 픽업에는 예약 수용량을 결합하지 않는다.
- 메뉴·수량·영업·픽업 구간이 모두 유효해야 하며 매장은 사용자에게 공개하지 않을 내부 준비 메모나 불필요한 신원정보를 요구하지 않는다.

### 정상 흐름·변경·취소·만료

- 메뉴·구간 선택 → 수량·사용자 제한 검증 → 임시 선점 → 필요한 확정 단계 → `픽업 대기` → `수령 완료` 순으로 처리한다. 중간 선점을 수령 보장처럼 표시하지 않는다.

- 반환은 `HOLD-008`의 원래 수량 풀과 메뉴·제공 구간에 귀속한다. 판매 종료·품절 정책 때문에 재판매할 수 없어도 원장상 반환과 공개 가용 여부를 구분한다.
- 홀드 생명주기의 `취소`·`거절`·`만료` 종결 상태는 바꾸지 않고, 별도 수량 반환 사건에만 `수량 복구 중`, `수량 복구 완료`, `수량 복구 실패` 하위 상태를 둔다. 복구 상태가 홀드를 재활성화하거나 `픽업 대기` 같은 거래 상태로 되돌리지 않는다.

### 정상 흐름·변경·취소·만료

- 취소 권한·정책 검증 → 홀드 종결 상태 확정 → 별도 반환 사건 `수량 복구 중` → 수량 반환 원장 기록 → `수량 복구 완료`와 처분 확정 → 알림 순으로 처리한다.

### 동시성·실패·복구·알림·감사·개인정보

- 반환 원장 ID별 처분 사건과 활성 승계 제안을 각각 하나만 허용하고 같은 수량 버전을 조건부 갱신한다. 제안 수락·만료·원래 취소의 재전송·경합은 `TRANSFER-004`·`TRANSFER-005`·`TRANSFER-009`의 멱등·순서 규칙으로 하나의 처분에 수렴한다.

## HOLD-012 메뉴 대기 자동 종결

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 메뉴 대기 항목은 로그인 사용자, 메뉴·제공 구간과 현재 대기 상태를 연결하며 홀드나 구매 권리를 그 자체로 보장하지 않는다. 자동 승계 활성 여부는 해당 반환 자원의 정책 입력이며 후보·우선순위는 `TRANSFER-002` 결과만 사용한다.
- 대기는 사용자의 명시적 취소, 성공한 홀드 전환, 메뉴 판매 종료·영구 중지, 제공 구간 종료 또는 계정 거래 권한 상실 때 종결한다. 구체적 시간 수치는 메뉴 정책 버전에서 가져온다.
- 식당 대표자는 자신의 메뉴 판매 상태를 변경할 수 있으나 개별 사용자의 대기를 임의 성공 처리하지 않는다. 운영자는 장애 복구 권한과 근거가 있을 때만 보정한다.

### 정상 흐름·변경·취소·만료

- 대기 생성 → `활성` → 반환 자원의 자동 승계가 활성이고 `TRANSFER-002`가 이 항목을 후보로 산출한 경우 `승계 제안 중(proposed)` → 사용자 수락과 원자 선점 성공 시 `승계 선점(claimed)` → 홀드 확정 시 `전환 완료`로 전이한다. 제안 유효기간이 지나면 `승계 제안 만료(expired)`를 남기고 다음 처분은 `TRANSFER-004`·`TRANSFER-005`·`TRANSFER-009` 결과를 따른다.
- 반환 자원의 자동 승계가 비활성이면 수량은 일반 가용으로 게시하고 특정 대기 항목에 제안을 만들지 않는다. 승계가 활성인 동안에는 `TRANSFER-002`가 정한 후보 정확히 한 명에게만 제안하고 같은 수량을 일반 가용으로 동시에 게시하지 않는다.
- 가용 알림만으로 대기를 `승계 선점`이나 `전환 완료`로 바꾸지 않고 실제 사용자 수락·수량 선점·홀드 성공을 순서대로 확인한다. 제안·응답 기한 수치는 `TRANSFER-004`, 중복 제안은 `TRANSFER-005`, 취소·수락 경합은 `TRANSFER-009` 결과를 소비하며 이 행에서 값을 정하지 않는다.
- 종결 상태는 재활성화하지 않는다. 메뉴 재판매·새 구간에는 사용자가 새 대기를 생성해야 하며 과거 순서·기회를 임의 복원하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 사용자·메뉴·구간 활성 대기 유일성, 상태 버전과 멱등 키로 중복 대기·종결을 막는다. 반환 원장 ID별 활성 `승계 제안 중`과 `승계 선점`을 하나만 허용하고, 제안·수락·만료·취소·홀드 성공 경합은 `TRANSFER-004`·`TRANSFER-005`·`TRANSFER-009`에 따른 조건부 전이 하나만 확정한다.
- 자동 종결 작업이 지연돼도 조회·전환 명령 시 중앙 메뉴·구간 상태로 만료를 판정한다. 알림 실패는 종결 상태를 되돌리지 않고 제한 재시도한다.
- 가용 안내·종결 사유를 사용자에게 알리고 대기·메뉴·구간·상태 전이·홀드 참조와 작업 결과를 감사한다. 다른 대기자의 수·신원은 노출하지 않는다.

### 관측 가능한 인수 조건

- 판매 종료·구간 종료·홀드 성공 후 대기가 한 번만 종결되고 중복 작업이 종결 알림을 중복 생성하지 않는다.
- 가용 알림 발송만으로 성공 종결되지 않으며 홀드 실패 시 정책이 허용하는 활성 상태를 유지한다.
- 반환 수량 하나에 일반 공개와 승계 제안이 동시에 존재하지 않고, 승계 활성 시 `TRANSFER-002` 후보 한 명만 `승계 제안 중`이 된다. 제안 수락·만료·대기 취소를 동시에 처리해도 `승계 선점`, `승계 제안 만료`, `취소` 중 중앙에 먼저 유효한 전이 하나만 남고 후속 처분은 지정된 TRANSFER 정책을 따른다.
- 종결 후 메뉴 재개가 과거 대기를 되살리지 않고 다른 대기자의 정보가 응답·이벤트에 나타나지 않는다.

## HOLD-013 수량 감사 로그와 대사

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건
