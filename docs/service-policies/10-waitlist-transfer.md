# 취소 자리·수량 자동 승계 정책

> 문서 상태: 세부 정책 확정
> 정책 범위: `TRANSFER-001`~`TRANSFER-009`
> 최종 변경일: 2026-07-23

승계 대기 등록, 우선순위, 제안 시간, 중복 제안, 결제 실패, 알림 공정성 및 동시성을 다룬다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 확정 | 핵심 | `TRANSFER-002` |
| 확정 | 필수 | `TRANSFER-001`, `TRANSFER-003`~`TRANSFER-006`, `TRANSFER-008`, `TRANSFER-009` |
| 확정 | 권장 | `TRANSFER-007` |

- `TRANSFER-002`: 반환된 인원·수량 이하를 요구하는 유효 대기 중 등록 순번이 가장 빠른 후보에게 제안한다. 예약 승계에서는 테이블·좌석 효율·예상 매출을 후보 선정에 사용하지 않는다.
- `TRANSFER-003`: `TRANSFER-002`에서 팀이 정한 우선순위만 기계적으로 반영한다. 독자적인 추첨·선착순·구독 우대 규칙을 발명하지 않는다.
- `TRANSFER-004`: 제안 응답 시간은 5분이고, 수락 뒤 예약·결제 완료 선점 시간은 10분이다.
- 아래 확정 세부는 `TRANSFER-002`가 산출한 후보·우선순위를 변경하지 않고, `TRANSFER-004`의 확정 시간과 안전한 경합 상태를 사용한다.

## 공통 자원 처분 불변식

- `RES-015`의 예약 점유 감소분과 `HOLD-010`의 복구 완료 수량은 반환 원장 ID별 중앙 처분 사건 하나로 연결한다. 예약 점유 감소분의 후보는 모든 관련 서비스 구간에서 필요한 인원·팀 수를 충족해야 한다. 자동 승계 비활성 또는 `TRANSFER-002`가 유효 후보 없음으로 산출한 때만 `일반 가용`으로 전이한다.
- 자동 승계가 활성이고 유효 후보가 있으면 정확히 한 후보의 `승계 제안 중(proposed)`으로 전이하고, 수락·원자 선점 성공은 `승계 선점(claimed)`, 유효기간 경과는 `승계 제안 만료(expired)`로 기록한다.
- 한 반환 사건에는 동시에 한 팀에게만 제안하고, 최종 전환은 한 팀까지만 허용한다. 그 팀의 전환이 완료되면 후보가 사용한 인원·수량을 제외한 잔여분은 `일반 가용`으로 분리 반환하며, 같은 반환 사건으로 다른 대기 후보에게 추가 승계하지 않는다.
- 최초 처분은 정확히 `복구 완료` → `일반 가용` 또는 `복구 완료` → `승계 제안 중` 중 하나다. `일반 가용`으로 공개된 자원은 같은 반환 처분에서 다시 회수해 승계 제안을 만들지 않으며, `승계 제안 중`·`승계 선점`인 자원은 일반 예약·판매에 게시하지 않는다. 제안, 선점, 만료·후속 처분은 같은 자원 버전을 조건부 갱신하여 한 상태만 유효하게 한다.

## TRANSFER-001 승계 대기 등록 조건

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 로그인한 일반 사용자만 공개된 매장·상품·회차/구간에 승계 대기를 등록한다. 원 거래와 동일한 인원·수량·자격·연령·중복 거래·결제 가능 조건을 서버가 검증하며, 대기 등록은 예약·홀드·구매나 우선권을 보장하지 않는다.
- 식당 대표자는 자신의 상품에 자동 승계 사용 여부와 등록 가능 범위를 게시할 수 있으나 개별 사용자를 임의 성공·우선 후보로 만들지 않는다. 플랫폼 운영자는 장애 복구 권한과 근거 없이 대기를 생성하지 않는다.
- 상태는 `활성`, `승계 제안 중`, `승계 선점`, `전환 완료`, `사용자 취소`, `자격 상실`, `상품 종료`로 구분한다. 사용자는 활성·제안 중 대기를 취소할 수 있고, 상품/구간 종료·계정 거래 권한 상실에는 종결하되 과거 기회를 자동 복원하지 않는다.

### 동시성·실패·알림·감사·개인정보

- 사용자-상품-회차/구간 활성 대기 유일성, 대기 버전과 멱등 키로 중복 등록·종결을 막는다. 자격 원본 장애에는 신규 등록을 실패 폐쇄하고 기존 대기는 성공으로 추정하지 않는다.
- 등록·취소·종결 작업 실패는 중앙 상태로 재시도하며 단일 인스턴스 목록이나 알림 구독 상태를 원본으로 쓰지 않는다.
- 등록·취소·종결과 현재 상태를 사용자에게 알리고 대기/정책 버전·자격 판정·행위자·사유를 감사한다. 다른 후보 수·신원, 동행자 연락처·기기 지문을 수집·노출하지 않는다.

### 관측 가능한 인수 조건

- 자격 미충족·중복 등록은 생성되지 않고 등록만으로 자원 선점·일반 공개 차단이 생기지 않는다.
- 병렬 등록·취소에도 활성 대기는 하나이고 작업 재시작 뒤 같은 중앙 상태로 수렴한다.
- 다른 후보 정보나 추가 개인정보 없이 등록 결과를 재현할 수 있다.

## TRANSFER-002 반환 인원·수량 이하 후보의 등록 순번 우선

> 정책 상태: 확정. 2026-07-23 팀 결정

### 확정 후보·순서·잔여분 규칙

- 예약 취소로 N명의 수용량이 반환되면 동일 매장·방문일·서비스 구간의 활성 예약 승계 대기 중 요청 인원이 N명 이하이고 원 거래 자격을 다시 충족하는 후보만 적격으로 판정한다. 메뉴 수량 반환도 동일하게 요청 수량이 반환 수량 이하인 유효 대기만 적격으로 판정한다.
- 적격 후보는 대기 등록 순번 오름차순으로 정렬하고 가장 빠른 후보 한 팀에게 먼저 제안한다. 순번이 같을 수 있는 구조에서는 대기 등록 시각, 대기 ID 오름차순을 차례로 최종 동점 기준으로 사용한다.
- 예약 후보 선정에는 물리적 테이블 크기·좌석 배치·좌석 효율·반환 인원과의 근접도·예상 매출을 사용하지 않는다. 따라서 4명의 수용량이 반환되고 1순위 유효 대기가 1명이면 1명 팀이 먼저 제안받는다.
- 요청 인원이 반환 수용량보다 많은 후보는 해당 반환 사건에서만 건너뛰며 대기 등록·기존 순번을 잃지 않는다. 현재 후보가 명시적으로 거절하거나 제안이 만료되거나 자격 재검증·결제가 확정 실패하면 같은 정렬 결과의 다음 적격 후보에게 제안한다.
- 한 후보가 최종 전환되면 그 후보가 사용한 인원·수량을 제외한 잔여분은 일반 가용량으로 반환한다. 잔여분으로 같은 반환 사건의 다음 대기 팀에게 추가 제안하지 않는다.

### 동시성·실패·알림·감사

- 후보 결과에는 반환 원장 ID, 대상 구간·자원 버전, 반환 인원·수량, 정렬 기준과 정책 버전을 고정한다. 모든 인스턴스는 중앙 대기 원장과 같은 스냅샷을 사용하며 DB의 우연한 조회 순서나 알림 도착 순서를 사용하지 않는다.
- 후보 계산 중 대기·자격 원본을 확인할 수 없거나 입력 버전이 맞지 않으면 특정 후보에게 제안하지 않고 재검증한다. 이미 유효한 제안의 후보는 정책 변경이나 늦은 대기 등록으로 소급 교체하지 않는다.
- 후보 선택·부적격 건너뜀·재제안·전환 완료·잔여 일반 반환을 감사하되 다른 후보의 신원·요청 세부를 제안 수신자에게 노출하지 않는다.

### 관측 가능한 인수 조건

- 4명 수용량이 반환되고 대기 순서가 `1명 → 5명 → 3명`이면 1명 팀에게 먼저 제안하며, 1명 팀의 거절·만료 뒤에는 5명 팀을 이번 반환 사건에서 건너뛰고 3명 팀에게 제안한다.
- 1명 팀이 4명 반환 사건의 승계를 완료하면 남은 3명 수용량은 일반 예약 가능 수량으로 반환되고 같은 반환 사건의 다른 대기 팀에게 추가 제안되지 않는다.
- 테이블 크기·좌석 효율·예상 매출 입력을 바꾸어도 후보 결과는 달라지지 않으며, 여러 인스턴스가 동시에 계산해도 같은 적격 후보 순서와 활성 제안 하나로 수렴한다.

## TRANSFER-003 추첨·선착순·구독 혜택 적용 경계

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 후보 목록, 순서, 동점 처리, 추첨·선착순 여부와 모든 자격 가중치는 오직 `TRANSFER-002`의 버전된 결과를 입력으로 받는다. 이 정책은 자체 정렬·무작위 추첨·등록 시각 우선·구독자 우대·얼리 액세스를 생성하지 않는다.
- 유효한 `TRANSFER-002` 정책 버전이 없거나 유효 후보를 산출할 수 없으면 특정 사용자에게 제안하지 않고 공통 자원 처분 불변식에 따라 `일반 가용`으로 보낸다. 임시 코드 순서·DB 조회 순서·알림 성공 순서를 우선순위로 사용하지 않는다.
- 제안 사건에는 사용한 `TRANSFER-002` 정책 버전, 후보 산출 결과의 무결성 참조와 반환 원장 ID를 고정한다. 정책 변경은 이미 활성인 제안의 수신자를 소급 교체하지 않고 이후 처분 사건에만 적용한다.
- 동일 입력 재평가는 같은 후보 결과를 소비하고, 입력 누락·버전 불일치는 제안 생성을 실패 폐쇄하여 일반 공개와 제안이 겹치지 않게 처분 상태를 원자 결정한다.
- 사용자에게 자신의 제안과 적용 정책 버전만 표시하고 타 후보·구독 상태 같은 개인정보를 노출하지 않는다. 후보 결과 수신·제안 선택·누락·실패를 감사한다.

### 관측 가능한 인수 조건

- 가능한 모든 `TRANSFER-002` 결과를 대입해도 정확히 그 후보 순서만 사용하며 별도 구독·추첨·선착순 가중치가 없다.
- 팀 결정 미확정·후보 없음에는 특정 제안이 생성되지 않고 자원은 일반 가용 또는 하나의 유효 처분으로만 남는다.
- 정책 변경·중복 평가·다중 인스턴스 경합에도 같은 반환 원장에 활성 제안은 하나이고 기존 수신자가 소급 변경되지 않는다.

## TRANSFER-004 승계 제안 응답·완료 선점 시간

> 정책 상태: 확정

### 확정 규칙·행위자·상태·흐름

- 유효 후보에게 보낸 승계 제안의 응답 시간은 중앙 시각 기준 5분이다. 제안에는 시작·응답 만료 시각, 후보·반환 원장·자원·정책 버전과 멱등 키를 고정하며 알림 도착·열람·재시도는 시간을 연장하지 않는다.
- 후보가 5분 안에 수락하면 자원을 `승계 선점`으로 원자 전이하고, 예약·결제 완료를 위한 선점 시간은 수락 시점부터 10분이다. 거절·자격 재검증 실패·검증된 결제 실패와 유효 만료만 후속 후보 또는 일반 가용 처분의 입력이 된다.
- 응답 만료와 수락이 경합하면 같은 제안/자원 버전의 중앙 조건부 전이 하나만 성공한다. 결제 결과가 불명확하면 10분 선점 종료 또는 다음 후보 처분을 진행하지 않고 `결과 대사/복구 필요`로 격리한다.

### 동시성·실패·복구·알림·감사·개인정보

- 제안 생성 → 5분 응답 → 수락 시 10분 선점 → 예약·결제 확정 또는 유효 종결 → 후속 처분 순으로 처리한다. 응답·수락·만료·결제 웹훅·재전송은 중앙 시각과 제안/자원/결제 버전을 매 명령 시 재검사한다.
- 다중 인스턴스에서 수락·응답 만료가 겹치면 승자 하나만 최신 상태를 확정하고 패자는 충돌 결과를 반환한다. 결제 결과 불명확·부분 장애는 공개·다음 후보·이중 청구를 막은 채 PAY·자원 원장 대사 후에만 한 후속 처분으로 수렴한다.
- 후보·매장에는 자기 제안의 응답/선점 만료와 결과만 알린다. 반환 원장·후보·정책 버전·기준 시각·상태 전이·결제 참조·멱등 키·대사 결과를 감사하되 다른 후보 정보와 결제수단 원문은 기록·노출하지 않는다.

### 관측 가능한 인수 조건

- 제안은 정확히 5분 경계에서 만료되고 경계 직전 수락·경계 직후 수락이 중앙 시각으로 일관되게 판정되며 알림 재시도로 시간이 늘어나지 않는다.
- 수락 뒤 예약·결제 완료는 10분 선점 안에서만 가능하고, 다중 인스턴스의 수락·응답 만료 경쟁에도 `승계 선점` 또는 `승계 제안 만료` 하나만 남는다.
- 결제 웹훅 결과가 불명확한 상태에서 10분 만료·다음 후보 선택을 병렬 실행해도 자원이 공개·재제안되지 않고 대사 뒤 정확히 한 후속 처분과 감사 기록으로 수렴한다.

## TRANSFER-005 중복 제안 방지

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 반환 원장 ID별 활성 제안은 정확히 하나이며 제안은 후보 대기 ID·자원 버전·`TRANSFER-002` 결과 버전과 연결한다. 같은 예약 점유 감소분 또는 메뉴 수량을 여러 후보에게 동시에 제안하거나 일반 가용으로 동시에 게시하지 않는다.
- 제안 생성은 아직 공개되지 않은 `복구 완료` 자원만 `승계 제안 중`으로 바꾸는 조건부 원자 전이와 함께 확정한다. `일반 가용`에서 `승계 제안 중`으로 가는 전이는 금지하며, 이미 공개된 자원을 회수해 제안하지 않는다. 중복 명령은 기존 제안을 반환하고 다른 후보·다른 요청 지문은 충돌로 거부한다.
- 만료·거절·자격 상실 뒤 후속 후보 재제안 또는 일반 공개는 이전 제안을 종결하고 같은 처분 사건의 새 버전으로만 진행한다. `TRANSFER-004` 시간 숫자는 사용하되 이 행에서 정하지 않는다.
- 제안·자원 버전·멱등 키·DB 유일성 제약으로 다중 인스턴스 중 하나만 성공한다. 부분 실패에는 자원을 공개하지 않은 채 `복구 필요`로 격리하고 대사가 활성 제안·자원 처분을 수렴시킨다.
- 제안·종결·재제안 결과를 해당 후보와 매장에 알리고 반환 원장·후보·정책 버전·전이·실패를 감사한다. 다른 후보의 개인정보는 노출하지 않는다.

### 관측 가능한 인수 조건

- 같은 반환 자원에 병렬 후보 선택을 보내도 활성 제안 하나만 생기고 공개 가용성과 공존하지 않는다.
- `일반 가용`으로 전이된 자원에는 이후 후보가 생겨도 승계 제안이 생성되지 않고 공개 재고를 회수하지 않는다.
- 만료·재제안·응답 유실 뒤에도 각 제안 버전은 한 후보에만 연결되고 중복 알림·선점이 없다.
- 부분 장애·작업 재시작 후 자원 처분과 활성 제안이 대사되며 고아 공개·고아 제안이 남지 않는다.

## TRANSFER-006 응답·결제 실패 때 다음 후보

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 결제 결과 불명확 중 자원이 다음 후보나 공개 판매로 넘어가지 않고 대사 뒤 정확히 한 후속 처분으로 수렴한다.

## TRANSFER-007 여러 대기 자동 정리

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 정리 대상 규칙은 거래 유형·시간 구간·수량을 사용하는 버전된 중앙 정책이며 동행자 연락처나 기기 지문으로 동일인을 추정하지 않는다. 사용자는 종결 전후 대상 목록을 확인하고 오판 이의를 제출할 수 있다.
- 원 거래 확정 사건 ID·대기 버전·정리 멱등 키로 중복 종결을 막고, 일부 실패에는 성공한 종결을 되돌리지 않은 채 남은 대기만 재처리한다. 원 거래가 보정 취소돼도 과거 대기를 자동 복원하지 않고 새 등록을 안내한다.
- 종결 대기와 이유를 사용자에게 알리고 원 거래·겹침 판정·대기 전후·작업 결과를 감사한다. 다른 후보·거래 정보는 노출하지 않는다.

### 관측 가능한 인수 조건

- 최종 확정 전에는 다른 대기가 종결되지 않고 확정 후 양립 불가한 동일 사용자 대기만 한 번 종결된다.
- 작업 중단·재실행 뒤 이미 종결된 대기가 중복 처리되지 않고 남은 대상이 수렴한다.
- 다른 사용자·매장·비겹침 대기가 영향받지 않으며 추가 식별정보를 수집하지 않는다.

- 중복 작업이 제안 시간을 연장하거나 여러 활성 제안을 만들지 않고 앱 내 상태는 중앙 원본과 일치한다.
- 알림 실패를 이유로 독자적 우대·유료 채널·추가 개인정보 수집이 생기지 않는다.

## TRANSFER-009 승계와 취소의 동시성

> 정책 상태: 확정

### 확정 규칙·흐름·복구

- 원 예약·홀드의 유효한 종결과 반환 원장 복구가 확정된 뒤에만 처분 사건을 만든다. 원 취소 재전송, 제안 수락, 제안 만료, 대기 취소, 일반 예약/판매는 같은 반환 원장 ID·자원 버전을 조건부 갱신한다.
- 최초 허용 전이는 정확히 `복구 완료` → `일반 가용` 또는 `복구 완료` → `승계 제안 중` 중 하나다. `일반 가용` → `승계 제안 중` 전이는 금지하고 공개 후 같은 반환 처분에서 자원을 회수하지 않는다. 제안 이후에는 `승계 제안 중` → `승계 선점`/`승계 제안 만료`/`후속 처분`만 허용하며 종결 상태를 과거 이벤트가 되돌리지 않는다. `승계 선점` 후 원 거래 취소를 취소하더라도 자원을 원 소유자에게 자동 복원하지 않는다.
- 수락과 만료가 경합하면 중앙에서 먼저 유효한 조건부 전이 하나만 성공하고 패자는 최신 상태를 반환한다. 수락 성공 뒤 결제·최종 확정 실패는 `TRANSFER-006` 보상 흐름을 거치기 전 공개하지 않는다.
- 처분 멱등 키, 원장 유일성 제약과 Outbox로 상태·후속 이벤트를 함께 기록한다. 결과 불명확·부분 실패는 `복구 필요`로 격리하고 RES/HOLD 원장 및 PAY 상태와 대사하며 로컬 락이나 단일 스케줄러에 의존하지 않는다.
- 원 취소 당사자, 제안 후보와 매장에 각자 필요한 결과만 알리고 원장 전후·경합 명령·승자 전이·복구를 감사한다. 서로의 신원·대기 순서를 노출하지 않는다.

### 관측 가능한 인수 조건

- 원 취소 재전송·수락·만료·공개 판매를 병렬 실행해도 자원은 일반 가용, 승계 제안 중, 승계 선점, 승계 제안 만료 중 하나의 중앙 처분만 가진다.
- 활성 제안/선점과 공개 가용이 동시에 존재하지 않고 응답 유실·작업 재시작 뒤 원장·제안·예약/홀드 상태가 대사된다.
- 공개된 `일반 가용` 자원은 늦게 도착한 후보 선택·재시도·역순 이벤트로 `승계 제안 중`에 전이하거나 회수되지 않는다.
- `TRANSFER-004`의 5분 응답·10분 수락 후 선점 시간을 소비하며 원 취소·금전 책임을 이 행에서 바꾸지 않는다.
