# 현장·원격 웨이팅 정책

> **1차 MVP 경계:** 로그인 사용자의 현장·원격 웨이팅 등록, 공통 3km 반경 검증, 같은 사용자·매장의 중복 등록 방지, 기본 FIFO 순번, 순번·예상 대기 정보 조회, 매장별 웨이팅 기능 온·오프와 자동·수동·일시중지 접수, 마감 전 일괄 종료, 같은 매장·영업 구간의 예약과 웨이팅 조건부 전환, 앞선 팀 두 팀 이하의 입장 임박 알림, 실제 입장 호출 한 번과 3분 도착 제한, 취소·입장 완료·미응답 처리 및 SSE 상태 전달을 제공한다. 매장별 1km~5km 반경 설정, 일행 계정 연결·대표자 변경, 서로 다른 매장의 동시 웨이팅 한도, 좌석 조건별 호환 팀 선택, 순번 미루기, 웨이팅 등록용 현장 증명 QR, 반복 미응답 자동 제재와 전체 다중 인스턴스 복구 자동화는 후속 구현 범위다. 웨이팅 보증금은 후속 구현이 아니라 현재 미도입으로 확정된 정책이다.

> 문서 상태: 세부 정책 확정(기존 확정·미래 TODO 경계 보존)
> 정책 범위: `WAIT-001`~`WAIT-017`
> 최종 변경일: 2026-07-23

## 1. 목적

사용자가 매장 근처에서 원격 웨이팅을 등록하고 일행과 대기 상태를 공유할 수 있게 하면서, 같은 일행이 여러 계정으로 중복 등록해 허수 대기 팀을 만드는 문제를 줄인다. 순번·미루기 규칙은 `WAIT-009`로 확정하고, 입장 임박 안내와 실제 호출 뒤 도착 제한을 분리하며 웨이팅 보증금은 도입하지 않는다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 확정 | 핵심 | `WAIT-009`, `WAIT-015` |
| 확정 | 필수 | `WAIT-001`~`WAIT-008`, `WAIT-010`~`WAIT-014`, `WAIT-016`, `WAIT-017` |

- `WAIT-015`: 현재 서비스와 최종 프로젝트 범위에서는 웨이팅 보증금을 도입하지 않는다.
- `WAIT-003`: 플랫폼 기본 반경은 3km이고 매장은 1km~5km 범위에서 설정한다.
- `WAIT-009`: 매장이 빈 테이블을 선택하면 시스템이 호환되는 활성 팀 중 등록 시각이 가장 이른 팀만 호출 대상으로 허용한다.
- `WAIT-010`·`WAIT-011`: 앞선 호환 팀이 두 팀 이하가 되면 입장 임박 알림을 한 번 보내고, 실제 입장 호출은 한 번만 하며 호출 뒤 3분 안에 도착해야 한다.
- `WAIT-001`: 현재 현장·원격 웨이팅은 모두 로그인한 일반 사용자에게만 허용한다. 향후 비회원 현장 예외는 채택하지 않은 별도 미래 `TODO`이며 원격 웨이팅으로 확대하지 않는다.

| `WAIT-017` | DB 원본, 중복 방지, 예약·웨이팅 조건부 전환, SSE 재연결 후 REST 복구 | Outbox 재생·원장 대사·전체 다중 인스턴스 자동 복구 |

- `후속 구현`은 정책 폐기나 `TODO` 전환이 아니다. 해당 정책은 확정 상태로 보존하되 1차 MVP의 UI·API·권한과 상태 진입 경로를 제공하지 않는다.
- 부분 적용 정책은 MVP 전용 두 번째 제품 규칙을 만들지 않는다. MVP가 제공하는 하위 범위에서도 아래 확정 정책의 권한·공정성·개인정보·멱등성 경계를 지킨다.

## 2. 기존 확정 정책

### WAIT-002 대표자 위치 검증

- 웨이팅 대표자는 매장이 허용한 반경 안에 있어야 신규 대기 팀을 만들 수 있다.
- 위치 정보는 등록 가능 여부 확인에 필요한 최소 범위로 사용한다.
- 정확한 원본 좌표는 불필요하게 장기 보관하지 않는다.

### WAIT-005 일행 초대와 거리 제한

- 대표자는 일행 참여 링크 또는 초대 코드를 공유할 수 있다.
- 초대받은 일행은 거리 제한 없이 기존 대기 팀에 참여할 수 있다.
- 일행 참여는 새로운 대기 팀을 만들지 않는다.
- 참여한 일행은 같은 순번과 예상 대기 시간을 확인한다.

## WAIT-001 현장·원격 웨이팅 지원 방식

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 현재 현장·원격 웨이팅은 모두 로그인한 활성 일반 사용자만 생성·참여할 수 있다. 식당 대표자와 플랫폼 운영자는 사용자 계정을 대신해 대기 팀을 만들지 않는다.
- 원격 생성은 `WAIT-002` 위치 검증, 현장 생성도 동일 원칙 또는 `WAIT-004`의 안전한 현장 증명 경로를 통과해야 한다. 매장의 웨이팅 기능·영업 상태·등록 접수 상태가 유효해야 한다.
- 비회원은 공개 대기 안내만 조회할 수 있고 상태를 만들거나 변경하지 못한다. 휴대전화 인증 비회원 현장 예외는 아래 미래 `TODO`가 채택되기 전에는 사용할 수 없다.

### 정상 흐름·변경·취소·만료

- 서버는 인증·매장 상태·위치·중복·동시 제한을 검증하고 하나의 대기 팀을 생성한 뒤 대표자를 첫 구성원으로 연결한다. 현장·원격은 등록 출처만 기록하고 같은 상태 기계를 사용한다.
- 사용자는 자신의 활성 팀을 취소할 수 있고 일행 변경은 `WAIT-006`, 호출·미응답은 `WAIT-011`·`WAIT-012`, 상태 전이는 `WAIT-016`을 따른다.
- 등록 실패·취소·만료 뒤 재등록의 순번은 `WAIT-009` 확정 규칙에 따라 계산하며 이 행에서 복원·우선권을 약속하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 생성 명령은 사용자·매장·요청 멱등 키와 중앙 중복 제약으로 처리한다. 여러 인스턴스의 현장·원격 동시 요청도 활성 팀 하나만 만든다.
- 인증·매장·위치·중복 원본을 확인할 수 없으면 신규 생성을 실패 폐쇄하고 부분 생성 팀을 제거한다. 응답 유실은 같은 멱등 키로 중앙 결과를 조회한다.
- 생성·취소·호출 결과를 대표자와 구성원에게 알리고 등록 출처, 정책 버전, 위치 판정 결과 범주와 상태 전이를 감사한다. 정확한 좌표·동행자 연락처를 로그에 남기지 않는다.

### 관측 가능한 인수 조건

- 로그아웃·정지 계정과 비회원은 현장·원격 팀을 만들거나 참여할 수 없고 공개 조회만 가능하다.
- 같은 사용자의 현장·원격 동시 제출과 응답 유실 재시도에도 활성 팀은 하나다.
- 원격 위치 실패와 매장 기능 중지에는 생성이 실패 폐쇄되며 미래 비회원 예외가 현재 API로 우회되지 않는다.

## WAIT-003 등록 반경

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 플랫폼 기본 등록 반경은 매장 기준 3km다. 식당 대표자는 자신의 매장에 한해 1km 이상 5km 이하의 값을 버전된 운영 설정으로 게시할 수 있으며, 이 범위 밖 값이나 운영자·사용자의 직접 입력은 거부한다.
- 신규 원격 등록은 중앙 시각에 적용 중인 반경 버전, 매장 기준점과 대표자의 현재 위치 정확도를 함께 검증한다. 위치 원문은 반경 판정에 필요한 최소 처리 범위에서만 사용하고 장기 저장하지 않는다.
- GPS 정확도 반경이 적용 반경보다 나쁘거나 위치를 신뢰할 수 없으면 즉시 탈락·취소하지 않는다. 대표자에게 재측정을 안내하고, 재측정도 불가하면 이미 확정된 매장 현장 QR 또는 회전 코드 증명 경로를 안내한다. 이 경로는 원격 반경을 우회하는 수단이 아니다.

### 정상 흐름·동시성·실패·복구

- 위치·정확도 검증 성공 → 등록 검증 → 대기 팀 생성으로 진행한다. 재측정 또는 현장 증명 전에는 대기 팀·순번을 생성하지 않으며, 성공한 증명은 현재 반경 정책 버전과 별도 사건으로 남긴다.
- 반경 설정 변경·등록·현장 증명이 경합하면 중앙 정책 버전과 팀 멱등 키로 하나의 결과만 확정한다. 위치 원본·정확도 원본을 확인할 수 없으면 신규 원격 등록을 실패 폐쇄하되 기존 활성 팀은 변경하지 않는다.
- 등록 거부·재측정·현장 증명 결과에 매장·반경 정책 버전·판정 시각·정확도 범주·멱등 키만 감사한다. 정확 좌표·이동 이력·동행자 정보를 감사 로그에 복제하지 않는다.

### 관측 가능한 인수 조건

- 매장 설정이 없으면 3km를 사용하고 1km·5km 경계 값은 게시·적용되며, 999m·5,001m 설정은 모든 인스턴스에서 거부된다.
- 적용 반경 안·밖 판정과 GPS 정확도 경계가 중앙 시각·같은 정책 버전으로 재현되고, 낮은 정확도는 즉시 탈락 대신 재측정 또는 현장 증명 안내를 반환한다.
- 반경 변경과 병렬 등록·현장 증명 재전송에도 팀 하나와 순번 사건 하나만 생성되며 정확 좌표가 응답·감사에 노출되지 않는다.

## WAIT-004 위치 권한 거부·GPS 오류 대응

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 원격 웨이팅은 신뢰 가능한 위치 판정이 없으면 생성하지 않는다. 위치 권한 거부, 정확도 부족, 시간 경과, 조작 의심을 반경 안으로 추정하거나 IP 위치로 대체하지 않는다.
- 로그인 사용자가 실제 현장에 있을 때만 매장이 통제하는 유효한 QR 또는 일회성·회전 가능한 매장 코드로 현장 존재를 증명할 수 있다. 이 대체 경로는 `WAIT-003` 반경 숫자를 우회하는 원격 등록 수단이 아니다.
- 식당 대표자는 자신의 매장 좌표 오류를 보정 요청할 수 있고 권한 있는 플랫폼 운영자가 공개 주소·기존 매장 자료로 검토해 새 좌표 버전을 승인한다. 운영자는 개별 사용자의 위치 검증을 수동 통과시키지 않는다.

### 정상 흐름·변경·취소·만료

- 위치 판정은 좌표 원문 대신 통과·정확도 부족·권한 거부·의심 등 결과 범주와 정책·매장 좌표 버전만 거래에 기록한다.
- 현장 토큰은 매장·용도·짧은 유효 구간에 묶고 한 등록에 한 번만 사용한다. 만료·재사용·다른 매장 토큰은 거부하며 구체적 수치는 버전된 보안 설정으로 운영한다.
- 매장 좌표 변경은 승인 효력 이후 요청에만 적용하고 과거 판정을 소급 변경하지 않는다. 실패한 사용자는 위치 재시도 또는 현장 대체 경로를 선택할 수 있다.

### 동시성·실패·복구·알림·감사·개인정보

- 토큰 소비와 대기 팀 생성은 하나의 멱등 처리 경계에서 확정한다. 여러 인스턴스가 같은 토큰을 받아도 한 번만 소비되고 생성 실패 시 재사용 여부는 중앙 사건 결과로 결정한다.
- 위치·토큰 검증 저장소 장애에는 실패 폐쇄한다. 좌표 보정 중에는 마지막 승인 버전을 사용하고 승인본이 없으면 신규 원격 등록을 중단한다.
- 사용자는 실패 범주와 현장 대체 방법만 안내받고 탐지 내부값은 노출하지 않는다. 좌표 원문은 판정 직후 불필요하게 보관하지 않으며 토큰·좌표 버전·보정 행위자·사유를 감사한다.

### 관측 가능한 인수 조건

- 위치 권한 거부·부정확·조작 의심 상태에서 원격 팀이 생성되지 않고 IP 추정으로 통과하지 않는다.
- 유효한 현장 토큰은 로그인 사용자에게 한 번만 성공하며 만료·재사용·타 매장 토큰은 모든 인스턴스에서 거부된다.
- 좌표 보정은 감사 가능한 승인 버전 이후에만 적용되고 개별 운영자 수동 우회와 정확한 위치 원문 장기 보관이 없다.

## WAIT-006 일행 합류·이탈·대표자 변경

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 대표자는 `WAIT-005` 초대 링크·코드로 로그인한 사용자를 같은 팀에 초대할 수 있고 구성원은 자신만 이탈할 수 있다. 대표자만 다른 구성원을 제거할 수 있으나 호출 후에는 현장 확인을 위해 변경을 제한할 수 있다.
- 대표자가 이탈하려면 팀을 취소하거나, 참여 중인 활성 구성원 한 명에게 대표자 역할을 제안하고 그 구성원이 수락해야 한다. 식당 대표자·운영자는 임의의 사용자를 대표자로 지정하지 않는다.

### 정상 흐름·변경·취소·만료

- 초대 수락은 팀 상태·초대 유효성·`WAIT-007`·`WAIT-008`을 검증하고 기존 팀에 구성원 관계만 추가한다. 새 대기 팀이나 별도 순번을 만들지 않는다.

- 팀 취소·체크인·만료 후에는 초대·이탈·이전을 허용하지 않는다. 구성원 이탈 뒤 재참여는 새 초대와 현재 검증을 거친다.

### 동시성·실패·복구·알림·감사·개인정보

- 팀 수정 번호·사용자 유일 관계·초대/이전 멱등 키로 중복 참여와 이중 대표자를 막는다. 대표자 전환과 팀 취소가 경합하면 하나만 확정한다.
- 인원 검증 또는 역할 전환 부분 실패 시 기존 구성과 대표자를 유지한다. 중앙 팀 상태를 읽지 못하면 변경을 실패 폐쇄한다.
- 합류·이탈·제거·대표자 제안과 결과를 팀 구성원에게 알리고 행위자·팀 버전·사유를 감사한다. 구성원의 연락처·정확한 위치는 서로에게 노출하지 않는다.

### 관측 가능한 인수 조건

- 초대 수락이 새 팀·새 순번을 만들지 않고 같은 사용자가 한 팀에 중복 연결되지 않는다.
- 대표자 수락과 팀 취소를 동시에 실행해도 활성 대표자는 정확히 한 명이거나 팀이 종결된다.
- 인원 부적합·장애 시 원래 구성과 대표자가 유지되고 구성원 연락처·위치 원문이 노출되지 않는다.

## WAIT-007 같은 매장 중복 등록 판정

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 같은 사용자가 대표자 또는 구성원으로 포함된 같은 매장의 활성 대기 팀은 하나만 허용한다. 대표자를 바꾸거나 초대 경로를 달리해도 같은 사용자 관계로 판정한다.

- 동행자 전화번호·기기 지문·정확한 위치를 모아 동일 일행을 추정하지 않는다. 확인된 계정 관계와 팀 구성만 판정 근거로 사용한다.

### 정상 흐름·변경·취소·만료

- 팀 생성·초대 수락·대표자 이전 직전에 사용자-매장 활성 관계를 검사하고 같은 처리 경계에서 관계를 점유한다.
- 기존 팀이 취소·체크인·만료 등 `WAIT-016` 종결 상태로 확정된 뒤 관계를 한 번 해제한다. 이후 재등록의 순번은 `WAIT-009`가 정한다.
- 오탐·고아 관계 신고는 운영자가 중앙 상태와 감사 이력으로 복구할 수 있으나 활성 팀을 임의 재정렬하거나 강제 확정하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 관계 확보 후 팀 생성 실패는 보상 해제하고, 불명확한 결과는 멱등 키로 조회한다. 대사 작업이 팀 없는 고아 관계를 격리·복구한다.
- 차단 사용자에게 자신의 기존 팀 최소 정보만 알리고 다른 구성원 정보는 숨긴다. 판정·충돌·해제·운영자 복구를 감사한다.

### 관측 가능한 인수 조건

- 같은 사용자가 대표·구성원 경로로 같은 매장에 동시에 요청해도 활성 팀 관계는 하나다.
- 팀 생성 중간 실패와 중복 작업 뒤 고아 관계가 남지 않고 종결 전에는 새 팀이 허용되지 않는다.
- 추가 개인 식별정보 없이 판정되고 차단 응답이 다른 구성원의 정보를 노출하지 않는다.

## WAIT-008 사용자별 동시 웨이팅 제한

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- `WAIT-007`에 따라 같은 사용자의 같은 매장 활성 웨이팅은 하나로 제한한다. 서로 다른 매장의 동시 활성 팀 수에는 플랫폼이 공개한 버전된 안전 한도를 적용하되 실제 한도 숫자는 운영·어뷰징 데이터로 조정하며 이 행에서 고정하지 않는다.
- 대표자·구성원 참여를 모두 활성 수에 포함하고, 식당 대표자나 운영자가 특정 사용자의 한도를 임의로 늘리지 못한다. 장애 복구로 생긴 고아 관계는 실제 활성 팀과 대사한 뒤에만 제외한다.
- 한도는 정상적인 여러 매장 탐색을 과도하게 막지 않는 어뷰징 방지 설정이며 대기 순번·우선권·보증금 혜택을 만들지 않는다.

### 정상 흐름·변경·취소·만료

- 팀 종결 확정 뒤 슬롯을 한 번 해제한다. 취소 뒤 재등록의 순번은 `WAIT-009`, 실제 호출 뒤 도착 제한 기준은 `WAIT-011` 결과를 따른다.
- 한도 정책 변경은 이후 요청부터 적용하고 이미 유효한 팀을 자동 취소하지 않는다. 초과 상태에서는 새 생성·합류만 차단한다.

### 동시성·실패·복구·알림·감사·개인정보

- 사용자별 중앙 카운터 또는 관계 집합을 조건부 원자 갱신하고 팀 상태와 대사한다. 다중 인스턴스 요청이 한도를 함께 통과하지 못하게 한다.
- 카운터 장애·불일치는 신규 생성을 실패 폐쇄하고 기존 팀은 유지한다. 대사로만 고친 뒤 재시도하며 로컬 카운터로 전환하지 않는다.
- 한도 차단·정책 변경을 앱에서 안내하고 정책 버전·활성 팀 식별자·점유·해제·대사 결과를 감사한다. 새 위치·기기 개인정보는 수집하지 않는다.

### 관측 가능한 인수 조건

- 같은 매장에는 하나만 허용되고 서로 다른 매장 동시 요청도 중앙의 현재 한도를 초과하지 않는다.
- 한도 축소가 기존 팀을 취소하지 않으며 종결과 생성 경합에도 카운터와 실제 활성 관계가 일치한다.
- 카운터 장애에는 신규 요청이 실패 폐쇄되고 복구 후 다중 인스턴스 대사로 같은 값을 본다.

## WAIT-009 순번·미루기·호출 대상 선택

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 플랫폼은 임의 팀을 자동 호출하지 않는다. 식당 대표자가 현재 빈 테이블을 선택해 호출을 실행할 수 있고, 시스템은 해당 테이블의 정원·좌석 조건을 충족하는 `대기 중` 활성 팀만 필터링한다.
- 호환 FIFO의 기본 전체 순서는 `(웨이팅 등록 시각, 원자적으로 발급된 단조 증가 queue sequence)` 오름차순으로 정의한다. 등록 시각이 같으면 queue sequence가 작은 팀이 선두이며, 필터된 팀 중 이 순서가 가장 앞선 팀만 호출 대상으로 허용한다. 단골·구독·예상 매출·결제액, 알림 수신 성공 등 상업적·비기록 신호는 일반 웨이팅 순번이나 호출 대상을 바꾸지 않는다.
- queue sequence는 대기 팀 생성과 같은 원자 처리에서 매장·대기열 범위의 중앙 원장이 발급한다. 생성 재시도는 같은 멱등 키로 기존 팀과 sequence를 반환하며, 발급된 sequence는 재사용하거나 상태 변경으로 수정하지 않는다. 종결 뒤 재등록은 새 팀과 새 sequence를 받고, `WAIT-009`의 1회 미루기만 이 불변 기본 순서에 기록되는 유일한 재배치 연산이다.
- 활성 팀 대표자는 등록 직후부터 호출 시작 직전까지 웨이팅 1건당 한 번만 미루기를 요청할 수 있다. 미루기는 같은 좌석 조건으로 입장 가능한 호환 팀 다섯 팀 뒤로 이동하며, 뒤에 호환 팀이 다섯 팀보다 적으면 현재 호환 대기열의 마지막으로 이동한다. 이후 새 등록 팀 때문에 다시 뒤로 밀리지 않는다.
- 식당 대표자와 운영자는 순번 값·정렬 키를 직접 쓰거나 기록되지 않은 사유로 앞 팀을 건너뛸 수 없다. 호출 시작 후에는 미루기를 허용하지 않는다.

### 정상 흐름·동시성·실패·복구

- 빈 테이블 선택 → 중앙 호환 팀 필터 → 등록 시각·queue sequence와 기록된 1회 미루기를 적용한 FIFO 대상 확정 → `대기 중`에서 `호출 중` 전이 순으로 처리한다. 호출은 선택한 테이블에 호환되는 최선두 팀이 없을 때 실패 폐쇄하며 다른 팀을 임의로 고르지 않는다.
- 미루기 요청은 팀의 미루기 사용 여부·대기열 버전·호환 팀 집합을 하나의 중앙 조건부 전이로 검증하고, 새 순번 원장 버전을 기록한다. 미루기와 호출이 경합하면 하나만 성공하며 패자는 최신 상태를 반환한다.
- 테이블 선택·필터 결과·등록 시각·queue sequence·미루기 전후 위치·호환 팀 수·정책/대기열 버전·멱등 키를 감사한다. 다른 팀의 신원·연락처·상업 정보는 사용자 응답이나 감사 조회 권한 밖에 노출하지 않는다.

### 관측 가능한 인수 조건

- 동일 테이블에 호환되는 활성 팀이 여럿이면 등록 시각이 가장 빠른 팀만 호출되고, 단골·구독·결제액을 바꿔도 호출 대상이 변하지 않는다.
- 여러 인스턴스가 같은 중앙 등록 시각에 호환 팀을 동시에 생성해도 서로 다른 단조 증가 queue sequence가 원자 발급되고 sequence가 작은 팀이 선두가 된다. 생성 재시도·이벤트 재생·상태 변경 뒤에도 모든 인스턴스가 같은 순서를 재현하며, 종결 팀의 sequence는 재등록 팀에 재사용되지 않는다.
- 한 팀은 호출 시작 전 한 번만 미룰 수 있으며 호환 팀 다섯 팀 뒤, 또는 호환 팀이 넷 이하이면 현재 호환 대기열 마지막으로 정확히 이동하고 이후 새 등록으로 추가 후퇴하지 않는다.
- 다중 인스턴스에서 미루기·호출·테이블 선택을 병렬 실행해도 하나의 대기열 버전과 하나의 승자 전이만 남으며 수동 순번 덮어쓰기·비기록 건너뛰기가 거부된다.

## WAIT-010 예상 대기 시간 계산·표시

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 예상 대기 시간은 현재 팀 상태, 매장이 제공한 운영 정보와 집계된 과거 처리시간을 입력으로 하는 추정 범위이며 예약 시각·입장 보장이 아니다. 모델·규칙 버전과 산출 시각을 함께 표시한다.
- 계산은 `WAIT-009`에서 정한 순번 결과를 입력으로만 사용하고 순서를 새로 결정하거나 조정하지 않는다. 입장 임박 판단은 같은 좌석 조건으로 앞선 활성 호환 팀 수가 두 팀 이하인지로 판정하며 `WAIT-011`의 실제 호출과 구분한다.
- 한 웨이팅 팀은 앞선 호환 팀이 처음 두 팀 이하가 된 때 입장 임박 안내 대상이 된다. 등록 시점부터 이미 두 팀 이하라면 등록 확정 직후 대상이 되며, 같은 웨이팅 팀에는 입장 임박 사건을 한 번만 만든다.
- 식당 대표자는 실제 운영 상태를 갱신할 수 있으나 특정 팀의 예상치를 직접 고쳐 우선권처럼 표시하지 않는다. 운영자는 장애 시 추정 표시를 중지할 수 있다.

### 정상 흐름·변경·취소·만료

- 등록 후와 팀 상태·운영 입력 변경 때 새 추정을 계산하고 구성원에게 동일한 값과 갱신 시각을 제공한다. 오래되거나 입력이 부족하면 숫자 대신 `산정 어려움`을 표시한다.
- 앞선 호환 팀이 두 팀 이하가 되면 중앙 원장 버전을 근거로 입장 임박 사건을 확정하고 `NOTI-004`와 실시간 전달 경로로 안내한다. 이 사건은 팀 상태·순번·호출 횟수를 변경하지 않는다.
- 알고리즘 변경은 버전 이후 산출에 적용하고 과거 표시를 소급 수정하지 않는다. 큰 변동도 확정 호출이나 순번 변경으로 표현하지 않는다.
- 팀이 호출·체크인·취소·만료되면 예상 대기 표시를 종료한다. 알림 지연 중에도 화면은 중앙 최신 상태를 우선한다.

### 동시성·실패·복구·알림·감사·개인정보

- 계산 작업은 팀·상태 버전으로 멱등 처리하고 구 상태 결과가 최신 값을 덮어쓰지 못하게 조건부 저장한다. 입장 임박 사건은 팀·알림 목적의 중앙 유일성 제약으로 한 번만 생성하며 여러 작업 인스턴스는 같은 입력 버전에 같은 표시 규칙을 적용한다.
- 계산·집계 장애는 팀 상태와 순번을 바꾸지 않고 추정만 숨긴다. 제한 재시도 후 다음 상태 갱신에서 다시 계산한다.
- 의미 있는 갱신은 기존 대기 상태 알림에 포함하되 과도한 개별 알림을 만들지 않는다. 입력 버전·결과 범위·실패를 감사하고 개인 이동경로나 다른 팀의 정보를 사용·노출하지 않는다.

### 관측 가능한 인수 조건

- 동일 팀 구성원은 같은 산출 시각·범위를 보고 화면에 비보장 추정임이 표시된다.
- 등록 당시 또는 순번 변경 뒤 앞선 호환 팀이 두 팀 이하가 된 팀에는 입장 임박 사건이 정확히 한 번 생성되고, 사건 생성 전후에 팀 상태와 순번은 바뀌지 않는다.
- 역순 계산 작업이 최신 추정을 덮어쓰지 않고 장애 시 순번·팀 상태는 유지된 채 `산정 어려움`으로 안전 저하된다.
- 알고리즘 버전 변경 전후 결과를 재현할 수 있고 개인 위치 이력이나 다른 팀 정보가 응답에 노출되지 않는다.

## WAIT-011 입장 호출 횟수·도착 제한 시간

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 실제 입장 호출은 팀당 한 번이며 도착 제한 시간은 중앙 호출 시각 기준 3분이다. 호출 사건에는 시작 시각, 도착 제한 시각, 정책 버전과 멱등 키를 고정하고 사용자 조회 응답에도 도착 제한 시각을 제공한다.
- `WAIT-010`의 입장 임박 안내, 알림의 전송·재시도·채널 변경은 실제 호출로 계산하지 않는다. 실제 호출은 매장 권한의 실행과 중앙 `대기 중` → `호출 중` 전이로만 시작하며, 시스템·운영자가 같은 팀에 두 번째 실제 호출을 만들 수 없다.
- 실제 호출 뒤 3분 안에 현장 도착이 확인되지 않으면 사용자가 취소한 것으로 기록하지 않고 `호출 실패`로 전이한다. 식당 대표자는 해당 호출 사건과 현장 상황을 확인한 뒤에만 `미응답 종료`를 확정하며, 확인 전에는 순번을 수동 재배치하거나 다른 팀을 호출하지 않는다.

### 정상 흐름·동시성·실패·복구

- 입장 임박 안내 → 실제 호출 한 번 → 3분 안에 현장 도착 확인 시 `도착 확인`, 미도착 시 `호출 실패` → 매장 확인된 `미응답 종료` 순으로 처리한다. 알림 실패는 호출 상태·도착 제한 시각을 되돌리거나 사용자 취소로 바꾸지 않는다.
- 호출·도착 확인·만료·매장 확인은 팀/호출 버전, 중앙 시각과 조건부 전이로 처리한다. 도착 제한 시각의 만료와 현장 도착 확인, 또는 호출 실패 확인이 경합하면 중앙에서 먼저 유효한 전이 하나만 성공한다.
- 입장 임박·호출·도착 확인·만료·확인·알림 재시도와 결과를 정책 버전·기준 시각·멱등 키·행위자와 함께 감사한다. 위치 원문·동행자 연락처·알림 콘텐츠 원문은 추가 수집하지 않는다.

### 관측 가능한 인수 조건

- 실제 호출은 정확히 한 번이고 호출 뒤 정확히 3분 동안 현장 도착 확인을 받으며, 만료 뒤에는 두 번째 실제 호출이나 자동 사용자 취소가 생성되지 않는다.
- 입장 임박·호출 알림 작업을 여러 번 재시도해도 실제 호출 횟수는 증가하지 않으며, 만료와 현장 도착 확인을 다중 인스턴스에서 병렬 처리해도 `도착 확인` 또는 `호출 실패` 중 하나만 남는다.
- `호출 실패`는 매장 확인 전 `미응답 종료`로 확정되지 않고, 확인·종결·경합 결과가 감사 가능하며 임의의 순번 복원이 없다.

## WAIT-012 대기 지속 확인·미응답·재등록

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 대기 지속 확인과 현장 도착 증거 제시는 활성 팀 대표자가 수행하며 구성원은 상태를 조회한다. 실제 호출은 `WAIT-011`의 한 번·도착 제한 3분을 적용하며, 정책 버전과 호출·도착 제한 시각을 해당 사건에 고정한다.
- 사용자의 명시적 취소와 호출 뒤 미도착 종결을 구분한다. 알림 전송 실패·앱 비활성·위치 미수신만으로 사용자가 도착했다고 보거나 즉시 제재하지 않는다.
- 재등록은 종결된 팀을 되살리는 것이 아니라 현재 자격·위치·중복 검증을 거친 새 팀이다. 순번·우선권은 `WAIT-009` 결과가 정하며 이 행이 복원 여부를 선택하지 않는다.

### 정상 흐름·변경·취소·만료

- 지속 확인 요청 → 사용자 응답 또는 명시적 취소 → 유효하면 대기 유지, 입장 임박 안내 뒤 실제 호출은 `호출 중` → 3분 안에 현장 도착 확인 시 체크인 진행, 도착 제한 시각 만료 시 `호출 실패` → 매장 확인 후 `미응답 종료`로 전이한다.
- 제한 시각은 중앙 시각으로 판정하고 스케줄러가 늦어도 명령 시점에 만료를 재검사한다. 현장 도착 확인과 만료가 경합하면 먼저 중앙 확정된 조건부 전이 하나만 성공한다.
- 미응답 판정 오류 신고는 `WAIT-014` 검토·복구를 사용한다. 운영자는 과거 팀을 직접 활성화하거나 순번을 수동 복원하지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 호출 사건·도착 확인·종결은 팀 버전과 멱등 키로 처리하고 작업 중복·순서 역전에도 한 결과만 만든다. 입장 임박·호출 알림 작업과 호출 상태를 분리해 발송 중복이 실제 호출 횟수를 늘리지 않게 한다.
- 알림 장애는 제한 재시도와 다른 사용 가능한 채널로 복구하되 상태 조회 경로를 유지한다. 중앙 상태 장애에는 미응답 종결을 실패 폐쇄하고 복구 후 다시 판정한다.
- 호출·응답·취소·미응답과 발송 결과를 구성원에게 알리고 정책 버전·기준 시각·행위자·전이를 감사한다. 새 위치나 동행자 연락처는 수집하지 않는다.

### 관측 가능한 인수 조건

- 중복 호출·도착 확인·만료 작업에도 호출 사건과 최종 전이는 하나이며 알림 재시도가 실제 호출 횟수를 증가시키지 않는다.
- 각 호출 사건은 적용 정책 버전과 고정된 단일 3분 도착 경계로 판정되며, 입장 임박·호출 알림 재시도는 호출 횟수를 바꾸지 않는다.
- 알림 장애만으로 즉시 미응답 종결·제재되지 않고 재등록이 과거 팀이나 순번을 임의 복원하지 않는다.

## WAIT-013 식당의 순번 조정 권한

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 팀의 체크인·호출·취소 등 사실 상태를 기록하고, 운영상 조정이 필요하면 사유 코드와 근거가 있는 조정 요청을 제출할 수 있다. 화면의 순번 숫자나 정렬 키를 직접 편집하지 못한다.
- 순번·호출 대상은 `WAIT-009`의 호환 팀 FIFO와 대표자 1회 미루기만으로 결정한다. 사실 상태 정정은 원장 재계산으로 처리하되 대표자·운영자가 순번 값을 직접 쓰거나 별도 사유로 앞 팀을 건너뛰게 하지 않는다.
- 플랫폼 운영자도 장애 복구 권한과 사건 근거 없이 순서를 변경하지 않는다. 복구는 원장 재계산이나 승인된 보정 사건으로 처리하고 임의 번호 입력을 금지한다.

### 정상 흐름·변경·취소·만료

- 테이블 선택·호출 또는 대표자 미루기 요청은 권한·팀 상태·`WAIT-009` 정책 버전·호환 팀 집합을 검증하고, 허용될 때만 새 순번 원장 버전을 생성한다. 사실 상태 보정은 새 순번을 직접 입력하지 않고 같은 원장을 재계산한다.
- 중복·부적합 요청은 거부하고 원래 순서를 유지한다. 조정 철회는 과거 버전을 덮어쓰지 않고 새 역보정 사건으로 처리하며 이미 체크인·종결된 팀을 재활성화하지 않는다.
- `WAIT-009` 정책 버전 변경은 효력 이후 요청부터 적용하고 과거 호출·미루기·재계산은 재현 가능하게 보존한다. `WAIT-015`의 보증금 미도입 원칙에 따라 금전 조건으로 우선순위를 만들지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 대기열 버전·조정 사건·멱등 키와 조건부 갱신으로 동시에 하나의 순번 원장만 확정한다. 조정과 호출 경합은 같은 팀 상태 버전을 검사한다.
- 부분 갱신은 전체 원장 버전을 게시하지 않고 롤백·재계산한다. 중앙 원장 또는 정책 버전을 확인할 수 없으면 조정을 실패 폐쇄한다.
- 승인된 조정으로 표시 순서가 바뀌면 영향 사용자에게 사유 범주와 결과를 알린다. 요청자·승인 규칙·근거·이전·새 원장·영향 팀을 감사하되 다른 팀 개인정보는 노출하지 않는다.

### 관측 가능한 인수 조건

- 대표자·운영자가 순번 값을 직접 쓰거나 `WAIT-009`에 없는 사유로 재정렬할 수 없다.
- 조정·호출 동시 처리와 중복 요청에도 하나의 원장 버전만 확정되고 부분 순서가 공개되지 않는다.
- 호출·미루기·사실 보정이 같은 `WAIT-009` 원장 규칙만 소비하고 두 번째 순번 규칙이나 수동 예외를 만들지 않는다.

## WAIT-014 허수 웨이팅 방지·제재

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 허수 위험 신호는 확인된 계정의 반복 중복 시도, 명시적 호출 미응답·노쇼와 초대 악용 같은 서비스 사건으로 제한한다. 정밀 기기 지문·상시 위치 추적·유료 탐지 서비스를 자동 도입하지 않는다.
- 단일 실패나 알림 장애만으로 계정을 자동 정지하지 않는다. 규칙 기반 경고 → 해당 기능의 일시 제한 후보 → 권한 있는 플랫폼 운영자 검토 순으로 단계화하고, 기간·횟수 임계치는 버전된 운영 정책과 관측으로 조정한다.
- 식당 대표자는 사실 사건을 신고할 수 있지만 사용자 제재를 확정하지 못한다. 플랫폼 운영자는 근거·사유·기간을 기록하고 계정 제재 공통 정책의 권한·이의 제기 절차를 따른다.

### 정상 흐름·변경·취소·만료

- 사건 기록 → 규칙 평가 → 사용자 경고 또는 검토 대기 → 운영자 유지·완화·취소 결정 → 제한 적용·만료 순으로 처리한다. 검토 중에는 새 영구 불이익을 자동 적용하지 않는다.
- 오탐이 확인되면 제한을 해제하고 위험 점수·파생 플래그를 새 보정 사건으로 수정한다. 이미 유효한 다른 대기 팀·순번은 `WAIT-009` 승인 경로 없이 직접 이동하지 않는다.
- 정책 버전 변경은 이후 사건 평가에 적용하고 과거 제재의 근거를 보존한다. `WAIT-015`에 따라 미응답·노쇼를 이유로 보증금이나 다른 금전 제재를 만들지 않는다.

### 동시성·실패·복구·알림·감사·개인정보

- 사건 ID·정책 버전·사용자별 활성 검토 유일성으로 중복 이벤트가 제재 단계를 여러 번 올리지 않게 한다. 역순 취소·보정 이벤트는 수정 번호로 처리한다.
- 탐지·평가 장애는 신규 자동 제재를 실패 폐쇄하고 원래 계정 상태를 유지한다. 대사 작업은 누락·중복 사건을 복구하고 수동 결정에는 권한을 재검증한다.
- 경고·제한·해제와 이의 제기 경로를 사용자에게 알린다. 사건·근거·행위자·정책 버전·결과를 감사하고 위치 원문·다른 구성원 연락처를 제재 데이터에 복제하지 않는다.

### 관측 가능한 인수 조건

- 같은 위험 이벤트 중복 소비가 경고·제한을 중복 적용하지 않고 단일 알림 실패만으로 제재되지 않는다.
- 식당 대표자는 신고만 가능하고 권한 없는 운영자나 자동 규칙이 영구 정지를 확정하지 못한다.
- 오탐 취소 뒤 기능 제한과 파생 플래그가 복구되며 불필요한 기기·위치 개인정보가 수집되지 않는다.

## WAIT-015 웨이팅 보증금 미도입

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 현재 서비스와 최종 프로젝트 범위에서는 현장·원격 웨이팅 등록, 입장 임박 안내, 실제 호출과 입장에 보증금·예약금·선결제를 요구하지 않는다.
- 예약금과 메뉴 홀드 결제 정책을 웨이팅에 적용하거나 미응답·노쇼를 이유로 금전 청구·자동 결제·환불 사건을 만들지 않는다.
- 식당 대표자와 플랫폼 운영자는 매장 설정이나 예외 처리로 웨이팅 보증금을 임의 활성화할 수 없다. 향후 도입하려면 금액, 결제 시점, 반환·몰취, 취소·미응답, 환불·분쟁 책임을 별도 정책 변경으로 확정해야 한다.

### 정상 흐름·실패·복구·감사

- 웨이팅은 결제 세션 없이 자격·위치·중복 검증을 통과하면 생성한다. 결제 시스템 장애는 웨이팅 등록·호출·입장 상태에 영향을 주지 않는다.
- 잘못 생성된 웨이팅 결제 요구나 금전 사건은 웨이팅 상태와 분리해 차단·대사하고, 사용자에게 청구하지 않은 상태를 유지한다.
- 적용 정책 버전과 보증금 미적용 결과를 거래·설정 경계에서 감사하되 결제수단 정보를 웨이팅 사건에 복제하지 않는다.

### 관측 가능한 인수 조건

- 현장·원격 웨이팅의 등록부터 입장·취소·미응답 종료까지 결제수단이나 결제 승인을 요구하지 않는다.
- 매장 설정·관리자 명령·오래된 클라이언트가 웨이팅 보증금 또는 미응답 금전 제재를 활성화하지 못한다.
- 예약금과 메뉴 홀드 결제 상태를 변경해도 웨이팅 순번·호출 자격·입장 결과가 바뀌지 않는다.

## WAIT-016 웨이팅 상태와 상태 전이

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 대기 팀 상태는 `등록 검증 중`, `대기 중`, `호출 중`, `호출 실패`, `도착 확인`, `체크인 완료`, `사용자 취소`, `매장 종료`, `미응답 종료`, `만료`로 구분한다. `호출 실패`는 매장 확인 전 보류 상태이고 종결 상태는 체크인 완료와 각 취소·종료·만료다.
- 활성 팀 대표자만 호출 응답·사용자 취소 명령을 제출하고 도착 증거를 제시하거나 제출할 수 있다. 구성원은 팀 상태·순번·예상 시간과 결과 알림을 조회하는 읽기 전용 참여자이며 대표자를 대신해 해당 명령을 제출하지 못한다. 식당 대표자는 자신의 매장 팀의 호출·도착 증거 확인·체크인·매장 종료 명령만 제출한다. 플랫폼 운영자는 기록된 장애 복구 사건과 권한 없이는 전이를 만들지 못하며 어느 행위자도 상태를 직접 덮어쓰지 않는다.
- `호출 중` → `도착 확인`은 활성 팀 대표자가 제시하거나 제출한, 팀·매장·호출 사건에 묶인 서버 발급 일회성 도착 토큰이나 호출 코드를 식당 대표자가 현장에서 확인한 경우에만 허용한다. 확인자, 증거 유형·토큰 지문, 호출 사건과 중앙 확인 시각을 남기고 위치 원문·동행자 신원은 추가 수집하지 않는다.
- `대기 중`의 표시 순번·미루기는 `WAIT-009`, 입장 임박 안내는 `WAIT-010`, `호출 중`의 한 번·도착 제한 3분과 `호출 실패` 처리는 `WAIT-011`, 보증금 미도입은 `WAIT-015`를 따른다.
- 웨이팅 운영 설정은 대표자 계정 전체가 아니라 대표자가 관리 권한을 가진 매장별로 둔다. 각 매장은 `웨이팅 기능 사용 여부`와 `현재 접수 방식`을 분리해 관리하며, 접수 방식은 `자동`, `수동 접수 중`, `접수 일시중지`다.
- 기능을 끄면 해당 매장의 자동 오픈 일정을 실행하지 않고 신규 웨이팅을 받지 않는다. 활성 팀이 있으면 즉시 종결하지 않고 대표자에게 기존 대기열 유지 또는 영향 팀 확인 뒤 일괄 `매장 종료` 중 하나를 선택하게 한다.
- 자동 접수는 영업 구간 시작 60분 전에 열리며 대표자는 매장별로 0분~180분 범위의 사전 오픈 시간을 설정한다. 수동 접수는 자동 오픈 시간 밖에도 해당 영업 구간이 시작되는 매장 현지 날짜 00:00부터 영업 구간 종료 시각까지 허용한다. 자정을 넘는 영업 구간은 시작 날짜의 영업일에 귀속하므로 자정 이후에도 구간 종료 전까지 유지할 수 있다.
- 휴점·폐점·긴급 중단 상태이거나 해당 영업 구간이 완전히 종료된 뒤에는 자동·수동 접수를 열 수 없다. `접수 일시중지`는 신규 등록만 차단하고 기존 팀의 순번·입장 임박·호출·도착 확인·종결 처리는 유지한다.

### 정상 흐름·변경·취소·만료

- 유효 등록은 `등록 검증 중` → `대기 중`, 호출은 `대기 중` → `호출 중`, 현장 증거 확인은 `호출 중` → `도착 확인`, 실제 입장 처리는 식당 대표자의 `도착 확인` → `체크인 완료`로 전이한다. 검증 실패는 팀을 활성화하지 않고 자원·관계를 해제한다.
- 팀 대표자는 `대기 중`·`호출 중`·`도착 확인`에서 사용자 취소를 요청할 수 있다. 매장 운영 종료는 같은 활성 상태에서 `매장 종료`로, 실제 호출 뒤 3분 도착 제한 만료는 `호출 중`에서 `호출 실패`로 전이하고 매장 확인 뒤 `미응답 종료`로 종결한다. 이 과정은 재대기·유예·순번 복원을 만들지 않는다.
- 등록 검증이나 대기 자격의 버전된 유효 조건이 끝나면 `등록 검증 중` 또는 `대기 중`에서 `만료`로 전이한다. 실제 호출 뒤 도착 증거 유효기간과 그 종료 결과는 `WAIT-011`의 한 번·도착 제한 3분 및 호출 실패 확인 흐름을 적용한다.
- 각 취소·종료·만료는 구분된 종결 상태와 사유를 남긴다. 종결 상태를 되돌리지 않고 오류 복구는 새 보정 사건과 필요한 재등록으로 처리하며, 재등록 순서는 `WAIT-009`를 따른다.
- 영업 종료 30분 전부터 대표자는 접수 마감과 잔여 웨이팅 일괄 종료를 실행할 수 있다. 실행 화면은 대상 영업 구간과 영향받는 활성 팀 수를 먼저 표시하고 재확인을 받으며, 신규 접수를 먼저 닫은 뒤 고정한 대상 팀을 사용자 취소가 아닌 `매장 종료`로 종결한다.
- 일괄 종료는 기존 순번을 재정렬하지 않고 사용자 미응답·노쇼·금전 제재 사건을 만들지 않는다. 모든 대상 팀에는 매장 마감으로 입장이 어렵다는 알림 작업을 생성하되 일부 알림 실패가 종료 상태를 되돌리지 않으며 알림만 제한 재시도한다.

### 동시성·실패·복구·알림·감사·개인정보

- 허용 전이표·팀 수정 번호·명령 멱등 키로 호출·도착 확인·체크인·취소·만료 경합에서 중앙에 먼저 유효하게 확정된 전이 하나만 성공하게 한다. 뒤늦은 명령은 최신 상태와 충돌 결과를 반환하고 상태·순번·관계를 다시 바꾸지 않는다.
- 상태 전이와 Outbox 후속 작업을 같은 처리 경계로 기록한다. 메시지 지연은 중앙 상태를 바꾸지 않고 소비자는 팀·전이 ID로 재처리한다.
- 호출·도착 확인·체크인·취소·종결을 구성원과 매장에 알리고 전이 전후·행위자·증거 참조·사유·정책 버전·상관관계 ID와 경합에서 거부된 명령을 감사한다. 이벤트에는 필요한 계정·팀 식별자와 비가역 토큰 지문만 둔다.
- 기능 설정·접수 방식·자동 오픈 작업은 매장·영업 구간·설정 버전·멱등 키로 조건부 갱신한다. 기능 끄기·수동 오픈·일시중지·자동 오픈이 경합하면 중앙에 먼저 유효하게 확정된 설정 하나만 적용하고 오래된 자동 작업은 신규 접수를 다시 열지 못한다.
- 일괄 종료는 대표자, 실행 시각, 대상 영업 구간, 영향 팀 수, 팀별 전후 상태와 알림 결과를 감사한다. 대상 스냅샷과 팀별 종결 명령을 멱등하게 기록하여 재시도·부분 장애에도 누락 팀을 대사하고 이미 종결된 팀을 다시 변경하지 않는다.

### 관측 가능한 인수 조건

- 허용되지 않은 상태 건너뛰기·종결 상태 재개·구 버전 명령이 모든 인스턴스에서 거부된다.

## WAIT-017 다중 인스턴스 순번 정합성

### 확정 규칙·행위자·권한·사전 조건

- 대기 팀, 구성원 관계, 상태와 `WAIT-009`가 산출한 순번 원장은 중앙 저장소가 원본이다. 인스턴스 로컬 목록·세션·고정 라우팅으로 순서를 확정하지 않는다.
- 모든 생성·합류·입장 임박·호출·취소·종결·조정은 대기열 버전과 팀 버전을 조건부 갱신하며 명확한 원장 또는 알림 사건을 만든다.
- 캐시와 SSE는 조회·전달 최적화일 뿐 쓰기 권한이나 최종 순서를 결정하지 않는다. 오래된 클라이언트 명령은 최신 버전과 함께 충돌로 반환한다.
- 같은 매장·같은 영업 구간의 활성 예약 보유자가 웨이팅을 신청하거나 활성 웨이팅 보유자가 예약을 요청하면 강제 차단만 하지 않고 기존 거래 유지 또는 기존 거래를 종결한 조건부 전환 선택지를 표시한다. 예약 취소가 필요한 경우 `RES-009`·`PAY-009`가 계산한 취소 수수료·환불 결과와 정책 버전을 먼저 표시하고 사용자의 명시적 확인을 받는다.

### 정상 흐름·변경·취소·만료

- 중앙 전이 또는 입장 임박 사건 확정 → Outbox 이벤트 → 캐시·실시간 표시·알림 반영 순으로 처리한다. 화면은 이벤트가 늦으면 중앙 조회로 최신 상태와 실제 호출의 도착 제한 시각을 회복한다.
- 팀 종결·새 팀 생성·테이블 선택 호출·대표자 1회 미루기의 순번 처리는 각각 정확히 한 번 중앙 원장에 반영한다. 재배치는 `WAIT-009`의 호환 팀 FIFO·다섯 팀 뒤 이동만 사용하고 다른 순서 규칙을 만들지 않는다.
- 배포 중 구·신 버전이 함께 실행돼도 호환되는 상태·이벤트 버전을 사용하며 지원하지 않는 새 명령은 실패 폐쇄한다.
- 예약에서 웨이팅으로 전환할 때는 현재 접수 가능성·운영 상태·영업 구간·사용자-매장 활성 웨이팅 유일성을 조건부로 확보한 뒤 예약 버전과 확인한 취소·환불 계산 버전을 다시 검사하여 예약 종결, 웨이팅 생성과 환불 작업 생성을 하나의 전환 사건으로 확정한다. 완료할 수 없으면 기존 예약과 점유를 유지한다.
- 웨이팅에서 예약으로 전환할 때도 예약 수용량 선점 가능성을 먼저 확보하고 웨이팅 종결과 예약 생성·선점을 같은 전환 사건으로 확정한다. 완료할 수 없으면 기존 웨이팅·순번을 유지하며, 외부 결제·환불 호출은 전환 사건의 Outbox 후속 작업으로 수행한다.

### 동시성·실패·복구·알림·감사·개인정보

- 중앙 유일성 제약, 조건부 갱신, 멱등 이벤트 소비와 Outbox로 다중 인스턴스 중복·순서 역전을 흡수한다. 입장 임박 사건은 팀·알림 목적별 한 번만 확정하고 작업 임대 만료 뒤 재실행돼도 같은 결과를 만든다.
- 캐시·메시지·SSE 장애는 중앙 원장을 되돌리지 않는다. 누락 이벤트 재생·원장 대사로 수렴시키며 원장 자체를 확인할 수 없으면 상태 변경을 실패 폐쇄한다.
- 사용자에게 다른 팀 상세가 아닌 자신의 최신 순번·상태만 제공한다. 원장 버전, 명령·이벤트·복구 상관관계와 지연을 감사·관측한다.
- 예약·웨이팅 전환은 원 거래 버전, 대상 자격·자원 버전, 취소·환불 계산 버전과 전환 멱등 키를 사용한다. 부분 실패나 외부 결과 불명에는 원 거래를 먼저 잃은 상태로 확정하지 않고 중앙 전환 원장을 대사하여 기존 거래 유지 또는 새 거래 확정 중 하나로 수렴시킨다.

### 관측 가능한 인수 조건

- 여러 인스턴스에서 생성·호출·취소를 병렬 실행해도 팀별 상태와 순번 원장에 중복·누락이 없다.
- 여러 인스턴스가 같은 순번 변화를 처리해도 입장 임박 사건은 팀별 한 번만 생성되고, 실제 호출 만료와 도착 확인이 경합해도 승자 전이 하나만 남는다.

- 구 버전·역순 이벤트가 최신 원장을 덮어쓰지 않고 다른 팀의 개인정보가 실시간 이벤트에 노출되지 않는다.
- 예약 취소와 웨이팅 생성 또는 웨이팅 취소와 예약 생성을 병렬·재시도해도 사용자가 두 거래를 모두 잃는 부분 결과가 남지 않고, 전환 실패 시 원 거래와 그 자원이 유지된다.
