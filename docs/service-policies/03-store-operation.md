# 매장 운영·영업시간·메뉴 정책

> 문서 상태: 세부 정책 확정
> 정책 범위: `OPER-001`~`OPER-010`
> 최종 변경일: 2026-07-22

모든 식당의 필수 기본 기능과 영업시간, 휴무, 메뉴 변경, 품절, 콘텐츠 표시, 영업 상태 및 긴급 중단의 안전한 운영 기본안을 확정한다. 매장 운영 변경은 중앙의 버전·상태를 기준으로 적용하고, 기존 확정 거래를 조용히 변경하거나 취소하지 않는다.

## 분류 현황

| 상태 | 중요도 | 정책 ID |
|---|---|---|
| 확정 | 필수 | `OPER-001` |
| 확정 | 필수 | `OPER-002`~`OPER-006`, `OPER-008`~`OPER-010` |
| 확정 | 권장 | `OPER-007` |

## OPER-002 정규 영업시간과 브레이크타임

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 입점이 승인되고 해당 매장 운영 권한이 유효한 식당 대표자만 정규 영업시간과 브레이크타임을 작성·변경·예약 적용할 수 있다. 플랫폼 운영자는 장애 복구 또는 분쟁 증거 보존을 위해 조회할 수 있지만 대표자 대신 임의의 영업시간을 만들지 않는다.
- 영업시간은 매장별 IANA 시간대, 요일, 현지 시작·종료 시각과 적용 시작일을 가진 불변 스케줄 버전으로 저장한다. 주소가 같은 시간대라는 추측이나 애플리케이션 인스턴스의 로컬 시간대를 사용하지 않는다.
- 요일별 영업 구간은 0개 이상 허용하며, 구간이 없으면 그 요일은 정규 휴무다. 브레이크타임은 같은 영업 구간 안에 완전히 포함되어야 하고 서로 겹치거나 길이가 0일 수 없다.
- 자정을 넘는 영업 구간은 시작 요일에 귀속된 하나의 구간으로 입력하고 종료 날짜를 명시적으로 계산한다. 같은 실제 시각의 영업 구간·브레이크타임이 중복되면 게시를 거부한다.

### 정상 흐름·변경·상태

- 대표자는 `초안`을 저장하고 검증을 통과한 버전을 `게시 예약` 또는 `게시`한다. 즉시 게시하면 중앙 확정 시각부터, 예약 게시하면 지정된 미래 시각부터 신규 조회·거래에 적용한다.
- 같은 효력 구간에는 활성 스케줄 버전 하나만 존재한다. 새 버전이 효력을 얻으면 이전 버전은 `종료`되지만 삭제하지 않으며, 과거 영업 여부와 거래 판정은 당시 버전으로 재현할 수 있어야 한다.
- 게시 예약은 효력 발생 전 취소하거나 새 버전으로 교체할 수 있다. 이미 효력이 발생한 버전은 소급 수정하지 않고 다음 버전을 게시한다. 정규 스케줄 자체에는 자동 만료를 두지 않는다.
- `현재 영업 중` 표시는 정규 영업시간만이 아니라 OPER-003 휴무·임시 휴점과 OPER-009 운영 상태를 함께 중앙 시각으로 계산한다. 정규 시간 안이라는 이유만으로 휴점·폐점 매장을 영업 중으로 표시하지 않는다.
- 스케줄 변경이 기존 예약·홀드·행사와 충돌하면 변경 저장은 허용하되 충돌 목록을 생성한다. 기존 확정 거래를 자동 이동·취소하지 않고 OPER-004·OPER-010과 해당 거래 정책으로 처리한다.

### 동시성·실패·복구·다중 인스턴스

- 저장·게시·취소 명령은 매장, 스케줄 버전과 멱등 키를 사용한다. 동시에 게시된 버전은 중앙 유일성·조건부 갱신으로 하나만 확정하고 충돌한 요청에는 최신 버전을 반환한다.
- 모든 인스턴스는 중앙 스케줄 버전과 공통 기준 시각으로 판정한다. 캐시는 읽기 최적화에만 사용하며 중앙 버전을 확인할 수 없는 쓰기 요청은 실패 폐쇄한다.
- 예약 게시 작업이 중복 실행되거나 인스턴스가 재시작돼도 같은 버전의 활성화와 이전 버전 종료는 한 번만 일어난다. 활성화 일부 실패는 중앙 상태를 기준으로 재시도하고 검색·캐시 갱신 실패는 기존 중앙 결과를 되돌리지 않는다.

### 알림·감사·개인정보·결제

- 일반 스케줄 변경만으로 모든 사용자에게 알림을 보내지 않는다. 기존 확정 거래와 충돌할 때만 해당 거래 당사자에게 영향과 후속 조치를 알리며, 최종 변경·취소 결과는 거래 도메인의 필수 알림을 따른다.
- 행위자, 매장, 이전·새 버전, 시간대, 영업·브레이크 구간, 효력 시각, 충돌 거래 수, 요청 식별자와 처리 결과를 감사 이력으로 남긴다.
- 영업시간에는 개인정보를 수집하지 않는다. 로그에는 예약자 연락처나 거래 상세를 넣지 않고 충돌 거래 식별자만 연결한다.
- 영업시간 변경 자체는 결제·환불·정산에 영향을 주지 않는다. 기존 거래를 취소하게 된 경우에만 PAY 및 거래 취소 정책을 따른다.

### 관측 가능한 인수 조건

- 겹침·0 길이·영업 구간 밖 브레이크타임은 게시되지 않고, 자정 경계와 시간대 변환 테스트에서 모든 인스턴스가 같은 영업 여부를 반환한다.
- 동시 게시 요청에도 같은 효력 구간의 활성 버전은 하나이며 재전송은 중복 버전을 만들지 않는다.
- 예약 게시 작업의 중복·재시작 후에도 지정 시각부터 새 버전이 한 번만 활성화된다.
- 기존 거래와 충돌하는 변경은 충돌 목록을 만들지만 해당 거래의 시간·상태·가격을 자동 변경하지 않는다.

## OPER-003 정기 휴무와 임시 휴점

> 정책 상태: 확정

### 확정 규칙·행위자·권한·사전 조건

- 식당 대표자는 매장별로 요일·날짜 규칙의 `정기 휴무`와 시작·종료 시각이 있는 `임시 휴점`을 생성·변경·취소할 수 있다. OPER-009의 폐점 매장에는 새 휴무를 만들 수 없다.
- 정기 휴무는 OPER-002 스케줄의 영업 구간보다 우선하고, 임시 휴점은 정규 영업·정기 휴무보다 우선한다. 여러 휴무가 겹치면 합집합으로 닫힌 시간대를 계산하되 각 원인과 이력은 별도로 보존한다.
- 임시 휴점에는 사용자에게 공개할 짧은 사유 범주와 선택적인 안내 문구를 두되, 직원·환자·사고 피해자 등 제3자의 개인정보나 내부 보안 내용을 입력하지 않게 한다.

### 정상 흐름·변경·만료

- 휴무·휴점은 `초안` → `예약` → `적용 중` → `종료`로 전이한다. 즉시 시작은 `예약`을 거치지 않고 중앙 확정 시각부터 `적용 중`이 될 수 있다.
- 시작 전에는 변경·취소할 수 있고, 적용 중에는 종료 시각을 앞당기거나 늘릴 수 있으나 이미 닫혀 있던 과거 시간을 소급해 영업으로 바꾸지 않는다. 종료 시각이 지나면 공통 기준 시각으로 즉시 미적용으로 판정하며 스케줄러 지연과 무관하게 `종료`된다.
- 임시 휴점의 종료 후 OPER-009가 `영업`이고 다른 휴무가 없을 때만 정규 영업시간을 다시 적용한다. 폐점·긴급 중단·다른 휴무를 무시하고 자동 영업 재개하지 않는다.
- 휴무가 새 거래 가능 시간과 겹치면 해당 시간의 신규 예약·웨이팅·홀드·결제를 차단한다. 기존 확정 거래는 `영향 확인 필요` 목록에 넣고 자동 취소하지 않는다.

### 동시성·실패·복구·다중 인스턴스

- 휴무 명령은 휴무 식별자·버전·멱등 키로 처리하고, 중복 생성·변경은 한 번만 반영한다. 겹치는 규칙은 중앙 시간 구간 계산으로 합성하며 마지막 요청 인스턴스의 로컬 상태에 의존하지 않는다.
- 예약 활성화·만료 작업은 중복 실행돼도 멱등해야 한다. 작업이 지연되어도 각 거래 명령은 중앙 시작·종료 시각과 최신 버전을 다시 검사하여 닫힌 시간의 거래를 허용하지 않는다.
- 중앙 휴무 상태를 확인할 수 없으면 신규 거래 생성은 실패 폐쇄한다. 공개 조회 캐시가 오래된 경우 `영업 중`을 확정 표시하지 않고 최신 상태를 다시 조회하게 한다.

### 알림·감사·개인정보·결제

- 휴무 생성·변경·취소로 기존 거래가 영향을 받으면 대상 사용자에게 휴점 사실, 영향 시간과 후속 처리 예정임을 알린다. 거래를 이행·변경·취소하기 전에는 확정 결과처럼 표현하지 않는다.
- 매장, 규칙·버전, 생성자, 시작·종료, 공개 사유, 영향 거래 식별자, 변경·취소 이유와 처리 결과를 감사 이력으로 남긴다.
- 휴무 정보에는 운영에 필요하지 않은 개인정보를 넣지 않는다. 결제·환불은 휴무 등록만으로 실행하지 않고 개별 거래의 최종 취소 결과에 따라 처리한다.

### 관측 가능한 인수 조건

- 정규 영업시간과 겹치는 정기 휴무·임시 휴점 동안 모든 인스턴스가 신규 거래를 거부한다.
- 겹치는 휴무 중 하나가 끝나도 다른 휴무가 남아 있으면 영업 중으로 바뀌지 않는다.
- 만료 작업을 중단해도 종료 시각 이후 요청 시점 판정으로 임시 휴점이 만료되고, 중복 작업은 상태를 두 번 전환하지 않는다.
- 기존 거래는 휴무 생성으로 자동 취소되지 않고 영향 목록·알림·감사 이력이 한 번씩 생성된다.

## OPER-004 예약 가능 시간과 실제 영업시간 관계

> 정책 상태: 확정

### 확정 규칙·행위자·사전 조건

- 공개할 예약 슬롯의 시작부터 서비스 종료까지의 전체 구간은 OPER-002의 영업 구간 안에 포함되고 브레이크타임, OPER-003 휴무·임시 휴점과 OPER-009 휴점·폐점을 겹치지 않아야 한다.
- 예약 기능 권한이 유효한 식당 대표자만 예약 가능 시간 규칙을 만들 수 있다. 슬롯 길이·회차·정원 등 예약 상품 규칙은 RES 도메인을 따르며, OPER-004가 임의의 회전 시간이나 버퍼 수치를 만들지 않는다.
- 예약 가능 시간 게시 전에 참조할 영업 스케줄·휴무·운영 상태 버전을 중앙에서 검증한다. 사용자가 본 화면이나 오래된 검색 색인을 승인 근거로 사용하지 않는다.

### 정상 흐름·변경·취소

- 유효한 슬롯만 공개·검색·예약 요청 대상으로 만든다. 예약 최종 확정 시 같은 영업·휴무·운영 상태와 슬롯 버전을 다시 검사한다.
- 영업시간 변경으로 미래 슬롯이 무효가 되면 신규 노출과 예약을 중단한다. 예약되지 않은 슬롯은 닫고, 이미 확정된 예약은 당시 스냅샷을 유지한 채 `영업시간 충돌`로 표시한다.
- 충돌 예약은 식당 대표자가 `기존대로 이행`, `변경 제안`, `매장 귀책 취소` 중 하나로 처리한다. 변경은 사용자의 명시적 동의 전까지 원래 예약을 유지하고, 취소·환불·보상은 RES-009·RES-013과 PAY 정책의 확정 결과를 따른다.
- 영업시간을 되돌려도 이미 사용자에게 확정된 변경·취소를 자동 복원하지 않는다. 필요한 경우 새 예약 또는 변경 동의 흐름을 사용한다.

### 동시성·장애·다중 인스턴스

- 예약 확정은 슬롯 버전과 매장 운영 버전을 조건부로 검사한다. 영업시간·휴점 전환과 예약 확정이 경합하면 먼저 중앙 확정된 버전에 따라 예약이 확정되어 영향 목록에 들어가거나 요청이 거부되는 둘 중 하나만 발생한다.
- 동일 예약 요청은 멱등 키로 한 번만 확정한다. 여러 인스턴스가 같은 무효 슬롯을 다시 열거나 충돌 사건을 중복 생성하지 못하게 중앙 유일성 제약을 둔다.
- 영업 상태 조회 장애 시 신규 예약 확정은 실패 폐쇄한다. 검색·알림 실패는 중앙 슬롯 폐쇄와 기존 예약 상태를 되돌리지 않고 재색인·재발송 작업으로 복구한다.

### 알림·감사·개인정보·결제

- 충돌 발생과 대표자의 이행·변경 제안·취소 결정, 사용자의 변경 동의 또는 거절을 각각 대상 사용자에게 알린다. 알림 실패가 동의나 취소를 대신하지 않는다.
- 슬롯·영업 스케줄·휴무·운영 상태 버전, 예약 식별자, 충돌 사유, 제안·동의·취소 행위자와 시각을 감사 이력으로 남긴다.
- 이 검증을 위해 새 개인정보를 수집하지 않는다. 사용자 연락처는 알림 도메인에만 전달하고 운영 로그에는 마스킹한다.
- 결제 영향은 충돌 자체가 아니라 확정된 예약 변경·취소 명령에서만 발생하며, OPER-004가 환불액·수수료·보상 주체를 새로 결정하지 않는다.

### 관측 가능한 인수 조건

- 브레이크타임·휴무·휴점·폐점과 겹치거나 영업 종료를 넘는 슬롯은 게시·확정되지 않는다.
- 영업시간 변경과 예약 확정을 동시에 실행해도 무효 슬롯의 초과 확정이 없고 충돌 예약 사건은 한 번만 생성된다.
- 이미 확정된 예약은 영업시간 변경만으로 시간·가격·상태가 바뀌지 않으며 변경 제안은 사용자 동의 없이는 적용되지 않는다.
- 중앙 운영 상태 장애 시 신규 예약은 거부되고 복구 후 최신 버전으로 다시 검증된다.

## OPER-005 메뉴·옵션·가격 변경 효력 시점

> 정책 상태: 확정

### 확정 규칙·행위자·사전 조건

- 해당 매장 운영 권한이 유효한 식당 대표자만 메뉴·옵션·가격의 새 버전을 작성·게시·예약 게시·철회할 수 있다. 플랫폼 운영자는 정책 위반 콘텐츠를 숨기거나 복구할 수 있지만 임의의 판매 가격을 정하지 않는다.
- 메뉴 상품은 변경되지 않는 식별자를 유지하고, 이름·설명·옵션 구성·표시 가격·세금 표시 입력·적용 시각을 불변 버전으로 저장한다. 게시된 버전의 값을 직접 덮어쓰지 않는다.
- 상태는 `초안` → `게시 예약` → `게시` → `종료`이며, 정책 위반이나 안전 문제에는 별도의 `관리자 숨김` 표시를 적용한다. 같은 상품·판매 맥락·효력 시각에는 게시 버전 하나만 활성화한다.

### 정상 흐름·변경·취소·스냅샷

- 즉시 게시한 버전은 중앙 확정 시각 이후 시작되는 신규 거래에, 예약 게시한 버전은 지정 효력 시각 이후 시작되는 신규 거래에 적용한다. 이미 시작된 홀드·예약·주문에는 거래 생성 시 캡처한 메뉴·옵션·가격·정책 버전을 유지한다.
- 결제 또는 최종 확정 직전에는 거래가 가진 가격 스냅샷과 서버 계산 금액을 검증한다. 메뉴 가격이 바뀌었다는 이유로 진행 중 거래 금액을 자동 교체하지 않으며, 스냅샷이 유효하지 않으면 사용자의 재확인 없이 새 가격으로 청구하지 않는다.
- 게시 예약은 효력 전 취소할 수 있고, 게시 후 되돌리려면 이전 내용을 복제한 새 버전을 게시한다. 종료·숨김 후에도 과거 거래가 참조하는 버전은 삭제하지 않는다.
- 옵션 제거·가격 변경이 기존 확정 거래의 구성과 이행 가능성에 영향을 주면 기존 거래를 그대로 이행하거나 별도 변경 제안·취소 절차를 사용한다. 거래 스냅샷을 소급 수정하지 않는다.

### 동시성·실패·다중 인스턴스

- 게시 명령은 상품·버전·멱등 키를 사용한다. 중앙 유일성·조건부 갱신으로 경합 버전 중 하나만 해당 효력 구간에 활성화한다.
- 거래 생성과 메뉴 전환은 중앙 효력 시각·활성 버전·거래 시작 시각을 일관되게 판정한다. 각 거래는 정확히 한 메뉴 버전을 참조해야 하며 인스턴스 캐시의 갱신 순서로 가격이 달라지지 않아야 한다.
- 검색·이미지·캐시 갱신 실패는 게시된 중앙 버전을 되돌리지 않는다. 이전 공개 버전을 임의로 최신처럼 유지하지 않고 재처리 상태를 표시하며, 결제·예약 쓰기는 중앙 원본을 직접 검증한다.

### 알림·감사·개인정보·결제

- 일반 메뉴 변경은 전체 사용자에게 알리지 않는다. 기존 확정 거래의 이행 내용이 달라질 때만 대상 사용자에게 변경 제안과 동의·거절·취소 결과를 알린다.
- 변경 전·후 버전, 변경 필드, 가격과 통화, 효력 시각, 행위자, 요청 식별자, 영향 거래와 처리 결과를 감사 이력으로 남긴다.
- 메뉴 버전에는 불필요한 개인정보를 포함하지 않는다. 가격 변경은 과거 결제·환불·정산 원장을 바꾸지 않고 PAY 도메인의 거래 스냅샷·대사 정책을 따른다. 세금 계산·반올림의 최종 규칙도 PAY-004를 따른다.

### 관측 가능한 인수 조건

- 메뉴 게시 전후 경계에서 시작한 거래가 각각 정확한 한 버전과 가격 스냅샷을 참조하며, 동시 게시에도 활성 버전이 중복되지 않는다.
- 동일 게시 요청 재전송은 새 버전이나 알림을 중복 생성하지 않는다.
- 게시 후 가격 변경·옵션 제거가 기존 홀드·예약·결제 금액을 소급 변경하지 않는다.
- 검색·이미지 갱신 장애 중에도 거래 쓰기는 중앙 게시 버전을 검증하고 복구 작업으로 공개 상태가 최종 수렴한다.

## OPER-006 품절·판매 중지·숨김 구분

> 정책 상태: 확정

### 확정 규칙·상태·권한

- 메뉴 가용성과 공개 노출을 분리한다. `판매 중`은 신규 선택 가능, `품절`은 일시적 수량 부족, `판매 중지`는 대표자가 판매를 중단한 상태다. 별도 노출 상태 `공개`·`숨김`은 검색·목록 표시 여부만 나타낸다.
- `품절`과 `판매 중지`는 신규 홀드·주문·예약 결합을 차단한다. `숨김`도 공개 화면·검색·직접 링크에서 신규 선택을 차단하지만 재고가 없다는 뜻으로 표시하지 않는다. 품절 메뉴는 대표자가 숨기지 않는 한 메뉴 목록에 품절로 표시할 수 있다.
- 식당 대표자는 상태를 변경하고 복구할 수 있다. 재고·홀드 시스템은 가용 수량이 0이 된 확정 결과에 따라 품절을 설정할 수 있으나, 플랫폼 운영자는 정책 위반 콘텐츠의 노출을 숨길 수 있을 뿐 판매 재고를 임의로 늘릴 수 없다.
- 대표 메뉴만 수량을 관리한다는 현재 서비스 경계를 유지하며 모든 메뉴의 POS 수준 재고 수집을 요구하지 않는다.

### 정상 흐름·변경·기존 거래

- 상태 전이는 `판매 중` ↔ `품절`, `판매 중`·`품절` ↔ `판매 중지`, `공개` ↔ `숨김`을 독립적으로 기록한다. 대표자가 재개하기 전까지 판매 중지에는 자동 만료를 두지 않는다. 품절 자동 해제는 중앙 가용 수량이 복구됐다는 확정 결과가 있을 때만 허용한다.
- 상태 변경은 중앙 확정 시각 이후 신규 거래에 적용한다. 이미 확정된 홀드·예약·주문의 메뉴와 수량을 자동 삭제·대체하지 않고 HOLD-009~HOLD-011 및 거래 취소 정책으로 이행·대체 동의·취소를 처리한다.
- 숨김을 해제해도 판매 상태가 품절·판매 중지이면 신규 선택을 허용하지 않는다. 판매를 재개해도 노출 상태가 숨김이면 공개하지 않는다.

### 동시성·실패·다중 인스턴스

- 상태 변경은 메뉴·판매 단위·상태 버전·멱등 키를 사용하고 중앙 조건부 갱신으로 처리한다. 홀드 확정과 품절 전환이 경합하면 수량 원자 갱신 결과가 최종 방어선이 되며 음수 수량이나 초과 판매를 허용하지 않는다.
- 모든 인스턴스는 중앙 판매·노출 상태와 가용 수량을 최종 확인한다. 검색 캐시나 직접 URL이 오래돼도 쓰기 요청은 차단한다.
- 재고 이벤트 중복·순서 역전은 이벤트 버전으로 무시하거나 재조회해 복구한다. 중앙 상태를 확인할 수 없으면 신규 수량 거래는 실패 폐쇄하고, 검색 색인·알림 장애는 판매 상태를 되돌리지 않는다.

### 알림·감사·개인정보·결제

- 단순 품절·숨김은 전체 사용자에게 알리지 않는다. 기존 확정 거래를 이행할 수 없을 때만 대상 사용자에게 대체 제안·취소 결과를 알리고, 대체는 명시적 동의를 받아야 한다.
- 행위자, 메뉴·판매 단위, 전후 판매·노출 상태, 수량 근거 버전, 사유, 효력 시각, 영향 거래와 복구 결과를 감사 이력으로 남긴다.
- 추가 개인정보 수집은 적용 대상이 아니다. 상태 변경만으로 결제·환불을 만들지 않고 기존 거래의 최종 취소·대체 결과가 PAY 정책을 호출한다.

### 관측 가능한 인수 조건

- 품절·판매 중지·숨김의 공개 표시와 신규 선택 가능 여부가 구분되고, 두 상태 축의 조합 테스트가 정의대로 동작한다.
- 품절 전환과 다중 인스턴스 홀드 요청을 동시에 실행해도 가용 수량이 음수가 되거나 초과 판매되지 않는다.
- 직접 링크·오래된 검색 결과로 품절·판매 중지·숨김 메뉴의 신규 거래를 만들 수 없다.
- 기존 확정 거래는 상태 변경만으로 삭제·대체·환불되지 않고 후속 처리 이력과 사용자 동의가 보존된다.

## OPER-007 이미지·설명 콘텐츠 기준

> 정책 상태: 확정

### 확정 규칙·행위자·사전 조건

- 식당 대표자는 자신이 관리하는 매장·메뉴의 이미지와 설명만 등록·변경할 수 있다. 플랫폼 운영자는 신고·명백한 정책 위반·보안 위험 콘텐츠를 숨기고 재검토할 수 있지만 대표자 명의로 새 홍보 문구를 작성하지 않는다.
- 이미지는 JPEG·PNG·WebP만 허용하고 파일 크기·가로·세로·전체 픽셀·디코딩·처리 시간은 중앙 보안 설정의 무료 내부 제한으로 검증한다. 실제 디코딩에 성공하고 방향 보정·메타데이터 제거·안전한 파생본 생성이 끝난 파일만 공개한다.
- 설명과 이미지에는 허위 가격·가용성, 타인 사칭, 불법·유해 콘텐츠, 악성 파일, 개인정보·연락처의 불필요한 노출, 사용 권한 없는 자료를 등록하지 않아야 한다. 자동 외부 유료 검수 서비스는 필수로 사용하지 않고 형식 검사·내부 금칙 기준·사용자 신고·운영자 검토를 조합한다.

### 상태·정상 흐름·변경·복구

- 콘텐츠 버전은 `초안` → `처리 중` → `게시`로 진행하며 검사 실패는 `처리 실패`, 신고·관리자 조치는 `숨김`, 대표자 교체는 `종료`로 기록한다.
- 새 이미지·설명 버전이 완전히 처리·검증되기 전에는 기존 게시 버전을 유지한다. 실패한 파일이나 중간 파생본을 공개하지 않는다.
- 대표자는 게시 버전을 새 버전으로 교체하거나 숨길 수 있다. 과거 거래·신고·심사가 참조하는 버전은 물리적으로 덮어쓰지 않고 식별자와 무결성 값을 보존한다.
- 관리자 숨김에 이의가 있으면 대표자는 새 자료와 사유로 재검토를 요청할 수 있다. 재검토 중에는 숨김을 유지하고 승인된 새 버전만 다시 공개한다.

### 동시성·장애·다중 인스턴스

- 업로드 확인·처리 작업은 콘텐츠 식별자·원본 무결성 값으로 멱등하게 수행하며, 중복 이벤트가 같은 공개 버전이나 파생 파일을 여러 번 만들지 않는다.
- 같은 콘텐츠의 동시 교체는 중앙 버전 조건으로 하나만 게시하고 오래된 처리 결과는 최신 버전을 덮어쓰지 못한다.
- 이미지 작업자·객체 저장소·검색 장애 시 새 버전을 `처리 중` 또는 `처리 실패`로 유지하고 제한 재시도한다. 재시도 소진 시 대표자에게 재업로드 경로와 운영자 경보를 제공하며 기존 정상 버전은 유지한다.

### 알림·감사·개인정보·결제

- 처리 실패·관리자 숨김·재검토 결과를 대표자에게 알리고, 신고자에게는 신고 처리에 필요한 최소 결과만 알린다. 내부 판정 규칙이나 다른 신고자의 정보는 공개하지 않는다.
- 콘텐츠·버전, 원본 무결성 값, 처리 결과, 게시·숨김 행위자, 사유, 신고·재검토 사건과 시각을 감사 이력으로 남기되 원본 URL·이미지 내용·개인정보를 로그에 복제하지 않는다.
- 원본 메타데이터를 공개본에서 제거하고 비공개 원본 접근을 최소 권한으로 제한한다. 정확한 원본 보관·파기 기간은 PRIV-005의 기존 법률·개인정보 결정을 따르며 OPER-007에서 새 기간을 정하지 않는다.
- 콘텐츠 변경 자체의 결제·환불·정산 영향은 적용 대상이 아니다. 가격·판매 상태를 바꾸려면 OPER-005·OPER-006의 별도 명령을 사용한다.

### 관측 가능한 인수 조건

- 잘못된 형식·디코딩 실패·제한 초과·악성 입력은 공개되지 않고, 게시 파일에는 원본 메타데이터가 남지 않는다.
- 중복 처리와 순서 역전에도 최신 승인 버전 하나만 공개되고 이전 정상 버전은 새 처리 실패로 사라지지 않는다.
- 대표자·권한 없는 운영자·다른 매장은 원본과 초안에 접근하거나 게시할 수 없다.
- 숨김·재검토·복구 과정의 행위자·사유·버전이 감사 가능하고 유료 검수 서비스 장애를 전제로 하지 않는다.

## OPER-008 알레르기·원산지·주류 표시

> 정책 상태: 확정

### 확정 규칙·책임·사전 조건

- 식당 대표자는 메뉴·옵션 버전을 게시하기 전에 알레르기 유발 가능 재료, 원산지 표시 값과 주류 여부를 구조화된 필드로 입력·확인할 책임이 있다. MiriYum은 입력·변경·노출 경로를 제공하지만 현장 조리 교차접촉이나 법적 표시의 사실성을 대신 보증하지 않는다.
- 알레르기 정보는 `포함`, `교차접촉 가능`, `확인 필요/정보 없음`을 구분하고 빈값을 안전함으로 해석하지 않는다. 사용자 화면에는 메뉴 선택·결제 전 확인 가능한 위치에 표시하고 정보 없음도 숨기지 않는다.
- 원산지는 대표자가 입력한 대상 재료·표시 값을 그대로 버전 관리한다. 시스템이 이미지·메뉴명·매장 주소로 원산지를 추론하거나 자동 보정하지 않는다.
- 주류 여부는 필수로 명시하며 주류 메뉴·옵션에는 주류 표시를 눈에 띄게 제공한다. 주류 거래는 AUTH-004의 온라인 연령 판정을 최종 명령에서 검사하고, 플랫폼 판정이 현장의 이용자 전원 나이·본인 확인을 대신하지 않음을 함께 안내한다.
- 관할 법령·표시 항목·문구가 변경되면 중앙 법률 표시 정책 버전으로 입력 필수 조건과 공개 형식을 갱신한다. 이 안전 표시 기본선은 최종 법률 의견이나 특정 세무·규제 책임을 미리 확정하지 않는다.

### 정상 흐름·변경·기존 거래

- 표시 정보는 OPER-005 메뉴 버전에 포함하여 `초안` → `게시` → `종료`로 관리한다. 중요한 표시 필드가 빠지거나 유효하지 않으면 해당 새 버전 게시를 실패 폐쇄하고 대표자에게 보완 항목을 반환한다.
- 게시 후 정정은 새 메뉴 버전으로 적용하고 과거 정보를 덮어쓰지 않는다. 즉시 안전 위험이 확인되면 대표자 또는 권한 있는 운영자가 OPER-006 판매 중지·숨김을 먼저 적용한 뒤 정정 버전을 게시한다.
- 기존 확정 거래가 정정 전 표시 버전을 참조하면 해당 스냅샷을 보존하고, 안전상 영향이 있는 정정은 영향 거래를 식별하여 사용자에게 최신 정보와 확인·변경·취소 경로를 제공한다. 사용자의 동의 없이 메뉴를 대체하지 않는다.

### 동시성·장애·다중 인스턴스

- 메뉴 게시와 표시 검증은 같은 메뉴·법률 정책 버전을 조건부로 확인하며, 여러 인스턴스가 서로 다른 필수 항목 기준으로 같은 버전을 게시하지 못하게 한다.
- 표시 정정·판매 중지·영향 거래 작업은 사건 식별자와 멱등 키로 한 번만 생성한다. 순서가 뒤바뀐 오래된 정정 이벤트는 최신 메뉴 버전을 덮어쓰지 않는다.
- 표시 정책이나 중앙 메뉴 원본을 확인할 수 없으면 새 메뉴 게시와 주류 거래를 실패 폐쇄한다. 공개 캐시·검색 장애는 중앙 판매 중지나 정정 결과를 되돌리지 않고 재색인으로 복구한다.

### 알림·감사·개인정보·결제

- 일반 표시 문구 변경은 전체 사용자에게 알리지 않는다. 안전에 영향을 주는 정정이 기존 거래와 연결될 때만 대상 사용자에게 변경 내용·확인 방법·거래 선택지를 알린다.
- 대표자 입력, 메뉴·표시·법률 정책 버전, 변경 전후 값, 게시·중지·숨김 행위자, 정정 사유, 영향 거래와 통지 결과를 감사 이력으로 남긴다.
- 알레르기 표시는 메뉴 정보이며 사용자의 건강정보를 추가 수집하지 않는다. 사용자가 별도로 제공한 알레르기 정보는 PRIV-003과 해당 거래 정책을 따르고 이 공개 메뉴 필드에 결합하지 않는다.
- 표시 정정만으로 가격·결제·환불을 바꾸지 않는다. 거래 변경·취소가 확정될 때 RES·HOLD·PAY 정책을 적용한다.

### 관측 가능한 인수 조건

- 필수 표시 누락·`정보 없음` 오해·주류 미표시 버전은 게시되지 않거나 명시된 보완 상태로 남고, 모든 공개 경로가 같은 표시 버전을 보여준다.
- 연령 미충족 사용자는 주류 정보 조회는 가능하지만 주류 홀드·예약·결제 명령이 모든 인스턴스에서 거부된다.
- 안전 정정 전후 거래가 각각 정확한 표시 스냅샷을 보존하며, 영향 거래 통지와 판매 중지 작업은 중복 생성되지 않는다.
- 시스템이 원산지·알레르기 안전성·현장 연령 확인 완료를 추론하거나 보증하는 표현을 생성하지 않는다.

## OPER-009 영업·휴점·폐점 상태 전이

> 정책 상태: 확정

### 확정 상태·권한·사전 조건

- 매장 운영 상태는 `영업`, `휴점`, `폐점` 세 가지 중앙 상태로 관리한다. 정규 영업시간 밖이라는 사실은 `휴점` 상태 전이가 아니며, `영업` 상태여도 현재 시각이 영업 구간 밖이면 `영업 중`으로 표시하지 않는다.
- 식당 대표자는 `영업` ↔ `휴점`을 즉시 또는 미래 시각으로 요청하고, 확인 절차를 거쳐 `영업`·`휴점`에서 `폐점`을 요청할 수 있다. 권한 있는 플랫폼 운영자는 정책 위반·소유권 사건·장애 복구 근거로 휴점·폐점을 적용하거나 명백한 오조작을 복구할 수 있으며 사유와 권한을 남겨야 한다.
- 폐점 전에는 현재 운영 권한, 매장 식별자, 기존 미래 거래와 미완료 결제·환불·분쟁 목록을 확인한다. 목록이 있다는 이유로 상태 전이를 숨기지 않되 각 거래의 처리 사건을 반드시 생성한다.

### 정상 전이·변경·만료

- `영업 → 휴점`은 신규 거래를 차단하고 기존 미래 거래를 영향 목록에 넣는다. `휴점 → 영업`은 입점·필수 기능 권한이 유효하고 폐업·긴급 중단·관리자 제한이 없으며 현재 적용할 영업시간·필수 표시가 유효할 때만 허용한다.
- `영업 또는 휴점 → 폐점`은 확인 화면에서 영향 거래 수와 비가역 기본 동작을 제시한 후 확정한다. 폐점은 공개 검색·신규 거래를 차단하고 사용자에게 폐점 상태를 표시하지만 과거 매장·메뉴·거래·감사 기록을 삭제하지 않는다.
- 폐점에는 자동 만료를 두지 않고 식당 대표자가 직접 영업으로 되돌릴 수 없다. 오조작 복구나 동일 사업자등록 단위의 재개는 `매장 상태 복구` 권한을 가진 운영자가 기존 귀속·입점·필수 표시를 다시 검증하고 감사 사건으로 복구한다. 운영 주체가 바뀌면 OPER-009 복구가 아니라 STORE-014를 사용한다.
- 종료 시각이 있는 예약 휴점은 OPER-003 규칙대로 만료될 수 있지만, 다른 휴점·긴급 중단·관리자 제한·폐점이 남아 있으면 자동 영업 전환하지 않는다.

### 동시성·실패·다중 인스턴스

- 상태 변경은 매장·상태 버전·멱등 키와 중앙 기준 시각으로 처리한다. 같은 버전에 대한 동시 전이는 최초의 유효한 하나만 확정하고 나머지는 최신 상태를 반환한다.
- 상태 전환과 권한 버전 갱신, 신규 거래 차단 기준, 영향 거래 사건·감사 이력 생성은 하나의 일관된 처리 경계에서 확정한다. 캐시·검색·알림은 후속 멱등 작업으로 수렴한다.
- 신규 거래 확정과 휴점·폐점이 경합하면 매장 상태 버전을 조건부로 검사하여 거래가 먼저 확정되어 영향 목록에 들어가거나 상태 전환이 먼저 확정되어 거래가 거부되는 둘 중 하나만 발생한다.
- 부분 실패 시 중앙 상태를 원본으로 재색인·캐시 무효화·영향 거래 생성을 재시도한다. 중앙 상태를 확인할 수 없으면 신규 거래와 영업 재개를 실패 폐쇄한다.

### 알림·감사·개인정보·결제

- 휴점·폐점과 복구를 공개 매장 화면에 반영하고, 기존 거래가 영향받으면 당사자에게 상태·후속 처리 예정·최종 거래 결과를 단계별로 알린다. 알림 실패가 상태 전환을 되돌리지 않는다.
- 행위자, 전후 상태·버전, 즉시·예약 효력 시각, 사유, 권한 근거, 영향 거래 수·식별자, 복구와 알림 결과를 감사 이력으로 남긴다.
- 상태 사유에는 불필요한 대표자·직원·고객 개인정보를 공개하지 않는다. 결제·환불·정산은 상태 전환만으로 소급 변경하지 않고 개별 거래 처리 결과와 PAY 정책을 따른다.

### 관측 가능한 인수 조건

- 영업시간 밖과 `휴점` 상태가 구분되고 휴점·폐점 매장은 오래된 캐시·직접 요청으로 신규 거래를 만들 수 없다.
- 동시 상태 전이와 거래 확정에도 상태 버전당 결과가 하나이며 누락된 영향 거래는 대사 작업으로 복구된다.
- 폐점 후 대표자 직접 재개가 거부되고, 운영자 복구 또는 STORE-014 전환은 권한·사유·검증 이력을 요구한다.
- 폐점 전 과거 거래·정산·감사 기록은 폐점 후에도 당시 귀속과 스냅샷을 유지한다.

## OPER-010 긴급 영업 중단 시 기존 예약 처리

> 정책 상태: 확정

### 확정 규칙·행위자·사전 조건

- 화재·시설·위생·안전 위험, 대표자 계정 탈취 의심 또는 플랫폼 장애처럼 신규 거래를 즉시 막아야 할 때 식당 대표자는 자신의 매장에 `긴급 중단`을 발동할 수 있다. `긴급 영업 중단` 권한이 있는 플랫폼 운영자도 근거와 범위를 기록하고 발동할 수 있다.
- 긴급 중단은 OPER-009 상태와 분리된 우선 차단 상태이며 `발동 중`인 동안 신규 예약·웨이팅·홀드·주류·결제 명령을 즉시 실패 폐쇄한다. 기존 거래 조회와 해결 작업은 계속 허용한다.
- 발동에는 공개 가능한 사유 범주, 내부 상세 사유, 시작 시각과 예상 영향 범위를 입력한다. 종료 시각을 모르는 경우 임의의 자동 만료를 두지 않으며 불필요한 제3자 개인정보를 수집하지 않는다.

### 기존 예약·웨이팅 기본 처리와 정상 흐름

- 발동 시 영향 시간대의 기존 예약·홀드·행사와 활성 웨이팅 항목을 중앙 스냅샷으로 고정하고 각각 `긴급 중단 확인 필요` 사건을 연결한다. 기존 거래 상태·시간·가격·결제 상태와 웨이팅 팀의 중단 직전 상태·순번·등록 시각·호출 이력을 보존하며 즉시 취소·이동·환불·삭제·재정렬하지 않는다.
- 사용자에게 먼저 긴급 중단 사실, 현재 예약·웨이팅이 자동 취소되거나 순번에서 제거된 것은 아니라는 점과 후속 안내 예정임을 알린다. 식당 대표자 또는 권한 있는 운영자는 예약·홀드·행사를 `기존대로 이행`, `변경 제안`, `매장 귀책 취소` 중 하나로 처리한다.
- 활성 웨이팅의 유지·호출 재개·종료·재등록 가능 여부와 순번 영향은 WAIT 도메인의 확정 정책에 따른 명시적 해결 명령으로만 정한다. OPER-010은 우선순위·재정렬·공정성 결과를 새로 정하지 않으며, WAIT 명령이 확정되기 전에는 중단 직전 중앙 순번과 상태를 그대로 보존한다.
- 변경 제안은 새 시간·구성·가격과 영향 내용을 제시하고 사용자의 명시적 동의를 받아야 효력이 생긴다. 동의 전에는 원래 거래를 유지하고, 거절·응답 없음의 최종 처리는 RES-009·RES-013과 각 거래 정책을 따른다.
- 매장 귀책 취소가 확정되면 거래 취소·자원 복구·환불 작업을 멱등하게 생성한다. 환불액·수수료·보상 책임은 PAY 및 해당 대표 팀 결정의 결과를 따르며 OPER-010이 새 금액이나 보상 약속을 만들지 않는다.
- 위험이 해소되면 발동자 또는 같은 권한을 가진 행위자가 재개 근거를 기록하고 긴급 중단을 해제한다. OPER-003 휴무, OPER-009 휴점·폐점, 관리자 기능 제한이 남아 있으면 해제만으로 영업·신규 거래를 재개하지 않는다.

### 동시성·멱등성·다중 인스턴스

- 매장에는 활성 긴급 중단 사건 하나만 허용한다. 반복 발동은 같은 사건에 멱등하게 연결하고 추가 사유·영향 범위는 버전으로 남긴다.
- 발동은 중앙의 매장 운영 세대 번호를 원자적으로 증가시킨다. 신규 거래 최종 확정은 시작 때 읽은 세대 번호를 조건부로 다시 검사하여, 거래가 먼저 확정되어 영향 스냅샷에 포함되거나 긴급 중단이 먼저 확정되어 거부되는 둘 중 하나만 발생하게 한다.
- 영향 거래·활성 웨이팅 스냅샷, WAIT 해결 명령, 알림·취소·환불·자원 복구 작업은 긴급 중단 사건과 거래·웨이팅 팀별 멱등 키를 사용한다. 여러 API·작업 인스턴스의 중복 실행과 이벤트 순서 역전이 웨이팅을 두 번 제거·종료·재정렬하거나 거래 결정·금액을 두 번 반영하지 못하게 한다.
- 모든 인스턴스는 중앙 긴급 중단 상태를 쓰기 전에 확인한다. 중앙 상태를 확인할 수 없으면 신규 거래는 실패 폐쇄하며, 로컬 캐시 만료를 기다리지 않고 상태 버전 무효화와 후속 대사로 최종 수렴한다.

### 실패·복구·알림·감사·개인정보

- 영향 거래·활성 웨이팅 검색이 부분 실패해도 긴급 중단 자체는 유지하고, 중앙 발동 시각·운영 세대와 거래·웨이팅의 중앙 생성·활성 상태를 기준으로 누락 항목을 반복 대사한다. 대사로 늦게 발견한 웨이팅도 중단 직전 순번·상태 증거를 복원해 같은 영향 사건에 멱등하게 연결하고, 자동 대사가 끝나지 않으면 운영자 경보와 수동 복구 큐로 보낸다.
- 사용자 알림 실패는 중단·거래·웨이팅 상태를 되돌리지 않는다. 예약 당사자와 WAIT 정책상 알림 대상에게 사건·대상별 멱등 키로 내구성 있는 알림을 제한 재시도하고, 최종 실패는 권한 있는 운영자의 수동 재발송 대상으로 남긴다. 알림 성공을 거래 변경 동의나 웨이팅 해결 명령으로 추정하지 않는다.
- 사건 ID, 발동·해제 행위자, 공개·내부 사유, 운영 세대, 영향 범위·거래 스냅샷, 웨이팅 팀 식별자와 중단 직전 상태·순번·등록·호출 증거, 각 거래 결정과 WAIT 해결 명령·결과, 동의·취소·환불 작업, 실패·대사·복구·알림 결과를 수정·삭제 불가능한 감사 이력으로 남긴다.
- 공개 알림과 상태에는 안전한 사유 범주만 사용하고 내부 사고 상세, 대표자·직원·고객 개인정보와 보안 증거를 노출하지 않는다. 새 건강·위치·신분정보 수집은 적용 대상이 아니다.

### 관측 가능한 인수 조건

- 긴급 중단 발동 직후 모든 인스턴스의 신규 거래가 거부되고 기존 거래 조회·해결은 가능하다.
- 발동과 예약 확정 경합에서 예약이 누락되지 않고 정확히 한 결과와 영향 사건을 가지며, 반복 발동·작업 재시도는 중복 취소·환불·자원 복구를 만들지 않는다.
- 기존 예약은 발동만으로 자동 취소·이동·가격 변경되지 않고, 변경은 사용자 동의, 취소·환불은 해당 거래 정책의 확정 명령을 요구한다.
- 활성 웨이팅이 있는 상태에서 긴급 중단을 발동하면 모든 팀의 중단 직전 상태·순번·등록·호출 증거와 영향 사건이 보존되고 조용한 제거·재정렬이 발생하지 않는다. 여러 인스턴스의 반복 해결·알림 명령에도 WAIT 정책이 정한 결과와 알림이 팀별 한 번만 반영된다.
- 영향 거래 조회·알림·검색 장애를 주입해도 중단은 유지되고 대사·재시도·운영자 큐로 복구되며, 해제 후에도 남은 휴무·휴점·폐점 제한을 우회하지 않는다.
