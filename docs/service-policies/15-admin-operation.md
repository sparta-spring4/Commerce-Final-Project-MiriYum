## 운영자 공통 불변식

- 운영자 계정은 일반 사용자·식당 대표자와 분리하고 모든 화면·API·작업에서 중앙 역할, 세부 권한, 사건 배정, 계정 상태와 추가 인증 상태를 다시 검증한다. 슈퍼관리자 여부만으로 업무상 필요가 없는 원문 개인정보·결제수단·비밀을 포괄 조회하지 않는다.
- 중요 작업은 현재 운영자 비밀번호 재인증과 세션 재확인을 요구하고, 성공 뒤 짧은 목적 결속 승인으로 한 번만 사용한다. `ADMIN-002`의 WebAuthn 도입 시점·범위·등록·복구는 TODO이며 현재 안전 기본선을 WebAuthn 확정으로 표현하지 않는다.
- 감사 사건은 행위자, 역할·권한 버전, 대상·변경 전후 상태, 사유, 승인자, 요청·사건 상관관계 ID와 시각을 내구성 원장에 남긴다. 일반 관리 API로 수정·삭제하지 못하고 정정은 원 사건을 보존한 보정 사건으로만 추가하며, 조회에도 별도 최소 권한과 조회 감사를 적용한다. 도메인 정책이 이미 보존기간을 확정한 감사 유형은 그 기간을 우선하고, 그 밖의 항목별 보존기간·최종 법률 문구만 `ADMIN-009` TODO로 둔다.
- 관리자 명령은 중앙 사건 ID와 멱등 키, 허용 상태 전이를 사용한다. 여러 운영 인스턴스의 중복·동시 요청에는 최초의 유효한 결정 하나만 확정하고 알림·후속 작업은 같은 사건에서 내구성 있게 한 번만 생성한다.
- 운영자 화면·알림·로그는 개인정보를 기본 마스킹하고 사건 처리에 필요한 필드만 단계적으로 공개한다. 일반 운영자 화면·업무 API에서는 자유형 대량 조회·내보내기, 비밀 조회, 직접 DB 수정과 공용 계정 사용을 허용하지 않는다. 개인정보 대량 조회·내보내기는 `PRIV-011`의 고도화된 관리자 통제 절차가 확정되기 전까지 비활성화하며, `ADMIN-010`의 일반 대량 작업 승인을 받았다는 사실만으로 허용하지 않는다.

## ADMIN-001 운영자 계정 발급

> 정책 상태: 확정. 아래 슈퍼관리자 발급·최초 비밀번호 변경·감사 하위 범위는 기존 확정으로 보존한다.

### 확정 사항

- 운영자는 공개 회원가입으로 계정을 만들 수 없다.
- 슈퍼관리자만 새로운 운영자 계정을 생성하고 초기 권한을 부여할 수 있다.
- 슈퍼관리자는 운영자 이메일과 임시 비밀번호를 등록한다.
- 신규 운영자는 최초 로그인에서 임시 비밀번호를 본인만 아는 새 비밀번호로 반드시 변경해야 한다.
- 비밀번호 변경이 완료되기 전에는 운영자 기능을 사용할 수 없다.
- 운영자 계정 생성, 초기 권한 부여 및 최초 비밀번호 변경 완료는 감사 로그에 남긴다.

### 다중 서버·다중 인스턴스 원칙

- 임시 비밀번호 상태와 최초 변경 완료 여부는 중앙 인증 상태로 관리한다.
- 여러 인스턴스에 최초 로그인 요청이 동시에 도착해도 한 번의 유효한 비밀번호 변경만 확정되도록 원자적으로 처리한다.
- 비밀번호 변경이 확정되는 즉시 기존 임시 자격 증명은 모든 인스턴스에서 더 이상 사용할 수 없어야 한다.

### ADMIN-001 남은 범위 — 확정 안전 기본선

- 운영자 역할은 심사, 고객지원, 계정·가게 제재, 결제 복구, 감사 조회, 장애 대응과 슈퍼관리자처럼 업무 단위로 분리한다. 계정 생성자는 요청된 최소 역할만 부여하고 서로 다른 업무 권한을 편의상 묶거나 공용 계정을 발급하지 않는다.
- 임시 비밀번호는 확인된 운영자에게만 제공하는 일회성 전달 절차를 사용하고 평문을 감사·로그·티켓에 남기지 않는다. 유효기간과 실패 한도는 중앙 보안 설정으로 제한하며 만료·한도 초과·재발급 시 이전 임시 비밀번호를 즉시 무효화한다.
- 역할 추가·회수, 정지·퇴사 처리와 고위험 권한 승격은 중앙 계정 버전의 조건부 변경으로 처리한다. 슈퍼관리자·감사 전체 조회·결제 복구 같은 고위험 권한은 요청자와 다른 승인자의 이중 승인을 요구하고, 임시 승격은 목적과 만료를 가진다.
- 계정 정지·역할 회수·비밀번호 재발급이 확정되면 중앙 세션 버전을 갱신하여 모든 인스턴스에서 기존 세션과 미사용 추가 인증 승인을 거부한다. 인증 저장소 장애에는 새 권한을 발급하지 않고 실패 폐쇄한다.
- 계정 발급·임시 자격 전달 완료·실패·재발급·역할 변경·정지·재활성화와 세션 철회를 감사하고, 대상 운영자에게 민감정보 없는 보안 알림을 내구성 작업으로 보낸다. 발송 실패가 권한 회수를 되돌리지는 않는다.

### 관측 가능한 인수 조건

- 공개 가입이나 일반 운영자가 새 운영자·슈퍼관리자를 만들 수 없고 고위험 역할 승격에는 서로 다른 두 계정의 승인 기록이 필요하다.
- 동시 최초 로그인·재발급·역할 회수 요청 뒤 유효한 자격·역할 버전은 하나이며 회수 전 세션은 모든 인스턴스에서 거부된다.
- 임시 비밀번호 원문이 로그·감사·알림에 없고 만료·실패 한도 초과·재발급 뒤 이전 자격으로 운영자 API를 사용할 수 없다.

## STORE-004 연계 — 매장 소유권 충돌 재확인

### 확정 사항

- 동일한 사업자등록번호의 중복 등록 신청은 중앙 `매장 소유권 충돌 심사` 사건으로 접수하며 `매장 소유권 충돌 심사` 권한을 가진 관리자 1명이 재확인한다.
- 관리자는 사업자 조회 결과, 신규 신청 정보와 기존 귀속·변경 이력을 확인하고 단순 중복·허위 신청, 같은 대표자의 계정 접근 상실, 실제 양도·대표자 변경 가운데 하나로 분류한다.
- 관리자는 이 심사 권한만으로 기존 가게를 중지하거나 매장을 다른 계정으로 이전할 수 없다. 계정 접근 상실은 AUTH-006, 실제 양도·대표자 변경은 STORE-014 절차로 연결한다.
- 신청자와 기존 가게에는 상대방의 연락처·계정 식별정보·제출 자료를 공개하지 않는다. 분류 결과와 근거, 처리 관리자와 처리 시각은 수정·삭제할 수 없는 감사 이력으로 남긴다.

### 다중 서버·다중 인스턴스 원칙

- 중앙 권한 저장소의 전용 심사 권한과 사건 배정을 모든 관리자 인스턴스가 동일하게 검증한다.
- 동시 분류 요청에는 최초의 유효한 결과 하나만 반영하고, 분류 확정·감사 이력·후속 사건 생성을 하나의 일관된 처리 경계에서 기록한다.

## STORE-005 연계 — 기본 입점 심사

### 확정 사항

- `기본 입점 심사` 권한을 가진 관리자 1명만 자동 검사를 모두 통과한 신청을 심사할 수 있다.
- 관리자는 신청자가 글로 입력한 사업자등록번호·상호·대표자명·개업일자·사업장 주소·업태·종목과 제출 이미지가 같은지 체크리스트로 확인하고 업태·종목의 지원 외식업 범위를 판정한다.
- 관리자는 국세청 자동 검사 실패나 미완료 상태를 우회 승인할 수 없다. 이미지 식별 불가·정보 불일치·업종 모호성은 보완 요청하고 명백한 허위 또는 지원 제외 업종은 사유를 기록해 거절한다.
- 사업자등록증 이미지는 배정된 사건의 심사에 필요한 기간에만 접근하고 다운로드·접근 이력을 감사 가능하게 남긴다. 승인·보완·거절 결과에는 사용한 신청 버전과 비교 결과를 연결한다.

### 다중 서버·다중 인스턴스 원칙

- 관리자 배정, 전용 권한과 신청 버전을 중앙에서 다시 검증하고, 이전 버전이나 이미 종결된 사건에 대한 결정을 거부한다.
- 승인·보완·거절의 동시 요청에는 최초의 유효한 결정 하나만 반영하고 결정·감사 이력·후속 기능 활성화 요청을 일관되게 기록한다.

## STORE-008 연계 — 제한 기능·테이스팅 행사 심사

> 적용 경계: 이 절차는 `STORE-008`과 `TASTE-001`의 정본이 해당 제한 기능 또는 테이스팅을 활성화한 경우에만 적용한다. 심사 권한과 큐를 정의했다는 사실만으로 현재 MVP 범위를 넓히거나 기능을 활성화하지 않는다.

### 확정 사항

- `제한 기능 심사` 권한을 가진 관리자만 가게별·기능별 승인과 테이스팅 행사별 승인을 처리할 수 있다.
- 예약금은 현재 MVP에서 제한 기능 심사 대상 예외이며, 관리자는 예약금 자체의 사용 승인·거절을 처리하지 않는다. 대신 `PAY-002`에서 정한 허용 최소·최대 범위 밖의 설정은 시스템이 저장·활성화하지 않는다.
- 가게·기능 승인 시 다른 제한 기능이나 다른 가게를 함께 승인할 수 없고, 테이스팅 기능 승인만으로 개별 행사를 승인할 수 없다.
- 관리자는 신청 버전, 심사 근거, 승인·보완·거절 결과와 처리 시각을 감사 이력에 남긴다.
- 테이스팅 행사의 중요 내용이 바뀌면 새 버전 심사를 요구하고, 새 버전 승인 전에는 기존 승인본만 유효한 것으로 처리한다. 표현·비핵심 이미지 변경은 재승인 없이 변경 이력만 남긴다.

### 다중 서버·다중 인스턴스 원칙

- 가게·기능 심사와 행사·버전 심사를 별도 중앙 사건으로 관리하고, 각각의 동시 결정에는 최초의 유효한 결과 하나만 반영한다.

## ADMIN-006 환불·보상 승인 한도와 감사

> 정책 상태: 확정. 2026-07-22 사용자 직접 결정

### 확정 승인 경계

- 원 결제금액 이하의 환불과 PAY 정본상 실행 가능한 보상은 같은 불변 거래 계보의 연결 요청 누계가 200,000원 이하이면 `결제 관리자` 1명이 승인할 수 있다.
- 원 결제금액 이하에서 연결 요청 누계가 200,000원을 초과하면 `결제 관리자`의 요청과 그 요청자와 다른 `슈퍼관리자`의 추가 승인이 모두 필요하다. 정확히 200,000원은 결제 관리자 단독 승인 경계에 포함한다.
- 승인 등급은 API 명령·승인 사건의 건별 금액이 아니라 PAY 정본이 같은 원 결제·주문·예약·사건·수취인·사유의 거래 계보로 연결한 환불·보상·정정 요청 누계로 판정한다. 거래 계보 ID는 시스템이 원 결제·주문·예약의 불변 루트 식별자로 생성하고, 요청 사유·명령 ID·사건 ID·정책 버전·운영자를 바꾸거나 금액을 나누어도 같은 루트 거래를 새 계보로 분리하지 않는다. 별도 계보는 독립된 원인 사건과 그 증거가 검증된 경우에만 생성한다.
- 원 결제금액을 초과하는 별도 보상은 환불이 아니며 금액과 무관하게 `슈퍼관리자` 승인이 필요하고 요청자와 승인자는 서로 달라야 한다. `ADMIN-006`은 관리자 승인 조건만 정하며, 실제 지급 가능 여부·금액 계산·지급 수단·원장·취소·회계 처리는 PAY 정본을 따른다. PAY 정본에 실행 가능한 별도 보상 정책이 확정되지 않았다면 관리자 승인만으로 지급을 생성할 수 없다.
- `PAY-014` 수동 복구에서도 같은 거래 계보의 연결 요청 누계가 원 결제금액 이하이면서 200,000원 이하이면 사건에 배정되고 `결제 복구`와 `결제 관리자` 권한을 모두 가진 같은 운영자가 요청·승인·실행할 수 있다. 이를 `단독 승인`으로 구분하며 자기 승인 거부 규칙은 추가 승인이 필요한 구간에만 적용한다.
- 승인 전에는 대상 거래·사건, 원 결제금액, 현재 환불·보상 누계, 요청 금액·사유·근거, 정책 버전과 승인자 자격을 중앙 원장에서 다시 검증한다. 이미 성공·취소된 같은 명령을 새 거래로 반복하지 않는다.

### 다중 인스턴스·감사·개인정보

- 요청·승인·실행은 중앙 승인 사건의 상태 버전과 멱등 키로 관리한다. 다중 인스턴스의 동시 승인·재시도·복구는 최초 유효한 승인과 외부 명령 하나만 확정하며, 결과 불명은 PG 재조회·웹훅·대사 뒤에만 종결한다.
- 금액이 있는 모든 승인 사건에는 최소한 대상 거래·사건 ID, 요청자와 승인자 계정, 요청·승인 시각, 원 결제금액, 요청·승인 금액, 승인 유형, 사유·근거 참조, 정책 버전, 멱등 키, PG 처리 결과와 실패·재시도 이력을 변경 불가능한 감사 기록으로 남긴다. 단독 승인에는 요청자와 승인자로 같은 계정을 기록하고, 추가 승인이 필요한 유형에는 서로 다른 계정을 기록한다.
- 감사·운영 화면에는 카드번호·계좌번호 등 결제수단 원문, 인증정보, 사건 처리에 불필요한 연락처·개인정보를 저장하거나 표시하지 않는다.

### 관측 가능한 인수 조건

- 연결 요청 누계 200,000원은 결제 관리자 1명으로 승인되며, `PAY-014` 수동 복구에서는 필요한 권한과 사건 배정을 가진 같은 운영자가 요청·승인·실행할 수 있다. 200,001원은 결제 관리자 요청과 서로 다른 슈퍼관리자 추가 승인 없이는 실행되지 않는다.

- 여러 인스턴스에서 같은 200,001원 승인·복구 요청을 동시에 처리하거나 작업자가 중단되어도 외부 환불·보상은 한 번만 실행되며, 모든 최소 감사 필드와 실패·재시도 이력이 남는다.

- 모든 break-glass 요청·승인·거부·발급·명령·데이터 조회·수동 보정·모드 변경·검증을 실시간 감사하고 사건 종료와 승인 만료 중 먼저 도달한 시점에 권한을 자동 회수한다. 사후 검토는 재발 방지 작업·담당·검증 상태를 추적하되 실행 전 승인 의무를 대신하지 않는다.

### 관측 가능한 인수 조건

- 한 인스턴스에서 장애 모드를 바꾸면 다른 인스턴스가 같은 중앙 버전을 적용하며 오래된 요청으로 해제되지 않는다.
- 일반 장애 복구용 break-glass는 사건·목적·대상·범위·만료·현재 재인증과 서로 다른 승인자 없이 발급되지 않는다. 승인 서비스 장애·계획 변경·버전 불일치·사건 종료·만료 뒤 새 명령은 거부되고, 원문 개인정보·개인정보 대량 내보내기·감사 전체 열람·광범위 민감정보 접근은 현재 정책 버전에서 항상 거부된다.
- 사전 정의된 가역 통제는 불필요한 권한 상승 없이 실행되지만, 같은 화면이나 API로 원문 PII·대량 내보내기·광범위 민감정보 접근을 우회할 수 없다.
- 복구 선언 전 원장·큐·캐시·외부 대사와 핵심 불변식 검증이 필요하며 실패 항목은 종료가 아닌 미해결 상태로 남는다.

### 확정 일반 확인·권한·공개 경계

- 고객지원 사건은 요청 목적, 대상 계정 유형, 허용 정보·작업과 필요한 소유 확인 수단을 버전 정책으로 정한다. 단순 공개 안내에는 본인 확인을 요구하지 않고 계정·거래·제재 상세 조회나 변경에만 목적에 필요한 확인을 요구한다.
- 등록 휴대전화 또는 등록 이메일의 소유 확인처럼 기존 확정 수단을 우선하고, 해당 목적이 둘 모두를 요구한다고 명시하지 않은 한 추가 신분증·정밀 위치·기기 지문·새 외부 본인확인을 수집하지 않는다. 기존 수단을 사용할 수 없으면 AUTH-006 복구로 전환하며 운영자 재량 질문으로 우회하지 않는다.
- 운영자는 인증번호·비밀번호를 보거나 대신 입력하지 않는다. 중앙 확인 서비스의 성공 여부, 대상 계정·목적·만료만 참조하고 확인 전에는 계정 존재·연락처·예약·제재·신고자 정보를 구분하지 않는 응답을 사용한다.
- 확인 성공은 사건·목적에 결속된 짧은 권한이며 다른 고객지원 사건, 결제·동의·로그인 또는 대리 세션으로 재사용하지 않는다. 민감 상태 변경에는 작업별 권한과 `ADMIN-002` 현재 재인증 기본선을 추가 적용한다.

### 동시성·실패·알림·감사·개인정보

- 코드 발급·검증 횟수와 소유 확인 상태는 중앙에서 원자 관리하고 연락처·IP·요청 경로별 제한을 모든 지원 인스턴스가 공유한다. 재발급은 이전 시도를 무효화하고 만료·소비된 증거는 재사용하지 않는다.
- 확인 서비스 장애·결과 불명에는 정보를 공개하거나 상태를 변경하지 않는다. 사건을 보류하고 안전한 재시도 또는 AUTH-006 복구 안내로 전환한다.
- 확인 요청·성공/실패·공개 필드·조회·변경·거부와 담당자를 감사하고 화면·알림은 연락처와 사건 정보를 마스킹한다. 중요한 변경 결과는 확인된 채널로 알리되 알림 실패가 완료된 보안 조치를 되돌리지 않는다.

### 관측 가능한 인수 조건

- 본인 확인 전 계정 존재·예약·제재 상세를 요청해도 공통 응답만 받고 운영자도 원 인증번호·비밀번호를 조회할 수 없다.
- 한 사건의 확인 증거로 다른 계정·목적·결제·동의를 실행하면 거부되며 만료·재발급 뒤 이전 증거도 사용할 수 없다.
