## AUTH-001 지원 회원 유형과 계정 관계

> 정책 상태: 확정

### 현재 확정 사항

- 일반 사용자 계정과 식당 대표자 계정을 서로 독립된 계정 유형으로 분리한다.
- 같은 사람이 일반 사용자 계정과 식당 대표자 계정을 각각 보유할 수 있다.
- 하나의 식당 대표자 계정은 여러 매장에 소속될 수 있다.
- 플랫폼 운영자 계정은 일반 사용자 및 식당 대표자 계정과 분리된 별도 계정 유형으로 관리한다.
- 식당 직원용 계정은 지원하지 않으며, 매장 관리 시스템에는 식당 대표자만 로그인할 수 있다.
- 식당 대표자는 해당 매장의 사업자등록증에 기재된 대표자 본인으로 정의한다.
- 개인사업자는 사업자등록증상 대표자 본인, 법인사업자는 사업자등록증상 법인 대표자 본인만 식당 대표자 계정을 사용할 수 있다.
- 식당 대표자 계정을 직원 또는 제3자와 공유하는 행위는 허용하지 않는다.
- 서로 다른 계정 유형의 인증 상태와 권한은 자동으로 공유하거나 승격하지 않는다.
- 비회원은 공개된 매장·메뉴·가격·예약 가능 여부를 탐색하고 조회할 수 있다.
- 예약, 웨이팅, 메뉴 홀드, 결제 및 리뷰 등 상태를 생성하거나 변경하는 기능은 일반 사용자 로그인이 필요하다.
- 현재 정책에서는 휴대전화 인증만으로 비회원 예약이나 비회원 웨이팅을 생성하는 방식을 지원하지 않는다.

### 정책 근거

- 고객 기능과 매장 운영 기능 사이의 보안 경계를 분명히 한다.
- 일반 사용자 계정의 탈취나 제재가 식당 운영 권한으로 자동 전파되는 것을 방지한다.
- 대표자의 다점포 운영을 계정 복제 없이 표현한다.
- 플랫폼 운영자의 고위험 권한을 서비스 이용 계정과 격리한다.
- 실제 작업자를 식별할 수 없는 매장 공용 계정의 생성과 사용을 방지한다.
- 공개 정보의 접근 장벽은 낮추되 거래성 요청은 계정 단위로 추적하여 중복 처리, 어뷰징 및 분쟁에 대응한다.

### 다중 서버·다중 인스턴스 원칙

- 계정 유형과 매장 소속은 특정 애플리케이션 인스턴스의 로컬 상태가 아닌 중앙의 권한 기준으로 판정한다.
- 모든 인스턴스가 같은 계정 유형과 소속 관계를 기준으로 권한을 검증해야 한다.
- 계정 유형 간 전환이나 권한 승격으로 해석될 수 있는 요청은 명시적인 인증 경계를 통과해야 한다.
- 공개 조회는 다중 인스턴스와 캐시 계층에서 확장할 수 있어야 하며, 비회원 요청에도 IP·기기 등 비계정 기준의 속도 제한을 적용할 수 있어야 한다.
- 거래성 요청은 인증된 계정 식별자를 기준으로 멱등성, 중복 제한 및 감사 추적을 적용한다.

## AUTH-002 이메일·소셜·휴대전화 로그인 수단

> 정책 상태: 확정

### 현재 확정 사항

- 일반 사용자와 식당 대표자에게 동일한 로그인 수단과 인증 강도를 일괄 적용하지 않는다.
- 일반 사용자는 이용 편의를 우선하여 소셜 로그인을 중심으로 하고 이메일을 보조 수단으로 제공하는 방향을 사용한다.
- 일반 사용자의 보조 이메일 로그인은 이메일 주소와 비밀번호를 사용하는 방식으로 제공한다.
- 이메일 일회용 코드나 휴대전화 OTP를 일반 사용자의 기본 보조 로그인 수단으로 사용하지 않는다.
- 최초 소셜 로그인 제공자는 카카오로 제한한다.
- 식당 대표자는 이메일 기반 로그인을 사용하고 휴대전화 본인 확인을 결합하는 방향을 사용한다.
- 식당 대표자의 로그인 자격 증명은 이메일 주소와 비밀번호로 구성한다.
- 식당 대표자의 휴대전화는 본인 확인 수단으로 사용하며 휴대전화 OTP를 기본 로그인 수단으로 사용하지 않는다.
- 플랫폼 운영자 계정의 발급과 초기 자격 증명은 AUTH-002의 공개 가입 범위에서 제외하고 ADMIN-001에서 관리한다.
- 일반 사용자와 식당 대표자의 이메일·비밀번호 계정은 이메일 소유 확인을 완료해야 가입이 확정되고 계정이 활성화된다.
- 카카오 로그인만 사용하는 일반 사용자는 이메일·비밀번호 로그인 수단을 추가할 때 해당 이메일 소유 확인을 완료해야 한다.
- 현재 로그인 방식은 카카오와 이메일·비밀번호로 확정했고, 계정 복구는 `AUTH-006` 확정 정책을 따른다. 후속 TODO는 추가 소셜 제공자와 그 제공자별 계정 연결·해제·중복 처리에 한정한다.

### 정책 근거

- 일반 사용자에게는 가입·로그인 마찰을 낮추고, 식당 대표자 계정에는 더 강한 신원 확인을 적용한다.
- 한 인증 수단이나 외부 사업자의 장애가 모든 계정 유형의 로그인을 동시에 중단시키지 않도록 경계를 분리한다.
- 소셜 인증 사업자를 사용할 수 없는 경우에도 일반 사용자가 플랫폼이 관리하는 이메일 자격 증명으로 로그인할 수 있게 한다.
- 식당 대표자의 이메일 로그인과 휴대전화 본인 확인을 분리하여 문자·본인확인 사업자의 일시적 장애가 모든 기존 대표자의 기본 로그인을 즉시 차단하지 않게 한다.

### 다중 서버·다중 인스턴스 원칙

- 로그인 수단별 인증 결과와 계정 유형은 중앙 인증 상태를 기준으로 검증하며 애플리케이션 인스턴스의 로컬 상태에 의존하지 않는다.
- 모든 애플리케이션 인스턴스는 중앙의 이메일 확인 상태를 기준으로 미확인 이메일·비밀번호 계정의 인증과 권한 부여를 차단한다.
- 외부 인증 응답의 중복 전달과 순서 역전을 고려하고, 동일한 외부 신원이 중복 계정을 생성하지 않도록 일관된 연결 기준을 사용한다.

## AUTH-003 휴대전화 본인 확인 시점과 재인증

> 정책 상태: 확정

### 현재 확정 사항

- 일반 사용자는 가입 완료 전에 휴대전화 본인 확인을 반드시 완료해야 한다.
- 카카오 가입과 이메일·비밀번호 가입에 동일한 기준을 적용한다.
- 휴대전화 본인 확인을 마치지 않은 가입 시도는 활성 계정으로 확정하지 않는다.
- 식당 대표자도 계정 가입 완료 전에 본인 휴대전화 인증을 반드시 완료해야 한다.
- 식당 대표자의 계정 본인 확인과 매장 추가 시 사업자등록증 인증은 서로 다른 절차로 관리한다.
- 휴대전화 본인 확인은 정기적으로 반복하지 않는다.
- 휴대전화 번호 변경, 계정 복구 또는 계정 탈취가 의심되는 경우에만 휴대전화 재인증을 요구한다.
- 장기 미사용, 매장 추가 또는 일반적인 민감 작업이라는 이유만으로 휴대전화 재인증을 반복하지 않는다.
- 본인 확인에 성공한 뒤에만 계정을 생성하고 가입을 완료한다.
- 본인 확인 실패·취소 또는 외부 본인확인 사업자 장애 시에는 계정을 생성하지 않고 가입 절차에서 재시도하게 한다.
- 본인 확인 전에 입력된 정보는 계정이 아닌 짧은 수명의 가입 세션으로만 유지한다.

- 가입 단계의 추가 마찰과 외부 본인확인 사업자 장애 시 가입 중단 위험을 감수하고 초기 계정 신뢰도를 우선한다.

- 휴대전화 본인 확인 상태와 가입 완료 여부는 중앙 인증 상태를 기준으로 원자적으로 확정한다.

## AUTH-004 주류 이용자의 연령 확인

> 정책 상태: 확정

### 현재 확정 사항

- 일반 사용자의 가입 시 필수 본인 확인 결과로 주류 이용 가능 연령 여부를 판정한다.
- 주류 이용 가능 연령 판정만을 목적으로 최초 주류 기능 이용이나 매 거래마다 본인 확인을 반복하지 않는다.
- 연령 기준은 요청 시점에 적용되는 대한민국 청소년 보호 법령을 따른다.
- 연령 기준을 충족하지 못한 사용자도 일반 식당 탐색·예약은 이용할 수 있다.
- 연령 기준을 충족하지 못한 사용자의 주류 상품 주문·홀드·결제와 주류 행사 예약·참여는 허용하지 않는다.
- 플랫폼의 연령 판정은 식당 현장에서 상대방의 나이와 본인 여부를 확인하는 절차를 대체하지 않는다.
- 주류 행사에서는 실제 참가자 전원, 일반 식사에서는 실제로 주류를 제공받는 사람 전원을 가게가 현장에서 확인한다.
- 현장 나이·본인 확인은 가게의 책임이며, MiriYum은 현장 확인을 직접 수행하거나 완료를 보증하지 않는다.
- MiriYum은 방문·예약·참가자별 현장 확인 완료 상태를 입력하도록 요구하거나 저장하지 않는다.
- 현장에서 제시된 신분증 이미지·사본·식별정보를 수집하거나 저장하지 않는다.
- 가입 본인확인 결과에서는 법적 기준에 따른 주류 이용 가능일과 확인 시점만 파생해 저장한다.
- 연령 판정 목적으로 생년월일 원본을 저장하지 않는다.
- 연령 기준 미충족 사용자에게도 주류명·가격·코스 구성 등 이용에 필요한 메뉴 정보와 일반 광고는 표시할 수 있다.
- 연령 기준 미충족 사용자에게는 배너·쿠폰·추천형 광고 등 주류 판촉 광고를 노출하지 않는다.

### 현재 확정 사항

- 비밀번호는 최대 64자까지 허용하며 제한을 초과한 값을 자동으로 잘라 저장하거나 검증하지 않는다.

- 수동 복구 후 재설정 코드에도 발급 후 10분 유효, 재발급 시 이전 코드 즉시 무효화, 60초 재전송 대기, 최초 발급 포함 동일 이메일 1시간 최대 5회, 코드별 누적 3회 입력 실패 시 해당 코드 무효화 기준을 동일하게 적용한다.
- 새 비밀번호 설정이 성공하면 복구 후 비밀번호 재설정 필요 상태를 해제하되 자동 로그인시키지 않고 새로 로그인하게 한다.
- 수동 복구가 승인되어 등록 이메일이 교체되면 교체 전 기존 이메일에도 등록 이메일 변경 사실을 알리는 보안 알림을 발송한다. 이 알림에는 복구 후 이메일 주소, 제출 증빙과 심사 세부정보를 포함하지 않는다.
- 교체 전 이메일의 변경 보안 알림 발송에 실패해도 수동 복구 승인, 등록 이메일 교체와 기존 세션 종료를 되돌리지 않는다. 최초 발송 실패 후 직전 실패 시점을 기준으로 1분·5분·15분 간격으로 최대 3회 추가 재시도한다.
- 추가 재시도까지 모두 실패하면 최종 실패 상태를 보관하고 권한 있는 플랫폼 운영자에게 경보와 감사 가능한 수동 재발송 경로를 제공하되, 복구된 계정의 이용은 제한하지 않는다.
- 승인과 거절 모두 심사 사건 식별자, 대상 계정 식별자, 담당 운영자 식별자, 증빙별 확인 결과, 결정 결과와 사유, 처리 시각을 중앙 감사 이력에 남긴다.
- 담당 운영자는 자신이 생성한 감사 이력을 수정하거나 삭제할 수 없다. 수동 복구 감사 이력은 심사 처리 완료 시점부터 3년간 보관한 뒤 파기한다.
- 완료된 수동 복구 감사 이력 전체는 슈퍼관리자 또는 별도의 `계정 복구 감사 열람` 권한을 가진 플랫폼 운영자만 열람할 수 있다. `계정 복구` 승인 권한만으로는 완료된 전체 감사 이력을 열람할 수 없다.
- 진행 중인 심사 담당자는 자신에게 배정된 사건을 처리하는 데 필요한 범위만 조회하며, 이 권한을 완료된 전체 감사 이력 열람 권한으로 확대하지 않는다.
- 수동 복구 신청은 접수 후 3영업일 이내에 승인, 거절 또는 보완 요청 가운데 하나로 1차 처리 결과를 안내한다. 보완 요청 후 최종 처리 기한은 보완 자료가 다시 접수된 시점부터 새로 계산한다.
- 이메일·비밀번호 로그인은 같은 계정에서 연속 5회 실패하면 1분 동안 지연하고, 지연 종료 후 다시 실패하면 5분, 이후 다시 실패하면 최대 15분까지 단계적으로 지연한다. 15분 단계 이후의 추가 실패에는 15분 지연을 반복 적용하며 영구 잠금으로 전환하지 않는다.
- 이메일·비밀번호 로그인에 성공하면 해당 계정의 연속 실패 횟수와 지연 단계를 초기화한다. 로그인 지연 중에도 카카오 로그인과 비밀번호 재설정·계정 복구는 허용한다.
- 계정 기준 지연과 별도로 IP·요청 경로의 반복 로그인 시도를 중앙 기준으로 제한하며, 실패 횟수와 지연 상태 또는 계정 존재 여부를 응답으로 노출하지 않는다. 별도 기기 지문 수집이나 유료 봇 방어 서비스는 현재 범위에 포함하지 않는다.
- 플랫폼 운영자의 임시 비밀번호와 최초 변경 기준은 ADMIN-001에서 별도로 관리한다.

### 정책 근거

- 최신 길이 중심 권고보다 사용자에게 익숙한 복잡도 규칙을 우선한다.
- 일반 사용자와 식당 대표자에게 같은 형식 기준을 적용하여 안내와 운영 기준을 단순화한다.
- 90일 주기의 변경 권고로 비밀번호 점검 계기를 제공하되, 강제 변경으로 인한 로그인 중단과 사용자 불편은 피한다.
- 취약한 비밀번호를 강제로 거부하기보다 위험을 알리고 사용자의 선택을 허용하여 설정 과정의 마찰을 낮춘다.
- 가입 때 소유 확인을 마친 이메일을 재사용하여 초기 재설정 절차를 단순화하고 외부 휴대전화 본인확인 사업자에 대한 상시 의존을 피한다.
- 숫자 코드는 이메일을 확인한 기기와 비밀번호를 재설정하는 기기가 달라도 입력할 수 있고 사용자에게 익숙한 흐름이다.
- 10분의 유효기간은 정상적인 이메일 확인 시간을 제공하면서 노출된 코드의 악용 가능 시간을 제한한다.
- 재발급할 때 이전 코드를 무효화하여 동시에 유효한 코드를 하나로 제한하고 코드 검증 상태를 단순화한다.
- 60초의 재전송 대기시간은 정상적인 이메일 도착을 기다리게 하면서 중복 발송과 발송 자원 남용을 줄인다.
- 1시간 최대 5회 제한은 정상적인 이메일 지연·유실에 대응할 여유를 두면서 반복 발송에 따른 괴롭힘·비용·트래픽을 억제한다.
- 코드당 3회 실패 제한으로 추측 기회를 줄이되, 계정 전체를 잠그지 않아 제3자가 고의 실패로 일반 로그인을 차단하는 상황을 피한다.
- 재설정 성공 시 계정 단위로 모든 세션을 종료하여 공격자가 이미 확보한 세션이 비밀번호 변경 뒤에도 유지되는 상황을 막는다.
- 등록 이메일 보안 알림으로 사용자가 본인이 요청하지 않은 비밀번호 재설정을 인지할 수 있게 한다.
- 이메일 발송 장애를 계정 복구 트랜잭션과 분리하여 외부 이메일 사업자 장애가 비밀번호 재설정을 중단시키거나 결과를 모호하게 만들지 않게 한다.
- 최대 3회의 추가 재시도로 일시적 장애를 흡수하되 무한 재시도와 대규모 작업 적체를 방지한다.
- 1분·5분·15분으로 늘어나는 간격은 짧은 장애에는 빠르게 대응하면서 장애가 지속될 때 이메일 사업자와 작업 큐에 부하가 집중되는 것을 완화한다.
- 최종 실패 기록과 운영자 경보·수동 재발송 경로를 두어 중요한 보안 알림의 영구 유실을 운영에서 추적하고 복구할 수 있게 한다.
- 등록 이메일 접근 불가 상황을 운영자 수동 심사로 분리하여 자동 복구 경로의 공격 표면을 줄이되, 운영 부담과 사회공학 위험은 구체적인 심사 기준으로 통제해야 한다.
- 등록 휴대전화·복구 후 사용할 이메일 소유 확인을 공통 기준으로 사용하고 식당 대표자에게 사업자등록증을 추가 요구하여 계정 유형별 위험에 맞는 고정 심사 기준을 둔다.
- 일반 사용자의 추가 비교 항목을 기존 등록 이메일 주소로 한정하여 신규·저빈도 사용자도 같은 기준으로 심사하되, 이메일 주소 단독의 낮은 확인 강도는 등록 휴대전화·복구 후 사용할 이메일 소유 확인과 운영자 최종 승인으로 보완한다.
- 기존 이메일 소유를 다시 확인한 경우에는 일반 재설정으로 돌리고, 확인할 수 없는 경우에만 다른 이메일로 교체하여 복구 후 로그인·재설정·보안 알림을 정상적으로 사용할 수 있게 한다.
- 수동 복구 시 모든 기존 세션을 계정 단위로 종료하여 이메일 접근권 상실 과정에서 탈취됐을 수 있는 세션이 복구 뒤에도 유지되는 상황을 막는다.
- 수동 복구 후 새 비밀번호를 반드시 설정하게 하여 기존 비밀번호가 노출됐을 가능성까지 차단하고, 그 전에는 카카오 로그인으로 우회하여 계정에 접근하지 못하게 한다.
- 교체 전 이메일에도 민감정보 없는 변경 알림을 남겨 잘못 승인된 복구를 기존 이메일 소유자가 인지할 가능성을 확보한다.
- 이메일 사업자의 일시적 장애와 계정 복구 결과를 분리하고 기존 보안 알림과 같은 재시도·최종 실패 관리 기준을 적용하여 운영 복잡도를 줄인다.
- 식당 대표자는 관리 중인 매장 1곳의 사업자등록번호와 대표자명을 모두 비교하여 핵심 사업자·대표자 관계를 확인하면서 다매장 계정의 서류 부담을 제한한다.
- 등록 이메일과 휴대전화를 모두 사용할 수 없는 상황에서는 대체 증빙을 받지 않아 민감정보 수집과 사회공학 공격 경로를 늘리지 않는다.
- 전용 계정 복구 권한과 변경 불가능한 승인·거절 이력으로 담당 운영자를 명확히 식별하고 단일 승인 방식의 책임소재를 확보한다.
- 복구 승인 권한과 완료된 전체 감사 이력 열람 권한을 분리하여 업무에 필요하지 않은 계정·증빙 결과의 노출 범위를 줄인다.
- 감사 이력을 처리 완료 후 3년간 보관하여 책임소재와 장기 보안 조사 가능성을 확보하면서 영구 보관에 따른 개인정보·저장 부담을 제한한다.
- 최대 64자와 공백·유니코드 지원으로 긴 암호문 사용과 비밀번호 관리자를 지원하되, 입력을 자동으로 자르지 않아 서로 다른 비밀번호가 같게 처리되는 상황을 막는다.
- 수동 복구의 3영업일 1차 처리 기한으로 승인 품질을 유지하면서 신청자가 처리 상태를 예측할 수 있게 한다.
- 점진적 지연과 기본 IP·요청 경로 제한을 함께 적용하여 분산 추측 공격을 늦추면서 공격자가 고의 실패로 계정 전체를 영구 잠그는 서비스 거부 위험을 줄인다. 기기 지문과 유료 봇 방어를 제외하여 추가 개인정보 수집·비용·구현 복잡도를 피한다.

### 다중 서버·다중 인스턴스 원칙

- 비밀번호 형식 기준과 정책 버전은 중앙 설정을 기준으로 모든 인스턴스에 동일하게 적용한다.
- 비밀번호 길이는 NFC 정규화 후 유니코드 코드 포인트 기준으로 중앙에서 동일하게 판정하고 64자를 초과한 입력은 잘라내지 않고 거부한다.
- 마지막 비밀번호 설정·변경 시점과 90일 경과 및 경고 대상 여부는 중앙 계정 상태와 공통 기준 시각으로 판정하며 애플리케이션 인스턴스의 로컬 시각이나 세션 상태에 의존하지 않는다.
- 90일 경과 경고 여부는 인증 성공과 서비스 권한 판정을 거부하는 조건으로 사용하지 않는다.
- 동시에 발생한 비밀번호 변경 요청은 계정 버전 또는 원자적 갱신 기준으로 하나만 확정되게 한다.
- 흔하거나 유출된 비밀번호의 판정 기준과 경고 문구 버전은 중앙에서 관리하여 모든 인스턴스가 동일한 경고 결과를 제공한다.
- 6자리 숫자 일회용 재설정 코드의 발급·사용·무효화 상태는 중앙에서 관리하고, 여러 인스턴스에서 동시에 사용되더라도 원자적으로 한 번만 성공하게 한다.
- 코드 만료는 중앙 발급 시각과 공통 기준 시각으로 판정하여 모든 인스턴스에서 동일한 10분 유효기간을 적용한다.
- 코드 재발급과 이전 코드 무효화는 중앙에서 하나의 원자적 처리 경계로 수행하여 동시 재발급 요청에도 최신 코드 하나만 유효하게 한다.
- 재전송 가능 여부는 중앙의 직전 발급 시각과 공통 기준 시각으로 판정하여 인스턴스를 바꾼 요청에도 동일한 60초 대기시간을 적용한다.

## AUTH-007 세션·토큰·기기 관리

> 정책 상태: 확정

### 현재 확정 사항

- 브라우저 인증은 Spring Security와 Spring Session Data Redis를 사용한 중앙 서버 세션 방식으로 처리한다.
- 브라우저에는 인증·권한 정보를 담지 않은 암호학적으로 안전한 무작위 세션 ID만 쿠키로 저장한다.
- 세션 쿠키에는 `HttpOnly`, `Secure`, `SameSite=Lax`를 적용한다.
- 브라우저 인증에는 액세스 JWT나 리프레시 JWT를 사용하지 않는다.
- 일반 사용자의 세션은 마지막 정상 요청 후 7일 동안 활동이 없으면 만료하고, 활동이 계속되더라도 최초 로그인 후 30일이 지나면 절대 만료한다.
- 식당 대표자의 세션은 마지막 정상 요청 후 8시간 동안 활동이 없으면 만료하고, 활동이 계속되더라도 최초 로그인 후 7일이 지나면 절대 만료한다.
- 플랫폼 운영자의 세션은 마지막 정상 요청 후 30분 동안 활동이 없으면 만료하고, 활동이 계속되더라도 최초 로그인 후 8시간이 지나면 절대 만료한다.
- 유휴 만료와 절대 만료 가운데 먼저 도달한 시점에 세션을 종료하며, 절대 만료 시점은 세션 갱신이나 활동으로 연장하지 않는다.
- Redis에서 중앙 세션을 조회하거나 갱신할 수 없으면 기존 세션을 유효하다고 추정하지 않고 신규 로그인과 인증이 필요한 기능을 일시 중단한다. 비회원 공개 조회는 인증 세션과 분리하여 계속 제공한다.
- Redis 장애 중 MySQL 세션 저장소나 애플리케이션 로컬 세션으로 자동 전환하지 않는다. Redis 복구 후 기존 세션 데이터가 유효하게 남아 있으면 다시 사용할 수 있고, 유실됐거나 만료됐다면 다시 로그인하게 한다.
- 로그인 성공, 권한 변경과 추가 인증 성공 시 세션 ID를 새 무작위 값으로 재발급하고 이전 세션 ID를 즉시 무효화한다.
- 브라우저의 상태 변경 요청에는 Spring Security의 CSRF 보호를 적용하며 `SameSite=Lax`만을 유일한 CSRF 방어 수단으로 사용하지 않는다.
- 로그아웃은 중앙 세션을 무효화하고 브라우저의 세션 쿠키를 삭제한다.
- 계정당 동시에 유지할 수 있는 세션은 일반 사용자 5개, 식당 대표자 3개, 플랫폼 운영자 1개로 제한한다.
- 식당 대표자의 세션 상한은 관리 매장 수와 무관하게 대표자 개인 계정 단위로 적용한다. 한 세션에서 권한이 있는 모든 매장을 전환하여 관리하며, 매장 수를 이유로 대표자 인증정보 공유를 허용하지 않는다.
- 새 로그인으로 상한을 초과하면 새 로그인을 거부하지 않고 생성 시각이 가장 오래된 기존 세션을 종료한 뒤 새 세션을 확정한다.
- 로그인된 상태에서 필요한 본인 확인을 거쳐 비밀번호나 등록 이메일을 정상 변경하면 현재 세션과 다른 기존 세션을 모두 유지한다. 현재 요청이 재인증 경계를 통과했다면 현재 세션 ID만 새 값으로 재발급하고 다른 세션은 종료하지 않는다.
- 비밀번호 분실 재설정과 수동 복구 시 모든 세션을 종료하는 AUTH-006 정책은 위 정상 변경의 예외로 그대로 적용한다.
- 모든 계정 유형은 자신의 활성 세션 목록에서 현재 세션 여부, 로그인 시각, 마지막 활동 시각과 대략적인 브라우저·운영체제 정보를 확인하고 개별 세션 또는 현재 세션을 제외한 모든 세션을 종료할 수 있다.
- 세션 표시를 위해 정밀 위치나 지속적인 기기 지문을 수집하지 않으며 IP 주소 원문을 사용자 화면에 노출하지 않는다.

### 정책 근거

- 기존 기술 스택의 Redis를 중앙 세션 저장소로 재사용하여 별도 유료 인증 서비스나 추가 토큰 체계를 도입하지 않는다.
- 서버가 세션의 유효성과 권한 상태를 중앙에서 통제하여 로그아웃, 비밀번호 재설정과 계정 복구 시 즉시 세션을 폐기할 수 있게 한다.
- 브라우저에 의미 없는 세션 ID만 노출하고 `HttpOnly`·`Secure`·`SameSite=Lax`를 적용하여 스크립트 접근, 평문 전송과 기본적인 교차 사이트 요청 위험을 줄인다.
- 계정 유형별 위험과 사용 패턴에 맞춰 세션 만료시간을 다르게 두어 일반 사용자의 편의성을 유지하면서 식당 대표자와 플랫폼 운영자의 장기 세션 탈취 위험을 줄인다.
- Redis 장애 시 인증을 실패 폐쇄하여 검증되지 않은 세션이나 오래된 권한으로 거래성 기능이 수행되는 것을 막고, 이중 세션 저장소와 동기화 복잡도를 피한다.
- 인증 경계에서 세션 ID를 재발급하여 세션 고정 공격을 줄이고, Spring Security의 기본 CSRF 보호를 사용하여 별도 유료 서비스 없이 쿠키 기반 인증의 교차 사이트 요청 위험을 통제한다.
- 계정 유형별 동시 세션 상한으로 정상적인 복수 기기 사용을 허용하면서 탈취·공유 세션과 Redis 저장량의 무제한 누적을 막는다. 다매장 운영은 세션 수가 아니라 하나의 대표자 계정에 연결된 매장 권한으로 처리한다.
- 정상적인 비밀번호·이메일 변경에서는 기존 세션을 유지하여 사용 편의를 우선하되, 사용자가 중앙 세션 목록에서 의심 세션을 직접 확인하고 종료할 수 있게 한다.

### 다중 서버·다중 인스턴스 원칙

- 모든 애플리케이션 인스턴스는 Spring Session Data Redis의 같은 중앙 세션 상태를 사용하며 로컬 메모리 세션이나 고정 세션 라우팅에 의존하지 않는다.
- 세션 생성·조회·갱신·만료·폐기는 중앙 Redis 상태를 기준으로 처리하여 어느 인스턴스에 요청해도 동일한 인증 결과를 사용한다.
- 세션의 마지막 정상 요청 시각, 최초 로그인 시각과 계정 유형은 중앙 세션 상태로 관리하고 공통 기준 시각으로 유휴·절대 만료를 판정한다. 여러 인스턴스의 동시 요청에도 절대 만료 시점은 연장하지 않는다.
- 모든 인스턴스는 Redis 세션 조회·갱신 실패를 동일한 인증 불가 상태로 처리하고, 로컬 캐시나 MySQL의 과거 상태로 인증을 우회하지 않는다. Redis 복구 후에는 중앙에 남은 세션의 만료·폐기 상태를 그대로 따른다.
- 세션 ID 재발급과 이전 ID 무효화는 중앙 세션 상태에서 일관되게 처리하여 동시 요청이나 다른 인스턴스에서 이전 ID가 다시 승인되지 않게 한다.
- 계정별 활성 세션 수 확인, 가장 오래된 세션 종료와 새 세션 생성은 중앙 Redis 상태를 기준으로 일관되게 처리하여 여러 인스턴스의 동시 로그인에도 일반 사용자 5개·식당 대표자 3개·플랫폼 운영자 1개 상한을 넘지 않게 한다.
- 정상 비밀번호·등록 이메일 변경 사건은 다른 세션의 만료 상태를 변경하지 않는다. 세션 목록 조회와 개별·일괄 종료는 중앙 Redis의 세션 식별자를 기준으로 처리하여 어느 인스턴스에서도 종료된 세션이 다시 승인되지 않게 한다.

## AUTH-010 휴면·탈퇴·재가입

> 정책 상태: 확정

- 탈퇴 대기 기간은 요청 접수 시점부터 7일로 한다.
- 7일 안에는 계정 소유자가 재인증을 완료한 경우에만 탈퇴를 취소하고 계정을 다시 활성화할 수 있다.

## AUTH-011 계정 정지·이의 제기

> 정책 상태: 확정

### 현재 확정 사항

- 계정 제재는 `경고`, `특정 기능 제한`, `기간 정지`, `영구 정지`의 단계로 운영한다.
- 일반적인 위반은 위반의 심각도·반복성·피해 범위에 따라 단계적으로 강화한다.
- 사기, 계정 탈취, 결제 악용 또는 심각한 안전 위협처럼 즉시 피해 확산을 막아야 하는 중대한 사건은 앞 단계를 생략하고 기간 정지나 영구 정지를 적용할 수 있다.
- 모든 제재에는 대상 계정, 제재 단계, 적용·종료 시각, 관리자, 사유와 근거를 연결하여 감사 가능한 중앙 이력으로 남긴다.
- 경고·특정 기능 제한·기간 정지는 중앙 권한 저장소에서 `계정 제재` 권한을 가진 관리자 1명이 확정할 수 있다.
- 사기·계정 탈취·결제 악용·심각한 안전 위협의 긴급 차단도 `계정 제재` 권한 관리자 1명이 즉시 기간 정지로 적용할 수 있다.
- 영구 정지는 `계정 제재` 권한 관리자가 제안하고 슈퍼관리자가 추가 승인해야 확정된다. 제안과 추가 승인은 서로 다른 관리자 계정으로 수행하며 슈퍼관리자 승인 전에는 영구 정지로 전환하지 않는다.
- 기간 정지 또는 영구 정지가 확정되면 로그인 방식과 관계없이 모든 기존 세션을 즉시 종료하고 이후 로그인을 완전히 차단한다. 정지된 계정에 제재 안내나 이의 제기를 위한 제한 로그인 화면은 제공하지 않는다.
- 기간·영구 정지 확정 직후 등록 이메일로 제재 단계, 적용·종료 시각 또는 영구 정지 여부, 사유 요약과 고객지원 이의 제기 방법을 안내한다. 신고자 신원이나 공개하면 안 되는 내부 증거는 이메일에 포함하지 않는다.
- 이메일 발송 실패는 확정된 제재를 취소하거나 로그인 차단을 지연시키지 않는다. 발송은 비동기로 처리하고 최초 실패 후 1분·5분·15분 간격으로 최대 3회 추가 재시도하며 최종 실패를 관리자 경보와 수동 재발송 대상으로 남긴다.
- 고객지원 이의 제기는 계정에 등록된 휴대전화 또는 등록 이메일 가운데 하나의 소유 확인을 완료하면 접수할 수 있다.
- 소유 확인 전에는 해당 연락처의 계정 존재 여부, 제재 단계·사유·근거와 신고자 정보를 공개하지 않는다. 두 등록 연락수단을 모두 사용할 수 없으면 AUTH-006의 계정 복구로 연락처 소유권을 먼저 회복해야 한다.
- 최초 제재를 확정하거나 제안한 관리자도 중앙의 `이의 제기 심사` 권한을 보유하면 해당 이의 제기를 심사할 수 있다.
- `이의 제기 심사` 권한 관리자는 경고·특정 기능 제한·기간 정지를 유지·감경·취소할 수 있다. 영구 정지의 유지·감경·취소는 심사 의견을 바탕으로 슈퍼관리자가 최종 확정한다.

- 심사 결정이 확정된 시점부터 유지·감경·취소 결과를 적용한다. 명백한 오정지는 별도의 임시 해제 상태를 만들지 않고 심사를 우선 처리하여 즉시 취소한다.

- 영구 정지 계정의 재가입 제한 토큰은 최종 삭제 완료 시점부터 3년간 보관한 뒤 파기한다. 3년이 지나면 같은 식별정보로 새 계정을 만들 수 있으며 삭제된 계정의 정보·이력·혜택은 복원하거나 연결하지 않는다.

- 하나의 제재 사건에는 동시에 하나의 이의 제기만 허용한다. 최종 결정 후에는 기존 심사에 없던 새로운 객관적 자료를 제출한 경우에만 다시 접수한다.
- 이의 제기는 접수 후 3영업일 안에 접수 확인·보완 요청·처리 결과 중 하나로 1차 응답하고, 7영업일 안에 최종 처리하는 것을 목표로 한다. 복잡한 사건으로 기한을 넘기면 지연 사유와 새 예상 처리일을 등록 이메일로 안내한다.
- 이의 제기 최종 결과와 변경된 제재 단계·기간은 등록 이메일로 통지한다. 발송 실패는 결정 결과를 되돌리지 않고 제재 통지와 같은 비동기 재시도·최종 실패 관리 원칙을 적용한다.

### 현재 확정 사항

- 낮은 위험은 중앙 보안 사건과 감사 이력만 기록하고 계정 이용을 제한하지 않는다.

- 사용자가 고객지원이나 보안 알림 경로에서 특정 활동을 본인의 활동이 아니라고 확인한 사건은 높은 위험으로 처리한다.
