# 07. 데이터 및 API 계약

## MySQL 원본 원칙

초기 영속 데이터의 원본은 MySQL이다. 애플리케이션은 Spring Data JPA를 통해 접근하고, 데이터베이스 구조 변경은 Flyway 마이그레이션으로 관리한다. 캐시나 후속 인프라가 도입되더라도 MySQL의 원본성 및 정합성 책임을 대체하지 않는다.

구현 전에는 테이블 이름, 컬럼, 인덱스, 관계와 물리 스키마를 확정하지 않는다. 이들은 도메인 유스케이스와 무결성 규칙이 구현될 때 마이그레이션과 함께 결정한다.

## 트랜잭션 경계

트랜잭션 경계는 service의 하나의 유스케이스에 둔다. 함께 성공하거나 실패해야 하는 데이터 변경은 하나의 트랜잭션으로 묶고, 긴 실행·외부 네트워크 호출·사용자 대기는 데이터베이스 트랜잭션 안에 유지하지 않는다.

도메인 간 협력이 필요한 경우에도 서비스가 변경 순서와 실패 처리를 명시적으로 조정한다. 외부 부작용의 전달 보장이 필요해지면 별도의 이벤트 도입 조건을 충족한 뒤 정책을 정한다.

## 유일 제약

중복을 허용하지 않는 비즈니스 식별자와 상태 전이는 애플리케이션 검사만으로 보장하지 않고 데이터베이스 유일 제약 또는 동등한 원자적 무결성 장치로 보호한다. 어떤 식별자와 조합이 유일한지는 각 도메인 규칙이 확정될 때 정한다.

제약 충돌은 재시도 가능 여부와 사용자에게 노출할 오류 의미를 구분해 일관된 API 오류로 변환한다. 아직 테이블 스키마나 제약 이름은 확정하지 않는다.

## 멱등 요청

재전송될 수 있는 상태 변경 요청, 특히 결제 확인이나 외부 시스템 호출이 수반되는 요청은 멱등 처리를 설계한다. 클라이언트가 제공하는 멱등 키 또는 도메인상 고유한 요청 식별자를 검증하고, 동일 요청의 결과를 안전하게 재현할 수 있어야 한다.

구체적인 키 전달 방식, 보관 기간, 요청 URL은 해당 기능의 계약을 구현할 때 정한다. 멱등성은 중복 요청이 새 상태 변경이나 중복 부작용을 만들지 않는다는 결과 기준으로 검증한다.

## 오류 응답 형식

오류 응답은 JSON 객체로 통일하고 최소한 기계 판독 가능한 `code`와 사람이 이해할 수 있는 `message`를 제공한다. 입력값 오류처럼 필드별 정보가 필요한 경우에만 `details`를 선택적으로 포함한다.

내부 예외 메시지, 스택 추적, 비밀 값은 응답에 포함하지 않는다. HTTP 상태 코드와 오류 코드는 전역 오류 처리에서 일관되게 매핑하며, 개별 도메인이 서로 다른 응답 형태를 만들지 않는다.

## API 버전

API 버전은 기존 소비자와 호환되지 않는 변경이 필요할 때 도입한다. 현재 초기 기준은 버전 표기 위치나 미구현 API의 URL을 확정하지 않으며, 외부 소비자가 생기거나 호환성 정책이 필요한 시점에 버전 방식과 지원 기간을 계약으로 결정한다.

호환 가능한 필드 추가와 호환되지 않는 변경을 구분하고, 후자의 경우에는 마이그레이션·폐기 일정·소비자 안내를 함께 정한다.

## 시간·금액 표현

서버의 시각 저장과 비교는 UTC 기준의 시점을 사용하고, API 시간값에는 오프셋 또는 UTC 의미가 분명한 ISO 8601 형식을 사용한다. 화면에서 필요한 지역 시간대 변환은 표시 계층의 명시적 책임으로 둔다.

금액은 부동소수점으로 계산하거나 전송하지 않는다. 통화 코드와 함께 정밀도를 보존할 수 있는 십진 표현을 사용하며, 반올림 규칙과 통화별 소수 자릿수는 결제 요구가 확정될 때 계약으로 정한다.

## 이벤트 도입 조건

이벤트는 동기 서비스 호출만으로 해결할 수 없는 실제 비동기 협력, 외부 부작용 분리, 재시도·추적 요구가 확인될 때만 도입한다. 도입 전에는 전달 보장, 중복 처리, 순서, 재처리, 관측성, 실패 복구의 책임을 명시한다.

초기 시스템은 이벤트 전달 인프라를 구성하지 않는다. 데이터 변경과 이벤트 발행의 원자성 요구가 확인되면 Outbox를 포함한 대안을 재평가하고, 결정된 방식은 별도 ADR과 기능 계약으로 기록한다.
