# TasteLock 공식 기술 방향

> 문서 상태: 정책 설계 진행 중
> 기준일: 2026-07-21
> 역할: 확정된 서비스 정책을 구현하기 위한 기술 원칙과 아키텍처 경계를 설명한다.
> 주의: 구체적인 제품·클라우드·프레임워크 선택은 별도 결정 전까지 확정하지 않는다.

## 1. 목표

- 실제 운영 가능한 품질을 갖춘다.
- 인기 예약과 한정 메뉴 오픈의 순간적인 대규모 트래픽을 방어한다.
- 다수 서버·다수 인스턴스에서 동일한 비즈니스 규칙과 데이터 정합성을 보장한다.
- 부분 장애와 외부 연동 실패가 전체 예약 시스템으로 확산되지 않도록 한다.
- 장애가 발생해도 예약, 결제 및 비동기 작업의 최종 상태를 복구할 수 있어야 한다.

## 2. 구조 원칙

- 애플리케이션 인스턴스는 가능한 한 무상태로 유지한다.
- 인증 세션, 예약 홀드, 대기열 등 공유 상태는 중앙 저장소에서 관리한다.
- 로드밸런서 뒤에서 애플리케이션 인스턴스를 수평 확장한다.
- 초기 배포 단위와 무관하게 도메인별 모듈 경계를 분명히 한다.
- 예약·대기열·알림 등 부하 특성이 다른 영역을 독립 확장할 수 있게 인터페이스를 둔다.
- 하나의 기술로 모든 정합성 문제를 해결하지 않고 DB 제약, 원자 연산, 멱등성 및 보상 처리를 조합한다.

## 3. 도메인 경계

- 회원·인증·권한
- 사업자·매장·직원
- 영업시간·메뉴·운영 설정
- 예약 자원·예약·홀드
- 웨이팅·승계
- 코스·테이스팅
- 결제·환불·정산
- 구독·혜택
- 리뷰·신뢰·신고
- 알림
- 광고·추천
- 분석·리포트
- 운영자·감사

각 도메인은 자신의 상태와 규칙을 소유하고, 다른 도메인에는 명시된 명령·조회·이벤트 인터페이스만 제공한다.

## 4. 핵심 데이터 흐름

### 4.1 복수 자원 예약

```text
요청 검증
→ 좌석·메뉴·회차 가용성 확인
→ 하나의 홀드 그룹으로 원자적 임시 선점
→ 결제 또는 승인 처리
→ DB 최종 확정
→ 확정 이벤트 발행
→ 알림·검색·분석 비동기 반영
```

일부 자원만 선점되면 전체 그룹을 해제한다. 제한 시간 만료, 결제 실패 및 최종 확정 실패에는 명시적인 보상 흐름을 둔다.

### 4.2 웨이팅

```text
위치·자격 확인
→ 중복 등록 검증
→ 대기 팀 생성
→ 일행 연결
→ 순번·예상 시간 갱신
→ 호출 및 응답 제한 시간
→ 체크인 또는 만료·취소
```

동일한 대기 팀은 하나의 대표 상태를 가지며, 일행 참여가 별도 대기 팀을 만들지 않도록 한다.

### 4.3 결제와 예약

```text
홀드 생성
→ 결제 대기
→ PG 승인 확인
→ 예약 확정
```

PG 승인 후 예약 확정이 실패하면 재시도하고, 확정이 불가능하면 결제 취소를 요청한다. 자동 복구가 실패한 건은 운영자 작업 큐로 이동한다.

## 5. 동시성과 정합성

- DB 유일 제약과 조건부 갱신을 최종 방어선으로 사용한다.
- 공유 카운터와 짧은 임시 선점은 원자 연산으로 처리한다.
- 분산 락은 소유권, 만료, fencing 및 실패 시나리오가 명확한 최소 구간에만 사용한다.
- 모든 쓰기 요청은 재시도와 중복 전송을 고려한다.
- 결제 요청, 웹훅, 이벤트 소비 및 관리자 복구 작업에 멱등성 기준을 둔다.
- 메시지는 중복되거나 순서가 바뀔 수 있다고 가정한다.
- 이벤트 발행 유실을 막기 위해 트랜잭션과 연계된 Outbox 계열 패턴을 사용한다.

## 6. 대규모 트래픽 방어

- 엣지와 애플리케이션 계층에 요청 속도 제한을 둔다.
- 인기 오픈에는 가상 대기열과 입장 토큰을 적용한다.
- 사용자·계정·기기·IP 및 대상 자원 기준의 제한을 조합한다.
- 읽기 캐시와 예약 쓰기 경로를 분리한다.
- 캐시 스탬피드, 핫키 및 특정 매장 쏠림을 부하 테스트에 포함한다.
- 처리 가능한 예약량에 따라 대기열 입장 속도를 조절한다.
- 과부하 시 부가 기능을 축소해도 예약·결제의 핵심 정합성은 유지한다.

## 7. 다중 인스턴스 운영

- 로컬 메모리만으로 세션, 예약 홀드, 작업 소유권 또는 스케줄 상태를 판단하지 않는다.
- 스케줄러와 배치 작업은 리더 선출, 작업 임대 또는 DB 기반 선점으로 중복 실행을 제어한다.
- 이벤트 소비자는 여러 인스턴스에서 동시에 실행되어도 동일 결과를 만든다.
- SSE·WebSocket 연결 상태와 이벤트 전달은 인스턴스 간 공유 또는 라우팅 전략을 사용한다.
- 배포 중 이전·신규 버전이 동시에 실행되는 상황을 고려해 API와 이벤트 스키마를 호환되게 변경한다.

## 8. 장애와 복구

- 외부 연동에는 제한 시간, 재시도 정책, 회로 차단 및 격리를 적용한다.
- 무제한 재시도를 금지하고 실패 이벤트를 별도 큐에 보관한다.
- Redis 장애 시 DB를 기준으로 예약 상태를 복원·재조정한다.
- 메시지 브로커 지연 시 예약 확정 자체와 후속 알림을 분리한다.
- 결제·예약 상태는 정기 대사 작업으로 불일치를 탐지한다.
- 중요한 수동 복구에는 권한 확인, 사유 입력 및 감사 로그가 필요하다.
- 복구 목표와 백업·복원 절차는 데이터 중요도별로 별도 확정한다.

## 9. 보안과 개인정보

- 최소 권한 RBAC를 적용한다.
- 결제수단 원문을 직접 저장하지 않는다.
- 민감정보는 전송·저장 시 보호하고 로그에서 제거한다.
- 관리자 중요 작업에는 추가 인증과 재확인을 적용한다.
- QR, 초대 링크, 승인 링크에는 만료와 재사용 방지를 적용한다.
- 위치 정보는 목적에 필요한 최소 범위로 처리하고 원본 좌표의 장기 보관을 피한다.
- 개인정보 보유 기간, 탈퇴, 삭제, 법적 보관 및 분석 익명화 정책을 상세 정책에서 확정한다.

## 10. 관측과 검증

- 모든 요청과 비동기 작업에 추적 가능한 상관관계 ID를 사용한다.
- 예약 성공률, 홀드 만료율, 초과 판매 탐지, 결제 불일치, 대기열 지연 및 알림 실패율을 측정한다.
- 단위, 통합, 계약, 동시성, 부하, 장애 주입 및 복구 훈련을 테스트 범위에 포함한다.
- 다중 인스턴스 동시 처리와 배포 중 혼합 버전 시나리오를 검증한다.
- 성공 기준과 경보 임계치는 예상 트래픽과 운영 목표를 정한 뒤 수치화한다.
