# MiriYum 공식 기술 방향

> 문서 상태: 정책 재분류·위임 추천 확정·사용자 직접 운영 목표 반영 완료·개발/시연 기술 방향 확정
> 기준일: 2026-07-22
> 역할: 확정된 서비스 정책과 대표 팀 결정에 종속되는 기술 원칙·아키텍처 경계를 설명한다.
> 주의: 이 문서의 확정 기술은 개발·시연 기준이다. 사용자 직접 결정된 서비스 목표는 반영하지만, 유료 사업자·운영 인프라 계약·최소 인프라 용량은 확정하지 않는다.

- 대규모 트래픽과 다중 서버·다중 인스턴스 전제는 유지하되, 유료 서비스 계약·불필요하게 복잡한 인프라·추가 개인정보 수집은 자동 기본값으로 채택하지 않는다.

## 1. 목표

- 실제 운영 가능한 품질을 갖춘다.
- 인기 예약과 한정 메뉴 오픈의 순간적인 대규모 트래픽을 방어한다.
- 다수 서버·다수 인스턴스에서 동일한 비즈니스 규칙과 데이터 정합성을 보장한다.
- 부분 장애와 외부 연동 실패가 전체 예약 시스템으로 확산되지 않도록 한다.
- 장애가 발생해도 예약, 결제 및 비동기 작업의 최종 상태를 복구할 수 있어야 한다.

## 2. 구조 원칙

- 애플리케이션 인스턴스는 가능한 한 무상태로 유지한다.
- 인증 세션, 예약 홀드, 대기열 등 공유 상태는 중앙 저장소에서 관리한다.
- 브라우저 인증은 Spring Security와 Spring Session Data Redis 기반의 중앙 서버 세션을 사용하고, 애플리케이션 인스턴스는 로컬 메모리 세션이나 고정 세션 라우팅에 의존하지 않는다.
- 로드밸런서 뒤에서 애플리케이션 인스턴스를 수평 확장한다.
- 초기 배포 단위와 무관하게 도메인별 모듈 경계를 분명히 한다.
- 예약·대기열·알림 등 부하 특성이 다른 영역을 독립 확장할 수 있게 인터페이스를 둔다.
- 하나의 기술로 모든 정합성 문제를 해결하지 않고 DB 제약, 원자 연산, 멱등성 및 보상 처리를 조합한다.

## 3. 도메인 경계

- 회원·인증·권한
- 사업자등록·매장·대표자
- 영업시간·메뉴·운영 설정
- 예약 자원·예약·홀드
- 웨이팅·승계
- 코스·테이스팅
- 결제·환불·정산
- 구독·혜택
- 리뷰·신뢰·신고
- 알림
- 광고·추천
- 분석·리포트
- 운영자·감사

각 도메인은 자신의 상태와 규칙을 소유하고, 다른 도메인에는 명시된 명령·조회·이벤트 인터페이스만 제공한다.

사업자등록 단위와 매장은 각각 독립 식별자를 가지되 원칙적으로 1:1 관계를 유지한다. 사업자등록번호의 중복 등록을 방지하고, 매장의 운영 주체가 변경될 때 과거 예약·결제·정산의 귀속 이력을 보존한다.

## 4. 핵심 데이터 흐름

### 4.1 복수 자원 예약

```text
요청 검증
→ 좌석·메뉴·회차 가용성 확인
→ 하나의 홀드 그룹으로 원자적 임시 선점
→ 결제 또는 승인 처리
→ DB 최종 확정
→ 확정 이벤트 발행
→ 알림·검색·분석 비동기 반영
```

일부 자원만 선점되면 전체 그룹을 해제한다. 제한 시간 만료, 결제 실패 및 최종 확정 실패에는 명시적인 보상 흐름을 둔다.

### 4.2 웨이팅

```text
위치·자격 확인
→ 중복 등록 검증
→ 대기 팀 생성
→ 일행 연결
→ 순번·예상 시간 갱신
→ 호출 및 응답 제한 시간
→ 체크인 또는 만료·취소
```

동일한 대기 팀은 하나의 대표 상태를 가지며, 일행 참여가 별도 대기 팀을 만들지 않도록 한다.

### 4.3 결제와 예약

```text
홀드 생성
→ 결제 대기
→ PG 승인 확인
→ 예약 확정
```

PG 승인 후 예약 확정이 실패하면 재시도하고, 확정이 불가능하면 결제 취소를 요청한다. 자동 복구가 실패한 건은 운영자 작업 큐로 이동한다.

## 5. 동시성과 정합성

- DB 유일 제약과 조건부 갱신을 최종 방어선으로 사용한다.
- 공유 카운터와 짧은 임시 선점은 원자 연산으로 처리한다.
- 분산 락은 소유권, 만료, fencing 및 실패 시나리오가 명확한 최소 구간에만 사용한다.
- 모든 쓰기 요청은 재시도와 중복 전송을 고려한다.
- 결제 요청, 웹훅, 이벤트 소비 및 관리자 복구 작업에 멱등성 기준을 둔다.
- 메시지는 중복되거나 순서가 바뀔 수 있다고 가정한다.
- 이벤트 발행 유실을 막기 위해 트랜잭션과 연계된 Outbox 계열 패턴을 사용한다.

## 6. 대규모 트래픽 방어

- 초기 용량 목표는 MAU 100,000명, 입점 매장 5,000개, 일 핵심 거래 50,000건이며, 전체 API는 정상 200 RPS와 1,000 RPS 10분 피크를 처리한다. 이는 유료 상품·인스턴스 수를 정하는 값이 아니다.
- 월간 가용성 목표는 99.9%이며 조회 API p95 500ms, 상태 변경 API p95 1초, 예약·선점 등 핵심 API p99 2초를 목표로 한다.
- 속도 제한은 중앙 공유 상태에서 원자 적용한다. 공개 조회는 IP당 분당 60회, 로그인된 조회는 계정당 분당 300회, 상태 변경은 계정당 분당 60회, 예약·웨이팅 생성은 계정당 분당 20회, 로그인·인증번호 요청은 10분당 5회다. 초과하면 `429`와 `Retry-After`를 반환한다.
- 비회원 공개 조회는 캐시 가능한 읽기 경로로 분리하고, IP 기준 속도 제한을 적용한다.
- 인기 오픈에는 가상 대기열과 입장 토큰을 적용한다.
- 사용자·계정·기기·IP 및 대상 자원 기준의 제한을 조합한다.
- 읽기 캐시와 예약 쓰기 경로를 분리한다.
- 캐시 스탬피드, 핫키 및 특정 매장 쏠림을 부하 테스트에 포함한다.
- 처리 가능한 예약량에 따라 대기열 입장 속도를 조절한다.
- 과부하 시 부가 기능을 축소해도 예약·결제의 핵심 정합성은 유지한다.

## 7. 다중 인스턴스 운영

- 로컬 메모리만으로 세션, 예약 홀드, 작업 소유권 또는 스케줄 상태를 판단하지 않는다.
- 스케줄러와 배치 작업은 리더 선출, 작업 임대 또는 DB 기반 선점으로 중복 실행을 제어한다.
- 이벤트 소비자는 여러 인스턴스에서 동시에 실행되어도 동일 결과를 만든다.
- SSE·WebSocket 연결 상태와 이벤트 전달은 인스턴스 간 공유 또는 라우팅 전략을 사용한다.
- 배포 중 이전·신규 버전이 동시에 실행되는 상황을 고려해 API와 이벤트 스키마를 호환되게 변경한다.

## 8. 장애와 복구

- 외부 연동에는 제한 시간, 재시도 정책, 회로 차단 및 격리를 적용한다.
- 무제한 재시도를 금지하고 실패 이벤트를 별도 큐에 보관한다.
- Redis 장애 시 DB를 기준으로 예약 상태를 복원·재조정한다.
- 메시지 브로커 지연 시 예약 확정 자체와 후속 알림을 분리한다.
- 결제·예약 상태는 정기 대사 작업으로 불일치를 탐지한다.
- 중요한 수동 복구에는 권한 확인, 사유 입력 및 감사 로그가 필요하다. 200,000원 이하는 결제 관리자가 승인하고, 초과 건은 슈퍼관리자 추가 승인이 필요하다. 원 결제금액을 넘는 보상은 금액과 무관하게 슈퍼관리자 승인을 요구한다.
- 암호화 백업, 운영계 분리와 정기 복원 검증을 적용한다. 거래 데이터 RPO/RTO는 5분/1시간, 설정·문서는 24시간/4시간, 분석 데이터는 24시간/24시간이다. 복원 검증 실패는 실패 원인·영향·다음 조치와 함께 운영 사건으로 보고한다.

## 9. 보안과 개인정보

- 최소 권한 RBAC를 적용한다.
- 일반 사용자, 식당 대표자 및 플랫폼 운영자 계정은 서로 분리된 인증·권한 경계로 취급한다.
- 동일인이 서로 다른 유형의 계정을 보유하더라도 세션과 권한을 자동으로 공유하거나 승격하지 않는다.
- 식당 대표자의 매장 소속은 중앙 권한 저장소를 기준으로 검증하여 모든 애플리케이션 인스턴스에서 같은 결과를 보장한다.
- 직원용 계정과 매장 공용 계정은 발급하지 않으며, 대표자 인증정보의 공유를 허용하지 않는다.
- 식당 대표자 권한은 사업자등록정보 진위확인 상태, 대표자 본인 자기확약과 매장 소속 관계를 함께 기준으로 검증한다. 현재 자기확약을 강한 실명 동일인 확인 결과로 표시하지 않는다.
- 예약·웨이팅·홀드·결제·리뷰 등 거래성 요청에는 인증된 일반 사용자 계정 식별자가 필요하다.
- 일반 사용자와 식당 대표자는 계정 위험도에 따라 서로 다른 로그인 수단과 인증 강도를 적용한다.
- 로그인 수단별 인증 결과와 계정 연결 관계는 중앙 저장소를 기준으로 검증하여 모든 애플리케이션 인스턴스에서 동일한 결과를 보장한다.
- 일반 사용자의 소셜 로그인과 이메일·비밀번호 로그인은 어느 경로로 인증하더라도 동일한 계정 상태와 제재를 적용한다.
- 초기 외부 로그인 연동은 카카오로 한정하되, 인증 제공자별 연결 정보를 분리하여 향후 다른 제공자를 추가해도 기존 계정 식별자와 도메인 데이터가 바뀌지 않게 한다.
- 일반 사용자 계정에 연결된 카카오와 이메일·비밀번호 인증은 어느 경로로 성공하더라도 동일한 중앙 계정 식별자와 서비스 데이터에 연결한다.
- 초기에는 사용자 요청으로 인증 제공자 연결을 해제하는 상태 전이를 제공하지 않는다.
- 현재 중복 계정 판정은 확인된 휴대전화 번호와 확인된 이메일 주소를 계정 유형별로 정규화하고 각각 중앙 유일성 제약을 적용하며, 둘 중 하나라도 중복되면 새 계정을 생성하지 않는다.
- 일반 사용자와 식당 대표자 계정은 서로 독립적으로 중복을 판정하므로 같은 확인 연락처를 서로 다른 두 계정 유형에서 사용할 수 있다.
- 중복 확인과 계정 생성을 하나의 원자적 경계로 처리하여 여러 인스턴스의 동시 가입과 중복 외부 인증 콜백에도 두 번째 계정이 만들어지지 않게 한다.
- 중복 가입 시도에서는 새 계정을 생성하지 않고, 기존 계정의 본인 인증을 완료한 경우에만 새 로그인 수단을 중앙 계정 연결 관계에 원자적·멱등하게 추가한다.

- 보안 알림의 최초 발송과 추가 재시도 횟수는 중앙 작업 상태에서 원자적으로 관리하여 다중 작업 인스턴스 환경에서도 추가 재시도가 최대 3회를 넘지 않게 한다.
- 세 번의 추가 재시도는 직전 실패 후 1분·5분·15분 간격으로 예약하고, 중앙의 다음 실행 시각과 원자적 작업 획득을 기준으로 여러 작업 인스턴스의 중복 실행을 방지한다.
- 모든 추가 재시도가 실패하면 중앙 작업을 최종 실패 상태로 전환하고 운영자 경보를 멱등하게 생성한다. 수동 재발송은 감사 가능한 별도 작업으로 기록하며 계정 상태와 완료된 재설정에는 영향을 주지 않는다.
- 등록 이메일 접근 불가 복구는 자동 SMS·외부 본인확인 흐름으로 처리하지 않고 중앙 심사 사건으로 생성한다. 중복 신청은 하나의 사건으로 멱등하게 통합하고 운영자 판단과 상태 변경 이력을 감사 가능하게 기록한다.
- 등록 휴대전화·복구 후 사용할 이메일 소유 확인, 일반 사용자의 기존 계정 정보 비교와 식당 대표자의 사업자등록증 검토 상태를 중앙 심사 사건에 연결하고, 필수 증빙 충족 여부와 운영자 최종 승인을 분리하여 자동 승인되지 않게 한다.
- 입력 이메일을 중앙 정규화 기준으로 기존 등록 이메일과 비교하여 같은 주소의 소유 확인이 완료되면 일반 재설정으로 전환한다. 다른 주소는 계정 유형별 이메일 유일성 검증을 통과한 경우에만 수동 심사에 사용할 수 있다.
- 일반 사용자가 제시한 기존 이메일 주소는 중앙의 정규화된 등록 이메일과 비교하고 그 결과를 심사 사건에 연결한다. 카카오 인증 상태와 예약·웨이팅·홀드 이력은 이 심사의 필수 입력으로 사용하지 않는다.
- 식당 대표자가 제출한 관리 매장 1곳의 사업자등록번호와 대표자명을 중앙의 기존 인증 정보와 비교하고 두 항목의 일치 결과를 심사 사건에 연결한다. 매장명·주소는 필수 일치 판정에 사용하지 않는다.
- 중앙의 등록 휴대전화 소유 확인 완료 상태를 수동 복구 신청의 필수 조건으로 사용하고, 미완료 상태에서는 어떤 운영 인스턴스에서도 대체 신분증·외부 본인확인 또는 운영자 재량으로 승인하지 못하게 한다.
- 수동 복구 승인은 중앙 권한 저장소에서 `계정 복구` 권한을 가진 운영자 1명에게만 허용한다. 심사 사건·대상 계정·담당 운영자·증빙 결과·결정 사유·처리 시각을 수정·삭제 불가능한 중앙 감사 이력으로 기록한다.
- 승인 상태 변경과 감사 이력 생성을 일관된 처리 경계에서 확정하고, 동시 승인·거절 요청에는 최초의 유효한 결정 하나만 반영하여 다중 인스턴스에서도 책임 주체와 결과가 바뀌지 않게 한다.
- 완료된 수동 복구 감사 이력 전체 조회는 중앙 권한 저장소의 슈퍼관리자 여부 또는 별도 `계정 복구 감사 열람` 권한으로 통제하며, 모든 운영 인스턴스에서 `계정 복구` 승인 권한과 구분한다. 진행 중인 담당 사건 조회는 배정 관계와 업무상 필요한 범위로 제한한다.
- 수동 복구 감사 이력의 3년 보관 만료 시점은 중앙의 심사 처리 완료 시각과 공통 기준 시각으로 계산한다. 파기 작업은 중앙 상태에서 대상을 원자적으로 획득하고 멱등하게 처리하여 여러 배치 인스턴스가 같은 이력을 중복 처리하지 않게 한다.
- 수동 복구의 3영업일 1차 처리 기한은 중앙 접수·보완 재접수 시각과 공통 영업일 기준으로 계산하여 모든 운영 인스턴스가 같은 기한을 사용한다.
- 수동 복구 승인, 등록 이메일 교체, 중앙 세션 버전 갱신, 복구 후 비밀번호 재설정 필요 상태 설정과 감사 이력 생성을 일관된 처리 경계에서 확정하여 이전 이메일과 복구 후 이메일이 동시에 유효한 등록 이메일이 되지 않고 로그인 방식과 무관하게 모든 기존 세션이 거부되게 한다.
- 모든 인증·서비스 인스턴스는 중앙의 복구 후 비밀번호 재설정 필요 상태를 확인하여 새 비밀번호 설정 전에는 카카오를 포함한 모든 로그인과 서비스 접근을 거부한다. 승인 직후 확인된 복구 후 이메일로 기존 정책과 동일한 6자리 숫자 일회용 코드를 발급하며, 10분 유효기간·재발급 무효화·60초 재전송·이메일별 1시간 5회 발급·코드별 3회 실패 제한을 중앙에서 동일하게 적용한다.
- 수동 복구 후 새 비밀번호 설정 성공 시 재설정 코드 소비, 비밀번호 갱신과 복구 후 비밀번호 재설정 필요 상태 해제를 일관된 처리 경계에서 확정하고 자동 세션은 발급하지 않는다.
- 수동 복구 승인 사건을 기준으로 교체 전 이메일에 보낼 등록 이메일 변경 보안 알림 요청을 생성하고, 중복 사건 처리에도 같은 요청이 한 번만 생성되도록 멱등성을 보장한다. 알림 데이터에는 복구 후 이메일 주소와 제출 증빙·심사 세부정보를 포함하지 않는다.
- 교체 전 이메일 변경 보안 알림 요청은 수동 복구 결과와 함께 유실되지 않도록 내구성 있게 기록하고 별도 작업자가 비동기로 처리한다. 발송 실패는 승인·이메일 교체·세션 철회를 되돌리지 않으며, 중앙 작업 상태와 원자적 작업 획득을 기준으로 직전 실패 후 1분·5분·15분 간격의 최대 3회 추가 재시도를 여러 작업 인스턴스에서 중복 없이 수행한다.
- 추가 재시도가 모두 실패하면 중앙 작업을 최종 실패 상태로 전환하고 플랫폼 운영자 경보를 멱등하게 생성한다. 수동 재발송은 감사 가능한 별도 작업으로 기록하며 복구된 계정 상태에는 영향을 주지 않는다.
- 계정별 이메일·비밀번호 연속 실패 횟수, 지연 단계와 다음 시도 가능 시각을 중앙 저장소에서 원자적으로 갱신하여 여러 인증 인스턴스의 동시 요청에도 연속 5회와 1분·5분·15분 기준을 일관되게 적용한다. 이메일·비밀번호 로그인 성공 시 상태를 초기화하며, 이 지연 상태를 카카오 로그인이나 비밀번호 재설정·계정 복구에 전파하지 않는다.
- IP·요청 경로별 로그인 속도 제한은 모든 인증 인스턴스가 공유하는 중앙 제한 상태와 조정 가능한 중앙 임계치를 사용하여 요청 분산으로 우회되지 않게 한다. 현재 범위에서는 별도 기기 지문 수집과 유료 봇 방어 서비스를 요구하지 않는다.
- 브라우저에는 인증·권한 정보를 포함하지 않은 암호학적으로 안전한 무작위 세션 ID만 `HttpOnly`·`Secure`·`SameSite=Lax` 쿠키로 전달하고, 브라우저 인증에는 JWT를 사용하지 않는다.
- 모든 애플리케이션 인스턴스는 Spring Session Data Redis의 같은 중앙 세션 상태를 조회하여 세션 생성·갱신·만료·폐기와 권한 판정을 일관되게 처리한다.
- 중앙 세션에는 최초 로그인 시각, 마지막 정상 요청 시각과 계정 유형을 관리한다. 유휴·절대 만료는 일반 사용자 7일·30일, 식당 대표자 8시간·7일, 플랫폼 운영자 30분·8시간으로 적용하고 공통 기준 시각으로 먼저 도달한 만료를 판정한다. 절대 만료 시점은 요청이나 세션 갱신으로 연장하지 않는다.
- Redis 세션 조회·갱신 장애 시 모든 애플리케이션 인스턴스는 인증을 실패 폐쇄하고 신규 로그인과 인증 필수 요청을 거부하되 비회원 공개 조회 경로는 유지한다. MySQL이나 로컬 메모리 세션으로 자동 전환하지 않으며, 복구 후 중앙 Redis에 남은 세션의 유효·만료·유실 상태를 그대로 따른다.
- 로그인·권한 변경·추가 인증 성공 시 세션 ID 재발급과 이전 ID 무효화를 중앙 세션 상태에서 일관되게 처리한다. 브라우저의 상태 변경 요청에는 Spring Security CSRF 보호를 적용하고, 로그아웃은 중앙 세션 무효화와 쿠키 삭제를 함께 수행한다.
- 계정별 활성 세션 수 확인, 생성 시각이 가장 오래된 세션 종료와 새 세션 생성은 중앙 Redis 상태를 기준으로 일관되게 처리하여 여러 인스턴스의 동시 로그인에도 일반 사용자 5개·식당 대표자 3개·플랫폼 운영자 1개 상한을 넘지 않게 한다. 식당 대표자의 상한은 관리 매장 수가 아닌 중앙 계정 식별자를 기준으로 계산한다.
- 정상 비밀번호·등록 이메일 변경은 다른 중앙 세션의 만료 상태를 변경하지 않는다. 현재 요청이 재인증 경계를 통과하면 현재 세션 ID만 재발급하며, 비밀번호 분실 재설정·수동 복구 사건은 기존 세션 버전을 갱신하여 모든 세션을 거부하는 AUTH-006 정책을 따른다.
- 활성 세션 조회와 개별·다른 세션 전체 종료는 Spring Session Data Redis의 계정별 세션 인덱스를 기준으로 처리한다. 표시 정보는 로그인·마지막 활동 시각과 요청에서 얻은 대략적인 브라우저·운영체제로 제한하고 정밀 위치·지속적 기기 지문을 추가 수집하지 않는다.
- 공개 닉네임은 변경되지 않는 중앙 계정 식별자와 분리하고 중복을 허용한다. NFC 정규화, 2~20자 길이, 허용 문자, 예약어·금칙어 버전과 7일 변경 제한을 중앙 상태로 검증하여 모든 인스턴스에서 같은 결과를 사용한다.
- 프로필 이미지 업로드는 JPEG·PNG·WebP와 5MB 이하로 제한하고 비동기 이미지 재처리로 원본 메타데이터와 비이미지 내용을 제거한 확정 버전만 공개한다. 프로필 변경 사건으로 분산 캐시를 무효화하며 유료 외부 콘텐츠 검수 API를 필수 의존성으로 두지 않는다.
- 일반 사용자 가입 가능 연령 기준과 정책 버전을 중앙에서 관리하고, 휴대전화 본인 확인 결과로 만 14세 이상 여부를 판정한다. 생년월일 원본은 저장하지 않고 가입 가능 연령 충족 여부·확인 시점·정책 버전만 중앙 인증 상태에 저장한다.
- 유효한 만 14세 이상 판정과 계정 생성은 하나의 일관된 처리 경계에서 확정하여 외부 확인 결과의 중복·순서 역전이나 다중 인스턴스 동시 요청으로 연령 확인 없는 계정이 생성되지 않게 한다. 미충족·판정 불가 상태는 운영자 재량이나 로컬 상태로 우회하지 않는다.
- 탈퇴 요청 시 중앙 계정 상태를 즉시 탈퇴 대기로 변경하고 중앙 세션 버전을 갱신하여 모든 인스턴스에서 기존 세션을 거부한다. 요청 시각과 7일 후 삭제 예정 시각을 중앙에 기록하고, 유예기간 안의 취소는 재인증 결과와 삭제 예정 시각을 원자적으로 검증한다.
- 유예기간 만료 작업은 계정 단위 멱등성과 작업 소유권을 보장하여 여러 작업 인스턴스가 같은 계정을 중복 처리하지 않게 한다. 실패 작업은 재시도·관측 가능하게 관리하고, 삭제·익명화 대상과 법정 보존 자료를 논리적으로 분리하여 보존 자료가 계정 복원이나 일반 서비스 처리에 사용되지 않게 한다.
- 탈퇴 전 거래 검사는 예약·웨이팅·홀드·결제·환불·분쟁의 중앙 확정 상태를 기준으로 수행한다. 웨이팅·무료 홀드·취소 비용 없는 미결제 예약의 취소 명령은 거래별 멱등성을 보장하고, 모든 자동취소 성공이 확인된 경우에만 중앙 계정 상태를 탈퇴 대기로 전환한다.
- 결제·보증금·취소 수수료·환불·분쟁 관련 상태가 발견되거나 자동취소가 실패하면 탈퇴 상태 전환을 거부하고 정리할 거래를 반환한다. 일부 거래만 취소된 뒤 실패한 경우 이미 확정된 취소를 되돌리지 않으며, 재요청은 남은 거래만 멱등하게 처리한다.
- 최종 삭제 시 정규화된 이메일·휴대전화·카카오 제공자 식별자는 원문이나 단순 해시로 재가입 제한 저장소에 남기지 않고 서버 비밀키 기반 식별 토큰으로 변환한다. 토큰에는 계정 유형과 최종 삭제 후 30일의 제한 만료 시각을 연결하고 만료 작업으로 삭제한다.
- 가입 확정은 중앙 재가입 제한 토큰 조회와 함께 일관되게 처리하여 다중 인스턴스의 동시 가입으로 제한이 우회되지 않게 한다. 제한 만료 후 가입은 새 중앙 계정 식별자를 발급하고 삭제된 계정의 서비스 이력을 연결하지 않는다.
- 30일 전 조기 해제는 휴대전화·이메일 소유 확인 결과와 `재가입 제한 해제` 권한을 가진 관리자 1명의 결정을 중앙 심사 사건에 연결한다. 제한 해제와 변경 불가능한 감사 이력은 하나의 일관된 처리 경계에서 확정하고 동시 결정에는 최초의 유효한 결과 하나만 반영한다.
- 장기 미사용을 위한 별도 휴면 상태·분리 저장소·자동탈퇴 배치를 두지 않는다. 모든 인스턴스는 중앙 계정 상태와 AUTH-007의 중앙 세션 만료를 그대로 적용하고, 계정 탈취 의심은 마지막 이용 시각만으로 자동 판정하지 않고 AUTH-012의 위험 사건으로 분리한다.
- 경고·특정 기능 제한·기간 정지·영구 정지의 제재 단계와 적용·종료 시각을 중앙 계정·권한 상태로 관리한다. 모든 인증·서비스 인스턴스는 같은 중앙 제재 상태를 권한 판정에 적용하고 캐시만으로 정지 여부를 승인하지 않는다.
- 제재 변경과 관리자·사유·근거 감사 이력은 하나의 일관된 처리 경계에서 확정한다. 기간 정지 만료는 중앙 기준 시각과 멱등한 상태 전환을 사용하여 다중 작업 인스턴스의 중복 해제를 막는다.
- 경고·기능 제한·기간 정지는 중앙의 `계정 제재` 권한을 검증한 관리자 1명이 확정한다. 영구 정지는 제안 관리자와 다른 슈퍼관리자의 추가 승인을 중앙 심사 사건에서 요구하며, 슈퍼관리자 승인 전에는 영구 정지 상태를 노출하지 않는다.
- 영구 정지 승인·거절의 동시 요청에는 최초의 유효한 결정 하나만 반영하고, 슈퍼관리자 권한 검증·제재 상태 전환·감사 이력 생성을 하나의 일관된 처리 경계에서 확정한다.
- 기간·영구 정지 확정과 중앙 세션 버전 갱신을 일관된 처리 경계에서 수행하여 모든 기존 세션을 거부하고, 인증 성공 전후의 중앙 제재 검사로 신규 로그인을 차단한다. 정지 계정용 제한 세션이나 별도 로그인 권한은 발급하지 않는다.
- 제재 이메일 발송 요청은 제재 확정 사건과 함께 내구성 있게 기록하고 멱등 키로 중복 생성을 막는다. 별도 작업자가 최초 실패 후 1분·5분·15분 간격으로 최대 3회 추가 재시도하며, 중앙 작업 상태의 원자적 획득으로 다중 작업 인스턴스의 중복 발송·재시도를 방지한다.
- 모든 추가 재시도가 실패하면 중앙 작업을 최종 실패로 전환하여 관리자 경보와 감사 가능한 수동 재발송 대상을 생성하되, 확정된 제재와 로그인 차단 상태는 변경하지 않는다.
- 이의 제기 접수는 중앙 심사 사건에서 등록 휴대전화 또는 등록 이메일 중 하나의 유효한 소유 확인 결과를 요구한다. 확인 전 응답은 계정 존재와 제재 상세를 구분하지 않으며, 연락처·IP·요청 경로별 발급·검증 횟수를 중앙에서 원자적으로 제한한다.
- 최초 제재 관리자와 이의 제기 심사자가 같아도 별도의 중앙 이의 제기 사건과 `이의 제기 심사` 권한을 검증한다. 경고·기능 제한·기간 정지의 유지·감경·취소는 상태 변경과 변경 전후 감사 이력을 일관되게 확정한다.
- 영구 정지 이의 제기는 심사 의견과 슈퍼관리자 최종 결정을 중앙 사건에 연결하고, 동시 승인·거절에는 최초의 유효한 결정 하나만 반영한다.
- 중앙 이의 제기 상태와 제재 상태를 분리하고 최종 결정 전에는 기존 제재와 만료 시각을 변경하지 않는다. 최종 감경·취소가 확정될 때 제재 변경과 감사 이력을 일관된 처리 경계에서 기록하여 모든 인스턴스가 같은 새 결과를 적용하게 한다.
- 제재 계정의 최종 삭제 시 정규화된 이메일·휴대전화·카카오 제공자 식별자를 서버 비밀키 기반 토큰으로 변환하고 원문 식별정보·프로필·거래내역·제재 상세 사유와 분리한다. 토큰에는 계정 유형, 제재 종류와 재가입 제한 만료 시각만 연결한다.
- 기간 정지 토큰의 만료는 정지 종료 시점과 AUTH-010의 최종 삭제 후 30일 중 늦은 시점으로, 영구 정지 토큰의 만료는 최종 삭제 완료 후 3년으로 계산한다. 제재 감경·취소 시 만료를 멱등하게 다시 계산한다.
- 가입 확정은 중앙 제재 토큰 조회와 함께 일관되게 처리하고, 토큰 만료·파기 작업은 원자적 작업 획득과 멱등성으로 다중 가입·작업 인스턴스의 우회와 중복 처리를 막는다.
- 제재 사건별 활성 이의 제기 하나만 허용하도록 중앙 유일성 기준을 적용하고, 종결 후 재접수는 기존 사건에 없던 새 자료 식별자를 요구한다. 동시 접수에도 하나의 사건만 확정한다.
- 이의 제기의 3영업일 1차 응답과 7영업일 최종 처리 목표는 중앙 접수·보완 재접수 시각과 공통 영업일 기준으로 계산한다. 최종 결정과 결과 이메일 발송 요청을 일관되게 기록하고 다중 작업 인스턴스의 중복 발송을 막는다.
- 계정 탈취 위험 사건의 낮음·중간·높음 단계, 판정 근거와 보안 잠금 상태를 중앙 계정 보안 상태로 관리한다. 기존 로그인 실패·세션의 대략적 브라우저·운영체제·연락처 변경·사용자 신고 사건만 입력으로 사용하고 유료 위험탐지·정밀 위치·지속적 기기 지문을 요구하지 않는다.
- 높은 위험 전환은 중앙 세션 버전 갱신과 보안 잠금 생성을 일관된 처리 경계에서 확정하여 모든 기존 세션과 신규 로그인을 거부한다. 같은 사건을 여러 인스턴스가 감지해도 계정별 멱등 키로 한 번만 전환한다.
- 중간 위험 이메일 알림 요청은 중앙 보안 사건을 기준으로 멱등하게 생성하고, 발송 실패가 위험 상태나 후속 판정을 되돌리지 않게 한다.
- 높은 위험 복구 사건에는 등록 휴대전화·등록 이메일 소유 확인 결과와 MiriYum 이메일 비밀번호 재설정 완료 상태를 연결한다. 모든 조건이 충족되기 전에는 일반 서비스 권한을 발급하지 않으며, 연락 수단을 사용할 수 없는 복구는 AUTH-006의 중앙 수동 심사 결과를 참조한다.
- 보안 잠금 해제는 중앙 복구 사건의 조건을 다시 검증하고 잠금 상태를 전환하는 방식으로 처리한다. 여러 인스턴스가 동시에 해제를 요청해도 최초의 유효한 전환 하나만 확정하고 모든 인증 인스턴스가 같은 중앙 결과를 적용한다.
- 외부 카카오 계정의 탈취 상태를 자체 판정하거나 복구하지 않으며 높은 위험 복구가 기존 카카오 연결 상태를 변경하지 않게 한다. MiriYum의 세션·이메일 비밀번호·중앙 계정 상태만 AUTH-012의 처리 경계에 포함한다.
- 휴대전화·이메일 소유 확인과 MiriYum 이메일 비밀번호 재설정이 모두 유효하면 관리자 승인 없이 잠금 해제와 복구 사건 종결을 하나의 일관된 처리 경계에서 자동 확정한다. 일부 조건이 실패하거나 동시 전환이 충돌하면 잠금을 유지하고 같은 복구 사건에 대한 재시도를 멱등하게 처리한다.
- 플랫폼 운영자 계정은 슈퍼관리자만 생성하며, 임시 비밀번호와 최초 변경 완료 상태를 중앙에서 관리하여 변경 전에는 운영자 권한을 사용할 수 없게 한다.
- 결제수단 원문을 직접 저장하지 않는다.
- 민감정보는 전송·저장 시 보호하고 로그에서 제거한다.
- 관리자 중요 작업에는 추가 인증과 재확인을 적용한다.
- QR, 초대 링크, 승인 링크에는 만료와 재사용 방지를 적용한다.
- 위치 정보는 목적에 필요한 최소 범위로 처리하고 원본 좌표의 장기 보관을 피한다.
- 개인정보 최소화·삭제 반영·접근 통제는 안전 기본선으로 적용한다. 항목별 보유·파기 기간, 탈퇴 후 법적 보관과 외부 제공·위탁 목록은 `PRIV-005`, `PRIV-006`, `PRIV-010`의 법률·계약 `TODO`를 따른다.

- 모든 요청과 비동기 작업에 추적 가능한 상관관계 ID를 사용한다.

## 11. 배포 환경과 비용 운영 방침

### 11.1 운영 확장 참조 구조

이 절은 대규모·다중 인스턴스 운영으로 확장할 때의 참조 구조다. 특정 AWS 관리형 상품의 유료 계약이나 최소 용량을 확정하는 정책이 아니며, 실제 도입은 트래픽·장애 영향·비용 관측과 별도 승인 후 결정한다.

- 백엔드는 Docker 이미지로 패키징하여 Application Load Balancer 뒤의 Amazon ECS Fargate에서 실행한다.
- 개발 환경의 백엔드 컨테이너는 기본 1개로 운영한다.
- 실제 운영 환경은 여러 가용 영역과 둘 이상의 백엔드 인스턴스로 단일 인스턴스 장애를 피하는 것을 목표로 한다. `SCALE-001`~`SCALE-003`·`SCALE-005`의 확정 서비스 목표를 충족하도록 검증하되, 정확한 최소 용량과 자동 확장 임계치·유료 상품은 별도 비용·운영 승인으로 정한다.
- 애플리케이션 인스턴스는 무상태로 유지하고 세션·홀드·대기열·파일 등의 공유 상태를 인스턴스 로컬 메모리나 로컬 디스크에 저장하지 않는다.
- Fargate 비용이 실제 사용량에 비해 지나치게 커지면 ECS의 실행 용량을 EC2로 전환하는 방안을 재평가한다.
- 독립 배포 서비스와 운영 팀이 크게 늘어나거나 Kubernetes 생태계가 필요한 경우에만 Amazon EKS를 재평가한다.

### 11.2 개발 환경

- 개발 기간에는 AWS 상시 자원을 만들지 않는다. 백엔드와 프론트엔드는 평소 IntelliJ에서 직접 실행하고, 기반 서비스는 로컬 Docker Compose로 실행하며 전체 통합 환경을 확인할 때 애플리케이션도 컨테이너로 실행한다.
- MySQL 8.4 LTS, Valkey, KRaft 모드의 Kafka, Kafka Connect와 Debezium, OpenSearch, S3 대체 환경인 LocalStack과 개발용 메일 서버 Mailpit을 로컬 컨테이너로 실행한다.
- 통합 테스트에서는 Testcontainers로 필요한 의존 서비스만 테스트 수행 중에 생성하고 종료한다.
- 로컬 환경도 운영과 동일한 제품 및 주요 버전을 사용하여 환경 차이를 줄인다.

### 11.3 시연 환경

- 시연 환경은 상시 운영하지 않고 시연 1~3일 전에 AWS에 임시로 생성하며, 시연 종료 직후 제거한다.
- 다중 인스턴스 동작을 검증하기 위해 Application Load Balancer 뒤에 ECS Fargate 백엔드 태스크 2개를 실행한다.
- 시연용 MySQL은 비용을 고려하여 작은 Amazon RDS for MySQL 8.4 Single-AZ 인스턴스를 사용한다.
- Valkey, KRaft 모드의 Kafka, Kafka Connect와 Debezium, OpenSearch와 시연용 관측 도구는 필요한 메모리를 확보한 보조 EC2 한 대에서 컨테이너로 실행할 수 있다.
- 프론트엔드 정적 파일과 사용자 업로드 파일은 각각 정한 S3 버킷과 CloudFront를 사용한다.
- NAT Gateway 등 실행 시간만으로 고정비가 발생하는 자원은 시연 요구 범위에서 생략하거나 최소화한다.
- 시연 구성은 애플리케이션의 다중 인스턴스 동작을 검증하기 위한 비용 절약형 환경이며, 데이터 계층까지 완전히 다중화된 실제 운영 환경을 의미하지 않는다.

### 11.4 시연 예산과 자원 정리

아래 금액은 1~3일 개발·시연을 위한 가변 계획값이며 운영 비용·용량 정책의 확정값이 아니다.

- 1~3일 시연의 AWS 예상 비용은 약 1만~8만 원으로 보고, 기본 예산은 10만 원, 비상 상한은 15만 원으로 둔다.
- 예상 금액은 트래픽, 환율, 세금 및 실행 시간을 반영한 대략적인 범위이며 OpenAI API, 문자 발송, 결제 및 외부 사업자 비용은 포함하지 않는다.
- 비용 경보를 설정하고 시연 종료 후 ECS 서비스와 태스크, ALB, EC2와 불필요한 EBS 볼륨, RDS, NAT Gateway, 퍼블릭 IPv4 주소 및 불필요한 로그가 남아 있지 않은지 확인한다.
- 보존이 필요한 DB 스냅샷과 S3 객체는 별도로 식별하고, 나머지 임시 자원은 삭제한다.

### 11.5 실제 운영 전환

- 실제 사용자가 예약과 결제를 사용하는 운영 환경으로 전환할 때 RDS Multi-AZ, Amazon ElastiCache for Valkey, Amazon MSK·MSK Connect, Amazon OpenSearch Service 같은 관리형 구성을 후보로 재평가한다. 유료 상품 계약과 구체 구성은 장애 영향·운영 인력·실측 부하·비용 근거 없이 자동 확정하지 않는다.
- 애플리케이션은 Valkey에 Spring Data Redis와 Lettuce로 접근하고, 짧은 원자적 Lua 스크립트와 Pub/Sub를 사용한다. 세밀한 메모리·퇴거 정책이 필요하거나 장기간 일정한 사용량에서 비용상 유리해지면 ElastiCache 노드형 Multi-AZ를 재평가한다.
- 현재 코드와 개발·시연 환경의 Kafka 연동에는 Spring Kafka를 사용한다. 실제 운영의 관리형 Kafka·Connect 제공자와 상품, Debezium Connector 작업자 용량은 확정하지 않은 `TODO`이며, 유료 계약·실측 부하·비용·운영 근거를 확인한 뒤 결정한다.
- 일반 검색에는 공식 OpenSearch Java Client를, AI 벡터 검색에는 Spring AI OpenSearch Vector Store를 사용하고 한국어 분석에는 Nori를 적용한다. MySQL을 원본으로 유지하고 Kafka 증분 갱신과 전체 재색인 경로를 제공한다.
- 초기부터 모든 관리형 자원을 상시 실행하지 않고 실제 트래픽, 장애 영향과 운영 지표를 근거로 단계적으로 전환한다.
- 비용 최적화 때문에 개발·시연 자원을 축소하더라도 운영 목표 구조와 다중 인스턴스 설계 원칙은 변경하지 않는다.

### 11.6 인프라 코드 관리

- AWS 인프라는 Terraform과 Terraform AWS Provider로 정의한다.
- 시연 환경은 Terraform으로 생성하고, 시연 종료 전 `terraform plan -destroy`로 삭제 대상을 검토한 뒤 `terraform destroy`로 임시 자원을 제거한다.
- Terraform 상태는 애플리케이션 자원과 분리된 전용 S3 버킷에 암호화하여 저장하고 S3 버전 관리와 Lockfile 기반 상태 잠금을 적용한다.
- Terraform 상태 저장용 S3 버킷은 별도 bootstrap 영역에서 관리하며 시연 환경 삭제 대상에 포함하지 않는다.
- demo와 production의 상태와 변수는 분리하고 Terraform 및 AWS Provider 버전을 고정한다.
- GitHub Actions가 Terraform을 실행할 때 장기 AWS 접근 키를 저장하지 않고 GitHub OIDC로 환경별 최소 권한 IAM 역할을 사용한다.
- RDS 삭제 방지, 보존 스냅샷과 로그 정책은 demo와 production에서 다르게 적용하며, Terraform 외부에서 만든 자원은 별도 정리 목록으로 관리한다.

### 11.7 비밀정보와 일반 설정

- RDS 자격증명, OpenAI API 키, 카카오 Client Secret과 PortOne API Secret 등 시연 환경의 비밀정보는 AWS Secrets Manager에 저장한다. 운영 전환으로 외부 문자·이메일 어댑터를 연결하면 해당 사업자 키도 같은 방식으로 관리한다.
- Spring Profile, 서비스 URL, 로그 수준, 캐시 TTL과 기능 설정 등 비밀이 아닌 환경별 설정은 AWS Systems Manager Parameter Store에 계층형 경로로 저장한다.
- 초기 암호화에는 AWS 관리형 KMS 키를 사용하고 실제 운영의 보안·감사 요구가 구체화되면 고객 관리형 KMS 키를 재평가한다.
- ECS Task Definition에는 비밀값을 평문으로 기록하지 않고 Secrets Manager 또는 Parameter Store의 ARN을 참조하며, Task Execution Role에는 필요한 값만 읽을 수 있는 최소 권한을 부여한다.
- 비밀정보가 변경되거나 회전되면 새로운 ECS 배포로 컨테이너를 교체한다. 실제 운영 전환 시 RDS 자격증명의 자동 회전을 적용한다.
- Terraform은 비밀 저장소와 접근 권한만 만들고 비밀값 자체를 코드·변수·상태 파일에 기록하지 않는다.
- GitHub Actions의 AWS 인증에는 OIDC를 사용하며 장기 AWS Access Key를 GitHub Secrets에 저장하지 않는다.
- 로컬 개발은 Git에서 제외한 `.env.local`에 개발 전용 값을 두고, 저장소에는 변수 이름과 예시만 포함한 `.env.example`을 제공한다.
- 운영 비밀정보를 로컬 개발자에게 배포하지 않고 환경별로 서로 다른 자격증명과 외부 서비스 키를 사용한다.

### 11.8 배포와 데이터베이스 변경

- 백엔드는 Amazon ECS Rolling Deployment를 사용하고 ECS Deployment Circuit Breaker와 자동 롤백을 적용한다.
- 배포 순서는 테스트, Docker 이미지 빌드, 커밋 SHA 태그로 ECR 푸시, 일회성 Flyway ECS Task 실행, Rolling Deployment, ALB 상태 확인 순서로 고정한다.
- Flyway 작업이 실패하면 새 애플리케이션 배포를 시작하지 않고, 새 태스크가 정상 상태에 도달하지 못하면 직전 정상 배포로 자동 롤백한다.
- Hibernate 스키마 자동 변경을 사용하지 않고 `ddl-auto=validate`로 애플리케이션과 스키마의 호환성만 검증한다.
- 혼합 버전이 동시에 실행되는 Rolling Deployment를 고려하여 데이터베이스 변경은 expand, migrate, contract 단계로 나누고 이전·신규 버전이 함께 사용할 수 있는 변경을 먼저 배포한다.
- 운영 위험과 무중단 전환 요구가 커질 때만 ECS 네이티브 Blue/Green Deployment를 재평가한다.

### 11.9 도메인과 엣지 구성

- `miriyum.com`과 `www.miriyum.com`은 CloudFront를 통해 OAC로 보호된 비공개 S3 프론트엔드 버킷을 제공한다.
- `api.miriyum.com`은 HTTPS Application Load Balancer를 통해 ECS 백엔드로 연결한다.
- DNS는 Route 53에서 관리하고 인증서는 ACM을 사용한다. CloudFront 인증서는 `us-east-1`, ALB 인증서는 `ap-northeast-2`에 둔다.
- HTTP 요청은 HTTPS로 전환하고, CORS는 정확한 프론트엔드 출처와 자격증명 사용만 허용하며 와일드카드 출처를 사용하지 않는다.
- 세션 쿠키는 가능한 한 `api.miriyum.com`의 host-only 쿠키로 유지한다.
- 비용 절약형 시연 환경에서는 AWS WAF를 생략한다. 실제 운영 전환 시에는 API ALB의 관리형 핵심 규칙·잘못된 입력 방어·속도 제한과 필요시 CloudFront WAF를 후보로 재평가하며, 유료 상품 도입과 임계치는 공격·정상 트래픽 실측 없이 확정하지 않는다.
- 도메인 등록과 Terraform 상태 저장소는 시연 환경 `terraform destroy` 대상에 포함하지 않는다.

## 12. 개발·시연 확정 기술 스택

### 12.1 프로젝트 실행 기준과 구조

- MiriYum은 실제 상시 운영 서비스가 아니라 개발·시연 프로젝트이며, 운영 계약과 유료 외부 서비스는 만들지 않되 코드와 아키텍처는 실제 운영으로 확장 가능한 수준을 목표로 한다.
- 백엔드 언어와 실행 기준은 Java 21, Spring Boot 4.0.x, Spring AI 2.0.x와 Jackson 3.x로 고정한다.
- 로컬 개발 IDE는 IntelliJ IDEA를 사용한다. IntelliJ가 관리하는 JDK 21을 Project SDK와 Gradle JVM으로 사용하며 로컬 JDK 공급업체는 고정하지 않는다.
- Gradle Kotlin DSL과 Gradle Wrapper를 사용하고 Java Toolchain의 언어 버전을 21로 고정한다.
- CI와 배포 컨테이너는 패치 버전 또는 이미지 digest가 고정된 Java 21 빌드·실행 이미지를 사용한다. 로컬 JDK와 컨테이너 JDK의 공급업체가 같을 필요는 없다.
- 백엔드는 하나의 코드베이스와 기본 배포 산출물을 유지하는 모듈러 모놀리스로 구성하고 Spring Modulith로 모듈 경계와 순환 의존을 검증한다.
- API와 비동기 작업의 부하가 분리될 때 같은 애플리케이션 이미지에 `api`와 `worker` Spring Profile을 적용하여 ECS 서비스를 독립 확장한다. 시연 비용을 줄일 때는 동일 태스크에서 실행할 수 있다.

### 12.2 백엔드와 데이터 접근

- 요청 처리는 Spring MVC와 내장 Tomcat의 동기식 모델을 사용하고 WebFlux는 사용하지 않는다.
- 기본 영속성 기술은 Spring Data JPA와 Hibernate로 정한다.
- 복잡한 동적 조회와 목록 검색에는 QueryDSL을 사용하고, 재고·홀드·작업 획득처럼 조건부 원자 갱신이 필요한 SQL에는 Spring JdbcClient를 사용한다. MyBatis는 초기 스택에서 제외한다.
- 데이터베이스 스키마는 Flyway SQL 마이그레이션으로만 변경하고 애플리케이션 시작 시 Hibernate `ddl-auto=validate`로 검증한다.
- 기본 데이터베이스는 MySQL 8.4 LTS이며 개발 Docker와 시연 RDS의 주요 버전을 일치시킨다.
- API 명세는 OpenAPI와 springdoc-openapi 3.x로 생성하고 Swagger UI를 개발·시연 환경에서 제공한다.
- 외부 HTTP API는 Spring RestClient 또는 해당 서비스의 공식 SDK로 호출한다.
- 외부 서비스별 연결·응답 제한 시간을 반드시 설정하고 Spring Cloud Circuit Breaker의 Resilience4j 구현으로 회로 차단, 제한된 재시도와 필요한 동시 실행 격리를 적용한다.
- 결제 승인·취소처럼 결과가 중복되면 안 되는 명령은 일반적인 자동 재시도 대상에서 제외하고 멱등 키와 상태 재조회로 복구한다.

### 12.3 프론트엔드

- 프론트엔드는 React 19.x, TypeScript strict mode, Vite 8.x 기반 SPA로 구성한다.
- Node.js 24 LTS와 pnpm 11.x를 사용하고 `packageManager` 설정과 `pnpm-lock.yaml`로 버전을 재현한다.
- UI에는 Bootstrap 5.3과 안정 버전의 React-Bootstrap을 사용한다.
- 라우팅은 TanStack Router의 파일 기반 라우팅, 서버 상태와 요청 캐시는 TanStack Query를 사용한다.
- 전역 클라이언트 상태 저장소는 처음부터 추가하지 않는다. 여러 화면에서 공유되는 순수 클라이언트 상태가 실제로 생기면 Zustand와 Redux Toolkit 중 요구에 맞는 도구를 재평가한다.
- 폼 상태는 React Hook Form, 입력 스키마와 검증은 Zod 4를 우선 사용한다. 문서 완성 후 구현 직전에 Yup이 폼 요구와 팀 생산성에 더 유리하다는 근거가 생기면 한 번 재평가한다.
- OpenAPI 기반 타입과 얇은 API 클라이언트에는 openapi-typescript, openapi-fetch와 openapi-react-query를 사용한다.
- API와 반복 Hook이 크게 늘어 수동 관리 부담이 커지면 TanStack Query Hook과 Mock을 생성할 수 있는 Orval로 구현 직전에 재평가한다.

### 12.4 인증, 결제와 외부 연동

- 인증은 Spring Security와 Spring Session Data Redis를 사용한다. 실제 중앙 세션 저장소는 Valkey이며 브라우저에는 의미 없는 무작위 세션 ID만 저장한다.
- 세션 쿠키는 HttpOnly, Secure, SameSite=Lax와 host-only를 기본으로 하고 상태 변경 요청에는 CSRF 방어를 유지한다. 브라우저 인증에 JWT를 사용하지 않는다.
- 일반 사용자 외부 로그인은 Kakao OAuth 2.0으로 시작한다.
- 개발·시연 결제 어댑터는 기존 계정과 키를 재사용하는 PortOne Browser SDK V2와 PortOne V2 REST API의 테스트 채널을 사용한다. 서버가 결제 금액·상태를 다시 조회하고 웹훅을 검증하며 멱등하게 처리한다. 이는 운영 PG·결제수단·계약을 정하는 `PAY-003`의 확정이 아니다.
- 휴대전화 본인확인 포트의 개발·시연 참조 연동은 PortOne V2와 다날 본인인증으로 두되 실제 유료 계약을 만들지 않고 개발용 어댑터를 사용한다. 운영 사업자와 계약은 `AUTH-005`·`PRIV-010`의 외부 의존 `TODO`로 별도 결정한다.
- 지도 표시는 Kakao Maps Web SDK, 주소·좌표 변환은 Kakao Local REST API를 사용한다. 식당 좌표는 MySQL에 저장하고 주변 검색은 OpenSearch `geo_point`로 처리하며 길찾기는 카카오맵 외부 링크로 연결한다.
- 이메일 시연은 Spring Mail과 Docker Compose의 Mailpit을 사용한다. 외부 SMS 사업자는 연결하지 않고 제공자 중립 개발용 SMS 어댑터를 로컬 싱크에 연결한다. 로컬 싱크에는 논리 알림 ID·템플릿 ID·처리 결과·시각 같은 최소 메타데이터와 마스킹한 수신자만 남기며, 메시지 본문과 원문 수신자는 애플리케이션 DB나 로그에 저장하지 않는다. 콘솔 출력이 필요해도 본문은 `[REDACTED]`로 표시한다.
- 이메일·문자·본인확인 구현은 포트와 어댑터로 분리하여 운영 전환 시 외부 제공자로 교체할 수 있게 하되, SES와 SOLAPI는 현재 사용 기술에 포함하지 않는다.

### 12.5 공유 상태, 메시징과 검색

- 중앙 세션, 캐시, 속도 제한, 홀드와 실시간 팬아웃에는 Valkey 8.1.x를 사용하고 Spring Data Redis와 Lettuce로 접근한다.
- 여러 키를 함께 원자적으로 변경해야 하는 짧은 작업에는 Valkey Lua Script를 사용하고 실시간 이벤트 팬아웃에는 Pub/Sub을 사용한다.
- 복잡한 분산 락과 장시간 동기화 요구가 구체화되면 구현 전에 Redisson을 재평가한다.
- 비동기 이벤트는 Apache Kafka 4.2.x와 Spring Kafka를 사용하고 개발 환경은 KRaft 모드로 실행한다.
- 소비는 at-least-once를 전제로 멱등하게 만들고 제한된 재시도 후 DLT로 이동시킨다.
- MySQL 트랜잭션과 이벤트 발행의 원자성은 Outbox 테이블, Debezium CDC와 Kafka Connect로 보장한다.
- 검색은 OpenSearch 3.7.x를 사용한다. 일반 검색에는 공식 OpenSearch Java Client, AI 벡터 검색에는 Spring AI OpenSearch Vector Store, 한국어 분석에는 Nori를 적용한다.
- MySQL을 검색 데이터의 원본으로 유지하고 Kafka로 증분 색인하며 전체 재색인 경로를 별도로 제공한다.
- 서버에서 브라우저로 보내는 실시간 갱신은 Spring MVC `SseEmitter`와 브라우저 `EventSource`를 사용한다. 명령은 REST로 처리하고 SSE는 상태 변화 알림에만 사용한다.

### 12.6 예약 작업과 대량 처리

- 단순 주기 실행은 Spring `@Scheduled`와 MySQL JDBC 기반 ShedLock을 사용한다. ShedLock은 데이터베이스 시간 `usingDbTime()`을 사용한다.
- 여러 ECS 인스턴스에서 같은 예약 작업이 동시에 실행되지 않게 잠금을 사용하되, 잠금 만료나 재시작에도 안전하도록 실제 작업 자체를 멱등하게 구현한다.
- 알림 재시도, 결제 보상과 복구 작업처럼 유실되면 안 되는 항목은 MySQL 작업 테이블과 JdbcClient의 조건부 원자 갱신으로 획득한다.
- 이벤트 발생 직후의 비동기 처리는 Kafka, 재시작 가능한 대량 재색인·이관·집계는 Spring Batch와 일회성 ECS Task를 사용한다.
- 복잡한 사용자별 달력과 영속 실행 일정 요구가 없으므로 Quartz는 초기 스택에서 제외한다.
- 만료 여부 같은 핵심 정합성은 스케줄러만 믿지 않고 명령 처리 시점에도 중앙 시각과 데이터베이스 상태를 다시 검증한다.

### 12.7 파일과 이미지

- 사용자 파일은 AWS SDK for Java 2.x로 발급한 presigned URL을 통해 비공개 S3 `incoming` 경로에 직접 업로드하고 개발 환경에서는 LocalStack을 사용한다.
- 공개 가능한 확정 파일만 CloudFront와 OAC를 통해 제공한다.
- 프로필과 식당·메뉴 이미지는 S3 업로드 확인, Outbox, Kafka, 전용 Image Worker, libvips, S3 `processed` 경로 순서로 비동기 처리한다.
- Image Worker는 동일한 Spring Boot 이미지의 `worker` Profile로 실행하고 API 서비스와 독립적으로 확장할 수 있게 한다.
- JPEG·PNG·WebP 입력을 실제 디코딩 결과로 검증하고 파일 크기, 가로·세로, 전체 픽셀 수, 메모리와 처리 시간을 제한한다.
- libvips로 방향 보정, 크기별 썸네일 생성, 메타데이터 제거와 WebP 변환을 수행한다. 변환이 모두 성공한 결과만 공개하고 같은 이미지 ID의 재처리는 같은 키에 기록하여 멱등하게 만든다.

### 12.8 테스트, 품질과 보안 검사

- 백엔드 테스트는 JUnit Jupiter, AssertJ, Mockito, MockMvc, Spring Modulith Test, Testcontainers, WireMock과 Awaitility를 사용한다. 데이터베이스 테스트에 H2를 사용하지 않는다.
- 프론트엔드 테스트는 Vitest, React Testing Library, `user-event`, MSW, Playwright와 jsdom을 사용한다.
- Java 코드 형식은 Spotless, 모듈 경계는 Spring Modulith 기본 검증으로 확인한다.
- TypeScript·React 정적 검사는 ESLint Flat Config와 typescript-eslint, 코드 형식은 Prettier를 사용한다.
- 저장소 공통 인코딩·줄바꿈·기본 들여쓰기는 EditorConfig로 통일한다.
- 한 달 프로젝트의 상시 보안 도구는 Trivy 하나로 제한하고 Docker 이미지, 파일시스템 의존성, Terraform·Dockerfile 설정과 비밀정보 노출을 CI에서 검사한다.
- Dependabot과 OWASP ZAP은 현재 스택에서 제외하고 장기 운영으로 전환할 때 재평가한다.
- 부하 테스트는 Grafana k6 CLI를 사용한다. 로컬에서는 작은 시나리오, 시연 전에는 별도 임시 실행 환경에서 핵심 검색·로그인·주문·SSE 흐름과 성능 임계치를 검증한다.
- 코드 품질 대시보드가 필요할 정도로 코드와 팀이 커지면 SonarQube를 재평가한다.

### 12.9 관측, CI/CD와 버전 고정

- 애플리케이션 관측은 Spring Boot Actuator, Micrometer, Prometheus, Grafana와 구조화 로그로 시작한다.
- 로그·트레이스 통합이 필요해지면 Grafana Loki, OpenTelemetry, Tempo, Alloy와 Alertmanager를 단계적으로 추가한다.
- CI/CD는 GitHub Actions, Docker 다단계 빌드와 Amazon ECR을 사용한다.
- PR에서는 백엔드·프론트엔드 빌드, 타입·형식·정적 검사, 테스트, Spring Modulith 검증과 Trivy 검사를 실행한다.
- 배포 이미지는 커밋 SHA로 태그하고 GitHub Environment 승인과 OIDC 기반 AWS 역할로 시연 배포와 Terraform을 실행한다.
- 프론트엔드 빌드 결과는 S3에 배포한 뒤 CloudFront 캐시를 갱신한다.
- 기준 버전 계열은 Java 21, Spring Boot 4.0.x, Spring AI 2.0.x, MySQL 8.4 LTS, Valkey 8.1.x, Kafka 4.2.x, OpenSearch 3.7.x, Node.js 24 LTS, React 19.x, Vite 8.x와 pnpm 11.x로 고정한다.
- 실제 프로젝트 생성 시 호환되는 최신 안정 패치와 모든 컨테이너 이미지를 정확히 고정하고 한 달 동안은 보안 문제나 치명적 결함이 없는 한 기능 업데이트를 위해 버전을 올리지 않는다.

## 13. 운영 목표와 남은 외부 조건

아래 사용자 직접 결정 목표는 개발·시연 기술스택 선택과 분리되어 모든 인스턴스가 공유하는 서비스 기준이다. 수치가 유료 사업자·최소 인프라 용량을 뜻하지 않으며, 제공자·계약은 별도 승인으로 정한다.

- `SCALE-001`~`SCALE-003`: MAU 100,000명, 입점 매장 5,000개, 일 핵심 거래 50,000건; 정상 200 RPS·1,000 RPS 10분; 월간 99.9%, 조회 p95 500ms·상태 변경 p95 1초·핵심 p99 2초
- `SCALE-005`: 공개 조회 IP당 분당 60회, 로그인 조회 계정당 분당 300회, 상태 변경 계정당 분당 60회, 예약·웨이팅 생성 계정당 분당 20회, 로그인·인증번호 요청 10분당 5회; 중앙 공유 제한·`429`·`Retry-After`
- `SCALE-013`: 거래 5분/1시간, 설정·문서 24시간/4시간, 분석 24시간/24시간 RPO/RTO와 정기 복원 검증·실패 보고

- `NOTI-009`, `PRIV-005`, `PRIV-006`, `PRIV-010`, `PRIV-012`: 보관기간, 법적 보관, 외부 제공·위탁 및 법정 통지의 최종 범위
