# MiriYum 1차 MVP 문서 정합성 수정 설계

## 목적

현재 확정된 제품 결정을 바꾸지 않고, 정책 정본과 MVP 요약 문서가 같은 상태·범위·책임을 설명하게 한다. 실제 PG 계약, 최종 법률 검토, API·DB 명세, 구현 코드와 GitHub Wiki는 이번 범위에 포함하지 않는다.

## 선택한 접근

기존 상태 체계와 상세 정책을 정본으로 사용하는 최소 동기화 방식을 적용한다.

- `개정 확정`은 새 상태로 추가하지 않고 기존 공식 상태인 `확정`으로 정규화한다.
- `STORE-007`과 `WAIT-015`는 현재 상세 정책과 결정 기록에 따라 확정 상태로 유지한다.
- 체크인과 웨이팅의 상세 정책에 이미 기록된 1차 MVP 하위 범위를 요약 문서가 그대로 따르게 한다.
- 과거 시점이 명시된 결정 기록의 역사적 수치는 유지하되, `현재`라고 표현된 상태·목록만 최신 정본과 맞춘다.

`개정 확정`을 별도 상태로 신설하는 방식은 상태 체계와 소비 규칙을 불필요하게 확장하므로 채택하지 않는다. 이미 확정된 정책을 다시 팀 논의 상태로 되돌리는 방식도 현재 결정 기록과 충돌하므로 채택하지 않는다.

## 정책 상태 정합성

정책 마스터의 217개 행을 공식 상태로 정규화한 현재 집계는 다음과 같다.

- `확정`: 194개
- `팀원 상의 필요`: 8개
- `자동 추천 예정`: 0개
- `TODO`: 15개

남은 대표 팀 결정은 `TASTE-003`, `TASTE-008`, `TASTE-012`, `SUB-001`, `SUB-002`, `SUB-003`, `SUB-007`, `ADS-001`의 8개다.

다음 문서의 현재 집계와 현재 논의 목록을 같은 값으로 맞춘다.

- `docs/service-policies/README.md`
- `docs/service-definition.md`
- `docs/technical-architecture.md`
- `docs/service-policies/00-policy-template.md`
- `miriyum-service-decisions.md`

## 체크인 MVP 경계

1차 MVP에 포함하는 체크인·노쇼 범위는 다음과 같다.

- 서버가 발급한 회전형 QR 검증
- QR 사용이 어려울 때 권한 있는 매장 운영자의 직접 방문 완료 처리
- 모든 매장에 공통 5분 지각 허용시간
- 지각 허용시간 경과 뒤 권한 있는 매장 운영자의 직접 노쇼 확정
- 사유 코드, 예약금 환불 정책 입력과 최소 상태 변경 이력

일회 확인번호, 매장별 체크인 창·지각시간 설정, 일부 인원 도착, 자동 노쇼 후보·확정, 반복 제재, 정식 이의 제기와 고급 복구는 후속 구현으로 유지한다.

`README.md`와 `docs/flowcharts/flowchart.md`에서 일회 확인번호와 정식 이의 제기를 현재 MVP처럼 표현한 문구를 제거한다. `docs/service-definition.md`와 `docs/service-policies/09-checkin-noshow.md`의 상세 경계는 변경하지 않는다.

## 웨이팅 MVP 경계

1차 MVP의 팀 모델과 호출 범위는 다음과 같다.

- 로그인한 대표자와 입력 인원수로 단일 대기 팀 생성
- 공통 3km 반경
- 좌석·테이블 조건 없는 단일 FIFO
- 전체 FIFO 선두 팀만 호출
- 실제 호출 한 번과 호출 뒤 10분 도착 제한

일행 계정 초대·연결, 일행 합류·이탈, 대표자 변경, 좌석·테이블 호환 호출과 순번 미루기는 후속 구현으로 유지한다.

`README.md`, `docs/service-definition.md`, `docs/service-policies/05-waiting.md`에서 일행 계정이 현재 MVP에 참여해 상태를 공유하는 것으로 읽히는 문구를 대표자와 입력 인원수 기반 팀 설명으로 바꾼다. 확정된 후속 정책 자체와 결정 기록은 삭제하지 않는다.

## 검증

수정 전에는 아래 검사가 현재 불일치를 탐지해야 하고, 수정 후에는 모두 통과해야 한다.

1. 정책 마스터 217행이 허용 상태만 사용하고 `확정 194`, `팀원 상의 필요 8`, `TODO 15`로 집계된다.
2. 현재 논의 목록에 `STORE-007`과 `WAIT-015`가 없고 남은 8개 ID만 존재한다.
3. 현재 상태를 설명하는 공식 문서가 모두 `194/8/0/15`를 사용한다.
4. README와 플로우차트가 일회 확인번호·정식 이의 제기를 1차 MVP로 표현하지 않는다.
5. MVP 요약에서 일행 계정 연결이나 좌석·테이블 호환 호출을 현재 기능으로 표현하지 않는다.
6. 저장소 내부 Markdown 링크가 모두 유효하고 `git diff --check`가 통과한다.
