# MiriYum README·GitHub Wiki 설계

> 작성일: 2026-07-23
> 상태: 사용자 승인
> 범위: 루트 README와 GitHub Wiki의 정보 구조, 소유권, 작성·검증·전환 규칙

## 1. 목표

MiriYum 저장소를 처음 방문한 평가자·채용 담당자, 새로 합류한 개발자, 현재 팀원이 각자 필요한 정보에 빠르게 진입할 수 있게 한다.

- 루트 `README.md`는 3분 안에 프로젝트의 목적, 현재 단계, 핵심 기능, 기술 방향과 다음 읽을 문서를 이해하게 한다.
- GitHub의 `Wiki` 탭은 독자별 온보딩과 주제별 상세 설명을 제공한다.
- 제품 정책과 기술 결정의 정본은 메인 저장소에 유지한다.
- Wiki가 정본 문서를 복제해 서로 다른 결론을 만들지 않게 한다.
- 프로젝트 문서 구조가 안정되면 자동 동기화 방식으로 전환할 수 있게 경계를 남긴다.

## 2. 현재 저장소에 대한 전제

현재 저장소는 구현 코드보다 제품·정책·아키텍처 문서가 중심인 설계 단계다. 따라서 실행 파일과 실제로 검증된 명령이 생기기 전에는 설치, 실행, 테스트, 배포 명령을 만들어 내지 않는다.

현재 정본은 다음과 같다.

| 주제 | 정본 |
|---|---|
| 현재 서비스 정의와 MVP 경계 | `docs/service-definition.md` |
| 현재 기술 방향 | `docs/technical-architecture.md` |
| 정책 행별 상태·중요도 | `docs/service-policies/README.md` |
| 도메인별 상세 정책 | `docs/service-policies/01-*.md` ~ `18-*.md` |
| 확정·변경·보류 이력 | `tastelock-service-decisions.md` |
| 최초 아이디어와 과거 구상 | `tastelock-service-blueprint.md` |

향후 `docs/00-index.md`, ADR, 기능 명세가 생성되면 README와 Wiki의 링크 대상을 갱신하되, 이 표의 의미상 소유권을 임의로 바꾸지 않는다.

## 3. 문서 계층과 책임

| 계층 | 책임 | 포함하지 않는 내용 |
|---|---|---|
| 루트 `README.md` | 프로젝트 소개, 현재 단계, 핵심 가치, 기술 개요, 독자별 진입점 | 상세 정책 원문, 긴 결정 이력, 존재하지 않는 실행 명령 |
| 메인 저장소 `docs/` | 제품·정책·아키텍처·ADR·기능 명세의 버전 관리되는 정본 | 독자별로 반복 작성한 설명 사본 |
| GitHub Wiki | 온보딩, 용어 해설, 흐름 중심 설명, 정본 문서 탐색 | 정책 상태표의 사본, 별도 기술 결정, 작업 상태의 정본 |
| Issue·Project·PR·CI | 범위, 담당자, lifecycle, 실행 결과와 검증 증거 | 제품 정책이나 장기 기술 기준 |

GitHub Wiki는 사진에서 확인한 저장소 상단의 `Wiki` 탭을 뜻한다. 현재 단계에서는 메인 저장소에 `wiki/` 디렉터리를 만들지 않고 GitHub Wiki 저장소에 직접 페이지를 작성한다.

## 4. README 설계

README는 외부 방문자에게도 이해되는 문장으로 시작하고, 세 독자층의 진입점을 분리한다. 권장 순서는 다음과 같다.

1. **프로젝트 제목과 한 줄 정의**
   - `MiriYum`
   - 식당 탐색, 방문 예약, 웨이팅과 한정 메뉴 정보를 하나의 흐름으로 연결하는 다이닝 플랫폼
2. **현재 프로젝트 상태**
   - 설계·정책 정리 단계임을 명시한다.
   - 구현되지 않은 기능을 완료된 것처럼 표현하지 않는다.
3. **문제와 해결 방향**
   - 사용자와 매장이 겪는 문제를 각각 짧게 설명한다.
   - MiriYum이 제공하려는 핵심 연결 흐름을 설명한다.
4. **핵심 경험**
   - 매장·메뉴 탐색
   - 방문 예약과 선택 메뉴 수량의 함께 처리
   - 현장·원격 웨이팅
   - 취소 자리·수량 승계
   - 매장 운영과 플랫폼 운영
   - 1차 MVP 제외 기능은 별도로 구분한다.
5. **핵심 기술 방향**
   - 단일 Spring Boot 애플리케이션의 도메인 경계
   - React·TypeScript 프론트엔드
   - MySQL 중심 정합성
   - 필요 조건 충족 후 Redis·Kafka·OpenSearch·AWS 등을 단계적으로 도입한다는 원칙
   - 현재 구현 완료로 오해할 수 있는 기술 배지는 사용하지 않는다.
6. **저장소와 문서 지도**
   - 현재 존재하는 공식 문서만 링크한다.
   - 정책 정본과 과거 블루프린트의 차이를 설명한다.
7. **독자별 다음 단계**
   - 평가자·외부 방문자: 프로젝트 개요와 핵심 차별점
   - 신규 개발자: Wiki 온보딩과 기술 방향
   - 현재 팀원: 정책 색인, 결정 이력, 작업 도구
8. **로드맵**
   - 문서·정책 정리
   - 실행 가능한 최소 백엔드·프론트엔드 스캐폴드
   - 핵심 기능과 검증 gate
   - 테스트 완료 후 배포
9. **Wiki 링크**
   - 상세 가이드가 GitHub Wiki에 있음을 명시한다.

README는 목차를 포함해도 전체를 과도하게 길게 만들지 않는다. 상세 설명은 Wiki나 정본 문서로 넘긴다.

## 5. GitHub Wiki 정보 구조

Wiki는 독자별 진입과 주제별 탐색을 결합한다. 초기 상세 구성은 다음 12개 페이지와 공통 사이드바·푸터다.

| 페이지 | 목적 | 주요 정본 |
|---|---|---|
| `Home` | 세 독자층의 시작점과 전체 탐색 지도 | README와 아래 페이지 |
| `Project-Overview` | 문제, 대상 사용자, 가치, MVP 경계 설명 | `docs/service-definition.md` |
| `Core-User-Journeys` | 탐색→예약, 웨이팅, 메뉴 홀드, 승계 흐름 설명 | 서비스 정의와 관련 정책 |
| `Getting-Started-for-Developers` | 현재 단계, 필수 읽기 순서, 구현 전제 안내 | 기술 방향과 문서 지도 |
| `Architecture-Guide` | 시스템 경계, 데이터 흐름, 정합성, 확장 방향 해설 | `docs/technical-architecture.md` |
| `Domain-Guide` | 회원, 매장, 예약, 결제, 리뷰, 알림 등 도메인 지도 | 서비스 정책 문서 |
| `Policy-Guide` | 정책 상태 의미, 정본 우선순위, 정책 찾는 법 | 정책 README와 결정 기록 |
| `Documentation-Map` | 목적별 공식 문서와 읽기 순서 | 메인 저장소 문서 전체 |
| `Decision-and-Change-History` | 주요 결정이 어디에 기록되는지와 확인 방법 | `tastelock-service-decisions.md` |
| `Team-Workflow` | Issue·Project·PR·CI의 역할과 문서 갱신 규칙 | 저장소 운영 계약 |
| `Glossary` | MiriYum의 도메인·기술 용어 설명 | 현재 정본 문서 |
| `FAQ` | 범위, 상태, 문서 위치와 자주 생기는 오해 해소 | 각 정본으로 연결 |

### 5.1 Home

Home은 다음 순서로 구성한다.

- MiriYum 한 줄 소개
- `현재 상태: 설계·정책 정리 단계`
- 독자별 카드에 해당하는 텍스트 링크
  - 프로젝트를 처음 보는 사람
  - 새로 합류한 개발자
  - 현재 팀원
- 주요 주제별 링크
- 공식 문서는 메인 저장소가 정본이라는 안내
- 마지막 확인일과 관련 정본 링크

### 5.2 독자별 경로

- 외부 방문자: `Home` → `Project-Overview` → `Core-User-Journeys`
- 신규 개발자: `Home` → `Getting-Started-for-Developers` → `Architecture-Guide` → `Domain-Guide`
- 현재 팀원: `Home` → `Policy-Guide` → `Documentation-Map` → `Team-Workflow`

### 5.3 공통 페이지 형식

각 Wiki 페이지는 다음 구조를 사용한다.

1. 이 페이지에서 알 수 있는 것
2. 핵심 설명
3. 관련 흐름 또는 문서 탐색
4. 공식 기준 문서
5. 페이지 상태와 마지막 확인일

정책 수치나 상태 목록처럼 자주 바뀌는 내용은 Wiki에 복사하지 않는다. 필요한 경우 “현재 값은 정책 색인에서 확인”이라고 안내하고 정본으로 연결한다.

### 5.4 Sidebar와 Footer

`_Sidebar.md`는 다음 그룹을 고정 순서로 제공한다.

- 시작하기
- 제품 이해
- 개발 이해
- 정책과 문서
- 팀 운영
- 참고

`_Footer.md`에는 다음을 둔다.

- 메인 저장소
- 루트 README
- 공식 문서 안내
- `Wiki는 안내 문서이며 제품·정책의 정본은 메인 저장소입니다.`

## 6. 링크와 콘텐츠 규칙

- README에서 메인 저장소 파일은 상대 경로로 연결한다.
- README에서 Wiki는 GitHub Wiki의 실제 URL로 연결한다.
- Wiki에서 메인 저장소 문서는 기본 브랜치의 파일 URL로 연결한다.
- Wiki 내부 링크는 GitHub Wiki 페이지 이름을 사용한다.
- 아직 존재하지 않는 파일, 실행 명령, CI, 배포 환경은 링크하거나 완료 상태로 표현하지 않는다.
- 정책의 상태·중요도·수량은 `docs/service-policies/README.md`에서만 관리한다.
- 긴 설명이 필요해도 Wiki는 정본의 결론을 바꾸거나 새 정책을 확정하지 않는다.
- 문서 간 충돌을 발견하면 Wiki 문구로 해결하지 않고 정본 소유 문서를 먼저 수정한다.

## 7. 작성과 갱신 흐름

### 7.1 초기 단계

1. 현재 정본 문서에서 README에 필요한 사실을 추출한다.
2. 루트 README를 작성하고 모든 로컬 링크를 검증한다.
3. GitHub Wiki 저장소의 현재 `Home`을 보존 가능한 상태로 확인한다.
4. 12개 페이지, `_Sidebar.md`, `_Footer.md`를 작성한다.
5. Wiki의 메인 저장소 링크와 내부 링크를 검증한다.
6. Wiki 변경을 한 커밋으로 기록하고 게시한다.
7. 게시된 Wiki 화면에서 Home, Sidebar, 주요 독자별 경로를 다시 확인한다.

### 7.2 이후 변경

- 제품 정책 변경: 정본 정책 문서 → 필요할 때 README → 관련 Wiki 설명 순서로 갱신한다.
- 기술 결정 변경: 기술 문서·ADR → 필요할 때 README → Architecture Wiki 순서로 갱신한다.
- 단순 온보딩 개선: 정본 의미가 바뀌지 않으면 Wiki만 갱신할 수 있다.
- 작업 상태 변경: Issue·Project·PR에서만 관리하며 README·Wiki에 일별 상태를 복제하지 않는다.

## 8. 자동 동기화 전환

초기에는 GitHub Wiki를 직접 관리한다. 다음 조건이 모두 충족되면 별도의 승인된 계획으로 자동 동기화를 도입한다.

1. 메인 저장소의 문서 계층과 Wiki 페이지 이름이 두 번의 연속된 마일스톤 동안 이동·개명 없이 유지된다.
2. 같은 변경을 메인 저장소와 Wiki에 수동으로 반영한 사례가 두 번 이상 기록된다.
3. 동기화가 덮어쓸 페이지와 사람이 직접 편집할 페이지의 소유권을 명확히 나눌 수 있다.
4. GitHub Actions 또는 선택한 배포 방식의 자격 증명과 보호 규칙을 팀이 승인한다.

전환 후에는 메인 저장소의 `wiki/`를 동기화 원본으로 둘 수 있다. 다만 `wiki/`는 Wiki 페이지의 원본일 뿐 제품 정책·기술 결정의 정본이 되지 않는다. 동기화는 단방향 `main repository → GitHub Wiki`로 하고, 게시 전에 링크 검사와 변경 미리보기를 통과해야 한다.

## 9. 실패 처리

- Wiki 저장소 접근 또는 push가 실패하면 게시 완료로 주장하지 않는다.
- 실패한 로컬 Wiki 변경은 임시 작업 경로에 유지하고 실패 원인과 재시도 방법을 보고한다.
- 정본에서 확인할 수 없는 사실은 추측해 채우지 않고 해당 설명을 생략한다.
- 깨진 링크, 존재하지 않는 실행 명령, 정본과 다른 정책 상태가 하나라도 있으면 게시를 중단한다.
- 이미 존재하는 Wiki 페이지와 충돌하면 기존 내용을 보존해 비교한 뒤 의미를 통합한다.

## 10. 검증

### README

- 모든 상대 링크의 대상이 존재한다.
- 프로젝트가 설계 단계라는 설명과 문서의 현재 상태가 일치한다.
- 구현되지 않은 실행·테스트·배포 명령이나 성공 배지가 없다.
- 외부 방문자, 신규 개발자, 현재 팀원 경로가 각각 한 번 이상 제공된다.

### GitHub Wiki

- `Home`과 11개 상세 페이지, `_Sidebar.md`, `_Footer.md`가 게시되어 있다.
- Home에서 세 독자별 경로가 모두 시작된다.
- Sidebar의 모든 내부 링크가 실제 페이지로 열린다.
- 각 상세 페이지에 공식 기준 문서와 마지막 확인일이 있다.
- 메인 저장소 링크가 실제 파일을 연다.
- 정책 상태표나 변경 가능 수치의 사본이 없다.

### 최종 일관성

- README, Wiki, 서비스 정의, 기술 방향, 정책 색인 사이에 상충하는 프로젝트 상태나 MVP 범위가 없다.
- `git diff --check`가 성공한다.
- 메인 저장소의 기존 사용자 변경은 이번 작업의 커밋에 포함되지 않는다.

## 11. 완료 기준

- 루트 README가 프로젝트의 현재 상태와 핵심 가치를 정확히 설명한다.
- GitHub Wiki가 12개 페이지와 공통 탐색 요소로 상세 구성된다.
- 세 독자층이 각자의 권장 경로를 따라 정본 문서까지 도달할 수 있다.
- 정본과 안내 문서의 책임이 명확하며 같은 정책 결론을 중복 소유하지 않는다.
- 자동 동기화는 현재 범위에 포함하지 않고, 객관적인 전환 조건만 남긴다.
