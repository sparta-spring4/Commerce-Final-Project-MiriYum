# MiriYum 1차 MVP 담당 영역 3 문서 정비 설계

**작성일:** 2026-07-22

**상태:** 사용자 승인 완료·품절 대체 추천 반경 3km 직접 변경 반영

**범위:** 검색·추천·메뉴 홀드·결제·마이페이지·매장 운영 통계·확장성의 1차 MVP 문서 정비

## 1. 목적

현재 MiriYum 정책 체계를 유지하면서 담당 영역 3의 1차 MVP 요구사항과 기존 문서 사이의 누락·모순·모호함을 해소한다. 이번 작업은 문서 정비만 수행하며 구현 코드, API 코드, 데이터베이스 스키마와 실제 인프라는 작성하지 않는다.

상세 정책의 정본은 `docs/service-policies/`다. `docs/service-definition.md`는 사용자에게 보이는 제품 범위를, `docs/technical-architecture.md`는 구현 경계와 구성요소 책임을, `tastelock-service-decisions.md`는 결정 이력을 요약한다. 같은 규칙을 여러 문서에 전문으로 복제하지 않고 정본을 참조한다.

## 2. 변경 범위와 정책 상태

이번 사용자 직접 결정으로 다음 정책을 `팀원 상의 필요`에서 `확정`으로 전환한다.

- `RES-001`: 1차 MVP는 즉시 확정 예약만 지원하고 매장 승인제는 향후 범위로 둔다.
- `RES-009`: MiriYum 공통 취소·환불 기준과 매장·플랫폼 귀책 기준을 적용한다.
- `HOLD-004`: 방문 예약 수용량과 선택 메뉴 수량을 하나의 10분 선점 그룹으로 결합한다.
- `PAY-001`: 예약금 사용 여부는 매장이 선택하고 예약 수용량·메뉴 홀드 확정 전에 결제한다.

## 3. 검색 카테고리와 용어

### 3.1 용어

| 용어 | 정의 |
|---|---|
| 매장 카테고리 | 매장 전체의 음식 종류와 영업 성격을 분류하는 MiriYum 자체 코드 |
| 메뉴 카테고리 | 개별 메뉴의 음식·음료 종류를 분류하는 MiriYum 자체 코드 |
| 주 카테고리 | 해당 매장이나 메뉴를 가장 대표하는 카테고리. 대상마다 정확히 1개 지정 |
| 보조 카테고리 | 주 카테고리만으로 표현되지 않는 추가 성격. 대상마다 1개 이상 5개 이하 지정 |
| 대표 메뉴 | 매장 대표가 대표 상품으로 지정하고 게시한 메뉴. 예약금 계산에 사용하는 평균값의 대상 |
| 태그 | 분위기·조리법·식이·운영 특성처럼 카테고리 검색을 보완하는 속성 |

매장과 메뉴는 각각 주 카테고리 1개와 보조 카테고리 1~5개를 가진다. 주·보조에 같은 코드를 중복 지정하지 않는다. 문맥이 불분명할 때만 `매장의 주 카테고리`, `메뉴의 보조 카테고리`처럼 대상을 풀어쓴다. `대표 메뉴`를 카테고리와 혼용하지 않는다.

### 3.2 초기 고정 분류

매장 카테고리의 주 분류 예시는 한식, 중식, 일식, 양식, 아시아 음식, 분식·간편식, 카페·베이커리, 디저트, 주점, 기타다. 보조 분류 예시는 고기·구이, 해산물, 면요리, 밥요리, 국물요리, 치킨, 피자·버거, 브런치, 커피·차, 베이커리, 디저트, 주류다.

메뉴 카테고리의 주·보조 후보는 밥요리, 면요리, 국·탕·찌개, 고기요리, 해산물요리, 치킨, 피자·버거·샌드위치, 분식·사이드, 베이커리, 디저트, 비알코올 음료, 주류, 기타를 사용한다. 하나의 메뉴에서 가장 대표적인 항목을 주 카테고리로, 함께 적용되는 항목을 보조 카테고리로 고른다.

조리법, 분위기, 식이·알레르기, 포장·픽업·한정 수량은 카테고리와 분리된 태그다. 매장 대표가 정해진 코드 안에서 선택하고 관리자가 검수한 값만 검색·추천에 반영한다. 사업자등록증 업태·종목은 입점 확인에만 사용하며 카테고리를 자동 추출하지 않는다.

### 3.3 알레르기 정보

식이 특성 안에서 일반 취향 태그와 알레르기 정보를 구분한다. 초기 알레르기 유발 가능 성분 코드는 난류, 우유, 메밀, 땅콩, 대두, 밀, 고등어, 게, 새우, 돼지고기, 복숭아, 토마토, 아황산류, 호두, 닭고기, 쇠고기, 오징어, 조개류, 잣이다.

메뉴별 상태는 `함유`, `혼입 가능`, `정보 미등록`으로 관리한다. `알레르기 없음`을 임의 선택하게 하지 않으며 정보 미등록을 안전으로 해석하지 않는다. 사용자가 현재 검색·추천 요청에서 제외 성분을 지정하면 세 상태 중 위험을 배제할 수 없는 후보는 안전 우선 추천에서 제외한다. 사용자 알레르기 정보를 계정에 영구 저장하지 않는다. 레시피·원재료 변경 시 검수를 다시 요구하며 안전 보장을 표현하지 않는다.

## 4. 검색·자연어·취향 추천

### 4.1 검색 계약

- 매장명, 메뉴명, 지역명 키워드를 검색한다.
- 지역, 매장·메뉴 카테고리, 예약 가능 여부를 조합한다.
- 서로 다른 필터는 `AND`, 같은 필터의 복수 값은 `OR`로 결합한다.
- 커서 페이징을 사용하고 기본 20개, 최대 50개를 반환한다.
- 필터·정렬을 바꾸면 기존 커서를 폐기한다.
- 키워드 검색은 관련도, 자연어 추천은 추천 점수, 주변 검색은 거리, 예약 시간 검색은 가장 빠른 가능 시간을 기본 정렬로 사용한다.
- 동률은 주 카테고리 일치, 보조 카테고리 일치 수, 실제 예약 가능 여부, 거리, 매장 ID 순으로 결정한다.
- 표시 직전과 예약·홀드 확정 직전에 중앙 원본의 영업·품절·수용량 상태를 다시 검증한다.

### 4.2 자연어 조건 해석기

`자연어 조건 해석기`는 MiriYum 백엔드 구성요소이며 외부 상품명이 아니다. 사용자 문장을 지역, 분위기, 매장·메뉴 카테고리, 거리, 가격대, 방문 시각, 예약 가능 조건으로 바꾼다.

처리 순서는 다음과 같다.

1. 숫자·단위·시각 표현을 정규화한다.
2. MiriYum 지역·카테고리·태그 사전과 결정적 규칙으로 명확한 조건을 먼저 추출한다.
3. 규칙으로 해석하지 못한 표현만 개발·시연용 Spring AI OpenAI 어댑터에 전달한다.
4. 반환값을 허용된 MiriYum 코드와 범위로 검증하고 잘못된 값은 버린다.
5. 검증된 조건으로 OpenSearch를 조회한다.
6. 이전 예약과 최근 메뉴 홀드 이력을 외부로 보내지 않고 MiriYum 내부에서만 재정렬에 반영한다.
7. 검색 결과를 MySQL 원본 상태로 재검증한다.

OpenAI에는 사용자 입력 문장과 허용된 MiriYum 코드만 전달한다. 사용자 ID, 연락처, 예약 원문, 홀드 이력, 결제정보와 정확한 위치는 전달하지 않는다. 원문 프롬프트와 응답을 운영 로그에 보관하지 않는다. 호출 수, 입력 길이와 토큰을 제한한다.

AI 호출 실패·시간 초과·형식 오류에는 규칙 기반 결과만 사용한다. 규칙으로도 조건을 만들지 못하면 입력 문장을 매장명·메뉴명·지역명 일반 키워드 검색으로 넘긴다. 운영용 AI 제공자·모델·비용 한도와 외부 처리 법률 검토는 TODO다.

OpenSearch는 조건 추출기가 아니다. 일반 검색, Nori 기반 한국어 분석, `geo_point`, 벡터 후보 검색을 담당한다. Meilisearch는 1차 MVP 기술에 추가하지 않는다.

### 4.3 위치

현재 위치는 사용자가 내 주변·거리 정렬·거리 기반 추천을 명시적으로 실행할 때만 요청한다. 백그라운드로 수집하지 않는다. 원본 좌표는 요청 처리 동안만 사용하고 계정·추천 이력·로그에 저장하지 않는다.

권한 거부·GPS 오류·정확도 부족에는 시·군·구 또는 동 단위 지역 직접 선택을 제공한다. 캐시는 사용자 ID와 원본 좌표 대신 익명화된 지역 셀을 사용한다. 거리 계산은 사용자 현재 좌표와 매장 대표 좌표의 직선거리다. 위치정보 외부 제공·장기 저장과 최종 법률 검토는 TODO다.

## 5. 품절 대체 추천

품절 대체 후보는 다음 순서로 찾는다.

1. 같은 매장에서 주 메뉴 카테고리가 같은 메뉴를 찾는다.
2. 보조 카테고리 일치 수가 많은 후보를 우선한다.
3. 원래 메뉴 가격의 ±20% 범위를 우선한다.
4. 실제 판매·재고·영업·제공 시간을 재검증한다.
5. 같은 매장에 후보가 없으면 원래 매장 좌표 기준 반경 3km 이내 다른 매장을 찾는다.
6. 다른 매장 후보에도 주 메뉴 카테고리, 가격대, 영업시간과 같은 방문 시각·인원의 예약 가능 여부를 적용한다.

알레르기 제외 조건을 위반하거나 안전을 판단할 수 없는 후보는 안전 우선 추천에서 제외한다. 대체 메뉴나 다른 매장을 자동 적용하지 않는다. 같은 매장 대체는 사용자 명시 수락 뒤 새 수량 선점, 원래 수량 해제와 홀드 교체를 하나의 원자·보상 경계로 처리한다. 다른 매장 후보는 새 예약 화면으로 연결하며 기존 예약을 자동 취소하지 않는다.

## 6. 방문 예약·메뉴 홀드·예약금

### 6.1 예약과 메뉴 홀드

1차 MVP는 즉시 확정 예약만 지원한다. 사용자가 방문 시각과 인원을 정하면 관련 서비스 구간의 예약 수용량을 10분간 임시 선점한다. 메뉴 홀드가 가능한 매장이면 최종 확정 전에 메뉴 홀드 화면을 표시한다.

- 건너뛰기: 예약금 결제 또는 결제 불필요 판정 뒤 예약 수용량만 확정한다.
- 메뉴 선택: 같은 선점 그룹에 모든 메뉴 수량을 결합하고 예약금 결제 또는 결제 불필요 판정 뒤 예약 수용량과 메뉴를 함께 확정한다.
- 실패·포기·만료: 그룹에 포함된 예약 수용량과 메뉴 수량을 모두 반환한다.
- 단독 픽업 홀드: 방문 예약과 독립된 기존 흐름을 유지한다.

매장은 현재 게시된 모든 메뉴를 한 번에 홀드 가능으로 설정할 수 있다. 각 메뉴의 제공 시간과 수량은 MiriYum에서 관리하며 전체 POS 실시간 재고 연동은 MVP에서 제외한다.

### 6.2 예약금 계산

매장은 예약금 사용 여부를 선택한다. 예약금을 사용하려면 하나 이상의 유효한 대표 메뉴가 있어야 한다.

- 대표 메뉴 평균값: 현재 게시된 대표 메뉴 기본 판매가의 단순 산술평균
- 포함: 일시 품절된 대표 메뉴
- 제외: 숨김·판매 종료·삭제 메뉴와 옵션 추가금
- 최솟값: `(해당 가게의 대표 메뉴 평균값 × 예약 인원수) × 10%`
- 최댓값: `(해당 가게의 대표 메뉴 평균값 × 예약 인원수) × 30%`
- 최솟값과 최댓값 모두 1,000원 단위로 올림
- 매장은 계산된 범위 안에서 1,000원 단위 금액을 선택

고정 소액 예약금이 객단가가 높은 매장의 예상 이용 규모와 노쇼 부담을 충분히 반영하지 못하는 문제를 줄이면서 과도한 금액 설정은 상한으로 제한하기 위한 방식이다. 예약금은 실제 손해액을 확정하거나 보장하지 않는다.

메뉴를 선택해도 메뉴 가격이나 별도 홀드 수수료를 청구하지 않는다. 예약금은 노쇼 방지용 보증금이며 정상 체크인 시 전액 환불한다. 실제 식사 금액은 매장에서 별도로 결제하고 MiriYum은 예약금과 현장 주문 금액을 상계하지 않는다.

### 6.3 결제 실패와 재시도

검증된 결제 성공만 예약 수용량·메뉴 홀드 최종 확정의 근거다. 사용자 성공 화면은 정본이 아니다.

- 명확한 결제 실패: 원래 10분 만료 시각까지 선점 그룹을 유지하고 재결제를 허용한다.
- 재결제: 만료 시각을 연장하지 않는다.
- 사용자 명시 포기: 즉시 전체 반환한다.
- 10분 내 성공하지 못함: 전체 반환한다.
- 결과 불명확·Webhook 지연: `결제 확인 중`으로 격리하고 PG 조회·검증된 Webhook으로 판정할 때까지 재판매하지 않는다.
- 결제 성공 후 자원 확정 실패: 멱등 재시도하고 최종 확정 불가 시 승인 취소·전액 환불 보상으로 수렴한다.

PortOne V2 테스트 채널은 개발·시연용이다. 운영 PG·결제수단·계약·수수료와 실제 식당 정산은 기존 TODO를 유지한다.

### 6.4 취소·환불

MiriYum 공통 기준을 사용한다.

- 예약 확정 후 10분 이내이고 방문 예정 시각 전인 경우: 예약금 100% 환불
- 방문 예정 시각 48시간 전까지: 100% 환불
- 방문 예정 시각 48시간 미만부터 24시간 전까지: 50% 환불
- 방문 예정 시각 24시간 미만 또는 확정 노쇼: 환불 없음
- 매장 귀책, MiriYum 장애, 중복 결제, 결제만 성공하고 예약 확정 실패: 100% 환불

예약금·취소·노쇼 안내와 최종 약관·법률 검토는 TODO를 유지한다. 환불 계산과 실행은 기존 PAY 멱등성·대사·실패 복구 정책을 따른다.

### 6.5 노쇼

- 예약 방문 시각에 매장에 방문 확인 알림을 보낸다.
- 유효한 QR 체크인 또는 매장 `방문 완료` 확인 시 정상 방문으로 확정하고 예약금을 전액 환불한다.
- 체크인 창과 예약 시 고정된 지각 허용 시간이 지난 뒤 매장은 노쇼를 직접 확인할 수 있다.
- 매장 직접 확인 시 노쇼를 확정한다.
- 방문 시각부터 6시간 동안 방문·노쇼 확인이 모두 없으면 노쇼 후보로 전환하고 매장과 사용자에게 재알림한다.
- 24시간 안에 방문 완료 정정 또는 사용자 이의가 접수되면 자동 확정을 중단한다.
- 24시간까지 아무 확인·이의가 없으면 노쇼를 자동 확정한다.
- 매장 취소·휴점, MiriYum 체크인·알림 장애 등 사용자 귀책으로 볼 수 없는 사건이 있으면 자동 확정하지 않는다.
- 오판은 기존 이의·정정 정책에 따라 반전하고 예약금을 환불한다.

## 7. 마이페이지

마이페이지는 내 정보, 예약 이력, 웨이팅 이력, 메뉴 홀드 이력, 예약금 결제·환불 이력과 알림 이력을 분리한다.

- `주문 이력`이라는 용어를 사용하지 않는다.
- 실제 식사 주문과 현장 결제는 MVP 이력에 포함하지 않는다.
- 현재와 과거 상태를 분리한다.
- 최신 발생 시각 내림차순, 동률이면 이력 ID 내림차순으로 정렬한다.
- 커서 페이징 기본 20개, 최대 50개를 적용한다.
- 사용자는 자신의 기록만 조회한다.
- 결제수단 원문, 내부 감사 메모와 다른 방문자 정보를 노출하지 않는다.
- 알림 이력은 실제 보관 중인 기록만 표시하고 보관 기간은 `NOTI-009`와 개인정보·법률 TODO를 따른다.

## 8. 매장 무료 운영 통계

기본 집계는 매장 Free의 1차 MVP 권한으로 제공하고 구독 권한에 종속시키지 않는다. 상권 비교, 전주·전월 대비 해석, 취소 원인 분석, 운영 추천과 자동 리포트는 향후 Pro 후보이며 MVP에서 제외한다.

모든 날짜·시간은 매장 시간대를 기준으로 한다.

| 지표 | 산식·범위 |
|---|---|
| 오늘 예약 팀 수 | 오늘 방문 예정인 확정·체크인·이행 완료·노쇼 후보·노쇼 확정 예약의 팀 수. 취소·거절·결제 실패·만료 제외 |
| 예약률 | 오늘 서비스 구간별 확정 예약 인원 점유량을 플랫폼 예약 제공 인원으로 나눈 값의 합산 비율 |
| 팀 수 상한 사용률 | 오늘 서비스 구간별 확정 예약 팀 수를 최대 팀 수로 나눈 값의 합산 비율 |
| 취소율 | 해당 기간에 방문 예정이었던 확정 예약 중 취소된 예약 팀 수의 비율 |
| 웨이팅 현황 | 현재 대기 중·호출 중인 팀 수와 인원수, 가장 오래 기다린 시간 |
| 노쇼 현황 | 노쇼 후보와 노쇼 확정을 분리한 팀 수 |

결제 실패로 확정되지 않은 거래는 예약·취소 통계에 넣지 않는다. 예약 변경은 최신 유효 버전만 사용한다. 사건 ID로 중복 반영을 막고 정정·삭제 사건을 다음 재집계에 반영한다. 화면에는 마지막 반영 사건 시각과 통계 생성 시각을 표시한다. 집계 지연·격리 중에는 정상 수치처럼 표시하지 않는다.

## 9. 다중 서버·캐시·신뢰성

MySQL은 예약·수량·결제·환불·노쇼의 최종 원본이다. Valkey는 세션·속도 제한·임시 캐시와 짧은 선점 보조, OpenSearch는 검색 읽기 모델, Kafka는 상태 변경 전달을 담당한다. 애플리케이션 인스턴스의 로컬 메모리를 정본으로 사용하지 않는다.

### 9.1 검색·추천 캐시

- 일반 검색 결과 캐시는 최대 30초다.
- 키에는 정규화한 조건, 정렬, 커서, 카테고리·태그 버전, 검색 인덱스 버전과 익명화된 지역 셀을 포함한다.
- 사용자 ID와 원본 좌표를 공용 캐시 키에 넣지 않는다.
- 개인화 추천은 공통 후보만 캐시하고 사용자 이력 기반 최종 순위는 요청마다 계산한다.
- 매장·메뉴·가격·카테고리·영업·품절 변경 사건으로 관련 캐시를 무효화한다.
- 캐시가 남아 있어도 표시·확정 단계에서 중앙 원본을 재검증한다.
- Valkey 장애에는 오래된 값을 정본처럼 쓰지 않고 OpenSearch 또는 제한된 원본 조회로 우회한다.

### 9.2 누락된 SCALE 상세

- `SCALE-006`: 모든 공유 상태의 정본·복제·캐시 책임과 장애 시 실패 폐쇄/우회 경계를 명시한다.
- `SCALE-009`: 사건 ID, 대상 버전과 소비자 Inbox 유일성으로 중복 효과를 막고 역순·누락 이벤트를 재조회·재색인한다.
- `SCALE-010`: 다중 인스턴스 중 하나만 작업을 획득하되 만료·대사·노쇼·통계 작업 자체도 멱등하게 만든다.
- `SCALE-014`: 마지막 수용량·재고 경합, 결제/만료 경합, Webhook 중복·역순, 수량 복구, 캐시 지연, 통계 재집계, 인스턴스 종료와 의존성 장애를 검증한다.

실제 서버 대수, 캐시 용량, 경보 임계치와 부하 시험 장비 규모는 실측 TODO로 둔다. 기존 1,000 RPS 10분 피크와 응답시간 SLO는 검증 목표로 유지한다.

## 10. 1차 MVP 제외 경계

다음 기능은 정책 문서를 삭제하지 않고 상단 또는 범위 섹션에 `1차 MVP 제외`를 명시한다.

- 코스·테이스팅과 관련 선결제·설문·행사 운영
- 사용자 구독과 매장 Pro
- 비교·해석·추천·자동화 리포트
- 광고 상품·광고 과금·광고 제거
- 구독자 우선 예약·선공개
- POS 주문·실시간 전체 재고 연동
- 실제 식사 주문·현장 결제 관리
- 외부 캘린더 연동
- 사업자등록증 자동 카테고리 추출

`13-ad-recommendation.md`의 일반 추천 안전 규칙은 광고 상품과 분리해 MVP에서 참조한다. `07-course-tasting.md`, `11-subscription.md`, `13-ad-recommendation.md`의 미래 후보를 현재 권한이나 필수 구현으로 해석하지 않는다.
