# MiriYum 운영 수치·웨이팅·결제·구독 정책 결정 설계

> 작성일: 2026-07-22
> 범위: 사용자와의 대화에서 직접 선택한 정책만 기록한다.
> 원칙: 외부 계약·법률 검토·아직 답하지 않은 제품 선택은 확정으로 간주하지 않는다.

## 1. 상태 반영 원칙

- 아래 `즉시 확정` 항목은 사용자가 구체적인 값을 직접 선택했으므로 상세 정책과 인수 조건을 작성한 뒤 마스터 상태를 `확정`으로 바꾼다.
- PG·지급대행처럼 외부 계약이 필요한 항목은 목표 사업자와 운영 방향을 선택했더라도 실제 계약·심사·법률 검토가 끝날 때까지 행 상태를 `TODO`로 유지한다. 선택한 하위 범위와 남은 조건을 분리해 기록한다.
- 사용자 구독은 상품 후보안을 보존하되 초기에는 출시하지 않는다. 가게 Pro는 혜택·가격·결제 정책까지 보존하지만 출시 시점을 아직 선택하지 않았으므로 관련 대표 정책의 남은 범위를 확정하지 않는다.
- 현재 합의와 무관한 팀 결정 및 TODO는 변경하지 않는다.

## 2. 즉시 확정할 운영 수치

### RES-006 예약 임시 선점

- 선점 시간은 10분이다.
- 만료 2분 전에 경고한다.
- 서버 중앙 시각을 기준으로 하며 사용자의 임의 연장은 허용하지 않는다.
- 만료되면 좌석을 정확히 한 번 반환한다. 결제 결과 불명확이나 서버 장애는 사용자 책임으로 간주하지 않고 대사·복구 흐름으로 격리한다.

### WAIT-003 등록 반경

- 플랫폼 기본 반경은 500m다.
- 매장은 300m 이상 1km 이하 범위에서 설정할 수 있다.
- GPS 정확도가 기준보다 나쁘면 즉시 탈락시키지 않고 재측정 또는 이미 확정된 현장 증명 경로를 안내한다.

### WAIT-011 호출 횟수와 응답 시간

- 호출은 최대 2회이며 각 호출의 응답 제한은 3분이다.
- 알림 재시도는 호출 횟수로 계산하지 않는다.
- 두 번째 제한 시간이 지나면 즉시 사용자 취소로 기록하지 않고 `호출 실패`로 전환해 매장이 최종 상태를 확인한다.
- 모든 호출·응답·만료 판정은 중앙 시각, 정책 버전과 멱등 키를 사용한다.

### HOLD-007 메뉴 임시 선점

- 메뉴 수량 임시 선점은 10분이다.
- 좌석과 메뉴를 함께 선점하면 만료시각을 하나로 맞춘다.
- 결합 선점은 전부 확정하거나 전부 반환하며 한 자원만 활성 상태로 남기지 않는다.

### CHECK-005 지각 허용시간

- 기본값은 10분이다.
- 매장은 3분 이상 15분 이하 범위에서 설정할 수 있다.
- 예약 생성 당시 값을 스냅샷으로 보존하고 이후 설정을 기존 예약에 소급하지 않는다.
- 허용시간이 지나도 즉시 노쇼로 확정하지 않고 `CHECK-006` 절차로 넘긴다.

### TRANSFER-004 승계 제안 시간

- 후보의 제안 응답 시간은 5분이다.
- 수락 후 예약·결제 완료를 위한 선점 시간은 10분이다.
- 응답 만료와 수락이 경합하면 중앙 상태 전이 하나만 성공하며 결제 결과 불명확 중에는 다음 후보에게 넘기지 않는다.

### ADMIN-004 신고·분쟁 처리기한

- 접수 확인: 1영업일 이내
- 일반 신고: 3영업일 이내
- 결제·환불 분쟁: 7영업일 이내
- 추가 증거·외부 회신이 필요한 복잡 사건: 최대 14영업일
- 최대 기한을 넘길 사유가 생기면 종결로 오표시하지 않고 당사자에게 사유와 다음 확인 시점을 알린다.

### ADMIN-006 환불·보상 승인 한도

- 20만원 이하는 `결제관리자`가 단독 승인할 수 있다.
- 20만원 초과는 결제관리자의 요청과 슈퍼관리자의 추가 승인이 필요하다.
- 원결제액을 초과하는 별도 보상은 금액과 관계없이 슈퍼관리자 승인이 필요하다.
- 모든 금액에서 대상 거래·사건, 승인자 계정과 당시 역할, 원결제액, 승인액, 사유·근거, 정책 버전, 시각, 멱등 키, PG 처리 결과와 실패·재시도 이력을 감사한다. 결제수단 원문과 불필요한 개인정보는 기록하지 않는다.

### SCALE-001·002·003·005·013 용량과 복구 목표

- 초기 용량 기준: 월간 활성 사용자 10만 명, 입점 매장 5천 곳, 예약·웨이팅·홀드 합계 하루 5만 건
- 전체 API 목표: 정상 200 RPS, 피크 1,000 RPS를 최소 10분간 처리
- 월 가용성 99.9%, 조회 API p95 500ms, 상태 변경 API p95 1초, 핵심 예약·선점 API p99 2초
- 속도 제한: 공개 조회 60회/분/IP, 로그인 조회 300회/분/계정, 상태 변경 60회/분/계정, 예약·웨이팅·홀드 생성 20회/분/계정, 로그인·인증번호 요청 5회/10분
- 백업·복구: 거래 데이터 RPO 5분·RTO 1시간, 설정·문서 RPO 24시간·RTO 4시간, 분석 데이터 RPO/RTO 24시간
- 속도 제한은 모든 인스턴스가 공유하는 중앙 상태를 사용하고 제한 초과 시 재시도 가능 시각과 함께 `429`를 반환한다.

## 3. WAIT-009 순번·미루기 제품 결정

- 플랫폼이 자동으로 임의 팀을 호출하지 않는다. 매장이 현재 빈 테이블을 선택하고 호출을 실행한다.
- 시스템은 테이블 정원·좌석 조건을 충족하는 활성 팀만 추리고, 그중 등록 시각이 가장 빠른 팀만 호출 대상으로 허용한다.
- 단골, 구독, 예상 매출, 결제액은 일반 웨이팅 순번에 영향을 주지 않는다.
- 활성 팀 대표자는 등록 직후부터 호출 직전까지 웨이팅 1건당 한 번 순번을 미룰 수 있다.
- 미루면 같은 좌석 조건으로 입장 가능한 팀 5팀 뒤로 이동한다. 현재 뒤에 호환 팀이 5팀보다 적으면 현재 호환 대기열의 마지막으로 즉시 이동하며 이후 새로 등록되는 팀보다 뒤로 계속 밀리지 않는다.
- 호출 시작 뒤에는 미루기를 허용하지 않는다. 미루기와 호출이 경합하면 중앙 조건부 전이 하나만 성공한다.
- 매장이나 운영자가 순번 값을 직접 쓰거나 기록되지 않은 사유로 앞 팀을 건너뛰지 못한다.

## 4. 결제·정산 목표안과 남은 외부 조건

### PAY-003 목표안

- 운영 결제 경계는 PortOne V2를 유지하고 첫 PG 채널은 토스페이먼츠로 선택한다.
- 초기 결제수단은 신용·체크카드와 토스페이·카카오페이·네이버페이다.
- 가상계좌, 휴대폰 소액결제, 상품권과 정기결제는 별도 계약과 제품 필요가 확인된 뒤 추가한다.
- 운영 활성화에는 PortOne·토스페이먼츠 계약, 업종·결제수단 심사, 수수료와 취소·분쟁 조건 확인이 필요하다. 완료 전 `PAY-003`은 `TODO`다.

### PAY-012 목표안

- 사용자는 MiriYum 결제 화면에서 결제하고 MiriYum은 PG 정산 자료와 내부 원장으로 식당별 지급액을 계산한다.
- 월요일부터 일요일까지 완료된 거래를 다음 주 수요일에 지급하며 공휴일이면 다음 영업일에 지급한다.
- 환불·분쟁이 있는 거래 금액만 개별 보류하고 다른 정상 거래의 지급을 함께 막지 않는다.
- 보류 사유와 예상 해제 조건을 식당에 표시한다.
- 지급대행 계약, 세무·법률 검토와 실제 자금 흐름 승인이 끝나기 전 `PAY-012`는 `TODO`다.

## 5. 사용자 구독 후보안

### 상품과 가격

- 단일 사용자 구독 상품이다.
- 월 3,900원, 연 39,000원이며 소비자 표시가격은 부가세 포함이다.
- 초기에는 출시하지 않는다. 매장·행사 공급과 사용자 규모가 확보된 뒤 다시 출시 여부를 결정한다.

### 무료·구독 혜택

- 무료 사용자는 관심 매장·메뉴·행사·취소 자리 알림을 최대 5개까지 활성화할 수 있고 구독자는 앱·웹 알림을 개수 제한 없이 등록할 수 있다. 생성 속도 제한과 중복 알림 묶기는 양쪽에 적용한다.
- 무료 추천은 현재 검색 조건, 대략적인 거리, 인기와 가용성을 사용한다. 구독 추천은 사용자가 직접 저장한 취향과 MiriYum 이용 이력을 사용한 상세 추천과 주간 큐레이션을 추가한다.
- 무료 사용자는 예정 예약·웨이팅 목록을 볼 수 있다. 구독자는 MiriYum 내부 통합 캘린더, 일정 충돌 경고, 맞춤 리마인더와 방문 기록 정리를 추가로 사용한다. 외부 캘린더 연동은 포함하지 않는다.
- 매장은 처음부터 구독자용으로 별도 개설한 행사·테이스팅을 제공할 수 있다. 구독은 해당 행사 신청 자격만 주며 당첨·좌석을 보장하지 않는다.
- 매장이 선공개 참여를 선택한 일반 공개 행사는 전체 정원의 최대 40%를 구독자에게 24시간 먼저 공개할 수 있다. 남은 선공개 수량은 일반 공개 시 합류하고 최소 60%는 처음부터 일반 사용자에게 공개한다.
- 일반 예약, 일반 웨이팅과 취소 자리에는 선공개·우선순위 혜택을 적용하지 않는다.
- 향후 광고 상품이 도입되면 구독자에게 유료 광고·스폰서 노출을 제공하지 않는다.

### 갱신·해지

- 결제 실패 유예기간은 3일이며 하루 한 번 재시도한다.
- 유예 중 기존 거래 권리는 유지하지만 새 구독 전용 행사 신청과 선공개 혜택은 제한한다. 복구되면 즉시 정상화하고 유예가 끝나면 자동 해지한다.
- 결제 전 `결제 즉시 혜택이 시작되며 단순 변심이나 중도 해지에 따른 환불은 제공하지 않는다`는 점을 명확히 표시한다.
- 해지는 다음 자동결제를 즉시 중단하고 현재 이용기간 종료일까지 혜택을 유지한다.
- 법정 청약철회, 중복·오결제, 서비스 미제공과 MiriYum 귀책에 따른 강행 예외는 제한하지 않는다. 최종 약관은 법률 검토를 거친다.

## 6. 가게 Pro 후보안

### 상품 경계

- 무료 기본형과 유료 Pro 한 단계로 구성하고 매장 1곳마다 구독한다.
- 예약·웨이팅의 등록·호출·관리 기능은 무료 핵심 기능으로 유지한다.
- 무료 매장은 예약·웨이팅·홀드·행사에서 실제 발생한 집계 수치와 기본 표·그래프를 조회한다.
- Pro는 집계를 비교·해석한 전환율, 취소율, 시간대·요일·메뉴별 추세, 전주·전월 비교, 성과 요약, CSV와 주간·월간 자동 리포트를 제공한다.
- Pro의 운영 추천은 집계에 근거한 설명 가능한 규칙으로만 생성한다. 가격·수량·순번을 시스템이 임의로 변경하지 않는다.

### 자동화

- 메뉴 판매 시작·종료 예약
- 매장이 직접 정한 수량 임계치에 따른 품절 전환
- 코스·테이스팅 반복 생성
- 행사 공개·마감 예약
- 주간·월간 분석 리포트 자동 생성
- 자동화는 매장이 승인한 범위와 버전만 실행하며 오류 시 실패 사실과 복구 경로를 제공한다.

### 가격·갱신·해지

- 공급가 월 29,000원과 부가세 2,900원, 실제 월 결제액 31,900원이다.
- 공급가 연 290,000원과 부가세 29,000원, 실제 연 결제액 319,000원이다.
- 결제 실패 유예기간은 7일이고 총 3회 재시도한다.
- 유예 중 기존 자동화는 실행하지만 새 자동화 등록은 제한한다. 유예 종료 후 무료형으로 전환하고 자동화를 중지하되 설정과 리포트 정의는 정책상 보존기간 안에서 유지한다.
- 해지는 다음 자동결제를 중단하고 현재 기간 종료일까지 Pro를 유지하며 단순 중도 해지 환불은 제공하지 않는다. 법정 예외, 중복·오결제와 MiriYum 귀책은 별도 처리한다.
- 가게 Pro의 출시 시점은 아직 선택하지 않았다. 이 남은 결정은 문서 반영 시 `팀원 상의 필요` 범위로 유지한다.

## 7. 브랜드 명칭

- 사용자에게 보이는 프로젝트 명칭과 문서 본문은 `MiriYum`으로 통일한다.
- 최초 blueprint에 남아 있던 이전 브랜드명 표기 4곳도 사용자의 후속 승인에 따라 `MiriYum`으로 변경한다.
- 문서 파일명과 이를 가리키는 링크는 후속 MiriYum 명칭 마이그레이션에서 `miriyum-service-decisions.md`, `miriyum-service-blueprint.md`로 변경됐다.

## 8. 문서 반영 및 검증 기준

- 마스터 상태·중요도, 상세 정책, 서비스 정의, 기술 아키텍처, 의사결정 기록과 다음 단계 문구가 같은 상태를 가리켜야 한다.
- 즉시 확정 정책에는 행위자·권한, 상태 전이, 동시성·멱등성, 실패·복구, 알림·감사, 개인정보 경계와 관측 가능한 인수 조건을 기록한다.
- 외부 계약이나 미결 제품 결정이 남은 행은 확정 하위 범위와 미결 범위를 분리하고 행 상태를 과장하지 않는다.
- 전체 Markdown에서 이전 브랜드명이 0개인지 대소문자를 구분해 검사한다. 소문자 이전 명칭 토큰은 실제 파일명과 링크에 한정한다.
- `git diff --check`, 마스터·도메인 요약 일치, 정책 ID 중복과 상태별 수량을 검증한다.
