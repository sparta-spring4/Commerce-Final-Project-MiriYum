# MiriYum 공식 서비스 정의

> 문서 상태: 정책 재분류·위임 추천 확정·사용자 직접 운영 결정 반영 완료
> 기준일: 2026-07-22
> 역할: 현재까지 합의된 서비스 전체 구상과 제품 경계를 설명한다.
> 참고: `tastelock-service-blueprint.md`는 최초 초안이며 이 문서의 공식 기준이 아니다.

## 1. 서비스 정의

MiriYum은 사용자에게 식당 탐색, 방문 예약, 웨이팅과 한정 메뉴 확보 경험을 제공하고, 식당에는 필요한 운영 기능을 선택적으로 구성할 수 있게 하는 양면 다이닝 플랫폼이다. 코스·테이스팅은 미래 확장 후보이며 1차 MVP에서는 제공하지 않는다.

이 프로젝트는 취업·포트폴리오 목적이지만 실제 사용자를 받을 수 있는 운영 수준을 목표로 한다. 기능 시연에 그치지 않고 동시성, 결제 정합성, 권한, 개인정보, 어뷰징, 장애 복구 및 운영자 개입까지 설계 범위에 포함한다.

## 2. 핵심 참여자

- 일반 사용자: 매장과 메뉴를 탐색하고 1차 MVP에서 예약·웨이팅·메뉴 홀드를 이용한다. 코스·테이스팅은 향후 활성화된 경우에만 이용한다.
- 식당 대표자: 해당 매장의 사업자등록증에 기재된 대표자 본인으로서, 매장을 등록하고 필요한 기능을 선택하여 예약과 운영 자원을 직접 관리한다.
- 플랫폼 운영자: 입점과 제한 기능을 심사하고 신고, 분쟁, 결제, 장애 및 어뷰징을 관리한다.
- 외부 사업자: 결제, 본인 인증, 지도, 메시지 및 향후 POS 연동을 제공한다.

### 2.1 계정 유형과 참여자 관계

- 비회원은 공개된 매장·메뉴·가격·예약 가능 여부를 탐색하고 조회할 수 있다.
- 예약, 웨이팅, 메뉴 홀드, 결제 및 리뷰 등 거래성 기능은 일반 사용자 계정으로 로그인해야 이용할 수 있다.
- 일반 사용자와 식당 대표자는 서로 독립된 계정 유형을 사용한다.
- 같은 사람이 두 계정 유형을 각각 보유할 수 있지만 인증 상태와 권한은 서로 공유하지 않는다.
- 정책상 같은 사람은 일반 사용자 계정과 식당 대표자 계정을 각각 최대 1개까지 보유하며, 같은 유형의 중복 계정은 허용하지 않는다.
- 현재는 계정 유형별로 확인된 휴대전화 번호와 확인된 이메일 주소를 각각 유일하게 관리하며, 둘 중 하나라도 기존 계정과 같으면 새 계정을 만들지 않는다.
- DI·CI 기반의 실제 동일인 중복 판정, 외부 본인확인 사업자 선정 및 유료 계약은 현재 범위를 넘는 미래·외부 의존 `TODO`다. 현재 확정된 연락처 유일성 기준을 그대로 적용한다.
- 중복 여부는 계정 생성 전에 확인하며, 중복이 발견되면 기존 계정 연결 흐름으로 전환하여 본인 인증 후 카카오·이메일 등 새 로그인 수단을 기존 계정에 추가한다.
- 일반 사용자 계정에 연결된 카카오와 이메일·비밀번호는 어느 방식으로 로그인해도 같은 계정과 서비스 데이터를 사용하며, 초기에는 로그인 방식의 개별 해제 기능을 제공하지 않는다.
- 입력한 연락처의 소유 확인 전에는 계정 존재 여부를 노출하지 않고, 확인 후에만 기존 계정을 찾는 데 필요한 최소 정보를 마스킹하여 안내한다.
- 확인된 휴대전화와 이메일이 서로 다른 기존 계정을 가리키면 어느 계정도 우선하지 않고 계정 생성·로그인 수단 연결을 중단한 뒤 복구 안내와 관리자 검토로 전환한다.
- 초기 서비스 범위에는 이미 생성된 두 계정의 데이터 통합 기능을 포함하지 않으며, 예외적인 중복 계정의 데이터 통합은 현재 범위를 넘는 미래 `TODO`로 둔다.
- 하나의 식당 대표자 계정은 여러 매장에 소속될 수 있다.
- 식당 직원용 계정은 제공하지 않으며, 대표자 계정의 공유도 허용하지 않는다.
- 개인사업자와 법인사업자 모두 사업자등록증상 대표자 본인의 개인 계정을 사용한다.
- 플랫폼 운영자는 일반 사용자 및 식당 대표자와 분리된 별도 운영자 계정을 사용한다.
- 현재 현장·원격 웨이팅은 모두 로그인한 일반 사용자에게만 허용한다. 휴대전화 번호를 인증한 비회원에게 현장 웨이팅만 예외적으로 허용하는 방안은 채택하지 않은 미래 `TODO`이며 원격 웨이팅으로 확대하지 않는다.
- 향후 정식 위임 책임자 1명의 개인별 운영 계정을 허용하는 방안은 위임 증빙·본인 확인·권한 회수 기준이 필요한 미래 `TODO`다.

### 2.2 계정 유형별 인증 방향

- 일반 사용자는 소셜 로그인을 중심으로 하고 이메일·비밀번호 로그인을 보조 수단으로 제공한다.
- 최초 소셜 로그인은 카카오만 지원하며 네이버·Google·Apple 추가는 미래 제공자 연동 `TODO`다.
- 일반 사용자는 소셜 또는 이메일·비밀번호 가입 여부와 관계없이 가입 완료 전에 휴대전화 본인 확인을 마쳐야 한다.
- 식당 대표자는 가입 완료 전에 휴대전화 본인 확인을 마쳐야 하며, 이후 이메일·비밀번호로 로그인한다.
- 휴대전화 본인 확인에 성공한 뒤에만 계정을 생성하며, 실패·취소·외부 본인확인 사업자 장애 시에는 계정을 만들지 않고 가입 절차에서 재시도하게 한다.
- 본인 확인 전 가입 입력 정보는 계정이 아닌 짧은 수명의 가입 세션으로만 유지한다.
- 일반 사용자와 식당 대표자의 이메일·비밀번호 계정은 이메일 소유 확인을 완료해야 가입이 확정되고 활성화된다.
- 일반 사용자와 식당 대표자의 비밀번호는 최소 8자, 최대 64자이며 영문 대문자·소문자·숫자·특수문자 중 3종 이상을 포함한다. 공백과 출력 가능한 ASCII·유니코드 문자를 허용하고 유니코드는 NFC로 정규화하며, 최대 길이를 넘는 입력은 자동으로 자르지 않는다. 설정 또는 마지막 변경 후 90일이 지나면 변경 권고 경고를 표시하되 로그인과 서비스 이용은 계속 허용한다.
- 흔하거나 알려진 유출 비밀번호에는 위험 경고를 표시하되 사용자가 원하면 설정할 수 있다.
- 일반 사용자와 식당 대표자가 비밀번호를 잊으면 소유 확인이 완료된 등록 이메일로 발급 후 10분 동안 유효한 6자리 숫자 일회용 코드를 받아 재설정한다. 코드를 재발급하면 이전 코드는 즉시 무효화하고 최신 코드 하나만 유효하게 하며, 재전송은 직전 발급 후 60초가 지나야 허용한다. 동일 이메일의 코드 발급은 계정 등록 여부와 무관하게 최초 발급을 포함해 1시간 최대 5회로 제한한다. 같은 코드 입력에 누적 3회 실패하면 해당 코드만 무효화하고 계정과 일반 로그인은 잠그지 않는다. 등록 이메일에 접근할 수 없는 경우에는 자동 복구 없이 플랫폼 운영자의 수동 심사로 처리한다.
- 비밀번호 재설정에 성공하면 로그인 방식과 무관하게 해당 계정의 모든 기존 세션을 종료하고 자동 로그인 없이 다시 로그인하게 한다.
- 비밀번호 재설정에 성공하면 계정의 등록 이메일로 즉시 보안 알림을 발송한다.
- 보안 알림 발송에 실패해도 비밀번호 재설정과 기존 세션 종료는 성공으로 유지하고, 직전 실패 후 1분·5분·15분 간격으로 최대 3회 추가 재시도한다.
- 추가 재시도까지 모두 실패하면 최종 실패 상태로 보관하고 권한 있는 플랫폼 운영자에게 경보와 수동 재발송 경로를 제공하되 계정 이용은 제한하지 않는다.
- 기존 등록 이메일과 같은 주소의 소유를 확인하면 수동 심사 없이 일반 비밀번호 재설정으로 전환한다. 기존 이메일의 소유를 확인할 수 없는 경우에만 다른 ‘복구 후 사용할 이메일’과 등록 휴대전화의 소유를 확인한 뒤 수동 복구를 신청한다. 일반 사용자는 기존 등록 이메일 주소의 일치 여부만 추가 심사하고 카카오 인증이나 이용 이력은 요구하지 않는다. 식당 대표자는 현재 관리 중인 매장 1곳의 사업자등록증을 제출하고 사업자등록번호와 대표자명이 기존 인증 정보와 모두 일치해야 하며, 매장명·주소는 필수 일치 조건으로 사용하지 않는다. 이후 플랫폼 운영자가 최종 승인한다.
- 등록 휴대전화도 사용할 수 없으면 해당 번호의 이용을 먼저 복구해야 하며, MiriYum은 신분증 제출이나 외부 본인확인을 대체 복구 절차로 제공하지 않는다.
- 수동 복구는 `계정 복구` 권한을 가진 플랫폼 운영자 1명이 승인한다. 승인·거절 시 심사 사건과 대상 계정, 담당 운영자, 증빙 확인 결과, 결정 사유와 처리 시각을 수정·삭제할 수 없는 중앙 감사 이력으로 남긴다.
- 완료된 수동 복구 감사 이력 전체는 슈퍼관리자 또는 별도의 `계정 복구 감사 열람` 권한을 가진 플랫폼 운영자만 열람한다. 진행 중인 심사 담당자는 자신에게 배정된 사건의 처리에 필요한 범위만 조회하며, `계정 복구` 승인 권한만으로 완료된 전체 이력을 열람할 수 없다.
- 수동 복구 감사 이력은 심사 처리 완료 시점부터 3년간 보관한 뒤 파기한다.
- 수동 복구 신청은 접수 후 3영업일 이내에 승인·거절·보완 요청 중 하나로 안내하며, 보완 자료가 다시 접수되면 그 시점부터 기한을 새로 계산한다.
- 수동 복구가 승인되면 확인된 복구 후 사용할 이메일로 등록 이메일을 교체하며 기존 이메일은 로그인·비밀번호 재설정·보안 알림 주소로 더 이상 사용하지 않는다.
- 수동 복구가 승인되면 로그인 방식과 무관하게 해당 계정의 모든 기존 세션을 종료한다.
- 수동 복구 승인 직후 확인된 복구 후 사용할 이메일로 기존과 동일한 6자리 숫자 일회용 재설정 코드를 발송한다. 새 비밀번호를 설정하기 전까지 카카오를 포함한 모든 로그인과 서비스 이용을 차단하며, 코드에는 기존의 10분 유효기간·재발급 시 이전 코드 무효화·60초 재전송·1시간 최대 5회 발급·코드별 3회 입력 실패 제한을 동일하게 적용한다. 새 비밀번호 설정 후에도 자동 로그인하지 않고 다시 로그인하게 한다.
- 수동 복구 승인으로 등록 이메일을 교체하면 교체 전 기존 이메일에도 변경 사실을 알리는 보안 알림을 발송하되, 복구 후 이메일 주소나 제출 증빙·심사 세부정보는 포함하지 않는다.
- 교체 전 이메일의 변경 보안 알림 발송에 실패해도 수동 복구 결과는 유지한다. 직전 실패 후 1분·5분·15분 간격으로 최대 3회 추가 재시도하고, 모두 실패하면 최종 실패 상태를 보관하여 권한 있는 플랫폼 운영자에게 경보와 수동 재발송 경로를 제공하되 복구된 계정의 이용은 제한하지 않는다.
- 같은 계정의 이메일·비밀번호 로그인이 연속 5회 실패하면 1분 동안 지연하고, 이후 실패가 계속되면 5분과 최대 15분으로 단계적으로 늘린다. 영구 잠금은 적용하지 않고 15분 지연을 반복하며, 성공 시 실패 상태를 초기화한다. 지연 중에도 카카오 로그인과 비밀번호 재설정·계정 복구는 허용하고 기본 IP·요청 경로별 속도 제한을 함께 적용한다. 별도 기기 지문 수집과 유료 봇 방어 서비스는 현재 범위에서 제외한다.
- 브라우저 인증은 Spring Security와 Spring Session Data Redis 기반의 중앙 서버 세션을 사용한다. 브라우저에는 `HttpOnly`·`Secure`·`SameSite=Lax`가 적용된 무작위 세션 ID 쿠키만 저장하고 브라우저 인증에 JWT를 사용하지 않는다.
- 세션은 마지막 정상 요청 이후 일반 사용자는 7일, 식당 대표자는 8시간, 플랫폼 운영자는 30분 동안 활동이 없으면 만료한다. 활동이 계속되더라도 최초 로그인 이후 일반 사용자는 30일, 식당 대표자는 7일, 플랫폼 운영자는 8시간이 지나면 절대 만료하며, 두 만료 시점 중 먼저 도달한 때 다시 로그인하게 한다.
- Redis에서 중앙 세션을 확인할 수 없으면 기존 세션을 인정하지 않고 신규 로그인과 인증 필수 기능을 일시 중단하되 비회원 공개 조회는 계속 제공한다. MySQL이나 애플리케이션 로컬 세션으로 전환하지 않으며, Redis 복구 후 세션 데이터가 유실됐거나 만료된 경우 다시 로그인하게 한다.
- 로그인·권한 변경·추가 인증 성공 시 세션 ID를 재발급하고 이전 ID를 무효화한다. 상태 변경 요청에는 Spring Security의 CSRF 보호를 적용하며, 로그아웃 시 중앙 세션을 무효화하고 세션 쿠키를 삭제한다.
- 계정당 동시 세션은 일반 사용자 5개, 식당 대표자 3개, 플랫폼 운영자 1개로 제한한다. 새 로그인으로 상한을 넘으면 가장 오래전에 생성된 세션을 종료하고 새 로그인을 허용한다. 식당 대표자의 상한은 관리 매장 수와 무관하게 개인 계정 단위로 적용하며 한 세션에서 권한이 있는 모든 매장을 전환하여 관리한다.
- 로그인 상태에서 필요한 본인 확인을 거쳐 비밀번호나 등록 이메일을 정상 변경하면 모든 기존 세션을 유지한다. 단, 비밀번호 분실 재설정과 수동 복구 시에는 AUTH-006에 따라 모든 세션을 종료한다.
- 모든 계정 유형은 현재 세션 여부, 로그인·마지막 활동 시각과 대략적인 브라우저·운영체제 정보가 표시된 자신의 활성 세션을 조회하고 개별 세션 또는 현재 세션을 제외한 모든 세션을 종료할 수 있다. 이를 위해 정밀 위치나 지속적인 기기 지문을 수집하지 않는다.
- 일반 사용자의 공개 닉네임은 내부 계정 식별자와 분리하고 중복을 허용한다. 닉네임은 한글·영문·숫자·공백·밑줄·하이픈으로 구성된 2~20자로 제한하고 7일에 한 번 변경할 수 있으며 연락처·URL·공식 계정 사칭 예약어는 사용할 수 없다.
- 일반 사용자의 이메일·휴대전화·본인확인 실명은 공개하지 않는다. 프로필 이미지는 선택 사항으로 JPEG·PNG·WebP, 5MB 이하만 받고 공개 전에 이미지 재처리로 원본 메타데이터를 제거한다. 닉네임·프로필 위반은 무료 내부 기준과 신고·플랫폼 운영자 검토로 처리한다.
- 일반 사용자 계정은 가입 시 만 14세 이상인 경우에만 생성한다. 만 14세 미만은 비회원 공개 탐색·조회만 이용하며 법정대리인 동의·확인 가입 절차는 제공하지 않는다. 본인 확인 결과에서 생년월일 원본 대신 가입 가능 연령 충족 여부·확인 시점·정책 버전만 저장한다.
- 만 14세 이상 미성년자는 일반 식당 탐색·예약·웨이팅·리뷰를 이용할 수 있지만 주류 상품·행사·판촉 기능은 AUTH-004에 따라 제한한다.
- 탈퇴 요청이 접수되면 계정은 즉시 탈퇴 대기 상태가 되고 모든 세션이 종료되며, 7일 동안 재인증을 거쳐 탈퇴를 취소할 수 있다. 7일이 지나면 계정 개인정보를 복구 불가능하게 삭제·익명화하고 법정 보존이 필요한 자료는 활성 계정과 분리해 목적과 기간을 제한한다.
- 탈퇴 요청 시 웨이팅·무료 홀드·취소 비용 없는 미결제 예약은 자동취소한다. 결제·보증금·취소 수수료·환불·분쟁이 연관된 거래가 남아 있거나 자동취소가 실패하면 탈퇴를 접수하지 않고 먼저 정리할 항목을 안내한다.
- 최종 삭제 후 30일 동안 같은 계정 유형에서 동일한 이메일·휴대전화·카카오 식별자로 재가입할 수 없다. 그 전에 재가입하려면 휴대전화·이메일 소유 확인을 모두 완료하고 `재가입 제한 해제` 권한을 가진 관리자 1명의 감사를 남기는 승인을 받아야 하며, 새 계정에 삭제된 계정의 이용 이력을 복원하지 않는다.
- 장기 미사용 계정을 별도 휴면 상태로 전환하거나 자동탈퇴시키지 않는다. 장기 미사용 후에도 기존 세션 만료와 일반 로그인 절차를 적용하며, 계정 탈취 의심은 미사용 기간이 아닌 별도의 위험 사건 정책으로 처리한다.
- 계정 제재는 경고·특정 기능 제한·기간 정지·영구 정지 단계로 운영한다. 일반 위반은 심각도·반복성·피해 범위에 따라 단계적으로 강화하되 사기·계정 탈취·결제 악용·심각한 안전 위협은 앞 단계를 생략하고 즉시 정지할 수 있으며, 관리자·사유·근거·기간을 감사 이력에 남긴다.
- 경고·기능 제한·기간 정지는 `계정 제재` 권한 관리자 1명이 확정하고, 영구 정지는 해당 권한 관리자의 제안과 서로 다른 슈퍼관리자 계정의 추가 승인이 모두 있어야 확정한다. 긴급 피해 차단을 위한 기간 정지는 추가 승인 없이 즉시 적용할 수 있다.
- 기간·영구 정지가 확정되면 모든 기존 세션과 신규 로그인을 차단하고 제한 로그인 화면은 제공하지 않는다. 등록 이메일로 제재 단계·기간·사유 요약과 고객지원 이의 제기 방법을 안내하며, 이메일 장애가 제재 적용을 지연하거나 취소하지 않는다.
- 고객지원 이의 제기는 등록 휴대전화 또는 등록 이메일 중 하나의 소유 확인 후 접수한다. 확인 전에는 계정 존재나 제재 상세를 노출하지 않으며 두 수단 모두 사용할 수 없으면 AUTH-006 계정 복구를 먼저 거친다.
- 최초 제재 관리자도 별도 `이의 제기 심사` 권한이 있으면 같은 사건을 심사할 수 있다. 경고·기능 제한·기간 정지는 심사 관리자가 유지·감경·취소하고, 영구 정지의 변경은 슈퍼관리자가 최종 확정하며 모든 판단의 근거와 변경 이력을 보존한다.
- 이의 제기 심사 중에는 기존 제재를 유지하고 접수만으로 자동 해제하지 않는다. 심사 결정이 확정된 시점부터 유지·감경·취소 결과를 적용하며 명백한 오정지는 별도 임시 상태 없이 우선 심사로 취소한다.
- 제재 중 탈퇴하더라도 기간 정지는 정지 종료 시점과 일반 30일 제한 중 더 늦은 시점까지, 영구 정지는 최종 삭제 후 3년까지 최소 식별 토큰으로 같은 유형의 재가입을 막는다. 기간이 지나면 토큰을 파기하고 새 계정 가입을 허용하되 과거 이력을 복원하지 않는다.
- 제재가 유효한 동안 이의 제기를 접수하고, 경고·종료된 제재는 통지 후 30일 안에 접수한다. 제재 사건당 이의 제기는 동시에 1건만 처리하고 최종 결정 뒤에는 새 자료가 있을 때만 재접수하며, 3영업일 안의 1차 응답과 7영업일 안의 최종 처리를 목표로 결과를 이메일로 안내한다.
- 계정 탈취 위험은 낮음·중간·높음으로 나눈다. 낮음은 기록, 중간은 등록 이메일 보안 알림과 세션 점검 안내, 높음은 모든 세션 종료와 신규 로그인·계정정보 변경을 막는 중앙 보안 잠금으로 처리하며 사용자의 활동 부인 신고는 높은 위험으로 본다. 유료 탐지 서비스·정밀 위치·지속적 기기 지문은 사용하지 않는다.
- 높은 위험 보안 잠금은 등록 휴대전화와 등록 이메일의 소유를 모두 확인하고 MiriYum이 직접 관리하는 이메일 비밀번호를 재설정한 뒤에만 해제한다. 어느 한 수단을 사용할 수 없으면 AUTH-006 계정 복구를 먼저 완료하며, 복구 조건을 충족하기 전에는 일반 서비스 이용과 계정정보 변경을 허용하지 않는다. 카카오 계정 자체의 탈취·복구는 카카오의 책임 범위로 두고 기존 카카오 연결을 변경하거나 재연결을 강제하지 않는다.
- 높은 위험 복구 조건을 모두 충족하면 별도의 관리자 심사 없이 보안 잠금을 자동 해제한다. 일부 조건이 완료되지 않았거나 검증 결과가 유효하지 않으면 잠금을 유지한다.
- 휴대전화 본인 확인은 정기적으로 반복하지 않고 번호 변경, 계정 복구 또는 계정 탈취 의심이 발생할 때만 다시 요구한다.
- 일반 사용자의 가입 시 본인 확인 결과로 주류 이용 가능 연령 여부를 판정하며, 같은 목적의 추가 본인 확인을 반복하지 않는다.
- 연령 판정을 위해 본인확인 결과에서 파생한 주류 이용 가능일과 확인 시점만 저장하고 생년월일 원본은 저장하지 않는다.
- 연령 기준을 충족하지 못한 사용자도 일반 식당 탐색·예약은 이용할 수 있지만, 주류 상품의 주문·홀드·결제와 주류 행사 예약·참여는 허용하지 않는다.
- 연령 기준을 충족하지 못한 사용자에게도 주류명·가격·코스 구성 등 이용 정보와 일반 광고는 표시할 수 있지만 주류 판촉 광고는 노출하지 않는다.
- 플랫폼의 연령 판정은 식당 현장의 나이·본인 확인을 대체하지 않는다.
- 실제 주류 이용자 전원의 나이·본인 확인은 가게의 현장 책임이며, MiriYum은 현장 확인을 직접 수행하거나 완료를 보증하지 않는다.
- MiriYum은 방문별 현장 확인 완료 상태나 현장에서 제시된 신분증 자료를 수집·기록하지 않는다.
- 플랫폼 운영자는 공개 가입 대상이 아니며, 슈퍼관리자가 운영자 이메일과 임시 비밀번호를 등록한다.
- 신규 운영자는 최초 로그인에서 임시 비밀번호를 반드시 변경해야 운영자 기능을 사용할 수 있다.
- 최소 권한·중요 작업 재인증·감사는 안전 기본선으로 제안하고, 운영자 WebAuthn 보안키·패스키의 도입 시점·적용 범위·등록·분실 복구는 `ADMIN-002`의 `TODO`로 둔다.
- 등록 이메일 접근 불가 수동 복구의 처리 기한과 감사 이력 보관·열람 기준은 AUTH-006을 따르고, 운영자 계정 발급·인증 정책은 ADMIN-001·ADMIN-002에서 관리한다.

## 3. 지원 범위

- 특정 지역 또는 한 업종으로 제한하지 않는다.
- 일반 식당, 베이커리, 브루어리, 오마카세, 팝업 등 여러 외식 업종을 지원한다.
- 식당을 고정 단계로 분류하지 않는다.
- 식당은 플랫폼이 제공하는 기능 모듈 중 필요한 조합을 선택한다.
- 잘못된 조합은 기능 의존성 검증으로 차단한다.

### 3.1 사업자등록 단위와 매장

- 입점 시 제출된 사업자등록증과 사업자등록번호를 사업자 확인 기준으로 사용한다.
- 식당 대표자가 매장을 추가할 때마다 해당 매장의 사업자등록증 인증 절차를 완료해야 한다.
- 식당 대표자는 사업자등록번호·상호·대표자명·개업일자·사업장 주소·업태·종목을 글로 입력하고 동일한 사업자등록증 이미지를 제출한다.
- 사업자등록번호·개업일자·대표자명과 사업자 상태는 국세청 무료 API로 자동 확인한다. 일치하고 허용된 상태이면 자동 검사를 통과한 것으로 기록하되 최종 인증·입점은 관리자 이미지·업종 확인 후 확정한다. 외부 조회 장애·정보 미반영·불일치는 승인이나 최종 거절로 단정하지 않고 확인 보류 후 재시도 또는 관리자 검토로 처리한다.
- 신청자는 사업자등록증상 대표자 본인임을 확약하고 허위 등록 책임에 동의한다. 현재는 신분증이나 유료 본인확인을 추가하지 않으므로 실제 신청자와 대표자의 강한 동일인 확인은 제공하지 않으며, 유료 사업자·추가 개인정보 수집 없이 적용할 수 있는 수단을 확인하기 전까지 미래 `TODO`로 둔다.
- 사업자등록 단위와 MiriYum 매장은 원칙적으로 1:1로 연결한다.
- 같은 개인 또는 법인이 여러 지점을 운영하더라도 사업자등록번호가 다르면 별도 매장으로 등록한다.
- 하나의 식당 대표자 계정은 여러 매장을 관리할 수 있다.
- 이미 귀속된 사업자등록번호의 중복 신청은 자동 차단하되 기존 가게의 운영은 중지하지 않는다. 관리자가 기존 계정 정보를 노출하지 않고 충돌 원인을 재확인하여, 같은 대표자의 접근 상실은 AUTH-006 계정 복구로, 실제 양도·대표자 변경은 STORE-014로 안내하며 STORE-004 심사만으로 직접 소유권을 이전하지 않는다.
- 국세청 자동 검사와 필수 정보·동의 검사를 통과한 신청만 관리자 기본 입점 심사로 넘긴다. 관리자는 제출 이미지와 글 정보의 일치 여부 및 업태·종목이 지원 외식업 범위인지 확인하며, 승인 전에는 가게를 공개하거나 운영 기능을 허용하지 않는다.
- 사업자 단위 과세와 본사·지점 그룹은 미래 확장·법률·정산 구조 확인이 필요한 `STORE-013`의 `TODO`로 관리한다.

### 4.2 1차 MVP 선택 운영 기능

- 방문 예약
- 현장 웨이팅
- 원격 웨이팅

- 예약금

매장은 현재 게시된 모든 메뉴를 한 번에 홀드 가능으로 설정할 수 있고 이후 개별 메뉴를 중지할 수 있다. 홀드 설정된 메뉴별 제공 시간과 수량은 MiriYum에서 관리하되 POS의 실시간 전체 재고 연동은 요구하지 않는다. `대표 메뉴`는 예약금 계산 평균에 사용하는 별도 지정이며 홀드 가능 메뉴의 범위를 뜻하지 않는다.

### 4.3 1차 MVP 제외 후보

- 코스 예약, 테이스팅 행사와 회차·정원 관리
- 메뉴·코스 대금 선결제와 실제 식사 주문·현장 결제 관리

### 4.4 활성화 정책

- 일반 기능은 식당이 즉시 활성화할 수 있다.
- 즉시 활성화할 일반 기능의 정확한 목록과 기본 활성화·선택 활성화 구분은 `STORE-007`의 `팀원 상의 필요` 대표 결정으로 확정한다.
- 향후 결제·정산·주류·테이스팅 등 위험도가 높은 제한 기능은 해당 정책이 활성화된 뒤 관리자 심사를 거친다. 1차 MVP 제외 기능은 이 절만으로 활성화되지 않는다.
- 제한 기능은 가게별·기능별 최초 승인을 받고 다른 기능이나 다른 가게에 승인을 전파하지 않는다. 테이스팅은 기능 승인 후에도 개별 행사를 열 때마다 별도 승인을 받아야 하며 행사 승인 전에는 공개·모집·예약할 수 없다.
- 승인된 테이스팅의 일정·장소·가격·결제·환불·정원·주요 구성·주류·연령 조건 변경은 새 버전 재승인을 요구하고, 표현·비핵심 이미지 수정은 기존 승인을 유지한다. 새 버전 승인 전에는 기존 승인본만 사용자에게 적용한다.
- 관리자는 승인, 보완 요청, 거절 또는 일시 중지를 처리할 수 있다.
- 기능 활성화와 설정 변경, 심사 결과는 감사 로그에 남긴다.

## 5. 핵심 사용자 가치

- 매장명·메뉴명·지역·카테고리와 자연어 조건으로 실제 이용 가능한 매장·메뉴를 찾고, 명시한 취향에 맞는 무료 기본 추천을 받는다.

- 마이페이지에서 예약·웨이팅·메뉴 홀드·예약금 결제/환불·알림 이력을 서로 혼동 없이 확인한다.

## 8. 현재 확정된 웨이팅 원칙

- 일행 참여는 별도 대기 팀을 만들지 않는다.
- `WAIT-003`은 플랫폼 기본 반경 500m와 매장별 300m~1km 설정 범위를 확정한다. GPS 정확도가 부족하면 즉시 탈락시키지 않고 재측정 또는 매장 통제 현장 증명 경로를 안내하되, 이 대체 경로는 원격 등록을 허용하지 않는다. 위치 권한 거부·GPS 오류에는 로그인 사용자의 현장 QR·매장 코드 대체와 실패 폐쇄를 적용하는 `WAIT-004`의 안전 기본선을 함께 적용한다. 실제 운영 데이터에 따라 향후 조정이 필요하면 새 정책 버전으로 검토하되, 현재 값은 `TODO`가 아니다.

## 10. 정책 분류 현황과 다음 단계

2026-07-22에 217개 정책을 재감사한 뒤, 기존 139개 비핵심 추천 후보를 정책 ID별로 상세화하고 독립 검토했다. 이후 사용자 직접 운영 결정과 담당 영역 3의 7개 직접 결정을 반영한 현재 행 단위 합계는 `확정` 191개, `팀원 상의 필요` 11개, `자동 추천 예정` 0개, `TODO` 15개이며, 중요도는 `핵심` 19개, `필수` 181개, `권장` 17개다. 기존 확정 정책과 세부 결정·이력은 보존한다.

- `팀원 상의 필요`: 제품 방향, 주요 사용자·가게 흐름, 금전 책임, 희소 기회의 공정성 또는 되돌리기 어려운 약속을 바꾸며 안전한 단일 기본값이 없는 대표 결정
- `자동 추천 예정`: 대표 팀 결정에서 파생되거나 제품 방향을 바꾸지 않는 새 표준 보안·권한·멱등성·동시성·상태 전이·실패·재시도·알림·감사 후보. 상세안을 작성하는 동안 이 상태를 유지하며, 명시적 위임·세부 완성·독립 검토·결정 기록 없이 확정되지 않는다. 현재 대기 행은 0개지만 미래 후보에 계속 사용한다.
- `TODO`: 유료·외부 사업자 계약, 최종 법률·개인정보·약관 검토, 미래 확장, 실측 수치 또는 현재 범위를 넘는 복잡성 때문에 후속 조건이 필요한 결정
- `확정`: 사용자가 직접 승인했거나, 비핵심 추천안이 명시적인 추천 위임 기준 안에서 세부 내용과 인수 조건까지 완성되고 독립 검토를 거쳐 정책 ID별 의사결정 기록에 `확정`으로 남은 현재 적용 정책. 자동 추천 후보를 문서화하거나 구현한 것만으로는 충분하지 않다. 하나의 정책 행에 확정된 안전 하위 범위와 미정인 `TODO` 조각이 함께 있을 수 있으며, 어느 쪽도 다른 쪽의 경계를 바꾸지 않는다.

남은 대표 팀 결정 11개는 `STORE-007`, `WAIT-015`, `TASTE-003`, `TASTE-008`, `TASTE-012`, `TRANSFER-002`, `SUB-001`, `SUB-002`, `SUB-003`, `SUB-007`, `ADS-001`이다. 이미 확정된 종속 기술·운영 세부는 안전한 상태·권한·정합성 경계만 정하며, 이 11개 결정의 버전된 결과를 기계적으로 소비하고 제품·금전·공정성 선택을 대신하지 않는다. `PAY-003`과 `PAY-012`는 각각 결제/정산 목표 후보만 선택됐을 뿐 계약·심사·수수료·분쟁/취소, 지급대행·세무·법률·실제 자금 흐름 조건이 남은 `TODO`다. 사용자 구독과 매장 Pro·광고·코스·테이스팅은 1차 MVP에서 제외하며, 구독 혜택·광고 제거·Pro 리포트·자동화·유료 청구 권한은 현재 활성화하지 않는다. 다음 단계는 11개 대표 결정과 15개 `TODO`의 계약·법률·미래 조건을 각각 해소하는 것이다. 유료 서비스 도입, 불필요하게 복잡한 인프라, 추가 개인정보 수집은 기본값으로 채택하지 않으며 대규모 트래픽과 다중 서버·다중 인스턴스 전제는 유지한다.

상태·중요도와 전체 `TODO` 목록의 기준은 `docs/service-policies/README.md`이고, 정책별 남은 범위와 안전 기본선은 해당 도메인 문서를 따른다.
