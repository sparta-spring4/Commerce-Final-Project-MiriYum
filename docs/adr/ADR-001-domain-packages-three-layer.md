# ADR-001: 단일 애플리케이션과 도메인별 3계층 패키지

- 상태: Accepted
- 결정일: 2026-07-24

## 배경

초기 MiriYum은 예약과 결제처럼 정합성이 중요한 흐름을 빠르게 구현하면서도 도메인 책임과 HTTP·유스케이스·영속성 경계를 분명히 해야 한다.

## 결정

초기 백엔드는 하나의 Spring Boot 애플리케이션으로 구성한다. 각 도메인은 `controller`, `service`, `repository`의 3계층을 기본 경계로 두고, 실제 필요가 생길 때 `entity`, `dto`, `exception` 등의 하위 패키지를 해당 도메인 안에 추가한다.

`controller`는 HTTP 경계, `service`는 유스케이스와 트랜잭션 경계, `repository`는 영속성 접근을 담당한다. 도메인 간 협력은 공개된 service 경계를 통해서만 이루어진다.

## 검토한 대안

- 초기부터 마이크로서비스로 분리: 배포·관측·데이터 정합성의 운영 복잡도가 초기 요구보다 크다.
- 기술 계층별 전역 패키지 구성: 도메인 책임과 변경 범위가 흩어진다.
- controller에서 repository 직접 호출: 유스케이스와 트랜잭션 정책의 위치가 불명확해진다.

## 긍정적 결과

- 하나의 트랜잭션 모델 안에서 핵심 흐름의 정합성을 구현할 수 있다.
- 기능 변경이 도메인 경계 안에 머물러 탐색과 테스트가 쉬워진다.
- HTTP, 유스케이스, 영속성 책임을 분리할 수 있다.

## 부정적 결과

- 애플리케이션 규모가 커지면 배포 단위가 커질 수 있다.
- 도메인 간 service 협력 경계를 지속적으로 점검해야 한다.

## 재검토 조건

독립 배포, 독립 확장, 별도 데이터 소유 또는 비동기 통합 요구가 실제로 검증되면 모듈러 모놀리스 유지 또는 분리를 별도 ADR로 재검토한다.
