# TasteLock 서비스 의사결정 기록

> 문서 상태: Working Draft
> 최초 작성일: 2026-07-21
> 목적: 서비스 구현 전에 확정한 정책, 잠정안, 팀 협의 항목을 지속적으로 기록한다.
> 원칙: 확정 사항과 미확정 사항을 구분하고, 정책이 변경되면 변경 이유와 날짜를 남긴다.

## 1. 프로젝트 기준

### 확정

- 취업·포트폴리오용 프로젝트이지만 실제 운영 가능한 수준으로 설계한다.
- 대규모 트래픽과 특정 예약 오픈 시점의 순간적인 트래픽 폭주를 방어할 수 있어야 한다.
- 다수 서버·다수 애플리케이션 인스턴스 환경에서 모든 핵심 기능이 정상적으로 동작해야 한다.
- 애플리케이션 인스턴스는 가능한 한 무상태로 운영한다.
- 예약·홀드·결제·이벤트 처리는 중복 요청과 순서 역전을 전제로 멱등성을 보장한다.
- 좌석·메뉴·회차의 초과 판매를 DB 제약, 조건부 갱신, 원자적 연산으로 방지한다.
- 서버 또는 작업 인스턴스가 중단·재시작되어도 예약과 비동기 작업이 유실되지 않아야 한다.
- 가상 대기열, 요청 속도 제한, 캐시, 비동기 처리, 장애 격리, 자동 복구 및 부하 테스트를 설계 범위에 포함한다.
- 초기 구현 형태와 관계없이 예약·대기열·알림 등 트래픽이 큰 영역을 독립적으로 확장할 수 있는 도메인 경계를 둔다.

## 2. 지원 매장 범위와 운영 방식

### 확정

- 특정 지역이나 단일 업종으로 제한하지 않는다.
- 일반 식당, 베이커리, 브루어리, 오마카세, 팝업 등 다양한 외식 업종을 지원한다.
- 식당을 1~3단계로 고정 분류하는 기존 방식을 폐기한다.
- 식당이 플랫폼이 제공하는 전체 기능 중 필요한 기능을 직접 선택하는 모듈형 방식을 사용한다.
- 일반 기능은 식당이 즉시 활성화할 수 있다.
- 결제·정산·주류·테이스팅처럼 운영 또는 법적 위험이 큰 기능은 관리자 심사를 거쳐 활성화한다.
- 관리자는 제한 기능에 대해 `승인`, `보완 요청`, `거절`, `일시 중지` 처리를 할 수 있다.
- 기능 활성화, 설정 변경 및 관리자 심사 이력을 감사 로그로 보관한다.
- 기능 간 필수 의존성을 시스템에서 검사하여 올바르지 않은 조합을 차단한다.

### 기능 의존성 예시

```text
대표 메뉴 홀드 → 메뉴 관리 필요
코스 예약 → 회차·정원 관리 필요
선결제 → 결제·환불 정책 및 정산 정보 필요
테이스팅 → 회차 관리 및 참가자 설문 필요
주류 테이스팅 → 연령 확인 필요
```

## 3. 모든 식당의 필수 기본 기능

### 확정

다음 기능은 선택 해제할 수 없으며 모든 입점 식당에 기본 제공한다.

- 매장·사업자 정보
- 영업시간·휴무일
- 메뉴·가격·이미지
- 직원 계정과 권한
- 공지·알림 수신
- 정책 및 약관 동의
- 운영 상태: 영업·휴점·폐점

- 방문 예약
- 현장 웨이팅
- 원격 웨이팅

각 기능의 정확한 제공 범위, 의존성, 요금 및 심사 여부는 후속 결정한다.

## 5. 웨이팅 일행 및 중복 등록 정책

### 확정

- 웨이팅 대표자는 매장이 정한 등록 가능 반경 안에 있어야 신규 웨이팅을 생성할 수 있다.
- 대표자는 일행 참여 링크 또는 초대 코드를 공유할 수 있다.
- 초대받은 일행은 현재 위치와 관계없이 기존 웨이팅에 참여할 수 있다.
- 일행 참여는 새로운 대기 팀을 만들지 않고 기존 한 팀에 합류하는 방식이다.
- 일행은 동일한 대기 순번과 예상 대기 시간을 확인한다.
- 위치 정보는 등록 가능 여부 확인에 필요한 최소 범위로 사용하며, 정확한 원본 좌표를 불필요하게 장기 보관하지 않는다.

### 함께 검토 중인 보완 정책

다음 항목은 허수 웨이팅과 중복 등록을 줄이기 위한 후보이며 아직 최종 확정되지 않았다.

- 가입 시 휴대전화 번호를 한 번 인증한다.
- 한 계정은 같은 매장과 겹치는 시간대에 웨이팅을 한 건만 보유할 수 있다.
- 해당 웨이팅의 일행으로 참여한 사용자는 같은 매장에 별도 웨이팅을 만들 수 없다.
- 차례가 가까워지면 대기 지속 의사를 확인하고, 미응답 시 자동 취소한다.
- 반복적인 미응답·노쇼 사용자에게 원격 웨이팅 제한 또는 일시 정지를 적용한다.
- 인기 매장은 필요한 경우 체크인 시 반환되는 소액 보증금을 사용할 수 있다.
- 계정·휴대전화·기기·행동 패턴으로 중복 가능성을 탐지하되, 오탐만으로 즉시 제재하지 않는다.

## 6. 팀 협의 필요 사항

### 6.1 웨이팅 등록 가능 반경

상태: **미정 — 팀 협의 필요**

비교할 선택지:

1. 모든 매장에 플랫폼 고정 반경을 적용한다.
2. 식당이 반경을 자유롭게 설정한다.
3. 플랫폼이 최소·최대 허용 범위를 정하고 식당이 그 범위 안에서 선택한다.

현재 추천안:

- 3번 방식을 사용한다.
- 예시 기본값은 300m로 둔다.
- 예시 설정 가능 범위는 100m~1km로 둔다.
- 실제 기본값과 허용 범위는 상권 밀도, GPS 오차, 도보 접근성 및 테스트 결과를 근거로 확정한다.

함께 결정해야 할 예외:

- 실내·지하·고층 건물 등 GPS 정확도가 낮은 환경의 대체 인증 방식
- 위치 권한 거부 사용자의 현장 QR 또는 매장 코드 등록 허용 여부
- 위치 조작이 의심되는 사용자의 추가 확인 방식
- 매장 이전 또는 지도 좌표 오류 시 운영자 보정 절차

### 6.2 식당 구독의 무료·유료 기능 범위

상태: **미정 — 팀 협의 필요**

현재 잠정 추천안:

- 핵심 예약 기능은 식당의 초기 진입 장벽을 낮추기 위해 기본 제공한다.
- 구독 식당에는 얼리 테이스팅 운영, 고급 수요 분석 및 운영 리포트를 제공한다.
- 정확한 무료 사용량, 유료 기능, 구독 가격은 별도로 확정한다.

## 8. 공식 문서 체계

### 확정

- `tastelock-service-blueprint.md`: 최초 아이디어 초안이며 수정하지 않는다.
- `tastelock-service-decisions.md`: 확정, 변경 및 보류 이력을 누적한다.
- `docs/service-definition.md`: 현재 공식 서비스 전체 구상과 제품 경계를 관리한다.
- `docs/technical-architecture.md`: 현재 공식 기술 방향을 관리한다.
- `docs/service-policies/README.md`: 전체 정책 항목과 진행 상태를 관리한다.
- `docs/service-policies/00-policy-template.md`: 상세 정책의 공통 검토 양식이다.
- `docs/service-policies/01-*.md`부터 `18-*.md`: 도메인별 상세 정책을 관리한다.

문서가 충돌하면 최신 확정 상세 정책, 공식 서비스 정의·기술 아키텍처, 의사결정 기록, 최초 블루프린트 순으로 적용한다.

정책 상태는 `미논의`, `논의 중`, `팀 협의 필요`, `확정`, `변경 검토`로 구분한다. 사용자가 명시적으로 승인하지 않은 추천안은 확정으로 기록하지 않는다.
